# 决策框架：架构选择的形式化方法论

> **定位**：架构决策不应是"拍脑袋"或"跟风"，而应是有结构的分析过程。本框架提供一套可复用的方法论，将定性判断与定量分析结合。
>
> **核心命题**：没有 universally 最优的架构，只有最适合当前约束的架构。决策框架的价值在于让约束显式化、让权衡透明化。

---

## 一、思维导图：架构决策框架

```text
架构决策框架（ADMF）
│
├─【阶段1：问题定义】
│   ├─ 功能需求（做什么）
│   ├─ 非功能需求（做到什么程度）
│   └─ 约束条件（不能做什么）
│
├─【阶段2：选项生成】
│   ├─ 至少3个可行选项
│   ├─ 包含"保持现状"选项
│   └─ 包含激进选项（拓宽思维）
│
├─【阶段3：评估矩阵】
│   ├─ 维度定义（性能、成本、复杂度等）
│   ├─ 权重分配（业务优先级）
│   └─ 评分（1-5或1-10）
│
├─【阶段4：风险分析】
│   ├─ 每种选项的风险识别
│   ├─ 缓解策略
│   └─ 回滚计划
│
├─【阶段5：决策记录】
│   ├─ ADR（Architecture Decision Record）
│   └─ 决策理由透明化
│
└─【阶段6：验证与学习】
    ├─ 假设验证
    ├─ 指标监控
    └─ 决策复盘
```

## 二、评估矩阵模板

```
架构选项评估矩阵

维度（可根据场景调整）：
  ┌─────────────────┬────────┬────────┬────────┬─────────┐
  │ 维度 / 权重     │ 选项A  │ 选项B  │ 选项C  │ 重要性  │
  ├─────────────────┼────────┼────────┼────────┼─────────┤
  │ 性能 / 20%      │  4     │  3     │  5     │  高     │
  │ 可扩展性 / 15%  │  3     │  5     │  4     │  高     │
  │ 成本 / 15%      │  5     │  2     │  3     │  中     │
  │ 复杂度 / 15%    │  5     │  2     │  3     │  高     │
  │ 团队技能 / 10%  │  5     │  3     │  2     │  中     │
  │ 上市时间 / 10%  │  5     │  3     │  2     │  高     │
  │ 可维护性 / 10%  │  4     │  3     │  4     │  中     │
  │ 生态成熟度 / 5% │  5     │  4     │  2     │  低     │
  ├─────────────────┼────────┼────────┼────────┼─────────┤
  │ 加权总分        │  4.35  │  3.15  │  3.45  │         │
  └─────────────────┴────────┴────────┴────────┴─────────┘

评分标准：
  5 = 优秀，明显优于其他选项
  4 = 良好，满足需求
  3 = 可接受，有改进空间
  2 = 勉强接受，有明显缺点
  1 = 不可接受

关键：权重必须反映业务优先级，而非技术偏好
```

## 三、风险分析模板

```
风险登记册

┌─────────────┬──────────┬──────────┬──────────┬────────────┐
│ 风险描述     │ 概率     │ 影响     │ 风险等级  │ 缓解策略    │
├─────────────┼──────────┼──────────┼──────────┼────────────┤
│ 技术选型过时 │ 中       │ 高       │ 高       │ 抽象层隔离  │
│ 团队学习曲线 │ 高       │ 中       │ 高       │ 培训+导师   │
│ 供应商锁定   │ 低       │ 高       │ 中       │ 多云抽象    │
│ 性能不达标   │ 中       │ 高       │ 高       │ 早期原型验证│
│ 安全漏洞     │ 低       │ 极高     │ 高       │ 安全Review  │
└─────────────┴──────────┴──────────┴──────────┴────────────┘

风险等级 = 概率 × 影响
  高：立即制定缓解计划
  中：监控并准备应急方案
  低：定期Review
```

## 四、ADR模板（Architecture Decision Record）

```markdown
# ADR-042: 采用事件驱动架构处理订单流程

## 状态
Accepted (2024-03-15)

## 上下文
订单处理涉及多个服务（库存、支付、物流），需要松耦合和独立演进。

## 决策
采用编排式Saga（Temporal）管理订单工作流。

## 后果
### 正面
- 服务间松耦合
- 可视化流程状态
- 自动补偿处理

### 负面
- 引入Temporal运维负担
- 调试复杂度增加
- 团队学习成本

## 备选方案
1. 编舞式EDA（太分散，难追踪）
2. 2PC分布式事务（性能差，不适合长事务）
3. 保持同步调用（耦合高，级联故障风险）

## 相关决策
- ADR-038: 消息队列选型（Kafka）
- ADR-041: 服务拆分边界
```

## 五、概念定义与属性关系

| 概念 | 定义 | 属性 | 示例 | 反例 |
|------|------|------|------|------|
| **ADR** | 记录架构决策及其理由的文档 | 可追溯、可学习、可复盘 | 上述模板 | 口头决策、无记录 |
| **评估矩阵** | 多维度比较选项的量化工具 | 结构化、可比较、权重反映优先级 | 7维度评分表 | 单一维度比较（仅性能） |
| **风险登记册** | 识别和跟踪项目风险的工具 | 概率×影响、动态更新 | 5风险项表格 | 无风险分析即决策 |
| **假设验证** | 验证决策前提条件的过程 | 早期、低成本、可调整 | 原型验证性能假设 | 直接全面实施 |
| **决策复盘** | 回顾决策结果与学习的过程 | 诚实、无责备、持续改进 | 季度架构Review | 永不回顾 |

## 六、形式化论证：评估函数的数学表达

```
架构决策评估的形式化模型：

  设决策空间 D = {d₁, d₂, ..., dₙ}（n个候选架构）
  设评估维度集合 C = {c₁, c₂, ..., cₘ}（m个评估维度）

  对每个维度 cᵢ：
    - 权重 wᵢ ∈ [0, 1]，满足 Σwᵢ = 1
    - 评分 sᵢ(dⱼ) ∈ [1, 5]（方案dⱼ在维度cᵢ上的得分）

  综合评分函数：
    Score(dⱼ) = Σᵢ₌₁ᵐ wᵢ × sᵢ(dⱼ)

  最优决策：
    d* = argmaxⱼ Score(dⱼ)

Arrow不可能定理的约束：
  若评估满足：
    1. 一致性（Unanimity）
    2. 无关独立性（IIA）
    3. 非独裁性（Non-dictatorship）
  则不存在完美的评分聚合机制
  → 工程启示：评分结果不应视为"客观真理"，而是"结构化讨论的基础"
```

## 七、批判性分析：框架自身的隐含假设

```
本框架的隐性前提与失效条件：

  假设1："评分是客观的"
    失效：锚定效应、确认偏误使评分带有主观色彩
    → 缓解：多人独立评分后取中位数

  假设2："权重反映真实业务优先级"
    失效：政治化权重分配（老板偏好的维度权重过高）
    → 缓解：权重需多部门协商，公开透明

  假设3："所有关键维度都被识别"
    失效：未知未知（Unknown Unknowns）被遗漏
    → 缓解：预留"其他因素"维度，预-mortem分析

  假设4："保持现状"是中性选项
    失效：沉没成本谬误使现状选项被低估
    → 缓解：强制要求"若从零开始，还会选现状吗？"
```
