# 场景应用与决策框架：模型到现实问题的映射方法论

> **来源映射**: View/01.md

> **模块定位**：本模块是知识体系的"应用整合层"。所有理论模型（CAP、Raft、CRDT、DDD）的价值，最终体现在对具体场景的正确选择上。
>
> **核心命题**：掌握模型的真正标志，不是背诵定义，而是能在具体约束下**形式化地论证为何选择A而非B**。

## 一、电商系统架构演进：四阶段场景树

```text
【根场景】电商平台（初创→独角兽→全球化）
│
├─【阶段1】初创期（DAU < 1万，团队 < 10人）
│   ├─ 约束：快速迭代 / 零运维成本 / 单数据库
│   ├─ 架构：模块化单体（Rails/Django/Spring Boot）
│   ├─ 计算模型：同步阻塞（线程池）→ 认知负载最低
│   ├─ 数据模型：单体RDBMS（ACID事务保证库存一致性）
│   └─ 一致性：单节点强一致（无分布式问题）
│
├─【阶段2】成长期（DAU 1万-100万，团队 10-50人）
│   ├─ 痛点：读压力大 / 促销峰值 / 单库瓶颈
│   ├─ 架构：模块化单体 + 读写分离（CQRS雏形）
│   │   ├─ 写路径：单体RDBMS（订单/支付）
│   │   └─ 读路径：Redis缓存 + Elasticsearch（商品搜索）
│   ├─ 计算模型：引入异步（Actor）处理非关键路径
│   ├─ 一致性：最终一致（缓存与DB同步延迟 < 1s可接受）
│   └─ 部署：单体多实例 + 负载均衡
│
├─【阶段3】扩张期（DAU 100万-1000万，团队 50-200人）
│   ├─ 痛点：团队并行开发冲突 / 某模块需独立扩展
│   ├─ 架构：提取高内聚服务（绞杀者模式）
│   │   ├─ 优先提取：支付服务（PCI合规隔离）
│   │   ├─ 优先提取：搜索服务（独立ES集群）
│   │   └─ 保留单体：商品管理/运营后台（变更低频）
│   ├─ 计算模型：微服务间 gRPC（CSP同步语义）+ 内部EDA（Kafka）
│   ├─ 分布式事务：Saga模式（编排式）
│   │   └─ 订单Saga：创建订单 → 扣库存 → 支付 → 发货
│   ├─ 一致性：订单CP（强一致）/ 库存AP（可超卖后补偿）
│   └─ 部署：K8s + Istio → 平台工程团队支撑
│
└─【阶段4】全球化（DAU > 1000万，多区域）
    ├─ 痛点：跨境延迟 / 数据主权（GDPR/中国法）/ 货币/税务
    ├─ 架构：单元化架构（Set-based Architecture）
    │   ├─ 用户分片：按地域路由至就近数据中心
    │   └─ 全局服务：仅用户/支付账户全局强一致；商品/订单区域自治
    ├─ 计算模型：Actor模型（地理位置透明）
    ├─ 一致性：PACELC
    │   ├─ 同区域：EL（延迟优先）→ 本地Quorum
    │   └─ 跨区域：PC（一致优先）→ 全局Paxos（仅账户余额）
    ├─ 数据模型：CRDT（购物车跨设备同步）
    └─ 部署：多活K8s联邦
```

## 二、决策框架汇总

### 一致性选择决策树

```text
开始：业务需要分布式数据存储？
│
├─ 否 → 单节点RDBMS（PostgreSQL/MySQL）→ 强一致+高可用
│
└─ 是 → 是否存在跨地域部署需求？
    │
    ├─ 否 → 单数据中心
    │   ├─ 读/写比 < 10:1 且 需事务？ → NewSQL（TiDB/CockroachDB）→ Raft强一致
    │   └─ 读/写比 > 10:1 且 可接受最终一致？ → Redis Cluster / 读副本
    │
    └─ 是 → 跨地域延迟 > 100ms？
        │
        ├─ 否（同大陆） → 同步复制多活 → CockroachDB Multi-Region
        │
        └─ 是（跨大洋） → 必须接受最终一致性？
            │
            ├─ 否（金融核心） → 2PC + 补偿事务（TCC）
            │   └─ 警告：2PC在分区时阻塞 → 需业务层容忍不可用窗口
            │
            └─ 是（社交/日志/IoT） → CRDT / 向量时钟
                ├─ 数值累加 → G-Counter / PN-Counter
                ├─ 集合合并 → OR-Set / LWW-Element-Set
                └─ 文本协作 → YATA / RGA
```

## 五、批判性总结

场景应用与决策框架是理论模型通往工程现实的"最后一公里"，但这最后一公里的距离常被低估。本模块展示的电商四阶段演进、一致性选择决策树等工具，其有效性依赖于一个关键假设——业务场景的稳定性和可预测性。然而，2026年的软件系统面临的常态恰恰是场景漂移：初创期的"快速验证"假设可能在数月内被打破，全球化阶段的"数据主权"约束可能因政策突变而重构。决策树的二元分支结构（是/否）在复杂现实中往往失效，因为许多工程决策是概率性的、多目标优化的、且受组织政治影响的。更深层的问题是，所有形式化决策框架都隐含了一个"理性行动者"假设——决策者拥有完整信息、明确偏好和无限认知资源。但真实的架构决策是在信息不完备、时间压力和团队动态中做出的。与纯理论模块相比，场景应用模块的价值不在于提供"正确答案"，而在于提供"可论证的选择空间"：它迫使决策者显式化约束、量化权衡、并记录假设。未来，AI辅助的场景模拟（基于数字孪生的架构沙箱）可能将决策从"经验直觉"推向"计算实验"，但人类对业务价值和风险偏好的最终判断不可替代。
