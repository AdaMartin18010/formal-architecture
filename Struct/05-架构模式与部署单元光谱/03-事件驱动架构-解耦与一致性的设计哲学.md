# 事件驱动架构：解耦与一致性的设计哲学

> **定位**：EDA不是"使用消息队列"，而是一种深层的系统组织哲学——通过事件而非命令来协调，通过异步而非同步来解耦。但解耦的代价是一致性的复杂性。
>
> **核心命题**：EDA的核心张力：松耦合带来弹性，但事件ual一致性将分布式系统的复杂性从运行时推到了设计时。

---

## 一、思维导图：EDA的核心模式

```text
事件驱动架构（EDA）
│
├─【核心模式】
│   ├─ Event Notification（事件通知）
│   │   └─ 轻量通知，消费者自行查询详情
│   ├─ Event-Carried State Transfer（事件携带状态）
│   │   └─ 事件包含完整状态，消费者无需查询
│   └─ Event Sourcing（事件溯源）
│       └─ 状态是事件的fold，事件是真相来源
│
├─【一致性模式】
│   ├─ Saga（编排式/协调式）
│   ├─ Outbox Pattern（发件箱）
│   └─ CQRS（命令查询分离）
│
├─【技术实现】
│   ├─ 消息队列（Kafka, RabbitMQ, Pulsar）
│   ├─ 事件总线（EventBridge, EventGrid）
│   └─ 流处理（Flink, Kafka Streams）
│
└─【风险】
    ├─ 事件顺序
    ├─ 重复消费
    ├─ 消费者滞后
    └─ 调试困难
```

## 二、EDA三种核心模式对比

| 模式 | **Event Notification** | **Event-Carried State Transfer** | **Event Sourcing** |
|------|----------------------|--------------------------------|-------------------|
| **事件内容** | ID + 变更类型 | 完整状态快照 | 状态变更增量 |
| **消费者行为** | 查询源服务获取详情 | 直接使用事件数据 | fold事件重建状态 |
| **耦合度** | 中（需知道源服务API） | 低（自给自足） | 低 |
| **数据一致性** | 最终一致 | 最终一致 | 强一致（事件流级别） |
| **网络负载** | 高（N消费者×查询） | 中（事件较大） | 低（事件紧凑） |
| **存储需求** | 低 | 中 | 高（无限事件流） |
| **查询能力** | 依赖源服务 | 消费者自有数据 | 需投影（Projection） |
| **适用场景** | 简单通知 | 数据复制 | 审计、复杂业务逻辑 |

## 三、Saga模式：分布式事务的EDA替代

```
Saga模式：将长事务拆分为本地事务序列，通过补偿处理失败

两种实现方式：

  1. 编排式Saga（Choreography）
     服务A完成 → 发布事件 → 服务B消费 → 完成 → 发布事件 → ...

     优点：松耦合、无需协调器
     缺点：流程分散、难以理解全局、循环依赖风险

  2. 协调式Saga（Orchestration）
     协调器 → 命令服务A → 等待完成 → 命令服务B → ...

     优点：集中管理、流程清晰、易于监控
     缺点：协调器成为单点、微服务退化为准SOA

补偿事务（Compensating Transaction）：
  正向流程：  预订酒店 → 预订机票 → 预订租车
  若机票失败：取消酒店
  若租车失败：取消机票 → 取消酒店

  关键约束：补偿非"回滚"，而是业务层面的"撤销操作"
    - 回滚：数据库事务的ACID保证
    - 补偿：业务逻辑，可能失败，可能有副作用
```

### 3.1 Saga与2PC对比

| 维度 | **2PC（两阶段提交）** | **Saga** |
|------|---------------------|----------|
| **一致性** | 强一致 | 最终一致 |
| **隔离性** | 全局隔离 | 无隔离（脏读可能） |
| **性能** | 差（锁持有时间长） | 好（无全局锁） |
| **故障恢复** | 协调者故障→阻塞 | 可继续执行补偿 |
| **实现复杂度** | 中（协议标准） | 高（业务补偿逻辑） |
| **适用场景** | 短事务、强一致必须 | 长事务、最终一致可接受 |

## 四、Outbox Pattern：可靠事件发布

```
问题：双写问题（数据库 + 消息队列）

  场景：
    1. 写入订单到数据库
    2. 发布"订单已创建"事件到Kafka

  风险：
    - 数据库写入成功，消息发送失败 → 数据不一致
    - 消息发送成功，数据库回滚 → 虚假事件

Outbox解决方案：

  数据库事务内：
    BEGIN;
      INSERT INTO orders ...;
      INSERT INTO outbox (topic, payload) VALUES ('orders', '{...}');
    COMMIT;

  独立进程（CDC或轮询）：
    读取outbox表 → 发布到消息队列 → 删除/标记已发送

  保证：
    - 数据库和事件在同一事务中 → 原子性
    - CDC确保至少一次投递
    - 消费者幂等确保恰好一次语义

技术实现：
  - Debezium（CDC）
  - Kafka Connect JDBC Source
  - 自定义轮询器
```

## 五、概念定义与属性关系

| 概念 | 定义 | 属性 | 示例 | 反例 |
|------|------|------|------|------|
| **EDA** | 通过事件传播状态变更的系统架构风格 | 异步、松耦合、最终一致 | Kafka生态 | RPC同步调用链 |
| **Event Sourcing** | 以事件流为真相来源，状态是事件的fold | 完整审计、可回放、存储成本高 | 银行账户、购物车 | CRUD直接更新状态 |
| **Saga** | 将长事务拆分为本地事务+补偿的分布式事务模式 | 最终一致、无全局锁、补偿复杂 | 电商订单流程 | 2PC分布式事务 |
| **Outbox** | 将事件与业务数据在同一事务中持久化的模式 | 可靠、至少一次、需CDC | Debezium + Kafka | 独立双写 |
| **CQRS** | 命令和查询使用不同模型的架构模式 | 读写分离、可独立扩展、复杂性增加 | 电商查询优化 | 统一模型 |
