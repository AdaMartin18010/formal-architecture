## 一、知识体系思维导图

```text
大数据与流处理
│
├─► 批处理 (Batch Processing)
│   ├─► MapReduce 模型
│   │   ├─ Map: (K₁, V₁) → list(K₂, V₂)
│   │   ├─ Shuffle: 按 K₂ 分组
│   │   └─ Reduce: (K₂, list(V₂)) → list(K₃, V₃)
│   │
│   ├─► 代表系统
│   │   ├─ Hadoop MapReduce (经典)
│   │   ├─ Apache Spark: RDD / DataFrame / Dataset
│   │   └─ Apache Hive / Presto / Trino: SQL on Hadoop
│   │
│   └─► 特点
│       ├─ 高吞吐、高延迟（分钟到小时）
│       ├─ 数据完整性: 输入固定，输出确定
│       └─ 容错: 任务级重算
│
├─► 流处理 (Stream Processing)
│   ├─► 时间语义
│   │   ├─ Event Time: 事件产生的时间
│   │   ├─ Processing Time: 事件被处理的时间
│   │   └─ Ingestion Time: 事件进入系统的时间
│   │
│   ├─► 窗口操作
│   │   ├─ Tumbling Window: 固定大小、不重叠
│   │   ├─ Sliding Window: 固定大小、可重叠
│   │   ├─ Session Window: 活动间隙触发
│   │   └─ Global Window: 全局单一窗口
│   │
│   ├─► 水印 (Watermark)
│   │   ├─ 定义: Event Time 的进度指标
│   │   ├─ 作用: 处理乱序事件、触发窗口计算
│   │   └─ 迟到数据: allowedLateness、侧输出流
│   │
│   └─► 代表系统
│       ├─ Apache Flink: 真正的流处理（非微批）
│       ├─ Apache Kafka Streams: 轻量流处理
│       ├─ Apache Spark Streaming: 微批 (Micro-batch)
│       └─ Apache Storm: 低延迟经典
│
├─► 批流一体 (Unified Batch & Stream)
│   ├─ Flink: DataStream API 统一批流
│   ├─ Spark: Structured Streaming 统一
│   └─ 理念: 同一套代码处理批和流
│
└─► 数据湖与实时数仓
    ├─ Delta Lake: ACID + 版本控制 + 时间旅行
    ├─ Apache Iceberg: 开放表格式、隐藏分区
    ├─ Apache Hudi: 增量处理、增量查询
    └─ 实时数仓: StarRocks, ClickHouse, Druid
```

## 二、流处理时间语义的形式化

```text
定义 (时间语义):
  设事件 e 的属性:
    T_event(e): 事件产生时间
    T_ingest(e): 事件进入系统时间
    T_process(e): 事件被处理时间

  理想情况: T_event = T_ingest = T_process
  实际情况: T_event ≤ T_ingest ≤ T_process (存在延迟)

  水印 (Watermark):
    W(t) = max(T_event) - allowed_delay
    含义: 系统相信所有 Event Time < W(t) 的事件都已到达

  窗口触发条件:
    当 Watermark ≥ 窗口结束时间 时，触发窗口计算

  迟到数据处理:
    若事件到达时 Watermark 已超过窗口结束:
      - 丢弃 (默认)
      - 允许迟到 (allowedLateness)
      - 侧输出流 (Side Output)
```

## 三、批处理 vs 流处理对比

| 维度 | 批处理 (Spark) | 流处理 (Flink) |
|------|---------------|---------------|
| **数据模型** | 有限数据集 | 无限数据流 |
| **延迟** | 分钟到小时 | 毫秒到秒 |
| **容错** | 任务重算 | Checkpoint + Savepoint |
| **状态管理** | 无状态 / 聚合结果 | 有状态算子 + Keyed State |
| **时间语义** | Processing Time | Event Time + Watermark |
| **适用场景** | 报表、ETL、离线分析 | 实时监控、风控、推荐 |

## 六、批判性总结

大数据处理的演进从 **MapReduce**（批处理鼻祖）到 **Spark**（内存计算）再到 **Flink**（真正的流处理），本质是对**延迟**的不断压缩。但批流一体的真正挑战不是技术，而是**思维模式的转换**——批处理工程师习惯"数据完整后再计算"，流处理要求"数据到达即计算"。

水印 (Watermark) 是流处理中最优雅的概念之一：它用**不完美的时间估计**来平衡**延迟**与**完整性**。这类似于分布式系统中的**超时机制**——都是"在信息不完备时做出最优决策"的工程智慧。
