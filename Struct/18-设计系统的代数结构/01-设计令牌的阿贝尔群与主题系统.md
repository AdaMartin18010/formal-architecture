## 一、知识体系思维导图

```text
设计系统的代数结构
│
├─► 设计令牌 (Design Tokens)
│   ├─► 定义: 设计决策的抽象变量
│   │   ├─ 颜色: --color-primary-500: #3B82F6
│   │   ├─ 间距: --space-4: 16px
│   │   ├─ 字体: --font-heading: "Inter", sans-serif
│   │   ├─ 圆角: --radius-lg: 8px
│   │   └─ 阴影: --shadow-md: 0 4px 6px -1px rgb(0 0 0 / 0.1)
│   │
│   ├─► 令牌的代数结构
│   │   ├─ 集合: Token 的命名空间集合 (分层结构)
│   │   ├─ 运算: 引用 (alias)、变换 (transform)、复合 (composite)
│   │   └─ 阿贝尔群性质: 交换、结合、存在单位元与逆元
│   │
│   └─► W3C 标准格式 (DTCG)
│       ├─ $value: 令牌值
│       ├─ $type: 值类型 (color, dimension, fontFamily, shadow)
│       ├─ $description: 语义说明
│       └─ $extensions: 工具特定元数据
│
├─► 主题系统 (Theming)
│   ├─► 主题作为同态映射
│   │   Theme: TokenSpace → ValueSpace
│   │   即: 同一语义令牌在不同主题下映射到不同值
│   │
│   ├─► 主题切换
│   │   ├─ 类名切换: .light / .dark / .high-contrast
│   │   ├─ CSS 变量覆盖: :root 变量重定义
│   │   ├─ 运行时切换: 无刷新主题切换 (CSS 变量热替换)
│   │   └─ 系统偏好监听: prefers-color-scheme 媒体查询
│   │
│   └─► 多维度主题张量积
│       ├─ 颜色主题: light / dark / high-contrast
│       ├─ 密度主题: compact / comfortable / spacious
│       ├─ 品牌主题: brand-A / brand-B / brand-C
│       └─ 平台主题: iOS / Android / Web
│
├─► 组件组合的幺半群
│   ├─ 单位元: 空组件或透明层
│   ├─ 二元运算: 组件组合 (composition)
│   ├─ 结合律: (A ∘ B) ∘ C = A ∘ (B ∘ C)
│   └─ 非交换: 组件顺序通常影响渲染结果
│
└─► 设计系统即代码
    ├─ 单一可信源: tokens.json / tokens.yaml
    ├─ 多平台输出: Style Dictionary → CSS / iOS / Android
    ├─ 版本控制: SemVer 管理设计变更
    └─ CI/CD: 设计变更触发构建流水线
```

## 二、核心概念的形式化定义

### 2.1 设计令牌的代数结构

```text
定义 (设计令牌空间):
  设 T 为所有设计令牌的集合

  令牌结构 (W3C DTCG 格式):
    t = ⟨name, value, type, description, aliases, extensions⟩
    name: 分层命名 (如 "color.primary.500")
    value: 具体值 (如 "#3B82F6") 或引用 (如 "{color.blue.500}")
    type: 值类型 (color | dimension | fontFamily | fontWeight | shadow | ...)

  运算:
    1. 引用 (Alias): alias(t₁) = t₂.name
       语义: t₁.value = resolve(t₂)
       形成引用图 G_alias = (T, E_alias)，要求无环

    2. 变换 (Transform): transform(t, f) = t'
       其中 t'.value = f(t.value)
       例: transform(color, toHex) = hex_color
           transform(dimension, toRem) = rem_value

    3. 复合 (Composite): composite(t₁, t₂, ..., tₙ) = t_c
       例: shadow = composite(x, y, blur, color)
           typography = composite(fontFamily, fontSize, fontWeight, lineHeight)

  阿贝尔群性质 (在值空间上):
    设 V_color 为颜色值集合，⊕ 为颜色混合运算 (如 alpha 混合):
      交换律: c₁ ⊕ c₂ = c₂ ⊕ c₁
      结合律: (c₁ ⊕ c₂) ⊕ c₃ = c₁ ⊕ (c₂ ⊕ c₃)
      单位元: transparent (透明色, alpha = 0)
      逆元: ∀c, c⁻¹ 为在 alpha 混合下的补色

    设 V_spacing 为间距值集合，⊕ 为加法:
      交换律: s₁ + s₂ = s₂ + s₁
      结合律: (s₁ + s₂) + s₃ = s₁ + (s₂ + s₃)
      单位元: 0px
      逆元: -s

    注: 设计令牌本身不构成群，但其值空间在特定运算下可具有群结构。
```

### 2.2 主题作为函数与同态

```text
定义 (主题 Θ):
  主题 Θ 是从令牌名到值的映射:
    Θ: Name → Value

  主题切换:
    Θ_light: "color.bg" ↦ "#FFFFFF"
    Θ_dark:  "color.bg" ↦ "#1F2937"
    Θ_high_contrast: "color.bg" ↦ "#000000"

  主题组合 (多维度张量积):
    设主题维度集合 D = {color, density, brand, platform}
    组合主题:
      Θ_combined = Θ_color ⊗ Θ_density ⊗ Θ_brand

    其中 ⊗ 为偏函数组合:
      Θ_combined(name) =
        if name ∈ dom(Θ_color):    Θ_color(name)
        elif name ∈ dom(Θ_density): Θ_density(name)
        elif name ∈ dom(Θ_brand):   Θ_brand(name)
        else: error (未定义令牌)

  主题同态性质:
    对于运算 ⊕ (如颜色混合):
      Θ(a ⊕ b) = Θ(a) ⊕' Θ(b)
      即: 先组合再映射 = 先映射再组合 (保持运算结构)
    这一性质保证了主题切换不会改变组件的布局和层级关系
```

### 2.3 CSS 变量作为群的线性表示

```text
定义 (CSS Custom Properties 的线性表示):
  设令牌群 ⟨T, ⊕, 𝟎⟩，标量域 𝕂 (通常为 ℝ 或 ℤ)

  线性表示:
    ρ: T → CSS_Variables
    ρ(t) = var(--{t.name})

  线性性质:
    ρ(t₁ ⊕ t₂) = calc(ρ(t₁) + ρ(t₂))   (对间距)
    ρ(k ⊗ t) = calc(k * ρ(t))           (标量乘法)

  例:
    --space-1: 4px;
    --space-2: 8px;
    --space-combined: calc(var(--space-1) + var(--space-2));  // = 12px

  这一表示将抽象的群运算转化为 CSS 计算表达式，
  实现了设计代数到样式语言的同态映射。
```

```json

      "500": {
        "$value": "#3B82F6",
        "$type": "color",
        "$description": "Primary brand color, used for CTAs and links"
      }
    }
  },
  "spacing": {
    "4": {
      "$value": "16px",
      "$type": "dimension"
    }
  },
  "shadow": {

        "y": "4px",

        "spread": "-1px",

      },
      "$type": "shadow"
    }
  }
}
```

### 5.3 主题切换实现

```css
:root {
  --color-bg: #FFFFFF;
  --color-text: #1F2937;
  --color-primary: #3B82F6;
  --spacing-unit: 4px;
}

[data-theme="dark"] {
  --color-bg: #1F2937;
  --color-text: #F9FAFB;
  --color-primary: #60A5FA;
}

[data-theme="high-contrast"] {
  --color-bg: #000000;
  --color-text: #FFFFFF;
  --color-primary: #FFFF00;
}

/* 使用令牌 */
.card {
  background: var(--color-bg);
  color: var(--color-text);
  padding: calc(var(--spacing-unit) * 4);
  border-radius: var(--radius-lg);
}
```

### 5.4 主题张量积 (多维度组合)

```typescript
// 多维度主题的类型定义
type ThemeDimensions = {
  color: 'light' | 'dark' | 'high-contrast';
  density: 'compact' | 'comfortable' | 'spacious';
  brand: 'brand-a' | 'brand-b';
};

// 主题组合函数 (偏函数组合)
function resolveToken(
  tokenName: string,
  dimensions: ThemeDimensions
): string {
  // 按优先级查找: brand > color > density
  const paths = [
    `brand.${dimensions.brand}.${tokenName}`,
    `color.${dimensions.color}.${tokenName}`,
    `density.${dimensions.density}.${tokenName}`,
    `base.${tokenName}`,
  ];

  for (const path of paths) {
    if (tokenExists(path)) return getToken(path);
  }
  throw new Error(`Token not found: ${tokenName}`);
}
```

## 六、批判性总结

设计令牌的本质是**设计决策的参数化**：将"主色是蓝色"这个决策从代码中抽离为可配置的变量，使同一套组件可以适应不同品牌、不同主题、不同平台。这是**单一职责原则**在设计领域的应用——组件负责结构和行为，令牌负责视觉表现。代数结构的引入（阿贝尔群、同态映射）并非纯粹的数学炫技，而是为设计系统提供了**可验证的一致性保证**：若令牌值空间满足群公理，则任何主题变换都不会破坏布局的稳定性。

但设计系统的"代数化"有**过度形式化的风险**。并非所有设计决策都适合用严格的数学结构描述。例如，"视觉层次"和"情感传达"依赖于设计师的直觉和经验，强行将其纳入令牌系统可能导致**创造力的僵化**。2026 年的最佳实践是"结构化但不僵化"——80% 的设计决策令牌化，20% 保留给设计师的自由裁量。令牌系统应定义"可接受的参数空间"，而非"唯一正确的值"。

Figma Variables 的推出标志着设计令牌从"开发工具"进化为"设计工具原生能力"，但这也带来了**双向同步**的挑战：设计师在 Figma 中修改变量后，如何确保代码中的令牌同步更新？Git-based Design Token 工作流（如 Tokens Studio 的 Git 同步、GitHub Actions 驱动的 Style Dictionary 构建）正在成为行业标准答案。未来的设计系统可能实现**实时双向绑定**——Figma 插件监听代码仓库的令牌变更，代码构建流程监听 Figma Variables 的更新，消除设计与工程之间的"最后一英里"鸿沟。

## 七、概念属性关系网络

```text
设计令牌阿贝尔群与主题系统概念属性关系网络
│
├─► 核心实体层
│   ├─ [令牌 t = ⟨name, value, type, description, aliases, extensions⟩]
│   │   ├─ name ──→ 属性: {分层命名, 语义路径, 平台后缀}
│   │   ├─ value ──→ 属性: {原始值, 引用(alias), 计算表达式}
│   │   ├─ type ──→ 属性: {color, dimension, fontFamily, fontWeight, shadow}
│   │   └─ aliases ──→ 属性: {引用图节点, 解析深度, 循环检测}
│   │
│   ├─ [值空间群 ⟨V, ⊕, 𝟎⟩]
│   │   ├─ V_color ──→ 属性: {sRGB, Display P3, 混合模式, alpha通道}
│   │   ├─ V_spacing ──→ 属性: {px, rem, em, 基准单位}
│   │   ├─ V_typography ──→ 属性: {字体族, 字重, 行高, 字间距}
│   │   └─ 𝟎 ──→ 属性: {transparent, 0px, inherit, normal}
│   │
│   ├─ [主题 Θ: Name → Value]
│   │   ├─ Θ_light ──→ 属性: {高亮度背景, 低亮度文本, 品牌色饱和度}
│   │   ├─ Θ_dark ──→ 属性: {低亮度背景, 高亮度文本, 降低饱和度}
│   │   └─ Θ_high_contrast ──→ 属性: {黑白极端对比, 去除灰度过渡}
│   │
│   └─ [引用图 G_alias = (T, E_alias)]
│       ├─ T ──→ 属性: {令牌节点, 解析状态, 值类型}
│       └─ E_alias ──→ 属性: {有向引用, 无环约束, 传递解析}
│
├─► 关系映射层
│   ├─ alias: T → T: 令牌引用形成有向图 (要求无环)
│   ├─ transform: T × F → T: 变换函数作用于令牌值
│   ├─ composite: Tⁿ → T: 复合令牌由原子令牌组合
│   ├─ Θ: Name → Value: 主题映射保持群运算结构
│   └─ ρ: T → CSS_Var: 线性表示连接设计代数与样式语言
│
├─► 属性约束层
│   ├─ 引用无环: G_alias 必须是有向无环图 (DAG)
│   ├─ 群封闭性: ∀a,b∈V, a⊕b ∈ V
│   ├─ 同态保持: Θ(a⊕b) = Θ(a) ⊕' Θ(b)
│   ├─ 线性表示: ρ(t₁⊕t₂) = calc(ρ(t₁) + ρ(t₂))
│   └─ W3C兼容: $value, $type, $description 符合 DTCG 规范
│
└─► 课程对齐层
    ├─ Stanford CS 247: 视觉设计系统与令牌管理
    └─ MIT 6.831: 颜色、排版与界面实现
```

## 八、形式化推理链

**推理命题 P**: 设计令牌的阿贝尔群结构保证了样式组合的可预测性和可逆性，而主题同态确保了多主题切换不会破坏视觉层级和布局稳定性。

**推理链**:

```text
前提 A1 (Jina Anne, 2018): "Design tokens are the single source of truth to store and
                             organize design decisions. They are the methodology that
                             bridges the gap between design tools and engineering
                             implementation."
前提 A2 (Danny Banks, 2019): "Style Dictionary is a build system that allows you to
                               define styles once, in a way for any platform or language
                               to consume. It transforms design tokens to any format
                               you need."
前提 A3 (W3C DTCG, 2023): "The $value, $type, and $description properties provide a
                            standardized format for tooling interoperability."

步骤 1: 由 A1，设计令牌作为"单一可信源"的数学基础
       设令牌集合 T 在运算 ⊕ 下构成阿贝尔群:
         封闭性: ∀a,b∈T, a⊕b ∈ T
         结合律: (a⊕b)⊕c = a⊕(b⊕c)
         交换律: a⊕b = b⊕a
         单位元: ∃𝟎, a⊕𝟎 = a
         逆元: ∀a, ∃a⁻¹, a⊕a⁻¹ = 𝟎
       例: 颜色 alpha 混合下，c₁⊕c₂ 的结果与混合顺序无关
       ∴ 群结构消除了样式组合的顺序不确定性

步骤 2: 令牌引用图的形式化约束
       alias: T → T 形成有向图 G_alias = (T, E_alias)
       解析函数: resolve(t) = t.value 若 t.value 为原始值
                          = resolve(alias(t)) 若 t.value 为引用
       要求: G_alias 为 DAG (无环引用)
       否则: resolve(t) 不终止 ⟹ 构建系统死锁

步骤 3: 由 A2-A3，Style Dictionary 变换管道的代数性质
       变换集合 𝓕 在复合运算 ∘ 下构成幺半群:
         Parse: JSON/YAML → G_token
         Transform: G_token → G_token' (值转换、引用解析)
         Format: G_token' → String (平台特定)
       封闭性: ∀f,g∈𝓕, f∘g ∈ 𝓕
       结合律: (f∘g)∘h = f∘(g∘h)
       单位元: id(t) = t
       ∴ 复杂的多平台输出可通过简单变换的组合实现

步骤 4: 主题同态的多维度扩展
       设单维度主题: Θ_i: Name → Value
       多维度张量积: Θ_combined = Θ_color ⊗ Θ_density ⊗ Θ_brand
       其中 ⊗ 为偏函数组合 (按优先级覆盖)
       同态保持条件:
         Θ_combined(a ⊕ b) = Θ_combined(a) ⊕' Θ_combined(b)
       该条件要求: 各维度主题在组合运算上的定义一致
       例: 若 Θ_color 和 Θ_density 都涉及间距值，优先级必须明确

步骤 5: CSS 变量线性表示的工程实现
       ρ: T → CSS_Variables
       ρ(t₁ ⊕ t₂) = calc(var(--t₁) + var(--t₂))  (对间距)
       ρ(k ⊗ t) = calc(k * var(--t))             (标量乘法)
       该表示的同态性质:
         ρ(t₁ ⊕ t₂) = ρ(t₁) ⊕_CSS ρ(t₂)
       即: 设计端的群运算 = CSS 端的 calc() 运算
       这是设计代数到样式语言的"保结构映射"

结论: 设计令牌的代数结构为设计系统提供了三层形式化保证:
       1. 令牌层: 阿贝尔群保证样式组合的可预测性
       2. 变换层: 幺半群保证多平台输出的可维护性
       3. 主题层: 同态映射保证主题切换的保结构性
       这三层结构使设计系统从"视觉规范"进化为"可验证的计算系统"。
```

## 九、推理判定树/决策树

```text
设计令牌命名与组织决策树
│
├─► Q1: 令牌抽象层级?
│   ├─ 基础/原始 (primitive) ──→ 直接值 (如 #3B82F6, 16px)
│   ├─ 语义 (semantic) ──→ 用途命名 (如 color-primary, spacing-md)
│   └─ 组件 (component) ──→ 组件专用 (如 button-bg-color, input-border-width)
│
├─► Q2: 命名空间策略?
│   ├─ 扁平 ──→ 简单项目，令牌数 < 100
│   └─ 分层 (category/type/item) ──→ 企业级设计系统，支持自动化文档和类型生成
│
├─► Q3: 是否使用别名引用?
│   ├─ 是 ──→ 建立引用图 G_alias，强制 DAG 检测 (避免循环引用)
│   └─ 否 ──→ 所有令牌为原始值 (简单但重复度高)
│
├─► Q4: 主题支持维度?
│   ├─ 单色主题 ──→ 单维度 Θ: Token → Value
│   ├─ light/dark ──→ 双主题映射 + CSS Variables 运行时切换
│   └─ 多维度 (颜色+密度+品牌) ──→ 张量积主题 + 分层解析优先级
│
└─► Q5: 平台输出范围?
    ├─ Web only ──→ CSS Custom Properties + JS 对象
    ├─ Web + iOS/Android ──→ Style Dictionary (CSS / Swift / XML)
    └─ 全平台 (含Flutter/Desktop) ──→ Style Dictionary v3+ 或 Theo

主题切换技术选型决策树
│
├─► Q1: 切换时机?
│   ├─ 构建时 ──→ Style Dictionary 生成多主题 CSS 文件，切换时加载不同文件
│   ├─ 运行时 (无刷新) ──→ CSS Custom Properties (:root 变量覆盖)
│   └─ 系统偏好跟随 ──→ prefers-color-scheme 媒体查询 + CSS 变量
│
├─► Q2: 是否需要动画过渡?
│   ├─ 是 ──→ CSS transition on background-color/color 等属性
│   └─ 否 ──→ 即时切换 (变量值突变)
│
├─► Q3: 多维度主题组合?
│   ├─ 单维度 (仅颜色) ──→ data-theme="light" 属性选择器覆盖
│   └─ 多维度 ──→ 组合类名 .theme-light.density-compact 或 JS 解析优先级链
│
└─► Q4: SSR/服务端渲染?
    ├─ 是 ──→ 服务端注入初始主题类名/变量，避免闪烁 (FART)
    └─ 否 ──→ 客户端 JS 检测并设置主题
```
