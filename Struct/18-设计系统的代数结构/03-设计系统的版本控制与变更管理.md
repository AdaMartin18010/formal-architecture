## 一、知识体系思维导图

```text
设计系统的版本控制与变更管理
│
├─► SemVer (语义化版本控制)
│   ├─ MAJOR: Breaking Change (不兼容的 API 修改)
│   ├─ MINOR: 向后兼容的功能新增
│   ├─ PATCH: 向后兼容的问题修复
│   └─ 预发布: 1.0.0-alpha.1, 1.0.0-beta.2
│
├─► 设计系统的 Breaking Change 分类
│   ├─► 令牌层面
│   │   ├─ 令牌删除或重命名
│   │   ├─ 令牌值变更 (颜色、间距的语义漂移)
│   │   └─ 令牌类型变更 (color → gradient)
│   │
│   ├─► 组件层面
│   │   ├─ 组件 API 变更 (props 删除/重命名)
│   │   ├─ 组件行为变更 (交互逻辑改变)
│   │   ├─ 组件移除 (deprecated → removed)
│   │   └─ DOM 结构变更 (影响 CSS 选择器)
│   │
│   └─► 主题层面
│       ├─ 主题变量映射变更
│       └─ 默认主题切换 (light → dark)
│
├─► 迁移策略
│   ├─ 自动化迁移 (Codemod)
│   ├─ 废弃周期 (Deprecation Timeline)
│   ├─ 兼容性层 (Adapter / Bridge)
│   └─ 并行版本共存 (npm dist-tag)
│
├─► 变更沟通
│   ├─ CHANGELOG (Keep a Changelog 规范)
│   ├─ 迁移指南 (Migration Guide)
│   ├─ 设计令牌 diff 报告
│   └─ 视觉回归测试 (Chromatic / Applitools)
│
└─► 发布工作流
    ├─ 独立版本 (Independent): 每个包独立 SemVer
    ├─ 固定版本 (Fixed): 所有包共享版本号
    └─ 金丝雀发布 (Canary): 自动预发布通道
```

## 二、核心概念的形式化定义

### 2.1 SemVer 作为设计空间的分层

```text
定义 (语义化版本):
  版本 v = (major, minor, patch, pre, build)
  
  版本序: v₁ < v₂ ⟺ 字典序比较 (major, minor, patch)
  
  兼容性谓词:
    Compatible(v₁, v₂) ⟺ 
      v₁.major = v₂.major ∧ v₁.major > 0
      ∨ v₁.major = v₂.major = 0 ∧ v₁.minor = v₂.minor
      
    即: 同一 MAJOR 版本内向后兼容 (MAJOR > 0 时)
        0.y.z 的 MINOR 视为 MAJOR (开发阶段)

  设计系统的版本空间:
    每个设计系统 DS 的版本历史是一个全序集合:
      Versions(DS) = {v₁, v₂, ..., vₙ | v₁ < v₂ < ... < vₙ}
    
    变更类型函数:
      Δ: Versions(DS) × Versions(DS) → {MAJOR, MINOR, PATCH}
      Δ(vᵢ, vⱼ) = 从 vᵢ 到 vⱼ 的最大变更级别
```

### 2.2 Breaking Change 的形式化

```text
定义 (Breaking Change):
  设设计系统 DS 的公共接口为 API(DS) = Tokens ∪ Components ∪ Themes
  
  变更 δ: API(DS_v) → API(DS_v')
  
  δ 是 Breaking Change ⟺ 
    ∃consumer, consumer 在 v 下合法的使用方式在 v' 下非法
    
  具体判定:
    1. 令牌删除: t ∈ Tokens(v), t ∉ Tokens(v')  → MAJOR
    2. 令牌重命名: t ↦ t', 无 alias  → MAJOR
    3. 令牌值漂移: 
       若值变更影响视觉/布局且消费者无法覆盖 → MAJOR
       若值变更为 bug fix (如对比度修正) → PATCH
    4. 组件 props 删除: prop ∈ Props(v), prop ∉ Props(v') → MAJOR
    5. 组件 DOM 变更: 
       若变更影响消费者 CSS 选择器 → MAJOR
       若仅为内部实现优化 → PATCH
    6. 默认行为变更:
       若变更影响无障碍/键盘导航 → MAJOR
       若仅为性能优化 → MINOR/PATCH
```

### 2.3 迁移成本模型

```text
定义 (迁移成本):
  对于消费者项目 P，从版本 v 迁移到 v' 的成本:
    Cost(P, v → v') = α × |BreakingChanges| 
                      + β × |FilesAffected| 
                      + γ × |ManualReviewHours|
    
    α: 每处 breaking change 的修复成本系数
    β: 每受影响文件的调整成本系数
    γ: 人工审查的小时成本

  自动化迁移 (Codemod):
    若 ∃ codemod: Code(v) → Code(v')，则:
      Cost_auto(P, v → v') = Cost(P, v → v') × (1 - automation_rate)
      
    其中 automation_rate ∈ [0, 1] 为 Codemod 的覆盖率
```

## 三、多维矩阵对比

### 3.1 版本策略对比

| 策略 | 版本模式 | 优点 | 缺点 | 适用场景 |
|------|---------|------|------|---------|
| **独立版本** | 每个包独立 SemVer | 精准表达变更，消费者按需升级 | 版本矩阵复杂，协调困难 | 大型 Monorepo (Material-UI) |
| **固定版本** | 所有包共享版本号 | 简单一致，易于沟通 | 小变更导致全量版本 bump | 小型设计系统 (Bootstrap) |
| **范围版本** | ^x.y.z 自动升级 | 自动获取修复和功能 | 意外引入 breaking change | 内部项目 (可控环境) |
| **锁定版本** | 精确版本 | 确定性构建 | 错过安全修复 | 生产关键系统 |
| **金丝雀通道** | x.y.z-canary.n | 早期反馈，持续集成 | 不稳定，不适合生产 | 活跃开发阶段 |

### 3.2 变更管理工具对比

| 工具 | 变更检测 | 可视化 diff | 自动化发布 | 回归测试 | 生态集成 |
|------|---------|------------|-----------|---------|---------|
| **Changesets** | ✅ Git + 手动描述 | ⚠️ 文本 | ✅ 自动 bump + 发布 | ❌ | 优秀 (pnpm/yarn/npm) |
| **Lerna** | ✅ Conventional Commits | ❌ | ✅ 自动发布 | ❌ | 中等 |
| **Beachball** | ✅ 变更文件 | ❌ | ✅ 自动 bump | ❌ | Microsoft 生态 |
| **Chromatic** | ⚠️ 手动 | ✅ 像素级 diff | ❌ | ✅ 核心功能 | Storybook 深度集成 |
| **Applitools** | ⚠️ 手动 | ✅ AI 辅助 diff | ❌ | ✅ 核心功能 | 多框架支持 |

## 四、权威引用

> **Tom Preston-Werner** (SemVer 作者, 2013):
> "SemVer is a simple set of rules and requirements that dictate how version numbers are assigned and incremented. These rules are based on but not necessarily limited to pre-existing widespread common practices in use in both closed and open-source software."

> **Nathan Curtis** ("Versioning Design Systems", EightShapes, 2019):
> "Design system versioning is not just about code. A change to a color token is a breaking change if it causes visual regression across 50 products."

> **Brad Frost** ("Structuring Design Systems", 2022):
> "A design system is a product serving products. Like any product, it needs a roadmap, versioning strategy, and deprecation policy."

> **Martin Fowler** ("API Evolution", 2008):
> "The key to evolving APIs is to never break existing consumers. When you must introduce a breaking change, do it by adding the new behavior alongside the old, and deprecating the old over a long timeline."

```markdown
Migrate to `scale` prop for consistent sizing across components.
```

### 5.2 Codemod 自动化迁移

```javascript
// codemods/button-size-to-scale.js
// jscodeshift transform
module.exports = function(fileInfo, api) {
  const j = api.jscodeshift;
  const root = j(fileInfo.source);

  // 查找 <Button size="..." /> 并替换为 scale
  root.find(j.JSXOpeningElement, { name: { name: 'Button' } })
    .find(j.JSXAttribute, { name: { name: 'size' } })
    .forEach(path => {
      path.value.name.name = 'scale';
    });

  return root.toSource();
};

// 执行
// npx jscodeshift -t codemods/button-size-to-scale.js src/
```

### 5.3 废弃周期策略

```typescript
// 阶段 1: 标记废弃 (MINOR 版本)
interface ButtonProps {
  /** @deprecated Use `scale` instead. Will be removed in v3.0.0 */
  size?: 'sm' | 'md' | 'lg';
  scale?: 'compact' | 'default' | ' spacious';
}

// 阶段 2: 运行时警告 (下一个 MINOR)
function Button(props: ButtonProps) {
  if (props.size) {
    console.warn(
      '[@myds/core] Button "size" prop is deprecated. Use "scale" instead.'
    );
    // 兼容映射
    const scaleMap = { sm: 'compact', md: 'default', lg: 'spacious' };
    props.scale = scaleMap[props.size];
  }
  // ...
}

// 阶段 3: 彻底移除 (MAJOR 版本)
```

## 六、批判性总结

设计系统的版本控制远比普通软件库复杂，因为其**"API" 同时存在于代码层、视觉层和交互层**。一个颜色令牌的值变更（如将错误红色从 #FF0000 调整为 #DC2626）在代码层面是完全向后兼容的（变量名未变、类型未变），但在视觉层面可能是 breaking change——如果下游产品的品牌规范要求使用特定的红色色值。这导致 SemVer 的机械规则在设计系统领域需要**视觉语义的人工判定**：变更的兼容性不仅取决于代码接口的稳定性，还取决于设计决策的稳定性。

更深层的问题是**版本传播的级联效应**。当一个基础设计令牌（如 --spacing-unit）发生 MAJOR 变更时，所有依赖该令牌的组件、主题、甚至消费者的自定义样式都会受到影响。在大型组织中，这种级联可能导致数十个产品团队同时进行迁移，产生巨大的协调成本。解决方案是**语义版本的多级分解**：基础令牌层、组件层、主题层各自独立版本化，通过依赖范围（如组件依赖令牌层的 ^2.x）控制变更的传播速度。这正是 Material-UI (MUI) 采用独立版本策略的代数动机。

视觉回归测试（Chromatic、Applitools）为设计系统的版本控制提供了**形式化的 diff 能力**——将"这个变更是否 breaking"从主观判断转化为像素级比较的客观度量。但视觉回归也有盲区：它只能检测渲染输出的变化，无法检测无障碍属性（ARIA 角色）或交互行为（键盘焦点）的变更。完整的设计系统版本控制需要**多模态回归测试**的矩阵：视觉 + 无障碍 + 交互 + 性能，而这在当前工具链中仍是碎片化的。未来的设计系统 CI/CD 可能集成**形式化设计规范验证器**，在构建时自动判定变更的 SemVer 级别，消除人工判定的主观性和不一致性。
