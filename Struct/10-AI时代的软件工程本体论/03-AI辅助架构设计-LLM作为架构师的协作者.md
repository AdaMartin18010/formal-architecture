# AI辅助架构设计：LLM作为架构师的协作者

> **定位**：LLM不会取代架构师，但会改变架构师的工作方式——从"手绘全部"到"与AI共同探索设计空间"。AI是强大的模式检索和生成工具，但架构决策的最终责任仍在于人类。
>
> **核心命题**：AI辅助架构设计的最佳模式是"人类主导决策，AI辅助生成和验证"——而非相反。

---

## 一、思维导图：AI辅助架构设计的工作流

```text
AI辅助架构设计
│
├─【输入】
│   ├─ 业务需求（自然语言）
│   ├─ 约束条件（性能、安全、成本）
│   └─ 现有系统上下文
│
├─【AI辅助阶段】
│   ├─ 模式检索（相似场景的最佳实践）
│   ├─ 方案生成（多种架构选项）
│   ├─ 权衡分析（Pros/Cons矩阵）
│   ├─ 代码骨架生成（接口、模块划分）
│   └─ 文档生成（ADR、架构图描述）
│
├─【人类决策阶段】
│   ├─ 方案选择（基于业务上下文）
│   ├─ 风险评估（AI无法感知的组织风险）
│   ├─ 演进路线规划（技术债务 vs 交付速度）
│   └─ 最终Review和确认
│
└─【验证阶段】
    ├─ AI生成测试方案
    ├─ 人类Review关键路径
    └─ 原型验证（PoC）
```

## 二、AI辅助架构设计的具体模式

| 任务 | **AI能力** | **人类角色** | **最佳实践** |
|------|-----------|------------|------------|
| **需求分析** | 提取功能/非功能需求、识别冲突 | 确认业务优先级、补充隐性需求 | 结构化prompt + 迭代澄清 |
| **模式推荐** | 基于约束匹配架构模式 | 评估模式适配度、考虑组织因素 | 提供详细上下文 |
| **方案对比** | 生成多方案矩阵、量化指标 | 加入定性因素（团队技能、政治） | 强制AI列出反方观点 |
| **接口设计** | 生成API规范、数据模型 | Review边界情况、演进性 | 迭代精化 |
| **技术选型** | 对比技术栈、提供基准数据 | 考虑生态成熟度、招聘难度 | 验证数据时效性 |
| **文档生成** | ADR、架构图文本、运行手册 | 审查准确性、补充决策背景 | 模板化 + 人工校验 |

## 三、决策树：何时信任AI建议

```
AI建议可信度评估：

  1. 问题是否有标准解？
     ├── 是（CRUD、标准微服务拆分）
     │   └── AI可信度高（80%+）
     │       → 快速采纳，人工抽样验证
     └── 否（创新架构、领域特定约束）
         └── AI可信度中低（30-60%）
             → 作为灵感来源，深度人工分析

  2. 失败的代价？
     ├── 低（内部工具、可快速回滚）
     │   └── 可接受AI建议 + 快速迭代
     └── 高（核心系统、安全关键）
         └── 严格人工审查 + 形式化验证

  3. 是否有可验证的客观标准？
     ├── 是（性能指标、测试通过率）
     │   └── AI建议 + 自动化验证
     └── 否（可维护性、可演进性）
         └── 依赖人类专家判断
```

## 四、AI辅助架构的陷阱

```
陷阱1：过度自信
  症状：AI生成看似专业的架构图，但包含不存在的云服务
  防御：验证所有技术引用，交叉检查官方文档

陷阱2：模式匹配偏差
  症状：AI推荐"微服务"因为训练数据中微服务最常见
  防御：明确约束条件，要求AI论证其他选项

陷阱3：上下文丢失
  症状：多轮对话后AI忘记早期约束
  防御：每轮重申关键约束，使用结构化prompt

陷阱4：安全盲点
  症状：AI生成的架构缺少安全纵深防御
  防御：强制安全Review清单，不依赖AI的安全建议

陷阱5：过时知识
  症状：AI推荐已弃用的技术或模式
  防御：验证技术版本，参考最新官方文档
```

## 五、概念定义与属性关系

| 概念 | 定义 | 属性 | 示例 | 反例 |
|------|------|------|------|------|
| **AI辅助架构** | 使用LLM辅助架构设计的实践 | 高效、需验证、有局限 | AI生成方案对比矩阵 | 完全依赖AI决策 |
| **模式检索** | 从训练数据中提取相似场景的最佳实践 | 快速、覆盖常见场景 | AI推荐电商系统的微服务拆分 | 创新领域无模式可检索 |
| **设计空间探索** | 与AI共同探索多种架构选项 | 系统性、需人类引导 | 生成3-5种方案并比较 | 只接受AI的第一个建议 |
| **幻觉** | AI生成看似合理但错误的信息 | 概率性、难检测 | AI虚构不存在的AWS服务 | 准确的技术引用 |
| **架构决策记录(ADR)** | 记录架构决策及其理由的文档 | 可追溯、可学习 | AI生成ADR初稿 | 无文档的口头决策 |
