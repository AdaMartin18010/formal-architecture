> **来源映射**: View/05.md §7.1
> **国际权威参考**: "Consumer-Driven Contracts: A Service Evolution Pattern" (Ian Robinson, 2006), Pact Documentation, "API Design Patterns" (JJ Geewax)

## 一、知识体系思维导图

```text
契约测试与类型同构
│
├─► 消费者驱动契约 (CDC)
│   ├─ 消费者测试: 定义期望的请求/响应
│   ├─ 契约文件: JSON 规约 (Pact 格式)
│   ├─ 提供者验证: 验证服务是否满足契约
│   └─ 工作流程: 消费者 → 契约 → 提供者 (TDD 循环)
│
├─► 类型同构 (Type Isomorphism)
│   ├─ 同构定义: A ≅ B ⟺ ∃f: A→B, g: B→A, f∘g = id_B, g∘f = id_A
│   ├─ 前后端类型映射: TS 类型 ⟺ 后端 DTO/Schema
│   ├─ OpenAPI → TypeScript: openapi-typescript, Orval
│   └─ GraphQL Codegen: Schema → TS 类型
│
├─► 契约漂移检测
│   ├─ 破坏式变更 (Breaking): 删除字段、修改类型、删除端点
│   ├─ 非破坏式变更 (Non-breaking): 添加可选字段
│   ├─ 兼容矩阵: 消费者版本 × 提供者版本
│   └─ CI 阻断: 契约失败 → 合并阻断
│
├─► 工具生态
│   ├─ Pact: 消费者驱动契约测试的事实标准
│   ├─ Pact Broker: 契约版本管理与兼容性矩阵
│   ├─ MSW (Mock Service Worker): 浏览器/Node 统一 Mock
│   ├─ OpenAPI: 描述优先的契约规范
│   └─ Prism: OpenAPI Mock 服务器
│
└─► 形式化边界
    ├─ 类型同构不能捕获运行时约束 (string 格式、范围)
    ├─ 契约不能表达非功能性需求 (延迟、可用性)
    └─ 契约测试 ≠ 功能测试: 只验证接口形状，不验证业务逻辑
```

## 二、核心概念的形式化定义

### 2.1 消费者驱动契约的形式化

```text
定义 (消费者驱动契约):
  设 API 端点 e = (method, path, params, requestBody, responseBody)
  
  消费者期望:
    C = ⟨matchers, min_examples⟩
    matchers: 对请求/响应各字段的匹配规则
      - 精确匹配: equals(value)
      - 类型匹配: like(type), eachLike(template)
      - 正则匹配: regex(pattern)
      - 枚举匹配: oneOf([...])
      
  契约文件 (Pact Format):
    {
      "consumer": { "name": "FrontendApp" },
      "provider": { "name": "UserService" },
      "interactions": [
        {
          "description": "get user by id",
          "request": { "method": "GET", "path": "/users/123" },
          "response": {
            "status": 200,
            "body": { "id": 123, "name": "Alice" }
          }
        }
      ]
    }
    
  契约验证:
    提供者 P 满足契约 C ⟺ ∀interaction i ∈ C:
      P(i.request) 的响应与 i.response 的 matchers 匹配
```

### 2.2 类型同构与 API 边界

```text
定义 (类型同构):
  设前端类型 A_frontend，后端类型 A_backend
  
  A_frontend ≅ A_backend (同构) ⟺
    存在序列化函数 serialize: A_backend → JSON
    存在反序列化函数 deserialize: JSON → A_frontend
    使得 deserialize ∘ serialize ≈ id (在业务语义下)
    
  注意:
    - JSON 是中间媒介，本身无类型
    - 同构在运行时可能被破坏 (如后端返回 string 而非 number)
    - 需要运行时验证 (Zod, io-ts, Yup) 来补充静态类型
    
  类型生成链:
    OpenAPI Schema ──openapi-typescript──► TS 类型 ──Zod──► 运行时校验
    
  同构失效的常见原因:
    1. 后端字段重命名但未同步前端
    2. 类型拓宽: backend int → JSON number → frontend number (可能溢出)
    3. 可选字段变成必填 (Breaking Change)
    4. 时间格式差异: ISO 8601 vs Unix timestamp
```

### 2.3 契约漂移的形式化分类

```text
定义 (变更兼容性):
  设 Schema S₁, S₂，兼容性判定 compat(S₁, S₂):
  
  向后兼容 (Backward Compatible):
    所有满足 S₁ 的数据也满足 S₂
    示例: 添加可选字段, 放宽枚举范围
    
  向前兼容 (Forward Compatible):
    所有满足 S₂ 的数据也满足 S₁
    示例: 添加可选字段 (从旧 schema 视角)
    
  非兼容 (Breaking):
    既不向后也不向前兼容
    示例: 删除字段, 修改必填性, 改变类型
    
  版本策略:
    URL 版本: /v1/users → /v2/users
    Header 版本: Accept: application/vnd.api.v1+json
    契约矩阵: 消费者版本 × 提供者版本的兼容性网格
```

| 维度 | Pact (CDC) | OpenAPI + Codegen | MSW | GraphQL Codegen | tRPC |

| **契约驱动** | **是** (消费者定义) | **是** (描述优先) | 否 (Mock 工具) | 是 (Schema 优先) | **是** (共享类型) |

## 四、权威引用

> **Ian Robinson** (Thoughtworks, CDC 模式提出者):
> "Consumer-driven contracts are a pattern for evolving services in a microservices architecture. The consumer defines the contract, and the provider must satisfy it."

> **Martin Fowler** ("Microservices" 文章):
> "Consumer-driven contracts give the service provider insight into how their services are used, allowing them to make changes with confidence."

> **Matt Fellows** (Pact 核心维护者):
> "Pact is not a replacement for functional testing. It's a replacement for integration testing at the API boundary."

## 五、工程实践与代码示例

### 5.1 Pact 消费者测试与契约生成

```javascript
// consumer/user.pact.spec.js
import { PactV3 } from '@pact-foundation/pact';
import { userService } from './userService';

const provider = new PactV3({
  consumer: 'FrontendApp',
  provider: 'UserService',
  dir: './pacts',
});

describe('UserService contract', () => {
  it('returns user by id', () => {
    provider
      .given('user with id 123 exists')
      .uponReceiving('a request for user 123')
      .withRequest({
        method: 'GET',
        path: '/users/123',
        headers: { Accept: 'application/json' },
      })
      .willRespondWith({
        status: 200,
        headers: { 'Content-Type': 'application/json' },
        body: {
          id: 123,
          name: 'Alice',
          email: 'alice@example.com',
        },
      });

    return provider.executeTest(async (mockserver) => {
      const user = await userService.getUser(123, mockserver.url);
      expect(user.name).toBe('Alice');
    });
  });
});
```

### 5.2 OpenAPI → TypeScript + Zod 运行时校验

```typescript
// 由 openapi-typescript 从 OpenAPI 生成
import type { paths } from './api-types';

// 由 Orval/Zod 生成运行时校验
import { UserSchema } from './api-schemas';

async function fetchUser(id: number): Promise<paths['/users/{id}']['get']['responses']['200']['content']['application/json']> {
  const res = await fetch(`/users/${id}`);
  const data = await res.json();
  
  // 运行时校验补充编译时类型
  const parsed = UserSchema.safeParse(data);
  if (!parsed.success) {
    console.error('API response violates contract:', parsed.error);
    throw new Error('Contract violation');
  }
  
  return parsed.data;
}
```

## 六、批判性总结

契约测试是**分布式系统接口治理**的关键基础设施，但它不是银弹。消费者驱动契约的核心洞察是**将集成测试的验证责任前移到单元测试阶段**——消费者在本地通过 Mock Server 定义期望，提供者在 CI 中验证是否满足所有消费者的契约。这比传统的"部署到测试环境再联调"模式快一个数量级，且能提前发现 Breaking Change。

但契约测试的**形式化边界**必须清醒认识：Pact 验证的是请求/响应的**结构匹配**，而非**业务正确性**。一个契约完全通过的 API 仍可能返回业务上错误的数据（如返回了另一个用户的信息）。契约测试与功能测试是**正交互补**的关系，而非替代关系。

类型同构的理想在全栈 TypeScript 项目中几乎达到了**工程极限**：tRPC 通过共享类型定义实现了从前端到后端数据库的端到端类型安全，编译时就能捕获接口不匹配。但在异构技术栈（前端 TS + 后端 Java/Python/Go）中，OpenAPI 作为**中间语言**仍是最好的 Lingua Franca。它的局限在于：JSON Schema 的表达能力弱于现代类型系统——无法表达 branded types、nominal typing、依赖类型等高级构造。这导致"类型同构"在实践中是**近似同构**而非数学意义上的严格同构。

2026 年的趋势是**契约即代码**（Contract as Code）——将 OpenAPI/Pact 契约纳入版本控制，与业务代码同等对待，通过 CI 自动检测漂移。这要求团队建立**API 治理委员会**，对 Breaking Change 进行严格的审批流程。在大型组织中，契约测试已经从技术工具进化为**组织协作协议**。
