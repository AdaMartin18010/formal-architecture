## 二、测试类型的形式化描述

| 测试类型 | 覆盖对象 | 形式化描述 | 前端实现 |
|---------|---------|-----------|---------|
| **单元测试** | 函数/纯逻辑 | `f: A → B` 的点覆盖: `∀a∈A, f(a)` 符合规范 | Jest, Vitest, renderHook |
| **集成测试** | 组件组合 | 验证 `∘` 操作的封闭性 | React Testing Library |
| **契约测试** | API 边界 | 类型同构验证 `A_frontend ≡ A_backend` | Pact, MSW, OpenAPI → TS |
| **E2E 测试** | 用户行为路径 | 状态机遍历: `∀s ∈ S, ∃路径 s₀ →* s_accept` | Playwright, Cypress |
| **视觉回归** | 渲染输出像素 | 图像度量空间: `d(P₁, P₂) < ε` | Chromatic, Percy |
| **可访问性** | DOM 语义结构 | 图可达性: 所有交互节点必须可达 | axe-core, Pa11y |
| **性能测试** | 运行时指标 | 概率分布验证: `P(LCP < 2.5s) > 0.95` | Lighthouse CI, Web Vitals |
| **模糊测试** | 输入空间边界 | 随机采样: `∀x ∈ Domain, P(crash) → 0` | fast-check, Property-Based |

## 三、视觉回归的度量空间

```text
定义: 像素空间 P = [0,255]^(W×H×3) (RGB 图像)
  度量 d: P × P → ℝ⁺

常用度量:
  1. L0 (汉明距离): d₀(p₁, p₂) = |{i | p₁[i] ≠ p₂[i]}|
     用于: 像素级精确比较 (过于严格)

  2. L1 (曼哈顿): d₁(p₁, p₂) = Σ|p₁[i] - p₂[i]|
     用于: 颜色差异累积

  3. L2 (欧几里得): d₂(p₁, p₂) = √(Σ(p₁[i] - p₂[i])²)
     用于: 感知颜色差异

  4. SSIM (结构相似): d_SSIM = f(亮度, 对比度, 结构)
     用于: 人眼感知相似度 (更符合视觉系统)

工程决策:
  精确像素比较 (L0) 是反人类的
  ∵ 子像素渲染、字体抗锯齿、GPU 差异导致 L0 差异 ≠ 视觉差异

  ∴ 应使用 SSIM + 阈值 ε:
    若 d_SSIM(p₁, p₂) < ε, 则判定为 "视觉等价"

  这与前端工程中的 "视觉快照测试" 实践一致:
    Chromatic 使用 1px 容差 + 忽略区域
    Percy 使用稳定化 (Stabilization) 预处理
```

## 六、批判性总结

测试的本质是对程序行为空间的采样。单元测试采样函数定义域的点；E2E 测试采样用户行为路径；视觉回归测试采样渲染输出空间。理解这些采样的覆盖率和盲区，是设计有效测试策略的基础。

视觉回归测试中，精确像素比较 (L0) 是反人类的，因为子像素渲染、字体抗锯齿、GPU 差异导致像素差异不等于视觉差异。应使用 SSIM + 阈值 ε 来判定"视觉等价"。
