## 一、知识体系思维导图

```text
前端性能测试
│
├─► 性能指标 (Web Vitals)
│   ├─ LCP (Largest Contentful Paint): 最大内容渲染时间
│   │   └─ 目标: < 2.5s (Good), < 4s (Needs Improvement)
│   ├─ FID / INP (Interaction to Next Paint): 交互响应延迟
│   │   └─ 目标: INP < 200ms
│   ├─ CLS (Cumulative Layout Shift): 累积布局偏移
│   │   └─ 目标: < 0.1
│   ├─ TTFB (Time to First Byte): 首字节时间
│   │   └─ 目标: < 600ms
│   └─ FCP (First Contentful Paint): 首次内容绘制
│       └─ 目标: < 1.8s
│
├─► 基准测试方法论
│   ├─► 统计模型
│   │   ├─ 多次运行取中位数 (Median) 而非平均数
│   │   ├─ 排除异常值: IQR (四分位距) 方法
│   │   ├─ 置信区间: 95% CI 确保结果稳定性
│   │   └─ 最小样本量: n ≥ 30 次运行
│   │
│   ├─► 控制变量
│   │   ├─ 硬件一致: 同一台机器/CI环境
│   │   ├─ 浏览器隔离: 新无痕窗口
│   │   ├─ 缓存清除: 每次测试前清除
│   │   └─ 网络模拟: Lighthouse 的节流模型
│   │
│   └─► 回归检测
│       ├─ 基线对比: 与主分支对比
│       ├─ 阈值告警: 性能退化 > 10% 阻断合并
│       └─ 趋势分析: 长期性能变化追踪
│
├─► 测试工具
│   ├─ Lighthouse: Google 综合性能审计
│   ├─ WebPageTest: 多地点、多设备测试
│   ├─ Chrome DevTools Performance: 火焰图分析
│   ├─ Benchmark.js: JavaScript 微基准
│   └─ k6 / Artillery: 负载测试
│
└─► 概率模型
    ├─ 用户性能分布: 并非所有用户体验相同
    ├─ P50 (中位数): 典型用户体验
    ├─ P75: 大多数用户 (3/4) 的体验
    ├─ P95: 边缘用户/慢速设备
    └─ 长尾效应: P99 往往比 P50 差 10 倍
```

## 二、核心概念的形式化定义

### 2.1 性能指标的统计分布

```text
定义 (性能指标分布):
  设性能指标 X (如 LCP) 的观测值集合为 {x₁, x₂, ..., xₙ}

  统计量:
    中位数: P50 = median(X)
    第 p 百分位: Pp = inf{x | P(X ≤ x) ≥ p/100}

  Web Vitals 阈值:
    Good:        X ≤ T_good
    Needs Improvement: T_good < X ≤ T_poor
    Poor:        X > T_poor

  示例 (LCP):
    Good: X ≤ 2.5s
    Needs Improvement: 2.5s < X ≤ 4.0s
    Poor: X > 4.0s

  用户体验分布:
    P(Good) = count(xᵢ ≤ T_good) / n
    目标: P(Good) ≥ 0.75 (即 P75 ≤ T_good)
```

### 2.2 基准测试的置信区间

```text
定义 (基准测试统计):
  设 n 次独立运行的结果: X₁, X₂, ..., Xₙ

  样本均值: X̄ = (1/n) Σ Xᵢ
  样本标准差: s = √[(1/(n-1)) Σ(Xᵢ - X̄)²]

  95% 置信区间 (t分布):
    CI = X̄ ± t₀.₉₇₅,ₙ₋₁ × (s/√n)

  样本量确定:
    若要检测 δ 的相对差异，显著性 α=0.05，功效 1-β=0.8:
      n ≈ 16 × (CV/δ)²
      其中 CV = s/X̄ (变异系数)

  实际建议:
    - 前端基准测试: n ≥ 30
    - CI 宽度 < 5% 的均值
```

### 2.3 性能指标的概率分布模型

```text
定义 (性能指标的长尾分布):
  设性能指标 X 服从对数正态分布或 Pareto 分布:

  对数正态分布:
    ln(X) ~ N(μ, σ²)
    P(X ≤ x) = Φ((ln x - μ)/σ)

  Pareto 分布 (长尾):
    P(X > x) = (xₘ/x)^α,  x ≥ xₘ

  前端意义:
    - P50 反映典型用户体验
    - P95/P99 反映边缘设备/弱网用户
    - 长尾比 = P99 / P50 > 10 时，说明存在系统性瓶颈

  Web Vitals 的 Field Data 分布:
    Good:        P75 ≤ threshold_good
    Needs Improvement: threshold_good < P75 ≤ threshold_poor
    Poor:        P75 > threshold_poor

  注意: Google 使用 P75 而非均值作为评判标准
```

### 2.4 性能预算的形式化定义

```text
定义 (性能预算):
  设资源类型 R = {js, css, images, fonts, third-party}

  预算函数:
    B: R → ℕ (字节数) ∪ Time (毫秒)

  示例:
    B(js) = 200 KB (gzip)
    B(css) = 50 KB
    B(images) = 1 MB
    B(LCP) = 2500 ms
    B(TTI) = 3500 ms

  预算约束:
    ∀r ∈ R: size(r) ≤ B(r)

  回归检测:
    设基线性能 P_base, 当前性能 P_current
    若 (P_current - P_base) / P_base > δ (通常 10%):
      触发 CI 失败，阻断合并
```

module.exports = {

    },
  },

### 4.2 性能基准测试

```javascript
// Benchmark.js 微基准
const Benchmark = require('benchmark');

const suite = new Benchmark.Suite();

suite
  .add('Array#indexOf', () => {
    [1, 2, 3, 4, 5].indexOf(3);
  })
  .add('Set#has', () => {
    new Set([1, 2, 3, 4, 5]).has(3);
  })
  .on('cycle', (event) => {
    console.log(String(event.target));
  })
  .on('complete', function() {
    console.log('Fastest is ' + this.filter('fastest').map('name'));
  })
  .run({ async: true });

// 输出示例:
// Array#indexOf x 50,123,456 ops/sec ±1.23% (91 runs sampled)
// Set#has     x 85,234,567 ops/sec ±0.89% (95 runs sampled)
// Fastest is Set#has
```

### 4.3 Web Vitals RUM 采集与监控

```javascript
// 使用 web-vitals 库采集真实用户性能数据
import { onLCP, onINP, onCLS, onTTFB } from 'web-vitals';
import { sendToAnalytics } from './analytics';

// 上报所有核心指标
onLCP(sendToAnalytics, { reportAllChanges: true });
onINP(sendToAnalytics, { reportAllChanges: true });
onCLS(sendToAnalytics, { reportAllChanges: true });
onTTFB(sendToAnalytics);

// analytics.js: 按百分位聚合
function sendToAnalytics(metric) {
  const body = JSON.stringify({
    name: metric.name,
    value: metric.value,
    rating: metric.rating, // 'good' | 'needs-improvement' | 'poor'
    delta: metric.delta,
    id: metric.id,
    navigationType: metric.navigationType,
  });

  // 使用 sendBeacon 确保可靠上报
  if (navigator.sendBeacon) {
    navigator.sendBeacon('/analytics/vitals', body);
  } else {
    fetch('/analytics/vitals', { body, method: 'POST', keepalive: true });
  }
}
```

## 六、批判性总结

性能测试的**概率本质**常被工程师忽视：Web Vitals 的"Good"阈值不是魔法数字，而是基于用户留存率大数据的统计拐点——Google 的研究表明，当 LCP 超过 2.5s 时，用户跳出率显著增加。但这不意味着 2.4s 就是"安全"的，性能是一个**连续分布**，每毫秒的改善都会带来用户体验的提升。

基准测试的**统计严谨性**在前端领域几乎被完全忽略：大多数开发者运行一次 Lighthouse 就得出结论，但单次运行受 CPU 调度、网络抖动、GC 暂停等随机因素影响， variance 可能高达 20%。真正的基准测试需要 ≥30 次运行，计算置信区间，排除异常值——这是科学研究的标准，却被工程实践视为"过度严谨"。

INP (Interaction to Next Paint) 取代 FID 是 Google 对交互性能认知的**重大升级**：FID 只测量首次交互的输入延迟，忽略了处理时间和渲染时间；INP 测量所有交互的完整响应周期（输入→处理→渲染），更真实地反映用户感知的"卡顿"。这印证了性能工程的核心原则：**测量用户感知的指标，而非系统内部的指标**。

性能预算（Performance Budget）是**将性能从"事后优化"转变为"设计约束"**的工程实践。它的形式化本质是为一组可测量指标设定硬边界，并在 CI 中强制执行。但预算的设定需要基于数据而非主观臆断——2026 年的最佳实践是使用**P75 Field Data**作为预算基线，而非 Lab Data。Lab Data 在受控环境下测量，可重复但无法反映真实用户的设备、网络和认知差异；Field Data（RUM）来自真实用户，是性能工程的"地面真相"。两者的差距有时会达到 2-3 倍，仅用 Lab Data 做决策会导致资源错配。

|---------|---------|---------|---------|-----------|
| **性能指标分布** | LCP/FID/INP/CLS/TTFB 百分位 | Web Vitals 阈值函数 | ★★★★★ | X ~ LogNormal(μ,σ²) |

| **Lab vs Field Data** | 受控环境 vs 真实用户 RUM | 性能指标分布 | ★★★★☆ | 偏差 Δ = E[X_lab] - E[X_field] |

## 八、形式化推理链

**推理链 R1：从性能分布到回归检测的完备性**

```text
前提:
  P1: 设性能指标 X 的观测样本 {x₁, x₂, ..., xₙ}, n ≥ 30
  P2: X 服从对数正态分布: ln(X) ~ N(μ, σ²)
  P3: 基线性能 P_base 的置信区间 CI_base = [L_base, U_base]

推导:
  Step 1: 样本统计量计算
    X̄ = (1/n) Σ xᵢ  (样本均值)
    s = √[(1/(n-1)) Σ(xᵢ - X̄)²]  (样本标准差)
    CV = s / X̄  (变异系数)

  Step 2: 置信区间构建
    95% CI = X̄ ± t₀.₉₇₅,ₙ₋₁ × (s/√n)
    要求: CI 宽度 < 5% × X̄  (统计稳定性准则)

  Step 3: 回归检测判定
    设当前性能 P_current, 基线 P_base, 阈值 δ = 10%
    回归警报 ⟺ (P_current - P_base) / P_base > δ
              ∨ P_current > U_base × (1 + δ)

  Step 4: 统计功效分析
    功效 1-β = P(拒绝 H₀ | H₁ 为真)
    样本量: n ≈ 16 × (CV/δ)²  (Cohen, 1988)

结论 (定理 T-PT1):
  性能回归检测的可靠性取决于:
    Reliability(n, δ, CV) = 1 - β(n, δ, CV)
    当 n ≥ 30, CV ≤ 0.15, δ = 0.10 时，Reliability > 0.80。
    前端工程中单次 Lighthouse 运行的检测功效不足 0.30，
    必须执行多次运行并计算置信区间。
```

**推理链 R2：从用户分布到性能预算的形式化**

```text
前提:
  P1: 设用户群体 U, 每个用户 u ∈ U 的体验指标 X_u
  P2: Google Web Vitals 使用 P75 作为评判标准
  P3: 性能预算 B 为硬约束: ∀r ∈ R, size(r) ≤ B(r)

推导:
  Step 1: 用户体验分布
    P(Good) = P(X ≤ T_good) ≥ 0.75
    即: P75 ≤ T_good

  Step 2: Field Data 作为预算基线
    设 Field Data P75 = x₇₅_field
    预算设定: B(LCP) = x₇₅_field × (1 - margin)
    其中 margin ∈ [0.10, 0.20] 为安全余量

  Step 3: Lab Data 验证
    Lab Data 应满足: P75_lab ≤ B(LCP)
    若 P75_lab > B(LCP): 预算被突破，阻断合并

  Step 4: 长尾风险量化
    长尾比 ρ = P99 / P50
    若 ρ > 10: 存在系统性瓶颈，需专项优化

结论 (定理 T-PT2):
  性能预算的最优设定基于 Field Data 的 P75 分位:
    B*(r) = P75_field(r) × (1 - margin)
    仅使用 Lab Data 设定预算会导致系统性低估真实用户体验，
    误差可达 2-3 倍 (Beyer et al., 2016)。
```

## 九、推理判定树/决策树

```text
性能测试策略选择决策树
│
├─► 测试类型判定
│   ├─ [综合审计] ──► Lighthouse CI (定期审计)
│   ├─ [真实环境验证] ──► WebPageTest (多地点/多设备)
│   ├─ [算法微基准] ──► Benchmark.js (函数级对比)
│   ├─ [API 负载测试] ──► k6 / Artillery (压力/容量)
│   └─ [开发调试] ──► Chrome DevTools Performance (火焰图)
│
├─► 数据采集方式判定
│   ├─ [Lab Data] ──► CI 环境中 controlled 测试
│   │   └─ 要求: n ≥ 30 次运行，取中位数，排除异常值 (IQR)
│   └─ [Field Data (RUM)] ──► web-vitals 库采集真实用户
│       └─ 要求: 按百分位聚合 (P50/P75/P95/P99)
│
├─► 回归检测判定
│   ├─ [性能退化 > 10%] ──► CI 失败，阻断合并
│   ├─ [性能退化 5%~10%] ──► WARNING，需人工审查
│   └─ [性能退化 < 5%] ──► 通过，允许合并
│
└─► 预算管理判定
    ├─ [资源大小超限] ──► 阻断构建 (如 JS > 200KB gzip)
    ├─ [核心指标突破] ──► 阻断合并 (如 LCP > 2500ms)
    └─ [预算内] ──► 通过，更新基线
```
