# AI辅助形式化验证：从代码到证明的自动化管道

> **定位**：AI正在进入形式化验证领域——从自动生成规范到辅助证明构造。虽然当前能力有限，但"AI生成候选 + 机器验证"的管道代表了软件可靠性工程的下一个前沿。
>
> **核心命题**：形式化验证的最大瓶颈不是计算能力，而是人工编写规范和证明的智力成本。AI可能降低这个门槛，但不会消除对人类专家的需求。

---

## 一、思维导图：AI辅助验证管道

```text
AI辅助形式化验证
│
├─【阶段1：规范生成】
│   ├─ 从自然语言需求生成TLA+/Coq规范
│   ├─ 从代码提取不变量候选
│   └─ 人类Review和修正
│
├─【阶段2：证明辅助】
│   ├─ LLM建议证明策略（tactics）
│   ├─ 自动引理生成
│   └─ 反例解释和修复建议
│
├─【阶段3：验证执行】
│   ├─ 模型检查器（TLC, UPPAAL）
│   ├─ 定理证明器（Coq, Isabelle, Lean）
│   └─ SMT求解器（Z3, CVC5）
│
├─【阶段4：反馈迭代】
│   ├─ 验证失败 → 反例分析
│   ├─ LLM解释失败原因
│   └─ 生成修复建议
│
└─【当前局限】
    ├─ LLM生成规范准确率：30-50%
    ├─ 复杂不变量仍需人工
    └─ 证明策略建议质量不稳定
```

## 二、AI辅助验证的具体模式

### 2.1 从代码生成规范

```python
# 输入：Python代码
def transfer(from_acc, to_acc, amount):
    if from_acc.balance >= amount:
        from_acc.balance -= amount
        to_acc.balance += amount
        return True
    return False

# LLM生成的不变量候选：
# 1. from_acc.balance + to_acc.balance == total（全局守恒）
# 2. from_acc.balance >= 0（无透支）
# 3. amount >= 0（非负转账）

# 人类验证后转化为TLA+：
# Invariant ==
#   \A a, b \in Accounts:
#     a /= b => account[a] + account[b] = initial_total
```

### 2.2 证明策略建议

```coq
(* Coq证明中，LLM建议tactics *)

Theorem add_commutative : forall n m, n + m = m + n.
Proof.
  (* LLM建议：使用归纳法 *)
  intros n m.
  induction n as [| n' IHn'].
  - (* 基础情况：0 + m = m + 0 *)
    simpl.
    (* LLM建议：使用已证明的引理 *)
    apply add_0_r.
  - (* 归纳步骤：S n' + m = m + S n' *)
    simpl.
    (* LLM建议：重写使用归纳假设 *)
    rewrite IHn'.
    (* LLM建议：对称性 *)
    symmetry.
    apply add_succ_r.
Qed.
```

## 三、工具链对比

| 工具 | **验证类型** | **AI集成** | **适用场景** | **学习曲线** |
|------|-----------|-----------|------------|------------|
| **TLA+/TLC** | 模型检测 | LLM生成规范草案 | 分布式协议 | 中 |
| **Coq** | 定理证明 | LLM建议tactics | 数学证明、程序验证 | 高 |

| **Dafny** | 自动验证 | LLM生成规范注解 | 程序正确性 | 中 |
| **Isabelle** | 定理证明 | 有限AI集成 | 大规模验证（seL4） | 高 |

## 四、历史人物原话

> "The key challenge in automated theorem proving is not finding a proof, but finding a proof that a human can understand and trust." —— Emily First et al., "Baldur: Whole-Proof Generation and Repair with Large Language Models", *arXiv*, 2023

> "Large language models can generate correct proofs when guided by a sketch... but the gap between generated proofs and human-verifiable understanding remains significant." —— Albert Q. Jiang et al., "Draft, Sketch, and Prove", *NeurIPS*, 2022

> "The dream of fully automated formal verification is still distant. Human insight remains indispensable for specifying what needs to be proved." —— Leslie Lamport, *Specifying Systems*, 2002

## 五、当前能力边界

```
LLM在形式化验证中的能力评估：

  ✓ 能做的：
    - 将自然语言需求翻译为简单TLA+规范（~40%一次正确）
    - 解释模型检查器的反例（~60%准确率）
    - 生成常见不变量候选（类型安全、边界检查）
    - 建议证明策略（简单归纳、案例分析）

  ✗ 不能做的：
    - 生成复杂时序逻辑属性
    - 构造非平凡的不变量（如分布式协议的安全属性）
    - 完成需要深度数学洞察的证明
    - 保证生成规范的正确性

  人机协作最佳模式：
    - AI生成候选 → 人类筛选和修正 → 机器验证 → 迭代
```

## 五、概念定义与属性关系

| 概念 | 定义 | 属性 | 示例 | 反例 |
|------|------|------|------|------|
| **AI辅助验证** | 使用LLM辅助形式化验证的过程 | 降低门槛、需人工审核、迭代式 | LLM生成TLA+草案 | 全自动无监督验证 |
| **证明策略** | 定理证明器中的证明步骤序列 | 可学习、可建议、需正确性 | induction, rewrite | 错误策略（循环证明） |
| **不变量合成** | 自动生成程序不变量的过程 | 难、关键、部分可自动化 | 数组边界不变量 | 复杂分布式不变量 |
| **反例解释** | 将验证失败的反例转化为人类可读描述 | 有价值、可AI辅助 | TLC反例→自然语言 | 原始状态序列 |
| **规范漂移** | AI生成的规范与真实意图的差异 | 累积性、需人工校验 | LLM误解"互斥"含义 | 精确的人工规范 |
