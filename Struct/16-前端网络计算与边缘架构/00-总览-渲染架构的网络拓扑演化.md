```text

│
├─► 渲染架构的网络拓扑演化
│   ├─► 拓扑 I: 经典 CSR (2015)
│   │   Client ←(HTML)→ CDN ←(API)→ Origin
│   │   L_total = L_cdn + L_js_exec + L_api × n
│   │
│   ├─► 拓扑 II: SSR + Hydration (2020)
│   │   Client ←(HTML)→ CDN ←(SSR)→ Origin
│   │   L_total = L_cdn + L_ssr + L_hydration
│   │
│   ├─► 拓扑 III: Islands (2022)
│   │   Client ←(HTML)→ CDN ←(静态生成)→ Origin(Build)
│   │   L_total = L_cdn + Σ(L_island × p_activate)
│   │
│   ├─► 拓扑 IV: RSC + Edge (2024-2026)
│   │   Client ←(RSC)→ Edge ←(API)→ Origin ←(SQL)→ DB
│   │   L_total = L_edge + L_rsc_stream + L_cc
│   │
│   └─► 拓扑 V: DPR (2026)
│       Client ←→ Edge(RSC+Cache) ←→ Origin(RSC+Revalidate) ←→ DB
│       S(t) = f(s(t₀), Δt), Δt = t - t₀
│
├─► 边缘渲染的一致性模型
│   ├─► SSG: s_e = s_o(t₀), ε = ∞
│   ├─► ISR: s_e = s_o(t₀), 后台刷新
│   ├─► DPR: s_e = s_o(t), KV 存储同步
│   ├─► Streaming SSR: s_e = ⊥, 每次流式渲染
│   └─► PPR: s = s_shell ⊕ s_dynamic
│
└─► 前端存储层次的形式化
    ├─► L1: CPU Cache → JS 变量/闭包
    ├─► L2: Heap → V8 Heap, WASM Memory
    ├─► L3-L8: SessionStorage → localStorage → IndexedDB → OPFS → SQLite WASM → Network
    └─► CAP 约束映射
```

## 二、五种渲染拓扑的形式化对比

| 拓扑 | 延迟公式 | 一致性 | 首屏性能 | 交互性能 | 2026年适用场景 |
|------|---------|--------|---------|---------|--------------|
| **CSR** | `L_cdn + L_js_exec + L_api × n` | 强 (客户端状态) | 差 (JS 阻塞) | 优 | 纯 SPA、后台管理系统 |
| **SSR** | `L_cdn + L_ssr + L_hydration` | 强 | 中 | 中 (注水重复执行) | 电商、内容站 |
| **Islands** | `L_cdn + Σ(L_island × p_activate)` | 强 (静态部分) | **优** | 按需激活 | 内容/营销站 |
| **RSC+Edge** | `L_edge + L_rsc_stream + L_cc` | 中 (边缘缓存) | 优 | 中 | 全栈应用 |
| **DPR** | `min(L_cache, L_revalidate)` | 可调 (TTL控制) | **最优** | 按需 | 大规模内容平台 |

## 三、边缘渲染一致性模型的形式化分类

| 策略 | 形式化描述 | 一致性级别 | 延迟特征 |
|------|-----------|-----------|---------|
| **SSG** | `s_e = s_o(t₀)`, 静态快照 | 最终一致 (ε = ∞) | 最低 (CDN 直连) |
| **ISR** | `s_e = s_o(t₀)`, 后台刷新 `s_e' = s_o(t₁)` | 容忍 `ε = \|s_o(t₀) - s_o(t₁)\|` | 低 |
| **DPR** | `s_e = s_o(t)`, 首次渲染后缓存 | 可调 (TTL) | 低-中 |
| **Streaming SSR** | `s_e = ⊥`, 无缓存 | 强一致 | 高 |
| **PPR** | `s = s_shell ⊕ s_dynamic` | 壳强一致 + 动态最终一致 | 壳极低 |

## 四、前端存储层次的 CAP 约束

```text
定理 T6: 前端持久化状态系统最多同时满足 CAP 中的两项

┌─────────────────┬───────────┬───────────┬───────────┐
│    存储方案      │ 一致性(C) │ 可用性(A) │ 分区容错(P)│
├─────────────────┼───────────┼───────────┼───────────┤
│ localStorage    │ ✅ 强一致  │ ✅ 高可用  │ ❌ 无分区  │
│ IndexedDB       │ ✅ 事务一致│ ✅ 高可用  │ ⚠️ 单标签  │
│ BroadcastChannel│ ⚠️ 因果一致│ ✅ 高可用  │ ⚠️ 同源分区│
│ Service Worker  │ ⚠️ 最终一致│ ✅ 高可用  │ ✅ 分区容错│
│ Web Locks API   │ ✅ 强一致  │ ⚠️ 锁竞争  │ ❌ 单标签  │
│ CRDT (Yjs)      │ ⚠️ 最终一致│ ✅ 高可用  │ ✅ 分区容错│
└─────────────────┴───────────┴───────────┴───────────┘

工程推论:
  离线优先应用 = Service Worker + BackgroundSync (AP 系统)
  金融级表单 = Web Locks + IndexedDB (CP 系统)
  协作编辑 = CRDT + BroadcastChannel (AP 系统)
```

## 七、批判性总结

RSC 试图在**服务器（单次渲染）**与**客户端（长期生命周期）**之间建立对称性，但这两者的计算本质不同：服务器是**批处理模型（Batch）**，客户端是**事件驱动模型（Event Loop）**。强制对称导致了概念泄漏（Conceptual Leakage）。

前端存储的 CAP 约束表明：浏览器是一个微型分布式系统，每个标签页是一个节点，每个 Storage API 是一种一致性协议。理解这些约束是设计离线优先应用的基础。
