## 一、知识体系思维导图

```text
前端存储层次: L1 到 L8 的延迟-容量权衡
│
├─► 易失性存储 (进程生命周期)
│   ├─ L1: CPU Register + L1 Cache (~1ns, ~64KB)
│   │   └─ 前端映射: JS 标量变量、V8 内部优化
│   ├─ L2: V8 Heap / WASM Memory (~100ns, ~1-4GB)
│   │   └─ 前端映射: 对象、数组、闭包、React Fiber 树
│   └─ L3: JS 上下文 / Realm (~1μs, 进程隔离)
│       └─ 前端映射: iframe、Web Worker、Service Worker
│
├─► 半持久性存储 (会话/标签页生命周期)
│   ├─ L4: SessionStorage (~1μs, ~5MB)
│   └─ L5: Memory Cache (fetch API) (~1μs, 内存限制)
│       └─ Cache-Control: no-store 可绕过
│
├─► 持久性存储 (用户授权/同源策略)
│   ├─ L6: localStorage (~10μs, ~5-10MB)
│   ├─ L7: IndexedDB (~1ms, ~1GB+, 结构化)
│   ├─ L8: Cache API (Service Worker) (~1ms, 磁盘/Quota)
│   ├─ L9: OPFS (Origin Private File System) (~5ms, 磁盘容量)
│   └─ L10: SQLite WASM (~10ms, ~2GB, SQL 语义)
│
├─► 网络存储层次
│   ├─ L11: CDN Edge Cache (~1-10ms, 区域分布)
│   ├─ L12: Origin Server Cache (~10-50ms, 集中式)
│   ├─ L13: Database Cache (Redis) (~1ms, 内存)
│   └─ L14: Persistent DB (~5-50ms, 磁盘/SSD)
│
└─► 权衡定律
    ├─ 延迟-容量反比: 越快的存储越贵、越小
    ├─ 持久性-性能反比: 持久化需要 I/O，慢于内存
    └─ 隐私-便利权衡: 持久存储受同源策略和配额限制
```

## 二、核心概念的形式化定义

### 定义 D16.7：存储层次的延迟-容量模型

设存储层级为 L_i，定义其特征三元组：

```
L_i = ⟨Latency_i, Capacity_i, Persistence_i⟩

其中:
  Latency_i: 随机读写访问延迟 (ns / μs / ms)
  Capacity_i: 最大可存储数据量 (KB / MB / GB)
  Persistence_i ∈ {Volatile, Session, Persistent, Network}

权衡定律 (Trade-off Law):
  ∀i < j: Latency_i < Latency_j  ∧  Capacity_i < Capacity_j

  即: 更快的层级容量更小 (成本约束)
```

### 定义 D16.8：前端存储访问成本函数

对于数据项 d 的访问总成本：

```
Cost(d) = Latency(L(d)) + Energy(L(d)) + PrivacyRisk(L(d))

其中:
  L(d): 存储 d 的层级
  Energy: 访问能耗 (内存 < 磁盘 < 网络)
  PrivacyRisk: 数据持久化带来的隐私泄露风险

最优放置问题:
  minimize Σ Cost(d_i)
  subject to: ∀i, Capacity(L(d_i)) ≤ Cap_max(L(d_i))
              Persistence(d_i) ≤ Persistence(L(d_i))
```

### 定义 D16.9：缓存替换策略的形式化

对于容量为 C 的缓存，请求序列 σ = r₁, r₂, ..., rₙ：

```
OPT (Belady, 1966):
  当缓存满且需要加载新项时，淘汰最远未来才会被访问的项
  这是理论最优但不可实现的 (需要预知未来)

LRU (Least Recently Used):
  淘汰最久未被访问的项
  实现: 双向链表 + 哈希表, O(1) 访问/更新

LFU (Least Frequently Used):
  淘汰访问频率最低的项
  缺点: 历史累积，对新项不友好

浏览器 Cache API 策略:
  - Cache-First: 先读 Cache, miss 则网络请求
  - Network-First: 先读网络, fallback 到 Cache
  - Stale-While-Revalidate: 返回 Cache, 后台更新
```

## 三、多维矩阵对比

### 3.1 前端存储层级完整对比

| 层级 | 存储介质 | 典型延迟 | 典型容量 | 持久性 | 同步/异步 | 同源隔离 | 前端 API |
|------|---------|---------|---------|--------|----------|---------|---------|
| **L1** | CPU Register/L1 | ~1ns | ~64KB | 易失 | 同步 | N/A | JS 变量 |
| **L2** | V8 Heap | ~100ns | ~1.4GB (64位) | 易失 | 同步 | 进程级 | 对象/数组 |
| **L3** | SessionStorage | ~1μs | ~5MB | 会话 | 同步 | ✅ 同源 | sessionStorage |
| **L4** | localStorage | ~10μs | ~5-10MB | 永久 | 同步 | ✅ 同源 | localStorage |
| **L5** | IndexedDB | ~1ms | ~1GB+ | 永久 | 异步 | ✅ 同源 | IndexedDB API |

### 3.2 存储选型决策矩阵

| 数据特征 | 推荐层级 | 理由 |
|---------|---------|------|
| UI 临时状态 (hover, focus) | L1/L2 (Heap) | 极低延迟，无需持久化 |
| 表单草稿 (未提交) | L3 (SessionStorage) | 页面刷新保留，关闭标签清除 |
| 用户偏好 (主题、语言) | L4 (localStorage) | 小数据量，同步读取，永久保留 |
| 离线文档缓存 | L5 (IndexedDB) | 结构化，大容量，索引查询 |
| 静态资源缓存 (JS/CSS/图片) | L6 (Cache API) | HTTP 语义集成，Service Worker 控制 |
| 大型文件处理 (视频编辑) | L7 (OPFS) | 类文件系统 API，接近原生性能 |
| 离线 SQL 查询 | SQLite WASM + OPFS | 成熟关系语义，事务支持 |
| 实时协作数据 | L9 + CRDT | 网络同步 + 冲突解决 |

## 五、工程实践与代码示例

### 5.1 分层存储封装：自动降级策略

```typescript
// 分层存储管理器: 优先内存 → IndexedDB → Network
class TieredStorage<T> {
  private memory = new Map<string, T>();
  private db: IDBDatabase | null = null;

  async init(): Promise<void> {
    this.db = await openDB('AppStore', 1, {
      upgrade(db) {
        db.createObjectStore('data');
      }
    });
  }

  async get(key: string): Promise<T | undefined> {
    // L1: 内存缓存
    if (this.memory.has(key)) {
      return this.memory.get(key);
    }

    // L2: IndexedDB
    if (this.db) {
      const value = await this.db.get('data', key);
      if (value !== undefined) {
        this.memory.set(key, value); // 回写内存
        return value;
      }
    }

    return undefined;
  }

  async set(key: string, value: T): Promise<void> {
    // L1 + L2 同时写入
    this.memory.set(key, value);
    if (this.db) {
      await this.db.put('data', value, key);
    }
  }

  // 根据数据热度自动驱逐 (LRU)
  private lruKeys: string[] = [];

  touch(key: string): void {
    const idx = this.lruKeys.indexOf(key);
    if (idx >= 0) this.lruKeys.splice(idx, 1);
    this.lruKeys.push(key);

    // 内存限制: 最多保留 1000 项
    while (this.lruKeys.length > 1000) {
      const evict = this.lruKeys.shift()!;
      this.memory.delete(evict);
    }
  }
}
```

### 5.2 Service Worker 缓存策略矩阵

```typescript
// service-worker.ts
const CACHE_NAME = 'app-v1';

self.addEventListener('fetch', (event) => {
  const { request } = event;
  const url = new URL(request.url);

  // 策略选择矩阵
  if (isStaticAsset(url)) {
    // L6: Cache-First (静态资源极少变化)
    event.respondWith(cacheFirst(request));
  } else if (isAPIRequest(url)) {
    // L9: Network-First (API 数据要求最新)
    event.respondWith(networkFirst(request));
  } else if (isImage(url)) {
    // L6: Stale-While-Revalidate (图片可容忍短暂陈旧)
    event.respondWith(staleWhileRevalidate(request));
  } else {
    // 默认: Network-Only
    event.respondWith(fetch(request));
  }
});

async function cacheFirst(request: Request): Promise<Response> {
  const cache = await caches.open(CACHE_NAME);
  const cached = await cache.match(request);
  if (cached) return cached;

  const response = await fetch(request);
  cache.put(request, response.clone());
  return response;
}

async function staleWhileRevalidate(request: Request): Promise<Response> {
  const cache = await caches.open(CACHE_NAME);
  const cached = await cache.match(request);

  // 后台刷新
  const fetchPromise = fetch(request).then(response => {
    cache.put(request, response.clone());
    return response;
  });

  // 立即返回缓存 (如果有)
  return cached || fetchPromise;
}
```

## 六、批判性总结

前端存储层次从 CPU 寄存器到网络数据库构成了一个跨越 8 个数量级延迟范围的复杂谱系。理解这一层次结构对于前端性能优化至关重要——一个将用户偏好存储在 IndexedDB 而非 localStorage 中的决策，可能将读取延迟从 10 微秒增加到 1 毫秒，对于高频访问的设置面板而言，这种 100 倍的性能下降是用户可感知的。

然而，前端存储工程面临的真正挑战不是延迟优化，而是**一致性、隐私和配额的三角约束**。localStorage 的同步 API 虽然便捷，但其 5-10MB 的配额限制和阻塞主线程的特性使其在现代大型应用中成为瓶颈；IndexedDB 虽然提供了强大的结构化存储和异步 API，但其基于事件的编程模型和版本升级复杂性让开发者望而却步；OPFS 和 SQLite WASM 代表了前沿方向，但它们对文件系统语义和 SQL 引擎的依赖增加了 bundle 体积和初始化成本。

更深层的批判在于：浏览器的存储策略深受**同源策略 (Same-Origin Policy)** 的制约，这在安全上是必要的，但在现代微前端和跨域协作场景中成为了架构障碍。Storage Access API 和 First-Party Sets 等提案试图缓解这一问题，但进展缓慢。2026 年的务实建议是：**将存储视为延迟光谱上的连续优化问题，而非离散的选择题**。热数据在 V8 Heap 中，温数据在 localStorage/IndexedDB 中，冷数据在网络中；使用 Service Worker 的 Cache API 作为 HTTP 响应的加速器，而非应用状态的主存储；对于需要离线功能的应用，采用 OPFS + SQLite WASM 提供接近原生的文件系统能力，但要为初始加载的 WASM 编译成本做好准备。存储层次的设计哲学应该是"数据找最适合的家"，而不是"所有数据都塞进 localStorage"。
