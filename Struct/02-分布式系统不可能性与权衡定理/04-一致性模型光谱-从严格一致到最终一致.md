# 一致性模型光谱：从严格一致到最终一致

> **定位**：本文件建立分布式系统一致性级别的完整谱系。一致性不是"有或无"的二元属性，而是**正确性、延迟、可用性三维空间中的帕累托前沿**。理解这一光谱，是做出精确工程权衡的前提。
>
> **核心命题**：选择一致性级别的正确方法不是"越强越好"，而是**"在满足业务语义的前提下，选择延迟最低、可用性最高的一致性级别"**。

---

## 一、思维导图：一致性模型完整谱系

```text
一致性模型光谱
│
├─【严格一致性】Strict Consistency
│   └─ 所有操作全局实时顺序 → 物理上不可能（光速限制）
│
├─【线性一致性】Linearizability ⭐ 工程黄金标准
│   ├─ 操作全序，与实时时钟一致
│   ├─ 可组合性（Composable）
│   ├─ 代价：高延迟（Quorum同步）
│   └─ 代表：etcd, ZooKeeper, Spanner, CockroachDB
│
├─【顺序一致性】Sequential Consistency
│   ├─ 操作全序，无实时边界
│   ├─ 允许重排，只要所有进程看到相同顺序
│   └─ 代表：多核CPU默认内存模型、某些内存数据库
│
├─【因果一致性】Causal Consistency
│   ├─ 因果相关操作有序（happens-before）
│   ├─ 无关操作可并发/重排
│   ├─ 可接受延迟显著低于线性一致
│   └─ 代表：COPS, ChainReaction, AntidoteDB
│
├─【会话一致性】Session Consistency
│   ├─ 单会话内读写自洽（读己之写、单调读）
│   ├─ 跨会话无保证
│   └─ 代表：MongoDB默认, DynamoDB会话, Azure Cosmos DB
│
├─【最终一致性】Eventual Consistency
│   ├─ 无新更新时所有副本收敛
│   ├─ 不保证收敛前读到的值
│   ├─ 最低延迟、最高可用
│   └─ 代表：Cassandra, DynamoDB, S3, DNS
│
└─【CRDT强最终一致性】Strong Eventual Consistency
    ├─ 最终一致 + 无冲突合并（半格代数保证）
    ├─ 无需协调即可合并并发更新
    └─ 代表：Riak, Redis CRDT, Yjs, Figma
```

## 三、形式化定义与层级关系

### 3.1 严格一致性（Strict Consistency）

```
定义：任何read操作必须返回最近一次write的结果（按全局物理时间）。

形式化：
  若 write(x, v) 在物理时间 t 完成，
  则任何在 t' > t 时刻开始的 read(x) 必须返回 v。

可行性：❌ 物理上不可能。
原因：信息传播速度受光速限制；跨1ms距离的节点不可能"实时"同步。
用途：仅作为理论基准（Baseline）。
```

### 3.2 线性一致性（Linearizability）

> **权威来源**：Herlihy & Wing, 1990

```
定义：每个操作似乎在调用的某个瞬间原子执行，
      且所有操作的全序与它们的实时调用-响应区间兼容。

形式化：
  对每个操作op，存在线性化点lp(op) ∈ [调用开始, 响应结束]，
  使得按lp(op)排序的操作序列满足顺序规约。

关键性质：
  1. 本地性质（Local Property）：系统线性一致 ⟺ 每个对象线性一致
     → 可组合！这是线性一致性的最大工程价值。
  2. 非阻塞（Non-blocking）：读操作不必等待并发写完成。

工程代价：
  - 单区域：1-10ms（取决于Quorum大小）
  - 跨地域：50-150ms（纯软件实现）或 5-10ms（Spanner TrueTime硬件辅助）
```

### 3.3 顺序一致性（Sequential Consistency）

```
定义：所有进程看到的操作顺序一致，但不必与实时顺序一致。

与线性一致的区别：
  - 线性一致：顺序必须尊重实时（若op₁在op₂开始前结束，则op₁ < op₂）
  - 顺序一致：允许重排，只要所有进程看到相同的重排结果

例子：
  P1: write(x, 1) ──────────────────────▶
  P2: ─────────────── write(x, 2) ──────▶
  P3: ─────────────────────── read(x)=? ──▶

  线性一致：P3读必须返回2（因为write(x,2)在read开始前完成）
  顺序一致：P3读可能返回1（只要所有进程同意此顺序）

代表：多核CPU的默认内存模型（x86 TSO比顺序一致弱，但可模拟）。
```

### 3.4 因果一致性（Causal Consistency）

```
定义：若操作op₁因果先于op₂（op₁ → op₂），
      则所有进程必须先看到op₁再看到op₂。

因果关系定义：
  1. 同一进程内：op₁在op₂之前执行 → op₁ → op₂
  2. 读写关系：op₁ = write(x,v), op₂ = read(x)=v → op₁ → op₂
  3. 传递性：op₁ → op₂ 且 op₂ → op₃ → op₁ → op₃

与线性一致的区别：
  - 线性一致要求所有操作（包括无关操作）全局有序
  - 因果一致仅要求因果相关操作有序，无关操作可并发

工程优势：
  - 无需全局协调 → 延迟显著降低
  - 自然符合社交/协作场景的语义（"回复"必须看到"原帖"）

代表系统：COPS (Geo-replicated Causal Consistency), ChainReaction, AntidoteDB。
```

### 3.5 会话一致性（Session Consistency）

```
定义：在单个客户端会话内保证特定一致性属性。

常见会话保证：
  1. 读己之写（Read Your Writes）：
     会话内写操作完成后，后续读必能看到该写。
  2. 单调读（Monotonic Reads）：
     会话内后续读不会看到比先前读更旧的值。
  3. 单调写（Monotonic Writes）：
     会话内写操作按发出顺序执行。
  4. 读后写（Writes Follow Reads）：
     若会话读到某值，后续写基于该读的因果关系。

工程实现：
  - 会话状态通常绑定到客户端连接或会话Token
  - 负载均衡器需保证会话粘性（Session Stickiness）
  - 跨会话无保证 → 两个用户可能看到不同视图

代表：MongoDB默认读偏好, DynamoDB会话一致性, Azure Cosmos DB。
```

### 3.6 最终一致性（Eventual Consistency）

```
定义：若无新更新，所有副本最终收敛到相同值。

特点：
  - 最弱的一致性保证（仅保证"最终会一致"）
  - 不保证何时收敛、不保证收敛前读到的值
  - 最低延迟、最高可用性、最易实现

收敛条件：
  - 系统必须保证更新最终到达所有副本
  - 需要冲突解决策略（LWW: Last-Write-Wins 最常见）

风险：
  - 读取过时数据（Stale Read）
  - 冲突丢失（LWW导致并发更新被覆盖）
  - 缺乏可预测性（无法告知用户"数据已同步"）

代表：Cassandra, DynamoDB, S3, DNS, Amazon SQS。
```

### 3.7 CRDT强最终一致性（Strong Eventual Consistency）

```
定义：最终一致 + 无冲突合并（由代数结构保证）。

核心机制：
  - 数据类型设计为满足交换律、结合律、幂等律
  - 并发更新可自动合并，无需协调（Coordination-Free）
  - 合并结果与更新接收顺序无关

与普通最终一致的区别：
  ┌─────────────────┬──────────────────┬──────────────────┐
  │                 │ 最终一致         │ CRDT强最终一致   │
  ├─────────────────┼──────────────────┼──────────────────┤
  │ 冲突处理        │ 需外部策略(LWW)  │ 内置代数合并     │
  │ 收敛保证        │ 依赖消息到达     │ 半格理论保证     │
  │ 并发写          │ 可能丢失数据     │ 不会丢失数据     │
  │ 适用数据类型    │ 任意             │ 需特殊设计       │
  └─────────────────┴──────────────────┴──────────────────┘

代表：Riak ( converge CRDTs ), Redis CRDT, Yjs (协同编辑), Figma。
```

## 四、层级关系图

```text
一致性强度层级（由上至下递减）

严格一致性 ──→ 物理上不可能
    │
    ▼
线性一致性 ──→ Herlihy & Wing, 1990
    │           可组合（Local Property）
    ▼           工程黄金标准
顺序一致性 ──→ Lamport, 1979
    │           全序但无实时边界
    ▼
因果一致性 ──→ 仅因果相关操作有序
    │           适合社交/协作场景
    ▼
会话一致性 ──→ 单会话内保证
    │           用户体验友好
    ▼
最终一致性 ──→ 最弱保证，最高性能
    │           "最终会一致"
    ▼
CRDT强最终一致 ──→ 代数保证无冲突合并
                无需协调（Coordination-Free）

注：箭头表示"前者蕴含后者"（更强的一致蕴含更弱的一致）。
    即：线性一致 ⟹ 顺序一致 ⟹ 因果一致 ⟹ 会话一致 ⟹ 最终一致
    （但逆命题不成立）
```

## 五、工程映射：按场景选择一致性

| 业务场景 | 推荐一致性 | 理由 | 代表实现 |
|---------|-----------|------|---------|
| **账户余额/支付** | 线性一致 | 任何不一致都直接造成资金损失 | Spanner, CockroachDB, TiDB |
| **库存扣减** | 线性一致 | 超卖导致履约问题 | etcd分布式锁 + RDBMS |
| **分布式配置中心** | 线性一致 | 配置分歧导致系统行为分裂 | etcd, ZooKeeper, Consul |
| **分布式锁/选举** | 线性一致 | 脑裂是灾难性的 | etcd, ZooKeeper |
| **社交Feed/时间线** | 因果一致 | 用户可接受"回复在原帖后"而非全局实时 | COPS, 自定义实现 |
| **评论/点赞** | 因果一致 | 因果顺序重要，实时性次要 | 大多数社交系统 |
| **用户购物车** | 会话一致 | 用户只需看到自己的最新操作 | MongoDB, DynamoDB |
| **用户偏好设置** | 会话一致 | 跨设备延迟同步可接受 | 大多数云服务 |
| **日志/监控数据** | 最终一致 | 海量写入，延迟敏感，可接受短暂不一致 | Cassandra, S3, Elasticsearch |
| **推荐结果** | 最终一致 | 本身就是概率性的，无需强一致 | Redis, Cassandra |
| **协同编辑** | CRDT强最终 | 无冲突合并是核心需求 | Yjs, Figma, Google Docs |
| **计数器（PV/UV）** | CRDT强最终 | 数值累加天然满足半格性质 | G-Counter / PN-Counter |
| **全局唯一ID** | 最终一致 | 仅需唯一性，无需顺序 | Snowflake, UUID |
| **DNS/CDN** | 最终一致 | 天生异步传播，TTL机制 | DNS协议, Cloudflare |

## 六、概念定义与属性关系

| 概念 | 定义 | 属性 | 示例 | 反例 |
|------|------|------|------|------|
| **线性化点**（Linearization Point） | 操作在逻辑上原子执行的瞬间 | 存在于[调用开始, 响应结束]区间内 | CAS操作成功瞬间 | 无原子保证的读-改-写 |
| **因果先序**（Happens-Before） | Lamport定义的偏序关系 | 传递性、非全序、可检测并发 | P1写x后P2读x → 写 happens-before 读 | 两个独立进程的无关写操作 |
| **可组合性**（Composability） | 系统整体性质可由组件性质推导 | Local Property、模块化验证的基础 | 线性一致性 | 顺序一致性（非Local） |
| **协调自由**（Coordination-Free） | 副本间无需同步即可处理更新 | 高可用、低延迟、无限扩展 | CRDT、 eventually consistent | 线性一致（需要Quorum协调） |
| **陈旧读**（Stale Read） | 读取到非最新已提交值 | 在最终一致系统中是必然现象 | 从Cassandra副本读到过期数据 | 从etcd线性一致读 |
