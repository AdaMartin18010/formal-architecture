## 二、核心概念的形式化定义

### 定义 D15.19：依赖类型

依赖类型是允许**类型依赖于项 (term)** 的类型系统：

```
简单类型: Vec A          (类型构造子)
依赖类型: Vec (n: Nat) A  (类型依赖于值 n)

函数示例 (Agda/Idris 风格):
  append: ∀{n m: Nat}{A: Set}. Vec n A → Vec m A → Vec (n+m) A

前置条件编码:
  head: ∀{n: Nat}{A: Set}. Vec (suc n) A → A
  // 类型保证: head 不可能在空向量上调用
```

TypeScript 的**受限模拟**:

```typescript
type Vec<N extends number, A> = A[] & { readonly length: N };
// 注: TS 无法在类型层面进行算术运算 (n+m)，这是根本限制
```

### 定义 D15.20：精炼类型 (Liquid Types / Refined Types)

精炼类型通过在基础类型上附加**逻辑谓词**来增强表达能力：

```
{x: T | φ(x)}  —— 满足谓词 φ 的 T 的子集

示例:
  {v: number | v > 0}           —— 正数
  {v: string | len(v) < 256}    —— 长度受限字符串
  {v: Array<T> | len(v) = n}    —— 定长数组

类型规则 (子类型):
  {v: T | φ} <: {v: T | ψ}   ⇔   φ ⇒ ψ  (逻辑蕴含)
```

### 定义 D15.21：类型驱动开发 (TDD) 工作流程

```text
步骤 1: 定义数据类型的代数结构
  type Result<E, A> =
    | { readonly _tag: "Ok"; readonly value: A }
    | { readonly _tag: "Err"; readonly error: E }

步骤 2: 定义核心操作的类型签名 (无实现)
  declare function map<E, A, B>(
    fa: Result<E, A>,
    f: (a: A) => B
  ): Result<E, B>;

步骤 3: 编译器检查类型一致性，指导实现填充
  function map(fa, f) {
    switch (fa._tag) {
      case "Ok": return { _tag: "Ok", value: f(fa.value) };
      case "Err": return fa; // 类型系统保证这是 Result<E, B>
    }
  }

步骤 4: 属性测试验证函子法则
  map(fa, x => x) === fa                    (恒等)
  map(map(fa, f), g) === map(fa, x => g(f(x))) (复合)
```

## 五、工程实践与代码示例

### 5.1 类型驱动设计：API 客户端的类型安全封装

```typescript
// 步骤 1: 定义数据类型 (不可能状态在类型层面排除)
type HttpMethod = "GET" | "POST" | "PUT" | "DELETE";

type Endpoint<P extends string, M extends HttpMethod, Req, Res> = {
  readonly path: P;
  readonly method: M;
  readonly _req: Req;  // phantom type
  readonly _res: Res;  // phantom type
};

// 步骤 2: 定义操作签名
declare function apiCall<
  P extends string,
  M extends HttpMethod,
  Req,
  Res
>(
  endpoint: Endpoint<P, M, Req, Res>,
  request: M extends "GET" ? void : Req
): Promise<Res>;

// 步骤 3: 具体端点定义 (类型即契约)
const GetUser: Endpoint<"/user/:id", "GET", void, { name: string }> = {
  path: "/user/:id",
  method: "GET",
} as const;

// 编译器保证:
// apiCall(GetUser, undefined) ✅
// apiCall(GetUser, { name: "x" }) ❌ GET 不接受 body
```

### 5.2 模板字面量类型：路由参数提取

```typescript
// 从路径模板提取参数类型 (类型级字符串解析)
type ExtractParams<Path extends string> =
  Path extends `${infer _Start}:${infer Param}/${infer Rest}`
    ? { [K in Param | keyof ExtractParams<`/${Rest}`>]: string }
    : Path extends `${infer _Start}:${infer Param}`
    ? { [K in Param]: string }
    : {};

// 使用
type UserParams = ExtractParams<"/user/:id/posts/:postId">;
// 推导为: { id: string; postId: string }

function buildUrl<P extends string>(
  path: P,
  params: ExtractParams<P>
): string {
  return path.replace(/:([^/]+)/g, (_, key) => (params as any)[key]);
}

buildUrl("/user/:id", { id: "123" }); // ✅
// buildUrl("/user/:id", { postId: "123" }); // ❌ 编译错误
```

### 5.3 Result 类型的代数构造

```typescript
// 和类型 (Sum Type) 编码错误处理
type Result<E, A> =
  | { readonly _tag: "Ok"; readonly value: A }
  | { readonly _tag: "Err"; readonly error: E };

// 函子实例 (Functor)
const map = <E, A, B>(fa: Result<E, A>, f: (a: A) => B): Result<E, B> =>
  fa._tag === "Ok" ? { _tag: "Ok", value: f(fa.value) } : fa;

// 应用函子 (Applicative)
const ap = <E, A, B>(
  fab: Result<E, (a: A) => B>,
  fa: Result<E, A>
): Result<E, B> =>
  fab._tag === "Ok" && fa._tag === "Ok"
    ? { _tag: "Ok", value: fab.value(fa.value) }
    : fab._tag === "Err"
    ? fab
    : fa;

// Monad 绑定 (bind)
const flatMap = <E, A, B>(
  fa: Result<E, A>,
  f: (a: A) => Result<E, B>
): Result<E, B> =>
  fa._tag === "Ok" ? f(fa.value) : fa;
```

## 六、批判性总结

类型驱动开发 (Type-Driven Development) 代表了软件工程方法论从"先写代码再修 bug"到"先定义契约再推导实现"的范式转变。其核心洞察在于：**一个精心设计的类型签名可以排除绝大多数错误状态，使得非法程序在编译阶段即被拒绝**。这在理论上极为优雅——Martin-Löf 的依赖类型论甚至将类型系统提升为构造性数学证明的载体——但在前端工程实践中，TypeScript 的表达力天花板构成了严重的制约。

TypeScript 的类型系统虽然图灵完备（可以模拟任意图灵机，因此类型检查可能不终止），但它并非依赖类型系统。TS 无法在类型层面表达"这个数组的长度等于那个数组的长度"、"这个字符串匹配正则表达式"或"这个数值大于零"等精炼属性。社区通过模板字面量类型、条件类型和 branded types 进行的"类型体操"，本质上是在一个不支持依赖类型的系统中**用类型级编程模拟依赖类型的部分能力**。这种模拟是精巧的、富有创造性的，但也是脆弱的——它导致编译时间指数级增长、错误信息晦涩难懂、且随时可能在 TS 版本升级中断裂。

Liquid Types 和依赖类型的真正价值在于**将不变量编码进类型本身**，使得"数组越界访问"或"除以零"等错误在类型层面即被证明不可能发生。前端领域目前最接近这一理想的是 Effect-TS 的 `Option`、`Either` 类型和 Zod 的运行时验证组合，但这仍然是一个"运行时排除"而非"编译时证明"的模型。2026 年的现实是：对于绝大多数前端团队，"类型驱动"意味着"使用 TypeScript 编写精确的类型签名，辅以 Zod 进行边界验证"，而非 Idris 或 Agda 意义上的"类型即证明"。这种折中虽然不够纯粹，但在工程交付压力和理论完备性之间取得了务实的平衡——毕竟，一个能在周五下午安全部署的系统，比一个需要三年形式化验证的完美系统更有业务价值。

## 八、形式化推理链

### 8.1 Curry-Howard 同构与类型驱动开发的理论根基

**命题 P15.7**: 在构造性逻辑中，类型 inhabitedness 等价于命题可证性，程序提取等价于证明构造。

```
Curry-Howard 对应的三重映射 (Curry 1934, Howard 1969, Lambek 1980):

  逻辑 (Intuitionistic Propositional Logic)
    命题 A ∧ B          命题 A ∨ B           命题 A ⇒ B
       │                  │                    │
       │ [对应]           │ [对应]             │ [对应]
       ▼                  ▼                    ▼
  类型 (Simple Type λ-Calculus)
    积类型 A × B        和类型 A + B         函数类型 A → B
       │                  │                    │
       │ [对应]           │ [对应]             │ [对应]
       ▼                  ▼                    ▼
  范畴 (Cartesian Closed Category)
    积对象 A × B        余积对象 A + B       指数对象 B^A

类型 inhabitation 问题:
  给定上下文 Γ 和类型 A, 是否存在项 M 使得 Γ ⊢ M : A ?

  由 Curry-Howard, 这等价于:
    给定假设集合 Γ 和命题 A, 是否存在 Γ 中 A 的构造性证明 ?

程序综合 (Program Synthesis):
  Γ ⊢ ?: A  ──[证明搜索]──► M
  M 即为从类型 A 自动推导出的程序。
```

> **Haskell B. Curry** (1934, "Functionality in Combinatory Logic", Annals of Mathematics):
> 最早观察到"类型与命题之间的对应"，奠定了 Curry-Howard 同构的基础。Curry 指出：一个组合子的类型可以被看作一个逻辑命题，而组合子本身即为该命题的证明。

> **William A. Howard** (1969, "The Formulae-as-Types Notion of Construction", unpublished manuscript; 后收入 1980 To H.B. Curry):
> 将 Curry 的观察扩展到自然演绎与简单类型 λ-演算的精确对应，即著名的"命题即类型，证明即程序" (formulae-as-types, proofs-as-programs)。

### 8.2 类型 inhabitation 与程序综合的形式化

**定理 T15.4**: 简单类型 λ-演算 (STLC) 的类型 inhabitation 问题是 PSPACE-完全的，但对于受限片段（如 ML 类型的原子实例化）是 EXPSPACE-完全的。

```
形式化定义:
  问题: Closed Program Synthesis for ML
  输入: 类型 A
  输出: 项 M 使得 ⊢_ML M : A, 或声明无解

复杂度层级 (Benke, Schubert & Walukiewicz-Chrzaszcz 2016):
  - STLC (无多态): PSPACE-complete ............ [Statman 1979, Urzyczyn 2016]
  - ML (有 let-多态, 原子实例化): EXPSPACE-complete
  - ML (无界实例化): 不可判定

Djinn 算法 (Augustsson 2005):
  基于 Dyckhoff 的收缩自由相继演算 (contraction-free sequent calculus),
  对直觉主义命题逻辑的定理证明进行证明搜索，将得到的证明项
  翻译为 Haskell 代码。

  核心规则 (LJT 演算):
    Γ, A→B ⊢ C      Γ ⊢ A
    ───────────────────────── (→L)
         Γ, A→B ⊢ C

    Γ, A ⊢ B
    ─────────── (→R)
    Γ ⊢ A → B
```

> **Lennart Augustsson** (2005, "Djinn, a Theorem Prover in Haskell, for Haskell"):
> "Djinn is a small program that takes a (Haskell) type and gives you back a function of that type if one exists." —— Djinn 将类型驱动开发从理想变为工具现实，展示了从类型签名自动综合程序的理论可行性。

> **Marcin Benke, Aleksy Schubert & Daria Walukiewicz-Chrzaszcz** (2016, "Synthesis of Functional Programs with Help of First-Order Intuitionistic Logic", FSCD):
> 将程序综合问题与一阶直觉主义逻辑的证明搜索严格对应，证明了 ML 类型综合在原子实例化限制下的 EXPSPACE-完全性。

### 8.3 TypeScript 类型系统的形式化边界

```
TypeScript 类型系统 ≈ System F_ω + 递归类型 + 条件类型

形式化限制:
  1. 非依赖类型: 无法表达 Vec(n+m, A) 中的 n+m 算术
  2. 无高阶类型: 无法量化类型构造子 (kind * → *)
  3. 无限递归: 类型检查可能不终止 (图灵完备)
  4. 无精炼谓词: 无法静态证明 "x > 0"

TS 对依赖类型的受限模拟:
  - 模板字面量类型: 字符串级约束
  - 条件类型 + 递归: 受限的 type-level 计算
  - branded types: 名义区分 (运行时不附加信息)
  - const type parameters: 值到类型的有限映射 (TS 5.0+)

安全性间隙:
  编译时保证: 良类型的程序内部一致
  运行时保证: Zod/io-ts 验证边界数据
  ─────────────────────────────────────
  未覆盖区域: any、类型断言 (as)、非 total 函数
```

### 10.2 核心参考文献

> **Per Martin-Löf** (1984, "Intuitionistic Type Theory", Bibliopolis):
> 依赖类型理论的奠基之作。Martin-Löf 将类型系统从"程序错误的过滤器"提升为"构造性数学证明的载体"，其 Π-类型和 Σ-类型分别为依赖函数类型和依赖积类型提供了形式化基础。

> **Edwin Brady** (2017, "Type-Driven Development with Idris", Manning Publications):
> 系统阐述了"先写类型，再填实现"的工程方法论。Brady 提出的"精化 (refinement)"工作流程——从粗略类型逐步添加约束直到实现被唯一确定——与本模块第 5.1 节的 API 客户端类型设计案例完全一致。

> **Patrick M. Rondon, Ming Kawaguchi & Ranjit Jhala** (2008, "Liquid Types", PLDI):
> 将 Hindley-Milner 类型推断与谓词抽象相结合，通过 Horn 约束求解自动推断精炼类型。Liquid Types 的理论框架表明：在受限条件下，高表达力类型系统的推断可以是完全自动的。

> **Lennart Augustsson** (2005, "Djinn — a coding wizard", Haskell Mailing List):
> Djinn 工具展示了从 Haskell 类型签名自动综合实现程序的可行性，其核心算法基于 Dyckhoff (1992) 的收缩自由相继演算。Augustsson 的工作证明：类型 inhabitation 不仅是理论问题，更是实用的程序综合技术。

### 10.3 课程作业对标

- **Stanford CS 242 (Programming Languages)**: Homework 1 要求学生实现一个简单类型 λ-演算的类型检查器，并证明其满足"Progress + Preservation"定理（即类型安全性）。这与本模块关于"类型排除不可能状态"的讨论直接对应——Progress 保证良类型项不会卡住，Preservation 保证求值保持类型。
- **CMU 15-312 (Foundations of Programming Languages)**: Assignment 5 的"Type-directed Synthesis"要求学生为一个受限的 ML 子集实现程序综合器，输入类型签名，输出满足该类型的项（若存在）。该作业的参考实现正是基于 Dyckhoff 的 LJT 演算——与 Djinn 的核心算法相同。
- **MIT 6.170 (Software Studio)**: 2024 年春季学期的 Final Project 引入"Type-safe Full-stack"赛道，要求学生使用 TypeScript 的模板字面量类型实现一个类型安全的路由器（URL 参数自动提取为类型），并讨论"这种类型级编程在何种边界下失效"。这与本模块第 5.2 节的路由参数提取案例及批判性总结完全一致。
