## 二、Curry-Howard-Lambek 三维对应表

| 逻辑命题 | TypeScript 类型 | 范畴对象 | 前端概念 |
|---------|----------------|---------|---------|
| A ∧ B (合取) | `[A, B]` 元组 | A × B 积 | Props 组合 (多属性) |
| A ∨ B (析取) | `A \| B` 联合 | A + B 和 | 条件渲染 (if/else) |
| A → B (蕴含) | `(a: A) => B` | B^A 指数 | 回调函数/事件处理 |
| ¬A (否定) | `never` (A→⊥) | 初始对象 0 | 不可达代码/错误边界 |
| ∀x.P(x) | `<T>(x: T) => P` | 依赖积 Π | 泛型组件/高阶组件 |
| ∃x.P(x) | `{x: T & P}` | 依赖和 Σ | 存在类型/Branded Types |
| ◇A (可能) | `Promise<A>` | Monad T(A) | 异步操作/Effect |
| □A (必然) | `() => A` | Comonad W(A) | 纯函数/同步计算 |

> **学术溯源**: William Howard (1969) 发现直觉主义逻辑与自然演绎和 λ 演算之间的对应关系。Lambek (1970s) 将其扩展到范畴论，形成**命题-类型-对象**的三位一体。

## 三、TypeScript 结构类型的形式化批判

### 定理 T3：结构类型系统存在语义安全漏洞

```text
前提: A5 定义 T₁ ≡ T₂ ⇔ shape(T₁) = shape(T₂)

构造反例:
  设 type Email = string
  设 type UserId = string

  根据 A5:
    shape(Email) = { length, toString, ... } = shape(string)
    shape(UserId) = { length, toString, ... } = shape(string)
    ∴ Email ≡ UserId ≡ string

  但语义上:
    sem(Email) = "有效的电子邮件地址格式"
    sem(UserId) = "系统用户的唯一标识符"
    sem(Email) ≠ sem(UserId)

  构造函数:
    sendEmail(e: Email): void
    getUser(u: UserId): User

  由于 Email ≡ UserId ≡ string, 以下调用类型合法:
    sendEmail("user_12345")  // 编译通过, 运行时错误
    getUser("not-an-email")  // 编译通过, 语义错误

∴ 结构等价性不能保证语义安全性    ∎
```

### 工程缓解策略

| 策略 | 形式化 | 复杂度代价 |
|------|--------|-----------|
| **Branded Types** | `type UserId = string & { readonly __brand: unique symbol }` | 中 |
| **Discriminated Unions** | `type Result<T> = { ok: true; value: T } \| { ok: false; error: Error }` | 低 |
| **Nominal 库** | `newtype` 模拟 (如 `newtype-ts`) | 中 |
| **运行时验证** | Zod / io-ts 在边界验证 | 低 |

## 四、线性类型与 React Compiler

```text
传统类型: A → A → A  (idempotent, 可多次使用)
线性类型: A ⊸ A ⊸ A  (资源必须恰好使用一次)

前端映射:
  useEffect 的依赖数组 ≈ 线性类型的 "使用上下文"

  违反线性性:
    const [count, setCount] = useState(0)
    useEffect(() => { console.log(count) }, [])
    // 线性错误: 使用了未在依赖数组中声明的资源

  React Compiler 的自动 memo ≈ 线性类型推断器
    自动推断 "资源使用图", 插入正确的依赖声明
```

## 七、批判性总结

TypeScript 选择结构类型是因为 JavaScript 是动态语言，运行时无类型信息。结构类型是"鸭子类型"的静态化，符合 JS 生态的渐进式采纳策略。但在大型系统中，需通过约定 (Branded Types) 或工具 (Zod) 来弥补语义安全性的缺失。

**2026 年趋势**: TypeScript 5.x 引入更多类型级编程能力，但名义类型仍非一等公民。社区通过 `#private fields` 和 `Symbol` 来模拟名义边界。
