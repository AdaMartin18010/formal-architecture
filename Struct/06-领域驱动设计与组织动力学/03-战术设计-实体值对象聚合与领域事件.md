# 战术设计：实体、值对象、聚合与领域事件

> **来源映射**: Struct/06-领域驱动设计与组织动力学/03-战术设计-实体值对象聚合与领域事件.md
>
> **定位**：DDD战术设计是将战略边界（Bounded Context）落地为可运行代码的模式语言。实体、值对象、聚合——这三个概念构成了领域模型的"语法"。
>
> **核心命题**：聚合是事务边界。违反聚合边界的事务设计，是分布式系统一致性灾难的根源。

---

## 一、思维导图：战术设计模式

```text
DDD战术设计
│
├─【领域对象类型】
│   ├─ Entity（实体）
│   │   └─ 有身份标识，随时间变化
│   ├─ Value Object（值对象）
│   │   └─ 无身份，不可变，由属性定义
│   └─ Domain Event（领域事件）
│       └─ 已发生的事实，不可变
│
├─【聚合（Aggregate）】
│   ├─ 根实体（Aggregate Root）
│   ├─ 内部对象（实体/值对象）
│   └─ 事务边界：一个事务只修改一个聚合
│
├─【仓储（Repository）】
│   └─ 聚合的持久化抽象
│
├─【服务（Domain Service）】
│   └─ 跨聚合的操作
│
└─【工厂（Factory）】
    └─ 复杂聚合的创建逻辑
```

## 二、核心概念的形式化定义

### 2.1 实体（Entity）

```
Entity E 定义为：
  E = (id, A, B)

  其中：
    id = 全局唯一标识符（不变）
    A = 属性集合（可变）
    B = 行为集合（操作A的方法）

  身份判等：E₁ = E₂ ⟺ id(E₁) = id(E₂)
  （两个属性完全相同的实体，若id不同，仍是不同实体）

  示例：
    Order (id=12345, status=CREATED, items=[...])
    Order (id=12346, status=CREATED, items=[...]) ← 不同实体
```

### 2.2 值对象（Value Object）

```
Value Object V 定义为：
  V = (A)  // 无id

  判等：V₁ = V₂ ⟺ ∀a ∈ A: a(V₁) = a(V₂)
  （完全由属性定义，相同属性即同一对象）

  约束：不可变（Immutable）
    - 创建后不可修改
    - 修改 = 创建新实例

  示例：
    Money(amount=100, currency=USD)
    Address(street="123 Main St", city="NYC", zip="10001")

    Money + 10 = new Money(110, USD)  // 非修改，而是新建
```

### 2.3 聚合（Aggregate）

```
Aggregate A 定义为：
  A = (Root, Entities, VOs, Invariants)

  其中：
    Root = 聚合根实体（外部引用的唯一入口）
    Entities = 内部实体集合
    VOs = 内部值对象集合
    Invariants = 不变量规则集合（必须始终为真）

  核心约束：
    1. 事务边界：一个事务只修改一个聚合
    2. 引用规则：外部只能引用聚合根
    3. 删除规则：删除聚合根 = 删除整个聚合

  不变量示例：
    Order聚合：
      - totalAmount = Σ(item.price × item.quantity)
      - items非空（已下单必须有商品）
      - status状态机转换合法
```

## 三、实体 vs 值对象决策树

```
是否需要身份标识？
  ├── 是 → Entity
  │       └── 生命周期中属性会变化？
  │             ├── 是 → 典型Entity（User, Order）
  │             └── 否 → 可能是Entity（如SSN不变但有身份）
  └── 否 → Value Object
          └── 属性组合是否可能重复？
                ├── 是 → 典型VO（Money, Address）
                └── 否 → 考虑是否为Entity

经验法则：
  - 优先使用Value Object（更简单、线程安全、易测试）
  - 仅在需要跟踪身份或生命周期时使用Entity
```

## 四、聚合设计规则

```
Rule 1: 聚合尽量小
  - 大聚合 = 大事务范围 = 并发冲突
  - 经验：一个聚合的加载应在毫秒级

Rule 2: 聚合间通过ID引用
  - Order聚合持有 CustomerId，而非Customer对象
  - 打破此规则 → 大聚合、跨聚合事务

Rule 3: 一个事务 = 一个聚合
  - 若需修改两个聚合 → 使用领域事件 + 最终一致
  - 例外：真正的业务不变量跨聚合时（极罕见）

Rule 4: 聚合根是唯一的入口
  - 外部代码不能直接修改聚合内部对象
  - 只能通过聚合根的方法

反模式示例：
  × Order.Items[0].Price = 100  // 直接修改内部对象
  ✓ Order.UpdateItemPrice(0, Money.Of(100, USD))  // 通过根方法
```

## 五、概念定义与属性关系

| 概念 | 定义 | 属性 | 示例 | 反例 |
|------|------|------|------|------|
| **Entity** | 有唯一身份标识的领域对象 | 可变、由id判等 | Order, Customer | Money（无id） |
| **Value Object** | 由属性定义的无身份对象 | 不可变、由属性判等 | Money, Address, DateRange | User（有id） |
| **Aggregate** | 以根实体为中心的对象集群 | 事务边界、一致性边界 | Order（含Items, ShippingAddress） | 跨聚合直接引用 |
| **Domain Event** | 领域中已发生的事实 | 不可变、时态性、可订阅 | OrderPlaced, PaymentReceived | 命令（意图，非事实） |
| **Repository** | 聚合的持久化集合抽象 | 领域层、不暴露存储细节 | OrderRepository | 直接使用ORM Session |

## 八、权威引用

> **Eric Evans** (2003): "An aggregate is a cluster of associated objects that we treat as a unit for the purpose of data changes. Each aggregate has a root and a boundary. The boundary defines what is inside the aggregate. The root is a single, specific entity contained in the aggregate."

> **Vaughn Vernon** (2013): "The aggregate pattern is perhaps the most crucial tactical pattern because it provides the transaction consistency boundary. If you get the aggregate wrong, you will struggle with transactional integrity and performance."

## 九、批判性总结

DDD战术设计（实体、值对象、聚合）提供了一套精确的代码级设计语法，但其模式化表达在实践中常被教条化。聚合作为事务边界的定义是战术设计的核心洞见，却隐含了一个关键假设：业务不变量能够在单一事务内完成维护。在分布式系统和微服务架构中，这一假设频繁失效——跨服务的业务规则要求最终一致而非ACID，而聚合边界与物理数据库分片之间的错位会导致性能瓶颈。值对象的不可变性虽然函数式优雅，但在ORM框架和JSON序列化主导的工业界引入了额外的转换成本。与函数式编程的代数数据类型（ADT）相比，DDD战术设计更强调行为封装和身份追踪，但在类型安全性和组合性上略逊一筹。未来趋势上，事件溯源聚合（Event-Sourced Aggregates）正在成为分布式DDD的主流实现，而记录式不可变聚合与CQRS投影的结合正在重塑传统战术设计的边界。

```
│   ├─ Aggregate ⊃ {Root Entity, Child Entities, Value Objects, Invariants}
│   ├─ Bounded Context ⊃ {Aggregates, Domain Services, Repositories, Factories}
│   └─ Domain Event ⊃ {EventId, Timestamp, AggregateId, Payload}
│
├─【对立关系】
│   ├─ Entity ⟺ Value Object (有身份/生命周期 vs 无身份/不可变)
│   ├─ 强一致性 ⟺ 最终一致性 (聚合内事务 vs 跨聚合事件)
│   ├─ 贫血模型 ⟺ 充血模型 (无行为 vs 行为封装)
│   └─ 同步调用 ⟺ 异步事件 (紧耦合 vs 松耦合)
│
└─【映射关系】
    ├─ Entity ↔ 数据库表行（O/R映射）
    ├─ Value Object ↔ 嵌入值/JSON列
    ├─ Aggregate ↔ 事务边界（ACID单元）
    ├─ Domain Event ↔ 消息队列事件
    └─ Repository ↔ DAO/DataMapper模式
```

## 十一、形式化推理链：聚合边界条件与事务一致性证明

> **权威来源**：Eric Evans (2003); Vaughn Vernon (2013); Scott Millett & Nick Tune (2015)

### 11.1 聚合边界的形式化定义

```
定义（聚合）：Agg = (Root, Entities, VOs, Invariants, Transactions)
  Root ∈ Entities = 全局唯一标识的实体
  Entities = {e₁, e₂, ..., eₙ} = 聚合内部实体集合
  VOs = {v₁, v₂, ..., vₘ} = 聚合内部值对象集合
  Invariants = {i₁, i₂, ..., iₖ} = 不变量规则集合
  Transactions = {t | t修改Agg的状态且保持所有Invariants为真}

定义（聚合边界条件）：
  1. 根唯一性：∀e ∈ Entities, ∃! path(Root → e) （根是外部引用的唯一入口）
  2. 事务封闭性：∀t ∈ Transactions, scope(t) ⊆ Agg （事务不跨聚合边界）
  3. 不变量局部性：∀i ∈ Invariants, vars(i) ⊆ Agg （不变量只涉及聚合内状态）
  4. 删除传递性：delete(Root) ⟹ ∀e ∈ Entities, delete(e) （删除根=删除整个聚合）
```

### 11.2 事务一致性的形式化证明

```
定理（聚合事务一致性）：
  若所有业务操作满足聚合边界条件1-4，则系统在聚合粒度上保持强一致性。

证明：
  设系统状态 S = ⋃ Aggᵢ，每个Aggᵢ独立演化。

  对于任何业务操作 Op：
  Case 1: Op 只修改单个Aggᵢ
    由边界条件2（事务封闭性），Op在Aggᵢ内完成。
    由边界条件3（不变量局部性），Op只需验证Aggᵢ内不变量。
    ∴ Op保持Aggᵢ一致性，不影响其他Agg。

  Case 2: Op 需要修改多个Agg
    由边界条件2，不允许直接跨聚合事务。
    必须通过Domain Event实现最终一致：
      Step 1: Op修改Agg₁，产生Event E
      Step 2: E被持久化到Event Store
      Step 3: 消费者接收E，修改Agg₂

    中间状态（Step 1后，Step 3前）是不一致的，
    但这是业务可接受的暂时不一致（业务上"已下单未发货"是正常状态）。

    最终所有事件被处理，系统达到一致状态。
    ∴ 系统在聚合内强一致，跨聚合最终一致。

推论（性能优化）：
  聚合边界条件使数据库分片（Sharding）策略天然清晰：
    ShardKey = AggregateId ⟹ 同一聚合的所有数据在同一分片
    ⟹ 聚合内事务无需分布式事务（2PC/Saga）
    ⟹ 系统吞吐量与分片数线性扩展
```

## 十二、新增思维表征

### 12.1 推理判定树：实体vs值对象vs领域服务选择决策树

```text
DDD战术模式选择决策树
│
├─ 该概念是否需要全局唯一标识？
│   ├─ 是 → Entity（实体）
│   │       └─ 生命周期中属性是否变化？
│   │             ├─ 是 → 典型Entity（Order, Customer, Account）
│   │             └─ 否 → 特殊Entity（如SocialSecurityNumber，身份不变但有标识）
│   └─ 否 → Value Object（值对象）
│           └─ 属性组合是否可能重复？
│                 ├─ 是 → 典型VO（Money{100,USD} = Money{100,USD}）
│                 └─ 否 → 重新评估：是否遗漏了隐藏标识？
│
├─ 业务逻辑是否自然属于某个Entity/VO？
│   ├─ 是 → 封装为对象方法（充血模型）
│   │       └─ 方法是否只涉及该对象状态？
│   │             ├─ 是 → 直接封装
│   │             └─ 否 → 考虑Domain Service
│   └─ 否 → Domain Service（领域服务）
│           └─ 是否涉及跨聚合操作？
│                 ├─ 是 → Domain Service + 领域事件（最终一致）
│                 └─ 否 → 考虑是否遗漏聚合边界
│
├─ 多个对象是否需要在同一事务内保持一致？
│   ├─ 是 → Aggregate（聚合）
│   │       └─ 聚合根选择标准：
│   │             ├─ 哪个Entity拥有全局唯一标识？
│   │             ├─ 哪个Entity控制业务不变量？
│   │             └─ 删除哪个Entity应级联删除整个集群？
│   └─ 否 → 独立对象，通过ID引用关联
│
├─ 对象创建逻辑是否复杂？
│   ├─ 是 → Factory（工厂）
│   │       └─ 创建是否涉及多个步骤/验证？
│   │             ├─ 是 → 专用Factory类
│   │             └─ 否 → 静态工厂方法
│   └─ 否 → 直接使用构造函数
│
└─ 是否需要持久化抽象？
    ├─ 是 → Repository（仓储）
    │       └─ 是否只操作单一聚合？
    │             ├─ 是 → 标准Repository（1 Repo = 1 Agg）
    │             └─ 否 → 违反边界，重新设计聚合
    └─ 否 → 值对象直接序列化，无需Repository
```

### 12.2 多维关联树：战术设计与架构/安全/组织的关联

```text
战术设计多维关联树
│
├─【与模块05：架构模式】
│   ├─ Aggregate ↔ 事务边界 → 本地事务/ACID
│   ├─ Domain Event ↔ 事件驱动架构（EDA）→ 消息队列/Kafka
│   ├─ Repository ↔ 数据访问层 → ORM/NoSQL/DataMapper
│   ├─ Value Object不可变 ↔ 函数式编程 → Record/Case Class
│   └─ 充血模型 ↔ DDD分层架构 → 领域层独立
│
├─【与模块09：安全模型】
│   ├─ Entity身份标识 ↔ 身份认证模型（Principal）
│   ├─ 领域事件不可变 ↔ 审计追踪完整性（Tamper-evident Log）
│   ├─ 聚合边界 ↔ 安全访问控制边界（Authorization Boundary）
│   ├─ 不变量规则 ↔ 安全策略规则（Policy as Code）
│   └─ Repository抽象 ↔ 数据层安全（加密/脱敏/访问控制）
│
└─【与模块30：安全架构】
    ├─ 聚合内数据流 → 同安全域，简化控制
    ├─ 跨聚合事件流 → 安全事件监控（SIEM集成点）
    ├─ 领域服务跨聚合操作 → 安全编排（SOAR触发点）
    └─ 值对象不可变性 → 供应链安全（SBOM完整性校验）
```
