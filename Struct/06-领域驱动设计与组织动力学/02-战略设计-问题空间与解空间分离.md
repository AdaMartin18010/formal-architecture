# 战略设计：问题空间与解空间分离

> **定位**：DDD战略设计的核心洞见是"先理解问题，再设计解决方案"。问题空间（业务领域）与解空间（软件模型）的分离，是避免"用解决方案寻找问题"的疫苗。
>
> **核心命题**：软件项目失败的首要原因不是技术能力不足，而是对业务问题的理解错误。战略设计通过系统化的领域分析来降低这种风险。

---

## 一、思维导图：战略设计的双空间模型

```text
DDD战略设计
│
├─【问题空间（Problem Space）】
│   ├─ 核心域（Core Domain）
│   │   └─ 竞争优势所在，最需投入
│   ├─ 支撑子域（Supporting Subdomain）
│   │   └─ 必要但非差异化
│   └─ 通用子域（Generic Subdomain）
│       └─ 可外包或购买现成方案
│
├─【解空间（Solution Space）】
│   ├─ Bounded Context（限界上下文）
│   ├─ Context Map（上下文映射图）
│   └─ 战术设计（实体、值对象、聚合）
│
├─【核心域识别】
│   ├─ 是什么让公司与众不同？
│   ├─ 客户为什么付钱？
│   └─ 如果竞争对手复制这个功能会怎样？
│
└─【解空间映射】
    ├─ 核心域 → 内部团队 + 最优资源
    ├─ 支撑域 → 内部团队 + 标准资源
    └─ 通用域 → 外包 / SaaS / 开源
```

---

## 二、问题空间的三域分类

```
核心域（Core Domain）：
  - 定义：公司的核心竞争力和差异化来源
  - 特征：复杂、不断演进、需要领域专家深度参与
  - 策略：投入最优秀的人才和资源
  - 示例：
    - Amazon：推荐引擎、物流优化
    - Google：搜索排名、广告拍卖
    - 银行：风险评估、欺诈检测

支撑子域（Supporting Subdomain）：
  - 定义：支持核心业务但非差异化的领域
  - 特征：必要但可标准化
  - 策略：内部开发但不过度优化
  - 示例：
    - 用户管理（大多数系统需要但非差异化）
    - 通知系统（重要但可标准化）

通用子域（Generic Subdomain）：
  - 定义：行业通用、无差异化价值的领域
  - 特征：成熟解决方案存在
  - 策略：购买/外包，不自研
  - 示例：
    - 身份认证（Auth0, Keycloak）
    - 支付处理（Stripe, PayPal）
    - 文件存储（S3, Azure Blob）
```

## 三、问题空间到解空间的映射

| 问题空间 | **复杂度** | **差异化** | **解空间策略** | **团队配置** |
|---------|----------|----------|--------------|------------|
| **核心域** | 高 | 高 | 内部精研、DDD、持续演进 | 最佳团队+领域专家 |
| **支撑域** | 中 | 低 | 内部实现、适度优化 | 标准团队 |
| **通用域** | 低 | 无 | 购买/SaaS/开源 | 集成团队 |

## 四、核心域识别的启发式方法

```
问题1：如果竞争对手在X上比你强10倍，你会担心吗？
  - 是 → X可能是核心域
  - 否 → X可能是支撑域或通用域

问题2：你的客户会因为X而选择你而不是竞争对手吗？
  - 是 → X是核心域
  - 否 → X可能不是核心域

问题3：X的复杂度是否带来了进入壁垒？
  - 是 → X是核心域
  - 否 → X可能是通用域

问题4：你是否在X上积累了难以复制的领域知识？
  - 是 → X是核心域
  - 否 → X可能需要重新评估

反模式：
  × 将所有领域都视为核心域（资源分散）
  × 将核心域外包（丧失竞争力）
  × 核心域团队没有领域专家（模型与业务脱节）
```

## 五、概念定义与属性关系

| 概念 | 定义 | 属性 | 示例 | 反例 |
|------|------|------|------|------|
| **核心域** | 公司核心竞争力和差异化来源的领域 | 高复杂度、高投入、持续演进 | Amazon推荐引擎 | 用户登录系统 |
| **支撑域** | 支持核心业务但非差异化的领域 | 必要、可标准化、适度投入 | 订单通知系统 | 核心风控系统 |
| **通用域** | 行业通用、可购买的领域 | 成熟方案、无差异化、外包 | 支付处理（Stripe） | 自研支付网关 |
| **问题空间** | 业务领域及其内在结构 | 客观存在、先于软件 | 电商业务模型 | 数据库表设计 |
| **解空间** | 软件的模型和设计 | 主观构建、映射问题空间 | Bounded Context | 直接映射到代码 |
