# 战略设计：问题空间与解空间分离

> **来源映射**: Struct/06-领域驱动设计与组织动力学/02-战略设计-问题空间与解空间分离.md
>
> **定位**：DDD战略设计的核心洞见是"先理解问题，再设计解决方案"。问题空间（业务领域）与解空间（软件模型）的分离，是避免"用解决方案寻找问题"的疫苗。
>
> **核心命题**：软件项目失败的首要原因不是技术能力不足，而是对业务问题的理解错误。战略设计通过系统化的领域分析来降低这种风险。

---

## 一、思维导图：战略设计的双空间模型

```text
DDD战略设计
│
├─【问题空间（Problem Space）】
│   ├─ 核心域（Core Domain）
│   │   └─ 竞争优势所在，最需投入
│   ├─ 支撑子域（Supporting Subdomain）
│   │   └─ 必要但非差异化
│   └─ 通用子域（Generic Subdomain）
│       └─ 可外包或购买现成方案
│
├─【解空间（Solution Space）】
│   ├─ Bounded Context（限界上下文）
│   ├─ Context Map（上下文映射图）
│   └─ 战术设计（实体、值对象、聚合）
│
├─【核心域识别】
│   ├─ 是什么让公司与众不同？
│   ├─ 客户为什么付钱？
│   └─ 如果竞争对手复制这个功能会怎样？
│
└─【解空间映射】
    ├─ 核心域 → 内部团队 + 最优资源
    ├─ 支撑域 → 内部团队 + 标准资源
    └─ 通用域 → 外包 / SaaS / 开源
```

## 二、问题空间的三域分类

```
核心域（Core Domain）：
  - 定义：公司的核心竞争力和差异化来源
  - 特征：复杂、不断演进、需要领域专家深度参与
  - 策略：投入最优秀的人才和资源
  - 示例：
    - Amazon：推荐引擎、物流优化
    - Google：搜索排名、广告拍卖
    - 银行：风险评估、欺诈检测

支撑子域（Supporting Subdomain）：
  - 定义：支持核心业务但非差异化的领域
  - 特征：必要但可标准化
  - 策略：内部开发但不过度优化
  - 示例：
    - 用户管理（大多数系统需要但非差异化）
    - 通知系统（重要但可标准化）

通用子域（Generic Subdomain）：
  - 定义：行业通用、无差异化价值的领域
  - 特征：成熟解决方案存在
  - 策略：购买/外包，不自研
  - 示例：
    - 身份认证（Auth0, Keycloak）
    - 支付处理（Stripe, PayPal）
    - 文件存储（S3, Azure Blob）
```

## 三、问题空间到解空间的映射

| 问题空间 | **复杂度** | **差异化** | **解空间策略** | **团队配置** |
|---------|----------|----------|--------------|------------|
| **核心域** | 高 | 高 | 内部精研、DDD、持续演进 | 最佳团队+领域专家 |
| **支撑域** | 中 | 低 | 内部实现、适度优化 | 标准团队 |
| **通用域** | 低 | 无 | 购买/SaaS/开源 | 集成团队 |

## 四、核心域识别的启发式方法

```
问题1：如果竞争对手在X上比你强10倍，你会担心吗？
  - 是 → X可能是核心域
  - 否 → X可能是支撑域或通用域

问题2：你的客户会因为X而选择你而不是竞争对手吗？
  - 是 → X是核心域
  - 否 → X可能不是核心域

问题3：X的复杂度是否带来了进入壁垒？
  - 是 → X是核心域
  - 否 → X可能是通用域

问题4：你是否在X上积累了难以复制的领域知识？
  - 是 → X是核心域
  - 否 → X可能需要重新评估

反模式：
  × 将所有领域都视为核心域（资源分散）
  × 将核心域外包（丧失竞争力）
  × 核心域团队没有领域专家（模型与业务脱节）
```

| **问题空间** | 业务领域及其内在结构 | 客观存在、先于软件 | 电商业务模型 | 数据库表设计 |
| **解空间** | 软件的模型和设计 | 主观构建、映射问题空间 | Bounded Context | 直接映射到代码 |

## 八、战略设计的形式化定义

```
定义（问题空间-解空间映射）：
  设业务领域为 D，问题空间 P(D) 是 D 的内在结构分解：
    P(D) = CoreDomain ∪ SupportingSubdomain ∪ GenericSubdomain

  设技术系统为 S，解空间 Sol(S) 是软件模型的结构化实现：
    Sol(S) = {BC₁, BC₂, ..., BCₙ} ∪ CM ∪ TD
      其中 BC = Bounded Context
            CM = Context Map（集成拓扑）
            TD = 战术设计（实体/值对象/聚合）

  映射函数 π: P(D) → Sol(S) 满足：
    1. 核心域映射：π(CoreDomain) 对应内部精研的BC，配备最佳团队
    2. 支撑域映射：π(SupportingSubdomain) 对应标准实现的BC
    3. 通用域映射：π(GenericSubdomain) 对应集成第三方系统的BC

  映射质量评估：
    Q(π) = α·对齐度(BC边界, 团队边界) + β·演化独立性(CoreDomain)
    其中 α, β 为组织相关的权重系数
```

## 十、批判性总结

DDD战略设计通过问题空间与解空间的分离，为软件项目提供了对抗"用解决方案寻找问题"倾向的认知疫苗。然而，这一框架的隐含假设——业务战略稳定、领域专家可及、且组织有耐心进行深度领域分析——在敏捷冲刺和市场快速变化的压力下日益脆弱。失效条件包括：核心域识别错误导致战略资源错配、业务频繁 pivot 使得精心构建的领域模型迅速过时、以及"通用域"与"支撑域"的边界随商业环境变化而漂移。与精益创业"快速试错"哲学相比，DDD战略设计偏重前期分析，在不确定性极高的探索期可能构成过度投资；与敏捷开发的"可工作软件优先"相比，它强调模型深度而非交付速度。未来趋势上，Event Storming等协作式领域发现方法正在缩短问题空间分析周期，而AI辅助的领域知识提取（从业务文档自动识别聚合边界）正在降低领域专家参与的门槛。

## 十一、概念属性关系网络

```
战略设计核心概念关系网络
│
├─【依赖关系】
│   ├─ Solution Space → Problem Space (解空间依赖问题空间)
│   ├─ Core Domain → Domain Expert (核心域依赖领域专家深度参与)
│   ├─ Supporting Subdomain → Core Domain (支撑域为核心域服务)
│   └─ Generic Subdomain → External Vendor/SaaS (通用域依赖外部供应商)
│
├─【包含关系】
│   ├─ Problem Space ⊃ {Core Domain, Supporting Subdomain, Generic Subdomain}
│   ├─ Solution Space ⊃ {Bounded Context, Context Map, Tactical Design}
│   ├─ Core Domain ⊃ {关键业务规则, 差异化逻辑, 竞争壁垒}
│   └─ Generic Subdomain ⊃ {认证, 支付, 通知, 文件存储}
│
├─【对立关系】
│   ├─ Core Domain ⟺ Generic Subdomain (差异化 vs 标准化)
│   ├─ Problem Space ⟺ Solution Space (业务现实 vs 技术实现)
│   ├─ 内部构建 ⟺ 外部采购 (Build vs Buy)
│   └─ 精研投入 ⟺ 快速交付 (深度建模 vs MVP)
│
└─【映射关系】
    ├─ Core Domain ↔ 最佳团队 + 最大资源投入
    ├─ Supporting Subdomain ↔ 标准团队 + 适度优化
    ├─ Generic Subdomain ↔ 集成团队 + 外包/SaaS
    └─ 问题空间复杂度 ↔ 解空间技术选型
```

## 十二、形式化推理链：问题空间到解空间的映射证明

> **权威来源**：Eric Evans (2003); Vaughn Vernon (2013); Alberto Brandolini (2017)

### 12.1 映射函数的形式化定义

```
定义（问题空间-解空间映射）：
  设业务领域为 D，问题空间 P(D) = CoreDomain ∪ SupportingSubdomain ∪ GenericSubdomain。
  设技术系统为 S，解空间 Sol(S) = {BC₁, BC₂, ..., BCₙ} ∪ CM ∪ TD。

  映射函数 π: P(D) → Sol(S) 的良定义条件：

  条件1（覆盖性）：
    ⋃ π(p) = Sol(S), ∀p ∈ P(D)
    （解空间的每个元素都对应问题空间的某个域）

  条件2（互斥性）：
    π(CoreDomain) ∩ π(GenericSubdomain) = ∅
    （核心域与通用域的解空间不重叠）

  条件3（资源单调性）：
    Resource(π(CoreDomain)) > Resource(π(SupportingSubdomain)) > Resource(π(GenericSubdomain))
    （资源投入随差异化价值递减）

  条件4（演化一致性）：
    ∀t₁ < t₂, if BusinessValue(p, t₂) > BusinessValue(p, t₁) then Complexity(π(p), t₂) ≥ Complexity(π(p), t₁)
    （业务价值增长应伴随技术复杂度非降）
```

### 12.2 映射质量的量化评估

```
定义（映射质量函数 Q(π)）：
  Q(π) = α·Alignment(BC_boundary, Team_boundary)
       + β·EvolutionIndependence(CoreDomain)
       + γ·(1 - ResourceWaste(GenericSubdomain))
       + δ·CommunicationEfficiency(UbiquitousLanguage)

  其中 α + β + γ + δ = 1，权重反映组织优先级。

定理（最优映射存在性）：
  在有限资源约束下，存在映射 π* 使得 Q(π*) = max Q(π)。

证明概要：
  设 |P(D)| = n, |Sol(S)| = m，则可能的映射数为 mⁿ（有限）。
  Q(π) 在有限离散域上有上界。
  由极值定理，最大值必存在。

  实际求解采用启发式算法：
    1. Event Storming识别聚合边界（Brandolini, 2017）
    2. 核心域识别四问法（见原文件第四节）
    3. 迭代验证：通过Conway定律检验团队-边界对齐度
```

## 十三、新增思维表征

### 13.1 推理判定树：核心域识别决策树

```text
核心域识别决策树
│
├─ 该功能是否直接产生收入？
│   ├─ 是 → 候选核心域（收入引擎）
│   │       └─ 客户是否为此功能付费？
│   │             ├─ 是 → 高概率核心域
│   │             └─ 否 → 间接收入，需进一步评估
│   └─ 否 → 继续评估
│
├─ 竞争对手复制该功能是否会威胁生存？
│   ├─ 是 → 核心域（竞争壁垒）
│   │       └─ 该功能是否有专利/算法壁垒？
│   │             ├─ 是 → 确定为核心域
│   │             └─ 否 → 通过运营/数据壁垒保护
│   └─ 否 → 继续评估
│
├─ 该领域是否积累了难以复制的领域知识？
│   ├─ 是 → 核心域（知识壁垒）
│   │       └─ 知识是否编码在系统中？
│   │             ├─ 是 → 系统化核心资产
│   │             └─ 否 → 专家依赖风险，需知识转移
│   └─ 否 → 继续评估
│
├─ 该功能失败是否会导致监管/合规灾难？
│   ├─ 是 → 支撑域（必要但非差异化）
│   │       └─ 是否有成熟合规解决方案？
│   │             ├─ 是 → 通用域（采购标准方案）
│   │             └─ 否 → 支撑域（内部构建）
│   └─ 否 → 继续评估
│
└─ 市场上是否有3+成熟SaaS解决方案？
    ├─ 是 → 通用域（采购优于自研）
    │       └─ 集成成本 < 自研成本？
    │             ├─ 是 → 确定采购
    │             └─ 否 → 重新评估为支撑域
    └─ 否 → 支撑域（市场不成熟，需内部构建）
```

### 13.2 多维关联树：战略设计与架构/安全/组织的关联

```text
战略设计多维关联树
│
├─【与模块05：架构模式】
│   ├─ Core Domain → 最精细的架构设计（事件溯源/CQRS/微服务）
│   ├─ Supporting Subdomain → 标准架构（MVC/分层/单体模块）
│   ├─ Generic Subdomain → 集成架构（API适配/SaaS代理）
│   └─ 解空间映射 → 部署单元光谱决策（单体→微服务→Serverless）
│
├─【与模块09：安全模型】
│   ├─ Core Domain → 最高安全等级（形式化验证/渗透测试/代码审计）
│   ├─ Supporting Subdomain → 标准安全（OWASP Top 10防护）
│   ├─ Generic Subdomain → 供应商安全责任转移（SOC2/ISO27001审查）
│   └─ 问题空间风险 → 解空间威胁建模映射
│
└─【与模块30：安全架构】
    ├─ 核心域数据 → 最高密级（加密/访问控制/审计）
    ├─ 支撑域数据 → 标准密级（角色权限/日志）
    ├─ 通用域数据 → 供应商密级（合同约束/DPA）
    └─ 跨域数据流 → 安全架构中的数据分类与流控
```
