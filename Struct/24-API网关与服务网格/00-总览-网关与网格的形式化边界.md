│   │
│   ├─► 核心能力
│   │   ├─ mTLS: 自动证书管理、双向TLS
│   │   ├─ 流量管理: 金丝雀、蓝绿、A/B测试
│   │   ├─ 可观测性: 自动 Metrics/Tracing/Logging
│   │   └─ 策略执行: 访问控制、配额、熔断
│   │
│   └─► Sidecar-less 演进
│       ├─ eBPF (Cilium): 内核层代理

└─► 网关 vs 网格的边界
    ├─ 网关: 集群边界、外部通信、L7路由

    └─ 融合趋势: 统一数据平面 (Envoy Gateway)

## 二、API 网关的形式化

```text
定义 (API 网关):
  Gateway = ⟨Routes, Filters, Upstreams, Plugins⟩

  Routes: {r | r = ⟨match, upstream, filters⟩}
    match: Path ⊕ Host ⊕ Header ⊕ Method
    upstream: {backend₁, backend₂, ...}
    filters: [auth, rate_limit, transform, circuit_breaker]

  负载均衡函数:
    lb: Upstreams × Request → Backend
    lb(U, req) = round_robin(U) | weighted(U) | least_conn(U) | hash(req.key)
```

| 架构 | 代表 | 优势 | 劣势 | 2026年趋势 |

## 六、批判性总结

服务网格的 Sidecar 模式是**透明代理**思想的极致实现：应用不感知网络的存在，所有通信通过 Sidecar 中介。但这种透明性的代价是**资源税**（每个 Pod 额外一个容器）和**延迟税**（额外一跳网络）。

Istio Ambient Mesh 的洞察是：**并非所有流量都需要 L7 代理**。zTunnel 处理 L4（加密+路由），waypoint proxy 按需处理 L7。这降低了 80% 场景下的资源开销，同时保留完整的服务网格能力。这是**分层抽象**原则在服务网格中的胜利。

## 八、深度增强：形式化推理链

### 8.1 公理

**公理 A1（透明代理公理）**
Sidecar 模式的核心假设：
forall traffic: Intercept(traffic) = transparent => Application(traffic) = Application(direct)
即代理不改变应用层语义。

**公理 A2（资源税公理）**
Sidecar 引入的资源开销与 Pod 数成正比：
Overhead = N_pods * (CPU_s + MEM_s)

**公理 A3（分层必要公理）** [Istio Ambient, 2022]
并非所有流量都需要 L7 代理：
forall traffic: NeedL7(traffic) = false => Waypoint(traffic) = null

### 8.2 引理

**引理 L1（延迟税）**
Sidecar 额外延迟：
T_sidecar = T_iptables + T_envoy_processing + T_network_hop
典型值：1-3ms

**引理 L2（网关爆炸半径）**
单网关模式的故障影响范围：
BlastRadius = all(UpstreamServices)
多网关模式：BlastRadius = subset(UpstreamServices)

### 8.3 定理

**定理 T1（Sidecar-less 资源节省）** [Istio Ambient, 2022]
设 L4 流量比例为 p，则 Ambient 模式资源节省：
SavedResources = p *N_pods* MEM_envoy
当 p = 0.8, N = 10000, MEM_envoy = 128MB 时，节省约 1TB 内存。

**定理 T2（统一数据平面一致性）**
单一 Envoy 同时处理 T_NS 和 T_EW 时：
Consistent(Policy) = Config(Edge) = Config(Mesh) 当使用同一 xDS 控制面
否则存在策略分裂风险。

### 8.4 推论

**推论 C1**：Gateway API 的标准化收益：
Portability = 1 - VendorLockIn(GatewayAPI) / VendorLockIn(IngressAnnotations)

**推论 C2**：eBPF 的 L7 功能受限性：
forall f in L7Features: Implementable(eBPF, f) = false 当 f 需要 regex/backtrack

## 十一、批判性总结（深度增强版）

API 网关与服务网格的边界划分是云原生网络架构中的核心张力点。网关处理南北向流量，关注外部安全、协议转换和全局策略；网格处理东西向流量，关注服务发现、细粒度路由和可观测性。这种划分的合理性建立在**信任域差异**之上——外部流量不可信，需要 WAF、Bot 检测和 OAuth；内部流量相对可信，重点在于加密和访问控制。

然而，这一边界正在迅速模糊。随着微服务架构的演进，越来越多的内部服务需要暴露给外部合作伙伴（B2B API），而外部移动客户端又需要直接调用内部服务（微前端架构）。这导致南北向和东西向流量的治理策略必须统一——同一个服务既可能被外部用户通过网关访问，也可能被内部服务通过网格访问，两者需要一致的认证、授权和限流策略。Envoy 作为统一数据平面的事实标准，其优势正在于此：运维人员只需理解 Listener-Route-Cluster 模型，即可同时管理边缘入口和服务间通信。

Sidecar-less 运动（Istio Ambient、Cilium eBPF）并非对服务网格价值的否定，而是对其**实现方式**的重新审视。Sidecar 模式将功能与部署单元过度耦合——为了获得 L7 能力，每个 Pod 都必须承担完整 Envoy 的资源代价，即使其中 80% 的流量仅需 L4 加密和路由。Ambient Mesh 的分层架构（zTunnel + Waypoint）提供了一个精妙的解决方案：按需求分层付费。这与操作系统中微内核 + 可加载模块的设计哲学异曲同工，代表了分布式系统从单体式代理向模块化、可组合代理的演进方向。

但统一数据平面也引入了新的风险。**RBAC 策略的精细设计**变得至关重要——一个错误的 VirtualService 规则可能同时影响外部用户访问和内部服务通信，其爆炸半径远超分离式架构。此外，eBPF 的 Verifier 对程序复杂度的严格限制意味着某些高级 L7 功能（如复杂的 WASM 插件）在内核层难以实现，这决定了 eBPF 不可能完全替代用户态代理，而只能作为其加速层。
