## 一、知识体系思维导图

```text
API网关与服务网格
│
├─► API 网关 (南北向流量)
│   ├─► 核心职责
│   │   ├─ 请求路由: Path/Host/Header 匹配
│   │   ├─ 负载均衡: 轮询/加权/最小连接
│   │   ├─ 认证鉴权: JWT/OAuth/API Key
│   │   ├─ 速率限制: 令牌桶/漏桶
│   │   ├─ 请求/响应转换: 协议转换、加解密
│   │   └─ 熔断降级: 断路器模式
│   │
│   ├─► 架构演进
│   │   ├─ 第一代: 硬件负载均衡 (F5, A10)
│   │   ├─ 第二代: 软件网关 (Nginx, Kong, Zuul)
│   │   └─ 第三代: 云原生网关 (Envoy, Traefik, APISIX)
│   │
│   └─► 网关模式
│       ├─ 单网关: 统一入口、单点瓶颈
│       ├─ 多网关: 按域/团队拆分
│       └─ Backend-for-Frontend (BFF): 客户端专属网关
│
├─► 服务网格 (东西向流量)
│   ├─► 核心抽象
│   │   ├─ Sidecar 代理: 与应用容器共部署
│   │   ├─ 数据平面: Envoy / Linkerd-proxy
│   │   ├─ 控制平面: Istiod / Linkerd-control-plane
│   │   └─ 服务注册与发现: Kubernetes DNS / Consul
│   │
│   ├─► 核心能力
│   │   ├─ mTLS: 自动证书管理、双向TLS
│   │   ├─ 流量管理: 金丝雀、蓝绿、A/B测试
│   │   ├─ 可观测性: 自动 Metrics/Tracing/Logging
│   │   └─ 策略执行: 访问控制、配额、熔断
│   │
│   └─► Sidecar-less 演进
│       ├─ eBPF (Cilium): 内核层代理
│       ├─ Istio Ambient Mesh: zTunnel + waypoint
│       └─ 共享代理模式: 每个节点一个代理
│
└─► 网关 vs 网格的边界
    ├─ 网关: 集群边界、外部通信、L7路由
    ├─ 网格: 服务间通信、内部治理、L4+L7
    └─ 融合趋势: 统一数据平面 (Envoy Gateway)
```

## 二、API 网关的形式化

```text
定义 (API 网关):
  Gateway = ⟨Routes, Filters, Upstreams, Plugins⟩

  Routes: {r | r = ⟨match, upstream, filters⟩}
    match: Path ⊕ Host ⊕ Header ⊕ Method
    upstream: {backend₁, backend₂, ...}
    filters: [auth, rate_limit, transform, circuit_breaker]

  负载均衡函数:
    lb: Upstreams × Request → Backend
    lb(U, req) = round_robin(U) | weighted(U) | least_conn(U) | hash(req.key)
```

## 三、服务网格的架构演进

| 架构 | 代表 | 优势 | 劣势 | 2026年趋势 |
|------|------|------|------|-----------|
| **Sidecar** | Istio (经典) | 无侵入、语言无关 | 资源开销高、延迟增加 | 逐步替代 |
| **eBPF** | Cilium | 内核层、极低延迟 | 内核版本要求、功能受限 | 快速增长 |
| **Ambient** | Istio Ambient | Sidecar-less + 功能完整 | 新架构、成熟度待验证 | **主流方向** |
| **Shared Proxy** | Linkerd | 简单、轻量 | 多租户隔离弱 | 小规模场景 |

## 六、批判性总结

服务网格的 Sidecar 模式是**透明代理**思想的极致实现：应用不感知网络的存在，所有通信通过 Sidecar 中介。但这种透明性的代价是**资源税**（每个 Pod 额外一个容器）和**延迟税**（额外一跳网络）。

Istio Ambient Mesh 的洞察是：**并非所有流量都需要 L7 代理**。zTunnel 处理 L4（加密+路由），waypoint proxy 按需处理 L7。这降低了 80% 场景下的资源开销，同时保留完整的服务网格能力。这是**分层抽象**原则在服务网格中的胜利。
