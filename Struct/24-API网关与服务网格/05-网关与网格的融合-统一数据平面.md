## 一、知识体系思维导图

```text
网关与网格的融合趋势
│
├─► 流量方向定义
│   ├─► 南北向 (North-South): 外部 → 集群入口 (Edge)
│   ├─► 东西向 (East-West): 服务间通信 (Mesh)
│   └─► 边界模糊化: API 网关也开始处理东西向流量
│
├─► 融合驱动力
│   ├─► 技术同质: Envoy 同时统治网关和网格数据平面
│   ├─► 配置分裂: 两套控制平面导致策略不一致
│   ├─► 运维复杂: 南北向 + 东西向需分别管理
│   ├─► 可观测性割裂: 跨边界追踪链路断裂
│   └─► 安全边界: 内外网策略语义不统一
│
├─► 统一数据平面方案
│   ├─► Envoy Gateway (EG): K8s Gateway API + Envoy
│   ├─► Istio + Gateway API: 单一 Istiod 管理双向流量
│   ├─► Cilium Gateway API: eBPF L4 + Envoy L7
│   └─► 统一身份: SPIFFE/SPIRE 跨边界传递
│
└─► 标准化: Kubernetes Gateway API
    ├─► GatewayClass: 实现抽象 (Istio, Envoy-Gateway, Cilium)
    ├─► Gateway: 监听器定义 (端口, 协议, TLS)
    ├─► HTTPRoute/TCPRoute: 路由规则
    ├─► ReferenceGrant: 跨命名空间引用
    └─► 目标: 一套 API 管理所有流量方向
```

## 二、核心概念的形式化定义

**定义 1 (流量方向的拓扑定义)**:
设网络拓扑为图 $G = (V, E)$，其中 $V$ 为节点（客户端、网关、服务），$E$ 为连接。

- **南北向流量**: $Traffic_{NS} = \{ (u, v) \mid u \in External, v \in Cluster \}$
- **东西向流量**: $Traffic_{EW} = \{ (u, v) \mid u, v \in Cluster, u \neq v \}$

**定义 2 (统一数据平面)**:
$$UnifiedDP = \langle Envoy, xDS, GatewayAPI \rangle$$
其中单一 Envoy 实例可同时承担:

- **Edge Proxy**: 处理 $Traffic_{NS}$，绑定公网 IP，执行 WAF、Bot 检测
- **Mesh Proxy**: 处理 $Traffic_{EW}$，执行 mTLS、细粒度路由、熔断

$$Function(Envoy) = EdgeFunctions \cup MeshFunctions$$

**定义 3 (Gateway API 资源模型)**:
$$GatewayAPI = \langle GatewayClass, Gateway, Route, BackendRef \rangle$$

- $GatewayClass$: 实现声明，$GatewayClass = \langle controller, parameters \rangle$
- $Gateway$: 网络端点，$Gateway = \langle listeners, addresses \rangle$
- $Route$: 路由规则，$Route = \langle parentRefs, rules, hostnames \rangle$
- $BackendRef$: 后端引用，$BackendRef = \langle service, port, weight \rangle$

**定义 4 (跨边界身份传递)**:
设外部用户身份为 $Identity_{ext}$，服务网格身份为 $Identity_{mesh}$:
$$Propagate(Identity_{ext}) = Identity_{mesh} \text{ via } XFCC \text{ (x-forwarded-client-cert)}$$
统一网关将外部 JWT/OAuth 身份转换为内部 SPIFFE 身份，实现**端到端零信任**。

|---------|---------|-----------|-----------|---------|-----------|----------|
| **分离式** | Nginx + Envoy | 2+ | 低(策略分裂) | 断裂 | 高 | 遗留系统 |
| **同构分离** | Envoy + Envoy | 2 | 中 | 部分统一 | 中 | 存量过渡 |
| **统一平面** | **单一 Envoy** | **1** | **高** | **统一** | **低** | **主流方向** |
| **内核统一** | eBPF + Envoy | 1-2 | 高 | 统一 | 低 | 新兴 |

|-----------------|---------|--------|--------|--------|-------------------|
| **Envoy Gateway** | CNCF | **优** | 中 | 良 | 中(可扩展) |
| **Istio + GW API** | Istio | 良 | **优** | 良 | 中 |
| **Cilium Gateway** | Cilium | 良 | **优** | 良 | 有限 |
| **NGINX Gateway** | NGINX | **优** | 有限 | 良 | **优** |
| **GKE Gateway** | Google | **优** | 中 | **强** | **优(Cloud Armor)** |

|---------|---------|---------|
| **协议差异** | 外部 HTTP/1.1 vs 内部 gRPC | Envoy 自动协议转换 |

| **团队边界** | 平台组管网关，SRE 管网格 | 统一控制平面 + RBAC |

## 五、工程实践与代码示例

```yaml
# 统一流量管理: Envoy Gateway 管理南北向 + Istio 管理东西向
# --- 南北向入口 (Envoy Gateway) ---
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: public-gateway
  namespace: ingress
spec:
  gatewayClassName: envoy-gateway
  listeners:
    - name: https-public
      protocol: HTTPS
      port: 443
      tls:
        mode: Terminate
        certificateRefs:
          - name: public-cert
---
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: api-public-route
  namespace: ingress
spec:
  parentRefs:
    - name: public-gateway
  hostnames:
    - api.example.com
  rules:
    - matches:
        - path:
            type: PathPrefix
            value: /api/v1
      backendRefs:
        - name: api-gateway-internal
          namespace: mesh
          port: 8080
---
# --- 东西向路由 (Istio VirtualService) ---
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
metadata:
  name: api-internal
  namespace: mesh
spec:
  hosts:
    - api-gateway-internal
  http:
    - route:
        - destination:
            host: user-service
            port:
              number: 8080
      match:
        - uri:
            prefix: /api/v1/users
    - route:
        - destination:
            host: order-service
            port:
              number: 8080
      match:
        - uri:
            prefix: /api/v1/orders
```

## 六、批判性总结

网关与网格的融合是云原生网络栈演进的必然终点，但这一融合并非简单的"用同一个软件处理所有流量"，而是**在统一的数据平面之上，保持控制平面的领域特定性**。

Envoy 作为统一数据平面的事实标准，其优势在于"一次学习，到处使用"——运维人员只需理解 Envoy 的 Listener-Route-Cluster 模型，即可同时管理边缘入口和服务间通信。然而，Envoy 配置的复杂性（一个完整配置动辄数千行 YAML）也由此被同时带到了南北向和东西向两个场景中。Envoy Gateway 项目通过 Kubernetes Gateway API 提供了一层更友好的抽象，但这种抽象在高级场景下（如自定义 WASM 插件、复杂的局部速率限制）仍然需要回落到原生 Envoy 配置。

更深层的挑战在于**组织治理**而非技术本身。南北向网关通常由平台团队或安全团队管理，关注点是 WAF、DDoS 防护、Bot 检测和合规审计；东西向网格通常由 SRE 或基础设施团队管理，关注点是服务发现、熔断、金丝雀发布和 mTLS。当统一数据平面将两者的配置合并到同一套系统中时，**RBAC 策略的精细设计**变得至关重要——一个错误的 VirtualService 规则可能同时影响外部用户访问和内部服务通信，其爆炸半径远超分离式架构。

Kubernetes Gateway API 的真正价值不在于取代 Ingress 或 Istio API，而在于提供了**跨实现的通用语言**。当团队可以说"我们使用 Gateway + HTTPRoute"而不是"我们使用 Nginx Ingress 的 annotations"时，云原生网络配置的可移植性和可维护性得到了质的提升。然而，标准化总是滞后于创新：当 Gateway API 还在 v1 阶段时，服务网格领域已经在探索 Sidecar-less 和 eBPF——标准与前沿之间的张力，将是这个领域未来五年的主旋律。
