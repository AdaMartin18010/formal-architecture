# 分布式事务：2PC、Saga与TCC的比较

> **来源映射**: Struct/04-数据一致性代数结构/04-分布式事务-2PC-Saga与TCC的比较.md
>
> **定位**：分布式事务是数据一致性在工程中的最复杂体现——从强一致的2PC到最终一致的Saga，每种模式都是对CAP权衡的不同回答。
>
> **核心命题**：没有"最好的"分布式事务模式，只有"最适合当前一致性需求和性能约束"的模式。

## 一、思维导图：分布式事务模式谱系

```text
分布式事务模式
│
├─【强一致性】
│   ├─ 2PC（两阶段提交）
│   │   └─ Prepare + Commit，全局锁
│   ├─ 3PC（三阶段提交）
│   │   └─ 增加CanCommit阶段，减少阻塞
│   └─ XA协议
│       └─ 数据库层面的2PC标准
│
├─【最终一致性】
│   ├─ Saga模式
│   │   ├─ 编排式（Orchestration）
│   │   └─ 编舞式（Choreography）
│   ├─ TCC（Try-Confirm-Cancel）
│   │   └─ 业务层补偿，资源预留
│   └─ 本地消息表 / Outbox
│       └─ 可靠消息投递
│
└─【选择维度】
    ├─ 一致性需求
    ├─ 隔离性需求
    ├─ 业务复杂度
    └─ 性能要求
```

## 二、2PC：两阶段提交

```
2PC协议：

  Phase 1: Prepare
    Coordinator → All Participants: PREPARE

    Each Participant：
      - 执行本地事务（不提交）
      - 锁定资源
      - 若可提交 → 写redo/undo日志 → 回复YES
      - 若不可提交 → 回复NO

  Phase 2: Commit / Abort
    若所有Participant回复YES：
      Coordinator → All: COMMIT
      Participants提交本地事务、释放锁

    若有Participant回复NO：
      Coordinator → All: ABORT
      Participants回滚本地事务、释放锁

问题：
  - Coordinator单点故障 → 参与者可能无限阻塞
  - 同步阻塞 → 性能差
  - 全局锁 → 并发度低

3PC改进：
  增加CanCommit阶段，引入超时机制
  → 减少阻塞，但复杂度增加
```

## 三、Saga模式

```
Saga模式：将长事务拆分为本地事务序列

  正向流程：
    T1 → T2 → T3 → ... → Tn

  补偿流程（若Tk失败）：
    C(k-1) → C(k-2) → ... → C1
    （执行Tk-1到T1的补偿操作）

两种实现：

  1. 编排式Saga（Orchestration）
     中央协调器控制流程
     → 流程集中可见
     → 协调器可能成为瓶颈

  2. 编舞式Saga（Choreography）
     各服务通过事件自主响应
     → 松耦合
     → 流程分散、难追踪

示例（电商订单）：
  T1: 创建订单（本地事务）
  T2: 扣减库存（预留）
  T3: 处理支付
  T4: 创建物流单

  若T3失败：
    C2: 释放库存预留
    C1: 取消订单
```

## 四、TCC：Try-Confirm-Cancel

```
TCC模式：业务层实现的三阶段补偿

  Try阶段：
    - 预留资源（非真正执行）
    - 示例：库存服务预留商品（不扣减）

  Confirm阶段：
    - 真正执行业务
    - 示例：将预留转为实际扣减

  Cancel阶段：
    - 释放预留资源
    - 示例：取消预留，库存恢复

与Saga的区别：
  ┌───────────┬─────────────┬─────────────┐
  │ 维度      │ Saga         │ TCC          │
  ├───────────┼─────────────┼─────────────┤
  │ 补偿时机   │ 失败后补偿    │ Try后Confirm  │
  │ 资源锁定   │ 业务逻辑      │ Try阶段预留   │
  │ 侵入性     │ 低           │ 高（需三接口）│
  │ 适用       │ 通用         │ 资源预留场景  │
  └───────────┴─────────────┴─────────────┘
```

## 五、分布式事务模式对比矩阵

| 维度 | **2PC** | **Saga** | **TCC** | **本地消息表** |
|------|--------|---------|--------|--------------|
| **一致性** | 强一致 | 最终一致 | 最终一致 | 最终一致 |
| **隔离性** | 全局隔离 | 无隔离 | Try阶段隔离 | 无隔离 |
| **性能** | 差（锁+同步） | 好 | 中（需预留） | 好 |
| **复杂度** | 中 | 高（补偿逻辑） | 高（三接口） | 中 |
| **回滚** | 自动 | 手动补偿 | 手动Cancel | 重试+死信 |
| **适用场景** | 短事务、强一致 | 长事务、业务流程 | 资源预留、高并发 | 异步、解耦 |
| **代表框架** | XA, Java JTA | Temporal, Axon | 自研 | RocketMQ, Debezium |

## 六、概念定义与属性关系

| 概念 | 定义 | 属性 | 示例 | 反例 |
|------|------|------|------|------|
| **2PC** | 两阶段提交协议 | 强一致、阻塞、协调者单点 | 数据库XA事务 | 高并发场景 |
| **Saga** | 长事务拆分为本地事务+补偿 | 最终一致、无全局锁、补偿复杂 | 电商订单处理 | 需要强隔离的场景 |
| **TCC** | Try-Confirm-Cancel三阶段 | 业务层预留、高并发友好 | 库存扣减 | 无预留概念的业务 |
| **补偿事务** | 撤销已完成操作的业务逻辑 | 非技术回滚、可能有副作用 | 退款、释放库存 | 数据库ROLLBACK |
| **Outbox** | 将消息与业务数据同事务持久化 | 可靠、至少一次、需CDC | 订单创建+事件发送 | 独立双写 |

## 九、分布式事务的形式化定义

```
定义（分布式事务）：
  设事务 T 跨越参与者集合 P = {p₁, p₂, ..., pₙ}，
  每个参与者维护本地状态 Sᵢ 和本地事务日志 Wᵢ。

  分布式事务要求满足扩展的ACID属性：
    原子性（Atomicity）：∀pᵢ ∈ P, T在pᵢ上提交 ⟹ T在所有pⱼ ∈ P上提交
    一致性（Consistency）：∪Sᵢ(T后) 满足全局不变量
    隔离性（Isolation）：调度等价于某个串行调度（可串行化）
    持久性（Durability）：提交后，∀pᵢ, Wᵢ已持久化

  2PC的阻塞形式化：
    若协调者在Phase 1后崩溃，参与者持有锁等待决策，
    ∃pᵢ: State(pᵢ) = " prepared" ∧ 决策 ∈ {COMMIT, ABORT} 未知
    → 全局阻塞直到协调者恢复

  Saga的最终一致形式化：
    Saga = 本地事务序列 ⟨T₁, T₂, ..., Tₙ⟩ 及补偿序列 ⟨C₁, C₂, ..., Cₙ₋₁⟩
    若 Tₖ 失败，则执行 Cₖ₋₁, ..., C₁
    保证：∀i < k, Tᵢ 的效果被 Cᵢ 逻辑撤销（非技术回滚）
```

## 十一、批判性总结

分布式事务模式（2PC/Saga/TCC）本质上是对CAP定理中一致性-可用性权衡的工程回应，而非普适解决方案。2PC的强一致性承诺以协调者单点和全局阻塞为代价，其隐含假设——网络分区罕见且协调者高可用——在跨区域部署中往往不成立。Saga和TCC通过放弃隔离性和即时一致性来换取可用性，但假设补偿操作语义上可逆且无副作用，这一假设在金融交易、物理库存等域中经常失效（例如"已发货"无法真正撤销）。与CRDT的协调自由哲学相比，分布式事务保留了更强的一致性语义，却引入了复杂的协调拓扑。未来趋势上，确定性数据库（如Calvin、FaunaDB）通过预排序消除分布式协调的运行时不确定性，而Saga编排的自动化（Temporal、Netflix Conductor）正在降低补偿逻辑的工程负担。

## 十二、深度增强：概念属性关系网络

```text
【分布式事务核心概念属性关系网络】

2PC (Two-Phase Commit)
  ├─ 依赖 → 本地ACID事务 (参与者必须具备本地事务能力)
  ├─ 包含 → 协调者 (Coordinator)
  ├─ 包含 → Prepare阶段 (投票)
  ├─ 包含 → Commit/Abort阶段 (决策)
  ├─ 对立 → Saga (强一致 vs 最终一致)
  ├─ 对立 → CRDT (全局协调 vs 协调自由)
  └─ 映射 → XA协议 (数据库层面的2PC标准)

Saga模式
  ├─ 依赖 → 本地事务 (每个步骤是独立本地事务)
  ├─ 依赖 → 补偿操作 (语义逆操作)
  ├─ 包含 → 编排式 (Orchestration: 中央协调器)
  ├─ 包含 → 协同式 (Choreography: 事件驱动)
  ├─ 对立 → 2PC (无全局锁 vs 全局锁)
  ├─ 对立 → 强隔离 (无隔离 vs 全局隔离)
  └─ 映射 → EDA (协同式Saga是事件驱动架构的实现)

TCC (Try-Confirm-Cancel)
  ├─ 依赖 → 业务层预留 (Try阶段)
  ├─ 包含 → 三阶段接口 (Try/Confirm/Cancel)
  ├─ 对立 → Saga (预留资源 vs 事后补偿)
  ├─ 对立 → 2PC (业务层实现 vs 协议层实现)
  └─ 映射 → Seata (阿里开源TCC框架)

3PC (Three-Phase Commit)
  ├─ 依赖 → 2PC (增加CanCommit阶段)
  ├─ 包含 → 超时机制 (减少阻塞)
  ├─ 对立 → 2PC (更低阻塞概率 vs 更高复杂度)
  └─ 映射 → 网络分区容错 (超时作为故障检测)
```

## 十三、深度增强：形式化推理链

### 13.1 一致性模型层级关系证明（分布式事务视角）

```
定理：分布式事务模式的一致性层级
  2PC ⟹ 3PC（一致性强弱）
  2PC > Saga > TCC > 本地消息表（一致性强度）

但注意：
  2PC提供强一致性（阻塞式）
  Saga/TCC/本地消息表均提供最终一致性
  它们之间的差异在于隔离性和实现机制，而非一致性级别

2PC的强一致性形式化：
  定义：事务T的原子性
    ∀pᵢ ∈ P: T在pᵢ上提交 ⟹ ∀pⱼ ∈ P: T在pⱼ上提交
    ∀pᵢ ∈ P: T在pᵢ上中止 ⟹ ∀pⱼ ∈ P: T在pⱼ上中止

  2PC保证：
    Phase 1后，所有参与者处于PREPARED状态
    Phase 2中，协调者发送统一决策（COMMIT或ABORT）
    由本地事务的ACID保证，参与者的执行是原子的

  缺陷：
    协调者单点故障时，参与者可能无限阻塞
    形式化：∃pᵢ: State(pᵢ) = PREPARED ∧ Decision未知
             ⟹ pᵢ持有锁等待，直到协调者恢复

Saga的最终一致性形式化：
  定义：Saga = ⟨T₁, T₂, ..., Tₙ⟩ 及补偿序列 ⟨C₁, C₂, ..., Cₙ₋₁⟩

  一致性保证：
    若所有Tᵢ成功 → 最终状态等价于所有Tᵢ提交的串行执行
    若Tₖ失败 → 执行 Cₖ₋₁, ..., C₁
    最终状态 = T₁, ..., Tₖ₋₁的效果被Cₖ₋₁, ..., C₁逻辑撤销

  关键差异：
     Saga无全局隔离性。在Tᵢ提交和Tᵢ₊₁开始之间存在时间窗口，
    其他事务可能观察到中间状态（脏读）。
    形式化：¬∃全局调度S: S ≡ 某个串行调度
```

### 13.2 Saga补偿语义的形式化分析

```
定理：Saga补偿操作不构成数学逆元

给定：
  正向操作 T: S → S' （状态转换）
  补偿操作 C: S' → S'' （撤销尝试）

理想逆元要求：
  C ∘ T = id （恒等映射）
  即：执行T后再执行C，状态完全恢复为原始状态

现实补偿的局限：
  (1) 信息丢失不可逆
      T: 扣减库存（库存100→90）
      C: 恢复库存（库存90→100）
      此例中C ∘ T = id，是可逆的

      反例：
      T: 发送邮件通知客户
      C: 发送"取消通知"邮件
      客户已阅读第一封邮件，C无法撤销这一事实
      ∴ C ∘ T ≠ id

  (2) 副作用累积
      T₁: 创建订单 → 订单号#12345
      T₂: 扣减库存
      T₃: 处理支付

      若T₃失败，补偿：
      C₂: 恢复库存
      C₁: 取消订单#12345

      但订单号#12345已被分配且记录于日志，
      取消后该订单号不可复用（审计要求）
      ∴ 系统状态 ≠ T₁执行前状态（订单号已消耗）

  (3) 补偿操作的补偿
      C操作本身可能失败，需要重试或人工介入
      这与2PC的自动回滚有本质区别

结论：
  Saga的"一致性"是业务层面的一致性，而非数学层面的原子性。
  补偿操作C满足：C(T(s)) ≈ s （近似恢复），而非严格相等。
```

### 13.3 TCC与2PC的代数对应关系

```
定理：TCC可视为业务层实现的"软2PC"

对比形式化：

┌─────────────┬─────────────────────┬─────────────────────────┐
│ 维度        │ 2PC                 │ TCC                     │
├─────────────┼─────────────────────┼─────────────────────────┤
│ 准备阶段    │ PREPARE             │ Try (预留资源)          │
│ 提交阶段    │ COMMIT              │ Confirm (真正执行)      │
│ 回滚阶段    │ ABORT/ROLLBACK      │ Cancel (释放预留)       │
│ 锁定机制    │ 数据库锁            │ 业务层预留（乐观锁）    │
│ 隔离级别    │ 取决于本地DB        │ 读未提交（预留可见）    │
│ 阻塞性      │ 同步阻塞            │ 非阻塞（预留期有限）    │
│ 实现层级    │ 数据库/协议层        │ 应用/业务层              │
└─────────────┴─────────────────────┴─────────────────────────┘

TCC的代数优势：
  Try阶段将资源从"可用"转为"预留"，不阻塞其他操作。
  其他事务仍可操作非预留资源。
  形式化：
    设资源池R，TCC预留子集R_try ⊆ R
    其他事务可操作 R \ R_try（剩余资源）
    2PC的锁则阻塞对所有R的并发访问

TCC的代数风险：
  若Try后Confirm迟迟不来，预留资源被长期占用。
  需要预留超时机制：若超时未Confirm，自动Cancel。
  形式化：
    ∀预留r ∈ R_try: ∃超时T, 当前时间 > 创建时间(r) + T
    ⟹ 自动执行Cancel(r)
```

### 14.1 推理判定树：如何选择分布式事务模式

```text
开始：面临跨服务/跨数据库事务需求
│
├─ 是否可接受最终一致性？
│   ├─ 否（必须强一致）→ 协调型事务
│   │   ├─ 事务短（< 1秒）且参与者少（< 5）？ → 2PC / XA
│   │   │   └─ 但需注意：协调者单点、全局阻塞风险
│   │   │
│   │   ├─ 需要全球分布 + 外部一致性？ → Spanner TrueTime
│   │   └─ 可接受乐观并发 + 冲突重试？ → Calvin确定性数据库
│   │
│   └─ 是（可接受最终一致）→ Saga/TCC/消息路径
│       │
│       ├─ 业务操作是否有明确的"撤销"语义？
│       │   ├─ 是（如取消订单、退款）→ Saga模式
│       │   │   ├─ 流程复杂、需可视化监控？ → 编排式Saga (Temporal/Camunda)
│       │   │   └─ 流程简单、追求松耦合？ → 协同式Saga (Kafka + 事件处理器)
│       │   │
│       │   └─ 否（操作不可逆或难以补偿）→ TCC或重新设计
│       │       ├─ 可实施资源预留？ → TCC (Try-Confirm-Cancel)
│       │       └─ 不可预留 → 考虑2PC或业务重构
│       │
│       ├─ 是否需要最大可用性（分区时仍可用）？
│       │   ├─ 是 → 协同式Saga（无单点协调器）
│       │   └─ 否 → 编排式Saga或TCC
│       │
│       └─ 异步解耦需求强？
│           ├─ 是 → 本地消息表 / Outbox + 消息队列
│           └─ 否 → Saga直接服务调用
│
└─ 性能要求？
    ├─ 极高吞吐 + 低延迟 → 避免2PC（阻塞）; 考虑Saga或CRDT
    ├─ 中等吞吐 + 强一致 → 2PC可接受
    └─ 可变负载 + 事件驱动 → Saga + 消息队列
```

## 十六、深度增强：权威来源与批判性总结

> **权威来源**：
>
> - **Hector Garcia-Molina, Kenneth Salem** (1987): "Sagas", ACM SIGMOD. 首次提出Saga概念，将长事务拆分为可补偿的本地事务序列，是分布式长事务处理的奠基性工作。
> - **Jim Gray** (1978): "Notes on Data Base Operating Systems", LNCS 60. 系统阐述事务概念，定义ACID属性，2PC协议的早期形式化。
> - **Pat Helland** (2007/2016): "Life beyond Distributed Transactions: an Apostate's Opinion", CACM. 论证分布式事务在大型系统中的局限性，倡导基于补偿的最终一致性。
> - **Chris Richardson** (2018): *Microservices Patterns*, Manning. 第4-6章详细讨论Saga模式（编排式与协同式）和TCC在微服务中的实现。
> - **James C. Corbett et al.** (2012): "Spanner: Google's Globally-Distributed Database", OSDI. TrueTime + 2PC实现外部一致性的工程实践。

> **批判性总结（300字以上）**：
> 分布式事务模式（2PC/Saga/TCC）本质上是对CAP定理中一致性-可用性权衡的工程回应，而非普适解决方案。2PC的强一致性承诺以协调者单点和全局阻塞为代价，其隐含假设——网络分区罕见且协调者高可用——在跨区域部署中往往不成立。Gray 1978年对事务概念的优雅定义（"用BEGIN和COMMIT括号化计算，系统保证原子性和持久性"）在分布式环境下遭遇了根本性挑战：当括号跨越网络边界时，原子性的实现成本呈指数级增长。Saga和TCC通过放弃隔离性和即时一致性来换取可用性，但假设补偿操作语义上可逆且无副作用，这一假设在金融交易、物理库存等域中经常失效——"已发货"无法真正撤销，"已通知客户"无法收回，"已消耗的订单号"无法复用。Garcia-Molina和Salem 1987年的原始Saga论文将补偿定义为"语义逆操作"，却未充分讨论不可逆副作用的系统性处理；Helland 2007年的"异端之见"正确指出了分布式事务的局限性，却将补偿的复杂性轻描淡写为"业务逻辑中的不确定性管理"。与CRDT的协调自由哲学相比，分布式事务保留了更强的一致性语义，却引入了复杂的协调拓扑和单点故障风险；与确定性数据库（Calvin、FaunaDB）相比，2PC的运行时协调开销可以通过预排序消除，却牺牲了事务灵活性。TCC作为业务层2PC，通过资源预留降低了阻塞性，但预留本身引入了新的竞争条件和超时管理复杂性。未来趋势上，确定性数据库通过预排序消除分布式协调的运行时不确定性，Saga编排的自动化（Temporal、Netflix Conductor）正在降低补偿逻辑的工程负担，而"Saga + 事件溯源"的组合正在成为长事务管理的工业标准。然而，最根本的洞见始终未变：**分布式事务没有免费午餐——要么支付协调成本（2PC），要么支付补偿复杂性（Saga/TCC），要么接受语义弱化（CRDT）**。架构师的选择必须基于业务对一致性窗口的容忍度、对补偿语义的精确理解，以及对组织协调能力的现实评估。
