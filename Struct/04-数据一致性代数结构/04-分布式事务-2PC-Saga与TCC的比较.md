# 分布式事务：2PC、Saga与TCC的比较

> **定位**：分布式事务是数据一致性在工程中的最复杂体现——从强一致的2PC到最终一致的Saga，每种模式都是对CAP权衡的不同回答。
>
> **核心命题**：没有"最好的"分布式事务模式，只有"最适合当前一致性需求和性能约束"的模式。

---

## 一、思维导图：分布式事务模式谱系

```text
分布式事务模式
│
├─【强一致性】
│   ├─ 2PC（两阶段提交）
│   │   └─ Prepare + Commit，全局锁
│   ├─ 3PC（三阶段提交）
│   │   └─ 增加CanCommit阶段，减少阻塞
│   └─ XA协议
│       └─ 数据库层面的2PC标准
│
├─【最终一致性】
│   ├─ Saga模式
│   │   ├─ 编排式（Orchestration）
│   │   └─ 编舞式（Choreography）
│   ├─ TCC（Try-Confirm-Cancel）
│   │   └─ 业务层补偿，资源预留
│   └─ 本地消息表 / Outbox
│       └─ 可靠消息投递
│
└─【选择维度】
    ├─ 一致性需求
    ├─ 隔离性需求
    ├─ 业务复杂度
    └─ 性能要求
```

## 二、2PC：两阶段提交

```
2PC协议：

  Phase 1: Prepare
    Coordinator → All Participants: PREPARE

    Each Participant：
      - 执行本地事务（不提交）
      - 锁定资源
      - 若可提交 → 写redo/undo日志 → 回复YES
      - 若不可提交 → 回复NO

  Phase 2: Commit / Abort
    若所有Participant回复YES：
      Coordinator → All: COMMIT
      Participants提交本地事务、释放锁

    若有Participant回复NO：
      Coordinator → All: ABORT
      Participants回滚本地事务、释放锁

问题：
  - Coordinator单点故障 → 参与者可能无限阻塞
  - 同步阻塞 → 性能差
  - 全局锁 → 并发度低

3PC改进：
  增加CanCommit阶段，引入超时机制
  → 减少阻塞，但复杂度增加
```

## 三、Saga模式

```
Saga模式：将长事务拆分为本地事务序列

  正向流程：
    T1 → T2 → T3 → ... → Tn

  补偿流程（若Tk失败）：
    C(k-1) → C(k-2) → ... → C1
    （执行Tk-1到T1的补偿操作）

两种实现：

  1. 编排式Saga（Orchestration）
     中央协调器控制流程
     → 流程集中可见
     → 协调器可能成为瓶颈

  2. 编舞式Saga（Choreography）
     各服务通过事件自主响应
     → 松耦合
     → 流程分散、难追踪

示例（电商订单）：
  T1: 创建订单（本地事务）
  T2: 扣减库存（预留）
  T3: 处理支付
  T4: 创建物流单

  若T3失败：
    C2: 释放库存预留
    C1: 取消订单
```

## 四、TCC：Try-Confirm-Cancel

```
TCC模式：业务层实现的三阶段补偿

  Try阶段：
    - 预留资源（非真正执行）
    - 示例：库存服务预留商品（不扣减）

  Confirm阶段：
    - 真正执行业务
    - 示例：将预留转为实际扣减

  Cancel阶段：
    - 释放预留资源
    - 示例：取消预留，库存恢复

与Saga的区别：
  ┌───────────┬─────────────┬─────────────┐
  │ 维度      │ Saga         │ TCC          │
  ├───────────┼─────────────┼─────────────┤
  │ 补偿时机   │ 失败后补偿    │ Try后Confirm  │
  │ 资源锁定   │ 业务逻辑      │ Try阶段预留   │
  │ 侵入性     │ 低           │ 高（需三接口）│
  │ 适用       │ 通用         │ 资源预留场景  │
  └───────────┴─────────────┴─────────────┘
```

## 五、分布式事务模式对比矩阵

| 维度 | **2PC** | **Saga** | **TCC** | **本地消息表** |
|------|--------|---------|--------|--------------|
| **一致性** | 强一致 | 最终一致 | 最终一致 | 最终一致 |
| **隔离性** | 全局隔离 | 无隔离 | Try阶段隔离 | 无隔离 |
| **性能** | 差（锁+同步） | 好 | 中（需预留） | 好 |
| **复杂度** | 中 | 高（补偿逻辑） | 高（三接口） | 中 |
| **回滚** | 自动 | 手动补偿 | 手动Cancel | 重试+死信 |
| **适用场景** | 短事务、强一致 | 长事务、业务流程 | 资源预留、高并发 | 异步、解耦 |
| **代表框架** | XA, Java JTA | Temporal, Axon | 自研 | RocketMQ, Debezium |

## 六、概念定义与属性关系

| 概念 | 定义 | 属性 | 示例 | 反例 |
|------|------|------|------|------|
| **2PC** | 两阶段提交协议 | 强一致、阻塞、协调者单点 | 数据库XA事务 | 高并发场景 |
| **Saga** | 长事务拆分为本地事务+补偿 | 最终一致、无全局锁、补偿复杂 | 电商订单处理 | 需要强隔离的场景 |
| **TCC** | Try-Confirm-Cancel三阶段 | 业务层预留、高并发友好 | 库存扣减 | 无预留概念的业务 |
| **补偿事务** | 撤销已完成操作的业务逻辑 | 非技术回滚、可能有副作用 | 退款、释放库存 | 数据库ROLLBACK |
| **Outbox** | 将消息与业务数据同事务持久化 | 可靠、至少一次、需CDC | 订单创建+事件发送 | 独立双写 |
