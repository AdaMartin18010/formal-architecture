# 可信计算：从形式验证到运行时可信

> **来源映射**: [09-总览] → 可信计算层次 → 形式验证与运行时可信

> **定位**：可信计算是一个从"证明正确"到"确保运行时行为符合预期"的连续谱。seL4证明了微内核可以形式化验证，TrustZone和SGX提供了硬件级别的隔离——两者代表了不同层次的可信保障。
>
> **核心命题**：形式验证的代码量有限（seL4 ~10K行），而现代系统数百万行代码无法全部验证。可信计算的策略是"验证关键小部分，隔离不可信大部分"。

---

## 一、思维导图：可信计算层次

```text
可信计算层次
│
├─【形式验证（最强保证）】
│   ├─ 数学证明代码符合规范
│   ├─ seL4微内核（~10K行C代码）
│   ├─ CompCert编译器
│   └─ 成本极高、规模受限
│
├─【硬件可信根】
│   ├─ TPM（Trusted Platform Module）
│   ├─ ARM TrustZone
│   ├─ Intel SGX / TDX
│   └─ 硬件保证隔离和认证
│
├─【运行时可信】
│   ├─ 安全启动（Secure Boot）
│   ├─ 远程证明（Remote Attestation）
│   ├─ 运行时监控
│   └─ 行为基线检测
│
└─【供应链安全】
    ├─ SBOM（软件物料清单）
    ├─ 签名验证
    └─ 依赖扫描
```

## 二、形式验证：seL4案例

> **权威来源**：Klein et al., "seL4: Formal Verification of an OS Kernel", *SOSP*, 2009

```
seL4成就：
  - 世界首个形式化验证的OS微内核
  - ~10,000行C代码
  - 使用Isabelle/HOL定理证明器
  - 证明了功能正确性 + 安全属性

验证范围：
  1. 功能正确性：C代码符合抽象规范
  2. 安全性：无未授权访问
  3. 完整性：无未授权修改
  4. 二进制验证：编译后的机器码符合规范

成本：
  - 开发：~25人年
  - 验证：~25人年（1:1比例）
  - 每行代码的验证成本极高

局限：
  - 规模限制（10K行 vs Linux 30M行）
  - 变更成本高（修改需重新验证）
  - 只验证内核，不验证上层应用
```

## 三、硬件可信根

| 技术 | **保证** | **隔离机制** | **代表应用** |
|------|---------|------------|------------|
| **TPM** | 密钥安全存储、平台认证 | 独立安全芯片 | BitLocker, 安全启动 |
| **TrustZone** | 安全世界 vs 正常世界 | CPU状态切换 | 移动支付、DRM |
| **Intel SGX** | 内存加密、远程证明 | Enclave（加密内存区域） | 隐私计算、密钥管理 |
| **AMD SEV** | 虚拟机内存加密 | 内存控制器加密 | 云虚拟机隔离 |
| **Apple Secure Enclave** | 生物特征、密钥安全 | 独立处理器 | Face ID, Apple Pay |

## 四、运行时可信机制

```
安全启动链（Secure Boot Chain）：
  Boot ROM（不可变）→ 验证 Bootloader签名
  Bootloader → 验证 OS签名
  OS → 验证应用签名

  任一验证失败 → 拒绝启动

远程证明（Remote Attestation）：
  1. 挑战者发送随机nonce
  2. 被证明者用TPM/TEE签名nonce + 当前配置哈希
  3. 挑战者验证签名和配置哈希
  4. 仅当配置受信任时建立连接

运行时监控：
  - 行为基线：学习正常行为模式
  - 异常检测：偏离基线时报警
  - 完整性检查：定期校验关键文件
```

## 九、批判性总结

可信计算的"层次化防御"策略——验证关键小部分、隔离不可信大部分——在哲学上是务实的，但在工程实践中面临严峻的规模化挑战。seL4的1:1开发与验证成本比（25人年+25人年）意味着10万行代码的验证需要2500人年，这在商业上几乎不可接受；而Intel SGX等TEE技术虽提供了硬件级隔离，却受限于侧信道攻击（如Foreshadow、Plundervolt）和性能开销（Enclave切换代价）。更为根本的是，"可信"的定义本身具有语境依赖性：形式验证的可信建立在规范正确的假设上，硬件可信根建立在供应链未被渗透的假设上，运行时监控建立在基线行为可学习的假设上——任何一环的失效都会级联破坏整个信任链。与完全形式化相比，运行时可信机制（远程证明、安全启动）提供了可扩展的 weaker guarantee；与纯软件安全相比，硬件隔离又引入了新的攻击面（微架构漏洞）。未来，机密计算（Confidential Computing）和形式化验证编译器（如CompCert）的成熟可能缩小"关键小部分"与"系统整体"之间的验证鸿沟，但"可信"永远是一个概率性概念，而非绝对状态。

## 十一、形式化推理链（深度增强）

```text
公理 1（精化传递公理）—— seL4验证框架:
  精化关系是传递的偏序关系：
    M_E ⊑ M_A ∧ M_C ⊑ M_E ⇒ M_C ⊑ M_A
  即：若C代码精化可执行规约，可执行规约精化抽象规约，
      则C代码精化抽象规约。

公理 2（前向模拟公理）:
  具体机器M_C精化抽象机器M_A，当且仅当存在模拟关系R，
  使得对M_C的每个转移，M_A存在对应转移保持R：
    ∀s,s',σ. R(s,σ) ∧ M_C: s→s' ⇒ ∃σ'. M_A: σ→σ' ∧ R(s',σ')

公理 3（硬件可信根公理）:
  TPM/TEE作为不可变信任锚，提供：
    - 安全密钥存储：密钥不可被软件提取
    - 平台配置寄存器（PCRs）：累积度量启动链
    - 隔离执行：Enclave/TrustZone内存对OS不可见

引理 1（seL4功能正确性引理）—— Klein et al. (2009):
  由公理1-2，seL4的C代码实现精化其抽象规约：
    ⊢_{Isabelle/HOL}  M_C ⊑ M_A
  这意味着：所有抽象规约的安全属性在C代码中保持。

引理 2（seL4导出安全属性引理）:
  由功能正确性（引理1）可导出：
    - 无缓冲区溢出：∀array_access. index ∈ bounds
    - 无空指针解引用：∀ptr_access. ptr ≠ NULL
    - 无内存泄漏：kernel never runs out of memory
    - 无代码注入：∀execution. behavior ∈ specification

引理 3（远程证明验证引理）:
  挑战者验证被证明者的签名和PCR值：
    Verify(Attestation, PK_TPM) = OK ∧ PCRs ∈ TrustedSet
    ⇒ 被证明者运行可信软件配置

定理 1（seL4功能正确性定理）—— Klein et al. (2009):
  seL4微内核的C实现（~10,000行）在Isabelle/HOL中
  被证明精化其高层抽象规约：
    ⊢_{Isabelle}  M_C ⊑ M_E ⊑ M_A
  证明规模：~200,000行证明脚本，25人年验证 + 25人年开发。
  证明技术：前向模拟 + 不变式归纳  ∎

定理 2（信任链完整性定理）:
  安全启动链的完整性由数字签名保证：
    Trust(Boot_ROM) ∧ Verify(Bootloader, PK_ROM) ∧
    Verify(OS, PK_Bootloader) ∧ Verify(App, PK_OS)
    ⇒ Trust(App)
  任一验证失败 ⇒ 信任链断裂，拒绝启动  ∎

定理 3（TEE隔离定理）:
  在硬件正确实现的假设下：
    ∀addr ∈ EnclaveMemory. OS_Access(addr) = DENIED
  即：操作系统无法读取或修改Enclave内存内容  ∎

推论 1（验证成本推论）:
  形式验证成本与代码量呈超线性关系：
    Cost_verify ∝ |TCB|^α,  α > 1
  seL4的1:1开发与验证成本比意味着：
    Linux (~30M行) 的完全验证在经济学上不可行。

推论 2（验证边界推论）:
  形式验证保证受限于假设集合：
    Guarantee = SpecCorrect ∧ CompilerCorrect ∧ HardwareCorrect
  seL4假设编译器（GCC）、硬件（ARMv6）和启动代码正确。
  任何假设失效 ⇒ 保证失效（如侧信道攻击利用硬件微架构）。

推论 3（TCB最小化推论）:
  策略：验证关键小部分（seL4 ~10K行），隔离不可信大部分。
    TCB_total = TCB_verified + TCB_hardware
    Security = f(|TCB_verified|⁻¹, Assurance_level)
  即：验证的TCB越小，可达到的信任度越高。
```

## 十四、批判性总结：形式化视角的深度分析（深度增强）

从形式化推理链审视可信计算，其核心张力在于**定理1（seL4功能正确性定理）代表的"最强保证"与推论1（验证成本推论）代表的"经济不可行性"之间的根本性冲突**。seL4的25人年验证投入对于10,000行代码而言意味着每行代码的验证成本约为0.9人月——以此线性外推，Linux内核30M行代码的完全验证将需要约2,250万人年，相当于当前全球软件工程师总数数十年的工作量。这一计算揭示了形式化验证在工业规模系统中的根本局限：它不是一种可任意扩展的质量保证技术，而是一种**只能在最小化TCB的前提下应用的精密手术工具**。定理2（信任链完整性定理）虽然在逻辑上是优美的，但其可靠性完全依赖于信任根（Boot ROM）的不可变性——而硬件供应链攻击（如恶意植入的芯片级后门）可以直接在信任根层面破坏整个链条，使形式化的数学证明在物理世界中失去意义。定理3（TEE隔离定理）假设硬件实现正确，但Foreshadow、Plundervolt和Spectre等攻击反复证明，Intel SGX的Enclave隔离在微架构层面存在可被利用的缝隙——形式化验证的规格未包含微架构时序行为，因此这些攻击在形式化意义上是"规格之外的"。推论2（验证边界推论）指出了一个更深层的认识论问题：形式化验证将信任从代码转移到规格、编译器和硬件，但**信任不能被消除，只能被重新分配**。运行时可信机制（远程证明、安全启动、行为监控）的价值恰恰在于它们不追求消除信任，而是通过密码学绑定和持续监测将信任维持在一个可检测的状态——当状态偏离预期时，系统可以触发撤销和恢复。未来，将轻量级形式化验证（如Rust的借用检查器）与硬件可信根和运行时监控编织成"分层证据链"，可能是大规模系统可信工程的可行路径：编译期消除内存错误类，硬件期保证隔离边界，运行期检测异常行为——三者互补而非竞争，共同构成一个虽不完美但可量化的可信框架。
