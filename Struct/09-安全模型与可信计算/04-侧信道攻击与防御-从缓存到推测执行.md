# 侧信道攻击与防御：从缓存到推测执行

> **定位**：侧信道攻击是安全的形式化边界之外的威胁——即使密码算法数学上正确，其实现仍可能通过时间、功耗、缓存等物理属性泄露密钥。理解侧信道是安全工程的完整图景所必需的。
>
> **核心命题**：侧信道攻击提醒我们：安全不仅是协议和算法的属性，也是实现的属性。形式化验证的代码也可能有侧信道漏洞。

```text
│

    └─ 硬件缓解
```

## 二、缓存时序攻击

```
Flush+Reload攻击：

  原理：
    1. 攻击者flush（驱逐）共享缓存行
    2. 等待受害者执行（可能访问该缓存行）
    3. 攻击者reload该缓存行，测量访问时间
       - 快 → 受害者访问过（缓存命中）
       - 慢 → 受害者未访问（从内存加载）
    4. 推断受害者的内存访问模式

  影响：
    - 跨VM攻击（同物理机的不同虚拟机）
    - 跨进程攻击（共享库）
    - 密钥恢复（AES、RSA实现）

防御：
  - 禁用共享缓存（性能代价高）
  - 恒定时间实现（无分支依赖密钥）
  - 缓存隔离（Intel CAT）
```

## 三、推测执行攻击：Spectre与Meltdown

```
Spectre（幽灵）：
  原理：利用分支预测 speculative execution 泄漏数据
  
  步骤：
    1. 训练分支预测器采取某路径
    2. 触发越界访问（speculative）
    3. 访问结果被用于加载某缓存行
    4. 通过Flush+Reload测量哪个缓存行被加载
    5. 推断出越界访问的值
  
  变种：
    - Spectre V1: 边界检查绕过
    - Spectre V2: 分支目标注入
    - Spectre V4: 存储绕过

Meltdown（熔断）：
  原理：利用乱序执行绕过权限检查
  
  步骤：
    1. 用户态程序访问内核内存（触发异常）
    2. 乱序执行在异常处理前完成内存访问
    3. 访问结果影响缓存状态
    4. 通过时序分析推断内核数据
  
  影响更严重：可直接读取任意内核内存

缓解：
  - KPTI（内核页表隔离）：Meltdown
  - Retpoline：Spectre V2
  - 微码更新：各类Variant
  - 硬件修复：新CPU设计
```

## 四、恒定时间编程

```c
// 非恒定时间（泄露信息）：
uint8_t secret_data[16];
uint8_t selection = secret_byte;  // 密钥相关
uint8_t result = lookup_table[selection];  // 访问模式依赖密钥

// 恒定时间实现：
uint8_t constant_time_lookup(uint8_t selection) {
    uint8_t result = 0;
    for (int i = 0; i < 256; i++) {
        uint8_t mask = constant_time_eq(i, selection);  // 0或0xFF
        result |= mask & lookup_table[i];
    }
    return result;
}

// 恒定时间比较：
int constant_time_eq(uint8_t a, uint8_t b) {
    return ((uint32_t)(a ^ b) - 1) >> 31;  // 0或0xFFFFFFFF
}

原则：
  - 无分支依赖密钥
  - 无内存访问模式依赖密钥
  - 执行时间恒定（与数据无关）
```

## 五、概念定义与属性关系

| 概念 | 定义 | 属性 | 示例 | 反例 |
|------|------|------|------|------|
| **侧信道** | 通过非预期信息通道泄露数据的攻击 | 物理层、实现相关、难以检测 | 缓存时序攻击 | 直接的密码破解 |
| **推测执行** | CPU预测分支结果并提前执行 | 高性能、有安全风险 | Spectre, Meltdown | 顺序执行 |
| **恒定时间** | 执行时间不依赖敏感数据的编程技术 | 防御时间侧信道、有性能代价 | 恒定时间比较 | 分支依赖密钥 |
| **Flush+Reload** | 通过缓存状态推断内存访问的攻击技术 | 跨进程/VM、高精度 | 恢复AES密钥 | 无共享缓存 |
| **KPTI** | 内核页表隔离（Meltdown缓解） | 有效但有性能代价 | Linux KAISER | 无缓解 |
