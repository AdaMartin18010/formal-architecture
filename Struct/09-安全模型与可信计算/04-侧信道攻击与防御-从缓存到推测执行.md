# 侧信道攻击与防御：从缓存到推测执行

> **来源映射**: [09-总览] → 安全的形式化边界 → 侧信道攻击与防御策略

> **定位**：侧信道攻击是安全的形式化边界之外的威胁——即使密码算法数学上正确，其实现仍可能通过时间、功耗、缓存等物理属性泄露密钥。理解侧信道是安全工程的完整图景所必需的。
>
> **核心命题**：侧信道攻击提醒我们：安全不仅是协议和算法的属性，也是实现的属性。形式化验证的代码也可能有侧信道漏洞。

## 二、缓存时序攻击

```
Flush+Reload攻击：

  原理：
    1. 攻击者flush（驱逐）共享缓存行
    2. 等待受害者执行（可能访问该缓存行）
    3. 攻击者reload该缓存行，测量访问时间
       - 快 → 受害者访问过（缓存命中）
       - 慢 → 受害者未访问（从内存加载）
    4. 推断受害者的内存访问模式

  影响：
    - 跨VM攻击（同物理机的不同虚拟机）
    - 跨进程攻击（共享库）
    - 密钥恢复（AES、RSA实现）

防御：
  - 禁用共享缓存（性能代价高）
  - 恒定时间实现（无分支依赖密钥）
  - 缓存隔离（Intel CAT）
```

## 三、推测执行攻击：Spectre与Meltdown

```
Spectre（幽灵）：
  原理：利用分支预测 speculative execution 泄漏数据

  步骤：
    1. 训练分支预测器采取某路径
    2. 触发越界访问（speculative）
    3. 访问结果被用于加载某缓存行
    4. 通过Flush+Reload测量哪个缓存行被加载
    5. 推断出越界访问的值

  变种：
    - Spectre V1: 边界检查绕过
    - Spectre V2: 分支目标注入
    - Spectre V4: 存储绕过

Meltdown（熔断）：
  原理：利用乱序执行绕过权限检查

  步骤：
    1. 用户态程序访问内核内存（触发异常）
    2. 乱序执行在异常处理前完成内存访问
    3. 访问结果影响缓存状态
    4. 通过时序分析推断内核数据

  影响更严重：可直接读取任意内核内存

缓解：
  - KPTI（内核页表隔离）：Meltdown
  - Retpoline：Spectre V2
  - 微码更新：各类Variant
  - 硬件修复：新CPU设计
```

## 四、恒定时间编程

```c
// 非恒定时间（泄露信息）：
uint8_t secret_data[16];
uint8_t selection = secret_byte;  // 密钥相关
uint8_t result = lookup_table[selection];  // 访问模式依赖密钥

// 恒定时间实现：
uint8_t constant_time_lookup(uint8_t selection) {
    uint8_t result = 0;
    for (int i = 0; i < 256; i++) {
        uint8_t mask = constant_time_eq(i, selection);  // 0或0xFF
        result |= mask & lookup_table[i];
    }
    return result;
}

// 恒定时间比较：
int constant_time_eq(uint8_t a, uint8_t b) {
    return ((uint32_t)(a ^ b) - 1) >> 31;  // 0或0xFFFFFFFF
}

原则：
  - 无分支依赖密钥
  - 无内存访问模式依赖密钥
  - 执行时间恒定（与数据无关）
```

## 九、批判性总结

侧信道攻击的存在揭示了计算机安全最深层的矛盾：数学上正确的算法和形式化验证的协议，仍可能通过物理实现层面的信息泄漏被攻破。这一发现动摇了"形式化验证=安全"的简单等式——seL4微内核虽被形式化验证，但其原始实现仍受Spectre变体影响，因为验证模型未包含微架构时序行为。恒定时间编程作为防御时间侧信道的主要手段，其代价不仅是性能（循环遍历整个查找表而非单次索引），更是工程复杂性——编译器优化可能"好心办坏事"地消除恒定时间构造，需要汇编级审查来确保。与密码分析相比，侧信道攻击的门槛更低（不需要破解算法，只需要测量物理属性）；与传统软件漏洞相比，侧信道又更难修补（往往需要硬件重新设计）。未来，随着机密计算和全同态加密的普及，侧信道的攻击面可能从应用层转移到运行时层；而形式化方法社区正在探索"常数时间验证"——将时序非干涉性纳入类型系统，这或许是从根本上消除时间侧信道的唯一路径。

## 十一、形式化推理链（深度增强）

```text
公理 1（计算非干涉性公理）:
  安全程序应满足时序非干涉性：
    ∀secret₁, secret₂. Time(Program, secret₁) = Time(Program, secret₂)
    ∧ MemoryAccessPattern(Program, secret₁) = MemoryAccessPattern(Program, secret₂)

公理 2（缓存状态泄露公理）:
  若攻击者可测量缓存访问时间，则缓存状态成为信息通道：
    CacheState(victim) → Timing(observable) → InformationLeakage

公理 3（推测执行语义公理）:
  CPU推测执行的结果在错误预测时回滚，但微架构状态（缓存）可能保留：
    if (x < array_len) { y = array[x]; } // speculative
    // 即使 x >= array_len，array[x] 可能已加载到缓存

引理 1（Flush+Reload信息提取引理）:
  攻击者周期性地flush共享缓存行，测量reload时间：
    reload_fast ⇒ 受害者访问过该缓存行
    reload_slow ⇒ 受害者未访问
  通过监控缓存行访问模式，可推断受害者的内存访问序列。

引理 2（Spectre信息泄漏引理）—— Kocher et al. (2019):
  训练分支预测器采取某路径后，触发边界外访问：
    1. 训练：for i in 0..100: access(array[safe_i])  // 训练分支为真
    2. 攻击：access(array[secret_byte * 4096])  // 推测执行
    3. 测量：probe_table[byte * 4096] 的缓存状态
    ⇒ 从缓存状态推断 secret_byte

引理 3（Meltdown权限绕过引理）—— Lipp et al. (2018):
  用户态访问内核内存触发异常，但乱序执行在异常处理前完成：
    1. user_access(kernel_ptr)  // 触发页表权限异常
    2. OoO: byte = *kernel_ptr  // 乱序执行已读取
    3. OoO: probe[byte * 4096]  // 缓存状态被修改
    4. Exception handler: kill process  // 但泄漏已完成
    ⇒ 从缓存状态推断内核数据

定理 1（时序非干涉性破坏定理）:
  若程序存在分支或内存访问依赖敏感数据，
  且攻击者可测量执行时间或缓存状态，
  则程序不满足时序非干涉性。
  证明：构造两个不同的敏感输入s₁≠s₂，
    使程序执行路径或内存访问模式不同，
    则 Time/CacheState 可区分s₁和s₂ ∎

定理 2（恒定时间安全性定理）:
  若程序满足：
    (a) 无分支依赖敏感数据
    (b) 无内存访问地址依赖敏感数据
    (c) 无指令执行时间依赖敏感数据
  则程序满足时序非干涉性（相对于时间侧信道）。
  证明：由(a)(b)(c)，对任意s₁,s₂：
    执行路径相同、内存访问序列相同、指令序列相同
    ⇒ Time(s₁) = Time(s₂) ∎

定理 3（推测执行漏洞普遍性定理）:
  所有支持推测执行和乱序执行的现代CPU，
  在微架构层面均存在信息泄漏通道，
  无论上层软件是否功能正确。
  证明概要：
    推测执行引入的 transient instructions
    影响缓存状态（以及其他微架构状态如TLB、分支预测器），
    而这些状态未被架构级回滚机制清除 ∎

推论 1（形式化验证盲区推论）:
  传统形式化验证（如seL4、RustBelt）的模型不包含微架构时序行为：
    ⊢ FunctionalCorrectness ↮ SideChannelFree
  即：功能正确性证明不排除侧信道漏洞。

推论 2（缓解层次推论）:
  侧信道缓解形成层次：
    Level 1（软件）：恒定时间编程 — 消除时间差异
    Level 2（微码）：Retpoline/微码补丁 — 限制推测执行
    Level 3（OS）：KPTI — 隔离内核/用户页表
    Level 4（硬件）：新CPU设计 — 推测执行时禁止缓存加载
  每层有性能代价，完整防御需多层叠加。

推论 3（恒定时间脆弱性推论）:
  恒定时间编程依赖编译器不优化掉恒定时间构造：
    编译器可能将恒定时间循环优化为条件分支
    ⇒ 恒定时间源代码 ↮ 恒定时间机器码
  需要汇编级审查或编译器支持（如`-mspeculative-load-hardening`）。
```

## 十四、批判性总结：形式化视角的深度分析（深度增强）

从形式化推理链审视侧信道攻击与防御，其核心矛盾在于**定理3（推测执行漏洞普遍性定理）揭示了一个令人不安的计算机科学事实：功能正确性与时序非干涉性在当代处理器架构中是不兼容的**。公理3（推测执行语义公理）指出，CPU为了性能而执行的推测性内存访问会在微架构层面留下持久痕迹（缓存状态），即使架构层面的状态被正确回滚。这意味着，即使我们证明了程序在抽象机器上满足功能正确性和内存安全（如seL4的功能正确性证明、Rust的借用检查），这些证明在真实硬件上对于侧信道威胁是**真空中的真理**。定理1（时序非干涉性破坏定理）进一步表明，只要程序存在任何依赖敏感数据的分支或内存访问，且攻击者能够测量时间或缓存状态，信息泄漏在数学上就是不可避免的——这不是实现缺陷，而是**信息论的基本后果**：不同的秘密输入必然产生不同的计算轨迹，而任何可观测的差异都构成一个信道。定理2（恒定时间安全性定理）提供了软件层面的防御路径，但推论3（恒定时间脆弱性推论）摧毁了这条路径的安全感：恒定时间编程要求编译器将高级语言的恒定时间构造忠实翻译为机器码，但现代编译器的优化器正是以消除"冗余"操作为己任——一个遍历整个查找表的恒定时间循环，在编译器看来正是可被优化为单次条件访问的"低效率代码"。这种"好心办坏事"的优化使得恒定时间编程成为一种与编译器对抗的 arms race。与密码分析相比，侧信道攻击的门槛显著更低——攻击者不需要破解算法的数学结构，只需要测量物理属性；与缓冲区溢出等传统软件漏洞相比，侧信道又更难修补——软件补丁（如KPTI、Retpoline）只能缓解特定变种，而根本解决方案需要重新设计处理器微架构，代价是放弃数十年累积的推测执行性能优势。未来，将时序非干涉性作为类型系统属性（如常量时间类型）纳入编译期验证，可能是从软件层面根本消除时间侧信道的唯一路径——但这要求硬件提供恒定时间原语，且编译器验证链延伸到机器码生成，其工程复杂性远超当前的类型系统。
