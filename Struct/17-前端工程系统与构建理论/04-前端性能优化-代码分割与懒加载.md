## 二、核心概念的形式化定义

### 2.1 代码分割的图论模型

```text
定义 (代码分割):
  给定模块依赖图 G = (V, E, w)，其中 w: V → ℝ⁺ 为模块大小

  分割函数 Partition: V → {1, 2, ..., k}
  将模块映射到 chunk 编号

  约束条件:
    1. 完备性: ∀v ∈ V, Partition(v) 有定义
    2. 依赖闭包: 若 (u, v) ∈ E 且 Partition(u) ≠ Partition(v)，
       则 chunk(Partition(v)) 必须先于 chunk(Partition(u)) 加载
    3. 入口归属: 每个入口点 e ∈ Entry 所在 chunk 为加载根

  优化目标 (最小化期望加载量):
    min Σᵢ₌₁ᵏ [Size(Cᵢ) × P(Cᵢ needed)]

    其中 Size(Cᵢ) = Σ_{v ∈ Cᵢ} w(v)
    P(Cᵢ needed) 为 chunk i 被当前页面需要的概率
```

### 2.2 懒加载的时序模型

```text
定义 (懒加载):
  设页面生命周期为时间轴 T = [0, ∞)

  模块加载时序函数:
    Load: V × T → {0, 1}
    Load(v, t) = 1 表示模块 v 在时刻 t 已加载

  关键路径加载 (Eager):
    ∀v ∈ CriticalPath, Load(v, 0) = 1

  懒加载 (Lazy):
    ∃ trigger(v) ∈ T, Load(v, t) = 1 ⟺ t ≥ trigger(v)

    trigger(v) 的触发条件:
      - 路由导航: URL 匹配正则
      - 用户交互: click / scroll / hover
      - 视口进入: IntersectionObserver
      - 空闲时间: requestIdleCallback
```

### 2.3 预加载的概率模型

```text
定义 (预加载收益):
  设资源 r 的加载延迟为 L(r)，用户下一步需要 r 的概率为 P(r)

  预加载收益:
    Gain(r) = P(r) × L(r) - (1 - P(r)) × Cost(r)

    Cost(r): 预加载 r 消耗的带宽和 CPU

  最优预加载策略:
    选择资源集合 R* ⊆ Resources，使得:
      R* = argmax_{R} Σ_{r ∈ R} Gain(r)
      subject to Σ_{r ∈ R} Size(r) ≤ BandwidthBudget
```

## 三、多维矩阵对比

### 3.1 代码分割策略对比

| 策略 | 触发时机 | 优点 | 缺点 | 适用场景 |
|------|---------|------|------|---------|
| **入口点分割** | 页面加载 | 隔离性强，缓存友好 | 公共代码重复 | MPA、多品牌站点 |
| **动态导入** | 运行时条件 | 精确控制，按需加载 | 加载延迟，需处理 Loading | 条件功能、大组件 |
| **SplitChunks** | 构建时 | 自动提取公共代码 | 配置复杂，可能过度分割 | SPA 通用优化 |
| **组件级分割** | 渲染时 | 细粒度，响应快 | 水合 (Hydration) 延迟 | React/Vue 组件树 |
| **Worker 分割** | 后台计算 | 不阻塞主线程 | 通信开销，无 DOM 访问 | 大数据处理、加密 |

### 3.2 资源提示对比

| 提示类型 | 优先级 | 时机 | 资源类型 | 对 FCP/LCP 影响 |
|----------|--------|------|---------|----------------|
| **preload** | 高 | 当前导航 | CSS、字体、关键 JS | ⬆️ 显著提升 |
| **prefetch** | 低 | 空闲时 | 下一页 JS/CSS | ⬆️ 提升下一页 |
| **preconnect** | - | 尽早 | 第三方域名 | ⬆️ 减少连接延迟 |
| **dns-prefetch** | 低 | 尽早 | 跨域资源域名 | ➡️ 轻微提升 |
| **modulepreload** | 高 | 当前导航 | ESM 模块 | ⬆️ 提升模块解析 |

## 五、工程实践与代码示例

### 5.1 React 路由级代码分割

```javascript
import { lazy, Suspense } from 'react';
import { BrowserRouter, Routes, Route } from 'react-router-dom';

const Home = lazy(() => import('./pages/Home'));
const Dashboard = lazy(() => import('./pages/Dashboard'));
const Analytics = lazy(() => import('./pages/Analytics'));

function App() {
  return (
    <BrowserRouter>
      <Suspense fallback={<LoadingSpinner />}>
        <Routes>
          <Route path="/" element={<Home />} />
          <Route path="/dashboard" element={<Dashboard />} />
          <Route path="/analytics" element={<Analytics />} />
        </Routes>
      </Suspense>
    </BrowserRouter>
  );
}
```

### 5.2 Webpack SplitChunks 配置

```javascript
// webpack.config.js
module.exports = {
  optimization: {
    splitChunks: {
      chunks: 'all',
      maxInitialRequests: 25,
      minSize: 20000,
      cacheGroups: {
        vendor: {
          test: /[\\/]node_modules[\\/]/,
          name: 'vendors',
          chunks: 'all',
        },
        common: {
          minChunks: 2,
          priority: -10,
          reuseExistingChunk: true,
        },
      },
    },
  },
};
```

### 5.3 预加载与预连接

```html
<!-- 预加载关键字体 -->
<link rel="preload" href="/fonts/inter.woff2" as="font" type="font/woff2" crossorigin>

<!-- 预连接第三方 API -->
<link rel="preconnect" href="https://api.example.com">
<link rel="dns-prefetch" href="https://cdn.example.com">

<!-- 预加载下一页 -->
<link rel="prefetch" href="/about.chunk.js">

<!-- ESM 模块预加载 -->
<link rel="modulepreload" href="/src/app.js">
```

### 5.4 IntersectionObserver 图片懒加载

```javascript
const imageObserver = new IntersectionObserver((entries) => {
  entries.forEach((entry) => {
    if (entry.isIntersecting) {
      const img = entry.target;
      img.src = img.dataset.src;
      img.classList.remove('lazy');
      imageObserver.unobserve(img);
    }
  });
});

document.querySelectorAll('img.lazy').forEach((img) => imageObserver.observe(img));
```

## 六、批判性总结

代码分割的本质是**时间-空间的权衡**：将单一庞大的 bundle 拆分为多个 chunk，以空间上的分散换取时间上的按需加载。但这种权衡并非总是正向的——过度分割会导致**请求瀑布**（Request Waterfall），每个 chunk 的独立请求引入了额外的 RTT 延迟，在 HTTP/1.1 环境下尤为严重。HTTP/2 的多路复用缓解了这一问题，但 TCP 慢启动和队头阻塞仍使大量小 chunk 的加载效率低于少量大 chunk。最优的分割策略需要在**chunk 数量**（并行度）和**chunk 大小**（请求开销）之间找到帕累托最优，而这个问题本身是一个 NP-hard 的图划分问题。

懒加载的心理学基础是**感知性能优于实际性能**。用户并不测量首字节时间（TTFB），而是感知到内容是否"足够快"地出现。React.lazy 和 Suspense 的边界设计正是利用了这种感知——在异步模块加载期间展示骨架屏或进度指示器，将等待时间转化为"系统正在工作"的积极信号。但懒加载的滥用会导致**交互延迟**（Interaction Delay）：当用户点击按钮触发懒加载组件时，若网络延迟高，用户会经历明显的卡顿。解决方案是**预测性预加载**——基于用户行为模型（如鼠标悬停时预加载点击目标），将懒加载转化为"几乎即时的加载"。

预加载策略（preload/prefetch）的数学本质是一个**在线背包问题**：在带宽预算有限的情况下，选择预加载哪些资源以最大化期望收益。浏览器当前的实现采用简单的启发式策略（如 prefetch 仅在空闲时执行），而非真正的概率优化。未来的构建工具可能集成**用户行为分析**，根据实际导航模式动态调整预加载策略，使资源提示从静态声明进化为**自适应的在线学习系统**。

## 八、形式化推理链

**推理命题 P**: 代码分割与懒加载的组合策略能够最小化首屏期望加载量，同时保证交互时的模块可用性。

**推理链**:

```text
前提 A1 (Addy Osmani, 2016): "PRPL stands for Push, Render, Pre-cache, and Lazy-load.
                               It's a pattern for structuring and serving Progressive Web
                               Apps with an emphasis on performance of application delivery."
前提 A2 (Ilya Grigorik, 2013): "Preload is a declarative fetch, allowing you to force the
                                browser to make a request for a resource without blocking
                                the document's onload event."
前提 A3 (React Team, 2018): "React.lazy and Suspense let you render a dynamic import as a
                             regular component, automatically loading the bundle when it's
                             first rendered."

步骤 1: 由 A1，PRPL 模式将加载问题分解为四个阶段
       设页面访问为随机事件，用户导航路径为马尔可夫链
       Push: 服务端将关键资源与 HTML 一同推送 (HTTP/2 Server Push)
       Render: 仅加载渲染首屏所需的模块子图 G_critical ⊆ G
       Pre-cache: 将预期资源注册到 Service Worker 缓存
       Lazy-load: 非关键模块延迟加载
       ∴ 首屏加载量 = |G_critical| ≪ |G|

步骤 2: 代码分割的形式化优化
       给定 G=(V,E,w)，分割函数 Partition: V→{1,...,k}
       优化目标: min Σᵢ₌₁ᵏ [Size(Cᵢ) × P(Cᵢ needed)]
       约束:
         - 完备性: ∀v∈V, ∃! Cᵢ, v∈Cᵢ
         - 依赖闭包: (u,v)∈E ∧ Partition(u)≠Partition(v)
           ⟹ chunk(v) 必须先于 chunk(u) 加载
       该问题等价于带约束的图划分问题 (Graph Partition with Precedence Constraints)
       对于路由级分割，最优解可按路由入口将 G 分解为不相交子图

步骤 3: 由 A2-A3，资源提示与懒加载的协同
       预加载收益: Gain(r) = P(r) × L(r) - (1-P(r)) × Cost(r)
       其中 P(r) 为用户下一步需要资源 r 的概率
       懒加载将非关键模块的加载延迟到 trigger(v) 时刻:
         Load(v,t) = 1 ⟺ t ≥ trigger(v)
       当预加载概率 P(r) 高且懒加载触发延迟大时，组合策略最优:
         预加载高概率下一页资源 + 懒加载低概率交互组件

步骤 4: 性能度量的形式化关联
       FCP (First Contentful Paint) ⟺ G_critical 的加载完成时间
       LCP (Largest Contentful Paint) ⟺ 最大可见元素的渲染时间
       TTI (Time to Interactive) ⟺ G_critical 解析执行完成 + 主线程空闲
       代码分割通过减小 |G_critical| 改善 FCP/LCP
       懒加载通过推迟非关键模块的执行改善 TTI

结论: 代码分割与懒加载的联合优化是一个"带约束的随机优化问题"。
       最优策略取决于用户行为概率模型 P(Cᵢ needed) 和网络环境参数 L(r)、Cost(r)。
       构建工具 (Webpack/Vite) 的分割算法是这一优化问题的启发式求解器。
```

## 九、推理判定树/决策树

```text
代码分割策略选择决策树
│
├─► Q1: 应用类型?
│   ├─ 多页面应用 (MPA) ──→ 按入口点分割 (每个页面独立 chunk)
│   └─ 单页面应用 (SPA) ──→ 继续 Q2
│
├─► Q2: 路由框架?
│   ├─ React Router ──→ React.lazy + Suspense 按路由分割
│   ├─ Vue Router ──→ defineAsyncComponent 按路由分割
│   └─ 自定义路由 ──→ 动态 import() 手动分割
│
├─► Q3: 第三方库体积?
│   ├─ 大型库 (lodash, moment, charting) ──→ SplitChunks vendor 分离 + 按需加载子模块
│   └─ 小型库 ──→ 继续 Q4
│
├─► Q4: 是否存在高频交互组件?
│   ├─ 是 (Modal, Drawer, Rich Editor) ──→ 组件级懒加载 (React.lazy / defineAsyncComponent)
│   └─ 否 ──→ 继续 Q5
│
├─► Q5: 是否需要 Worker 计算?
│   ├─ 是 ──→ 独立 chunk + new Worker() 动态加载
│   └─ 否 ──→ 默认路由级分割已足够
│
└─► Q6: 资源提示策略?
    ├─ 首屏关键字体/图片 ──→ <link rel="preload">
    ├─ 下一页高概率路由 ──→ <link rel="prefetch">
    ├─ 第三方 API 域名 ──→ <link rel="preconnect">
    └─ 跨域 CDN ──→ <link rel="dns-prefetch">

性能问题诊断决策树
│
├─► 症状: 首屏加载缓慢 (FCP/LCP 高)
│   ├─► Q1: 初始 chunk 大小 > 200KB?
│   │   ├─ 是 ──→ 启用代码分割，提取 vendor 和非关键路由
│   │   └─ 否 ──→ 继续 Q2
│   │
│   ├─► Q2: 是否存在渲染阻塞资源?
│   │   ├─ 是 ──→ 将 CSS 内联关键样式，JS 加 async/defer
│   │   └─ 否 ──→ 继续 Q3
│   │
│   └─► Q3: 服务端响应时间 (TTFB) 高?
│       ├─ 是 ──→ 优化 SSR/边缘缓存/CDN
│       └─ 否 ──→ 检查图片/字体加载，启用 preload
│
├─► 症状: 交互延迟 (TTI/TBT 高)
│   ├─► Q1: 主线程长任务 > 50ms?
│   │   ├─ 是 ──→ 代码分割 + 懒加载非关键逻辑，移至 Worker
│   │   └─ 否 ──→ 继续 Q2
│   │
│   └─► Q2: 是否存在强制同步布局?
│       ├─ 是 ──→ 批量化 DOM 读写，使用 requestAnimationFrame
│       └─ 否 ──→ 检查第三方脚本影响
│
└─► 症状: 懒加载卡顿
    ├─► Q1: 是否使用骨架屏 / Suspense fallback?
    │   ├─ 否 ──→ 添加 loading 状态，避免空白闪烁
    │   └─ 是 ──→ 继续 Q2
    │
    └─► Q2: 网络延迟高且 chunk 大?
        ├─ 是 ──→ 进一步拆分 chunk，或预加载高概率交互组件
        └─ 否 ──→ 检查 chunk 是否存在瀑布请求 (并行加载优化)
```
