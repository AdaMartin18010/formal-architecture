## 二、核心概念的形式化定义

### 2.1 代码分割的图论模型

```text
定义 (代码分割):
  给定模块依赖图 G = (V, E, w)，其中 w: V → ℝ⁺ 为模块大小
  
  分割函数 Partition: V → {1, 2, ..., k}
  将模块映射到 chunk 编号
  
  约束条件:
    1. 完备性: ∀v ∈ V, Partition(v) 有定义
    2. 依赖闭包: 若 (u, v) ∈ E 且 Partition(u) ≠ Partition(v)，
       则 chunk(Partition(v)) 必须先于 chunk(Partition(u)) 加载
    3. 入口归属: 每个入口点 e ∈ Entry 所在 chunk 为加载根

  优化目标 (最小化期望加载量):
    min Σᵢ₌₁ᵏ [Size(Cᵢ) × P(Cᵢ needed)]
    
    其中 Size(Cᵢ) = Σ_{v ∈ Cᵢ} w(v)
    P(Cᵢ needed) 为 chunk i 被当前页面需要的概率
```

### 2.2 懒加载的时序模型

```text
定义 (懒加载):
  设页面生命周期为时间轴 T = [0, ∞)
  
  模块加载时序函数:
    Load: V × T → {0, 1}
    Load(v, t) = 1 表示模块 v 在时刻 t 已加载
    
  关键路径加载 (Eager):
    ∀v ∈ CriticalPath, Load(v, 0) = 1
    
  懒加载 (Lazy):
    ∃ trigger(v) ∈ T, Load(v, t) = 1 ⟺ t ≥ trigger(v)
    
    trigger(v) 的触发条件:
      - 路由导航: URL 匹配正则
      - 用户交互: click / scroll / hover
      - 视口进入: IntersectionObserver
      - 空闲时间: requestIdleCallback
```

### 2.3 预加载的概率模型

```text
定义 (预加载收益):
  设资源 r 的加载延迟为 L(r)，用户下一步需要 r 的概率为 P(r)
  
  预加载收益:
    Gain(r) = P(r) × L(r) - (1 - P(r)) × Cost(r)
    
    Cost(r): 预加载 r 消耗的带宽和 CPU
    
  最优预加载策略:
    选择资源集合 R* ⊆ Resources，使得:
      R* = argmax_{R} Σ_{r ∈ R} Gain(r)
      subject to Σ_{r ∈ R} Size(r) ≤ BandwidthBudget
```

## 三、多维矩阵对比

### 3.1 代码分割策略对比

| 策略 | 触发时机 | 优点 | 缺点 | 适用场景 |
|------|---------|------|------|---------|
| **入口点分割** | 页面加载 | 隔离性强，缓存友好 | 公共代码重复 | MPA、多品牌站点 |
| **动态导入** | 运行时条件 | 精确控制，按需加载 | 加载延迟，需处理 Loading | 条件功能、大组件 |
| **SplitChunks** | 构建时 | 自动提取公共代码 | 配置复杂，可能过度分割 | SPA 通用优化 |
| **组件级分割** | 渲染时 | 细粒度，响应快 | 水合 (Hydration) 延迟 | React/Vue 组件树 |
| **Worker 分割** | 后台计算 | 不阻塞主线程 | 通信开销，无 DOM 访问 | 大数据处理、加密 |

### 3.2 资源提示对比

| 提示类型 | 优先级 | 时机 | 资源类型 | 对 FCP/LCP 影响 |
|----------|--------|------|---------|----------------|
| **preload** | 高 | 当前导航 | CSS、字体、关键 JS | ⬆️ 显著提升 |
| **prefetch** | 低 | 空闲时 | 下一页 JS/CSS | ⬆️ 提升下一页 |
| **preconnect** | - | 尽早 | 第三方域名 | ⬆️ 减少连接延迟 |
| **dns-prefetch** | 低 | 尽早 | 跨域资源域名 | ➡️ 轻微提升 |
| **modulepreload** | 高 | 当前导航 | ESM 模块 | ⬆️ 提升模块解析 |

## 五、工程实践与代码示例

### 5.1 React 路由级代码分割

```javascript
import { lazy, Suspense } from 'react';
import { BrowserRouter, Routes, Route } from 'react-router-dom';

const Home = lazy(() => import('./pages/Home'));
const Dashboard = lazy(() => import('./pages/Dashboard'));
const Analytics = lazy(() => import('./pages/Analytics'));

function App() {
  return (
    <BrowserRouter>
      <Suspense fallback={<LoadingSpinner />}>
        <Routes>
          <Route path="/" element={<Home />} />
          <Route path="/dashboard" element={<Dashboard />} />
          <Route path="/analytics" element={<Analytics />} />
        </Routes>
      </Suspense>
    </BrowserRouter>
  );
}
```

### 5.2 Webpack SplitChunks 配置

```javascript
// webpack.config.js
module.exports = {
  optimization: {
    splitChunks: {
      chunks: 'all',
      maxInitialRequests: 25,
      minSize: 20000,
      cacheGroups: {
        vendor: {
          test: /[\\/]node_modules[\\/]/,
          name: 'vendors',
          chunks: 'all',
        },
        common: {
          minChunks: 2,
          priority: -10,
          reuseExistingChunk: true,
        },
      },
    },
  },
};
```

### 5.3 预加载与预连接

```html
<!-- 预加载关键字体 -->
<link rel="preload" href="/fonts/inter.woff2" as="font" type="font/woff2" crossorigin>

<!-- 预连接第三方 API -->
<link rel="preconnect" href="https://api.example.com">
<link rel="dns-prefetch" href="https://cdn.example.com">

<!-- 预加载下一页 -->
<link rel="prefetch" href="/about.chunk.js">

<!-- ESM 模块预加载 -->
<link rel="modulepreload" href="/src/app.js">
```

### 5.4 IntersectionObserver 图片懒加载

```javascript
const imageObserver = new IntersectionObserver((entries) => {
  entries.forEach((entry) => {
    if (entry.isIntersecting) {
      const img = entry.target;
      img.src = img.dataset.src;
      img.classList.remove('lazy');
      imageObserver.unobserve(img);
    }
  });
});

document.querySelectorAll('img.lazy').forEach((img) => imageObserver.observe(img));
```

## 六、批判性总结

代码分割的本质是**时间-空间的权衡**：将单一庞大的 bundle 拆分为多个 chunk，以空间上的分散换取时间上的按需加载。但这种权衡并非总是正向的——过度分割会导致**请求瀑布**（Request Waterfall），每个 chunk 的独立请求引入了额外的 RTT 延迟，在 HTTP/1.1 环境下尤为严重。HTTP/2 的多路复用缓解了这一问题，但 TCP 慢启动和队头阻塞仍使大量小 chunk 的加载效率低于少量大 chunk。最优的分割策略需要在**chunk 数量**（并行度）和**chunk 大小**（请求开销）之间找到帕累托最优，而这个问题本身是一个 NP-hard 的图划分问题。

懒加载的心理学基础是**感知性能优于实际性能**。用户并不测量首字节时间（TTFB），而是感知到内容是否"足够快"地出现。React.lazy 和 Suspense 的边界设计正是利用了这种感知——在异步模块加载期间展示骨架屏或进度指示器，将等待时间转化为"系统正在工作"的积极信号。但懒加载的滥用会导致**交互延迟**（Interaction Delay）：当用户点击按钮触发懒加载组件时，若网络延迟高，用户会经历明显的卡顿。解决方案是**预测性预加载**——基于用户行为模型（如鼠标悬停时预加载点击目标），将懒加载转化为"几乎即时的加载"。

预加载策略（preload/prefetch）的数学本质是一个**在线背包问题**：在带宽预算有限的情况下，选择预加载哪些资源以最大化期望收益。浏览器当前的实现采用简单的启发式策略（如 prefetch 仅在空闲时执行），而非真正的概率优化。未来的构建工具可能集成**用户行为分析**，根据实际导航模式动态调整预加载策略，使资源提示从静态声明进化为**自适应的在线学习系统**。
