## 二、模块依赖图的图论特征

| 图特征 | 形式化描述 | 工程问题 | 解决方案 |
|--------|-----------|---------|---------|
| **环 (Cycle)** | `∃v₁→v₂→...→v₁` | 循环依赖、初始化顺序不确定 | Madge 检测、重构为 DAG |
| **割点 (Cut)** | 移除后图不连通 | 关键模块单点故障 | 重点测试、代码审查 |
| **桥 (Bridge)** | 移除后连通分量+1 | 唯一依赖路径 | 冗余导入考虑 |
| **强连通分量 (SCC)** | SCC 内部双向可达 | 紧密耦合模块 | 合并或提取公共接口 |
| **拓扑排序** | DAG 的线性扩展 | 构建/打包顺序 | Webpack/Vite 依赖解析 |
| **最小生成树** | 连接所有模块的最小权重子图 | 初始加载路径优化 | Tree Shaking、入口点优化 |
| **图着色** | 相邻节点不同色 | 代码分割、并行加载 | SplitChunks、按路由分块 |

## 三、Tree Shaking 的形式化分析

```text
问题: 给定模块图 M, 标记所有从入口可达的节点 (Live Code)
      删除不可达节点 (Dead Code)

形式化:
  设入口集合 E ⊆ V
  可达集 R = {v ∈ V | ∃e ∈ E, e →* v} (从入口可达)
  死亡代码 D = V \ R

算法 (标记-清除):
  1. 标记: 从 E 开始 BFS/DFS, 标记所有可达节点
  2. 清除: 删除未标记节点及其导入边

副作用分析 (Side Effect):
  问题: 某些模块有全局副作用 (polyfill, CSS inject)
        即使无显式导出, 也不能删除

  形式化: 设副作用函数 side: V → {0, 1}
          若 side(v) = 1, 则 v ∈ R (强制保留)

ES Module 的 pure 标记:
  /*#__PURE__*/ 提示编译器: 此调用无副作用
  形式化: side(v) = 0 (可被 Tree Shaking)

ESM vs CSM:
  ESM: 静态导入, 编译期可确定依赖图
  CSM: 动态 require, 运行时确定, 不可静态分析

  ∴ ESM 的 Tree Shaking 是编译期优化 (AOT)
     CSM 的 Tree Shaking 是运行期近似
```

## 六、批判性总结

构建系统的本质是对模块依赖图的操作。理解图论概念（环检测、拓扑排序、最小生成树）是优化构建产物的理论基础。Tree Shaking 的局限性在于副作用分析的不完备性——编译器无法静态确定所有副作用，这导致"保守保留"策略下的产物膨胀。

## 八、形式化推理链

**推理命题 P**: 构建系统的所有优化问题均可归约为模块依赖图 M=(V,E,w) 的图论操作。

**推理链**:

```text
前提 A1 (Tobias Koppers, 2019): "Webpack's module graph is the central data structure
                                  that everything else revolves around."
前提 A2 (Rich Harris, 2016): "Tree-shaking relies on ES modules' static structure—import
                              and export declarations are static and analyzable at build time."
前提 A3 (Addy Osmani, 2018): "The cost of JavaScript is not just download time. Parsing
                              and compilation on the main thread can be 2-5× the download
                              cost on mobile devices."

步骤 1: 由 A1，Webpack 的 ModuleGraph 和 ChunkGraph 是构建过程的核心数据结构
       ModuleGraph: Map<Module, ModuleGraphModule>，跟踪模块间 Connection
       ChunkGraph: WeakMap<Chunk, ChunkGraphChunk>，跟踪 chunk 归属
       ∴ 任何构建优化都必须表达为对这两个图的操作

步骤 2: 由 A2，Tree Shaking 的精确性取决于依赖图的静态可分析性
       设入口集合 Entry ⊆ V，可达集 R = ⋃_{e∈Entry} DFS_reachable(e)
       死亡代码 D = V \ R
       副作用约束: side(v)=1 ⟹ v ∈ R (强制保留)
       ∴ Tree Shaking = 图可达性分析 + 副作用谓词过滤

步骤 3: 由 A3，性能优化的目标是减少主线程的解析+编译时间
       代码分割将 G 划分为 {C₁, C₂, ..., Cₖ}，使首屏仅加载必要子图
       懒加载将非关键子图延迟到触发条件满足时加载
       ∴ 性能优化 = 图划分问题 + 加载时序调度

步骤 4: 综合 A1-A3，构建系统的图论统一性:
       Tree Shaking   ⟺ 可达子图提取
       Code Splitting ⟺ 图划分 (Graph Partition)
       Lazy Loading   ⟺ 动态子图加载
       Caching        ⟺ 子图哈希冻结
       HMR            ⟺ 增量子图更新

       所有操作共享同一数学对象: M=(V,E,w)

结论: 模块依赖图 M 是构建系统的"通用代数"，所有优化策略都是 M 上的图算法实例。
      理解图论概念（环、SCC、拓扑排序、最小生成树）是掌握构建工具原理的充分必要条件。
```

## 九、推理判定树/决策树

```text
构建系统图论问题诊断决策树
│
├─► 症状: 构建产物体积过大
│   ├─► Q1: 是否启用了 Tree Shaking?
│   │   ├─ 否 ──→ 启用 ES Module 输出，设置 sideEffects: false
│   │   └─ 是 ──→ 继续 Q2
│   │
│   ├─► Q2: 是否存在大量未使用的导出?
│   │   ├─ 是 ──→ 使用 webpack-bundle-analyzer 定位死代码来源
│   │   └─ 否 ──→ 继续 Q3
│   │
│   └─► Q3: 是否包含大量第三方库?
│       ├─ 是 ──→ 启用 SplitChunks (vendor 分离) + 按需加载 (lodash-es)
│       └─ 否 ──→ 检查是否存在重复依赖 (npm dedupe / pnpm 优化)
│
├─► 症状: 构建速度缓慢
│   ├─► Q1: 项目模块数 > 1000?
│   │   ├─ 是 ──→ 迁移到 SWC/esbuild (O(n) 替换 O(n log n))
│   │   └─ 否 ──→ 继续 Q2
│   │
│   ├─► Q2: 是否使用大量 Babel 插件?
│   │   ├─ 是 ──→ 精简插件链，用 SWC 替代 Babel
│   │   └─ 否 ──→ 启用增量构建 / persistent cache
│   │
│   └─► Q3: 是否为 Monorepo?
│       ├─ 是 ──→ 配置 Turborepo/Nx 远程缓存
│       └─ 否 ──→ 优化 resolve.extensions 减少文件查找
│
├─► 症状: 运行时循环依赖错误
│   ├─► Q1: 使用 Madge 检测是否存在环?
│   │   ├─ 是 ──→ 提取公共接口层打破循环 (A→B→C→A 改为 A→I, B→I, C→I)
│   │   └─ 否 ──→ 运行 npx madge --circular src/
│   │
│   └─► Q2: 环是否存在于 SCC 内部?
│       ├─ 是 ──→ 使用 Tarjan 算法分解 SCC，逐个重构
│       └─ 否 ──→ 检查动态 require 导致的运行时环
│
└─► 症状: 缓存频繁失效
    ├─► Q1: 依赖声明是否完整?
    │   ├─ 否 ──→ 补全 dependsOn / inputs 声明
    │   └─ 是 ──→ 继续 Q2
    │
    └─► Q2: 是否存在非确定性构建?
        ├─ 是 ──→ 消除时间戳/随机数等不稳定输入
        └─ 否 ──→ 检查 lockfile 是否被意外修改
```
