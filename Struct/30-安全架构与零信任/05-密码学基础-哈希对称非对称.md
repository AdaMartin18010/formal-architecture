## 一、知识体系思维导图

```text
密码学基础体系
│
├─► 哈希函数 (Hash Functions) — 单向变换
│   ├─ 密码学哈希要求
│   │   ├─ 抗原像 (Preimage Resistance): 给定 h, 难找 m 使 H(m)=h
│   │   ├─ 抗第二原像 (Second Preimage): 给定 m₁, 难找 m₂ 使 H(m₁)=H(m₂)
│   │   └─ 抗碰撞 (Collision Resistance): 难找任意 m₁≠m₂ 使 H(m₁)=H(m₂)
│   ├─ 算法演进
│   │   ├─ MD5: 128-bit, 已完全破解 (秒级碰撞)
│   │   ├─ SHA-1: 160-bit, 已实际碰撞 (SHAttered, 2017)
│   │   ├─ SHA-256: 256-bit, 当前标准
│   │   ├─ SHA-3 (Keccak): 与 SHA-2 不同设计范式
│   │   └─ BLAKE2/BLAKE3: 快速、并行、现代设计
│   ├─ 密码哈希 (Password Hashing)
│   │   ├─ 需求: 慢计算、加盐、内存-hard
│   │   ├─ bcrypt: 基于 Blowfish, 自适应成本因子
│   │   ├─ scrypt: 内存-hard, 抗 GPU/ASIC
│   │   ├─ Argon2: 2015 密码哈希竞赛冠军, 推荐标准
│   │   │   └─ Argon2id (混合模式): 抗侧信道 + 抗 GPU
│   │   └─ PBKDF2: 旧标准, NIST 推荐但计算成本可调有限
│   └─ 消息认证码 (HMAC)
│       └─ HMAC-SHA256: 密钥 + 哈希 = 完整性 + 认证
│
├─► 对称加密 (Symmetric Encryption) — 单密钥
│   ├─ 分组密码 (Block Ciphers)
│   │   ├─ AES (Advanced Encryption Standard)
│   │   │   ├─ 密钥长度: 128/192/256 bit
│   │   │   ├─ 分组大小: 128 bit
│   │   │   ├─ 轮数: 10/12/14
│   │   │   └─ 操作: SubBytes, ShiftRows, MixColumns, AddRoundKey
│   │   ├─ 操作模式
│   │   │   ├─ ECB: 已废弃 (相同明文 → 相同密文)
│   │   │   ├─ CBC: 需随机 IV，串行处理，需填充
│   │   │   ├─ CTR: 计数器模式，并行加密，流式
│   │   │   ├─ GCM: CTR + GMAC，认证加密 (AEAD)
│   │   │   │   └─ 推荐: AES-256-GCM (TLS 1.3 标配)
│   │   │   └─ ChaCha20-Poly1305: 流密码 + MAC，软件实现高效
│   │   └─ 填充 oracle 攻击: CBC 模式实现缺陷导致解密
│   └─ 流密码 (Stream Ciphers)
│       └─ ChaCha20: 20 轮 ARX 设计，软件优化，抗时序攻击
│
└─► 非对称加密 (Asymmetric Cryptography) — 密钥对
    ├─ 整数分解类
    │   ├─ RSA (Rivest-Shamir-Adleman)
    │   │   ├─ 基于大整数分解困难性
    │   │   ├─ 密钥长度: 2048-bit (最低) / 3072-bit / 4096-bit
    │   │   ├─ 操作: 加密、签名、密钥交换
    │   │   └─ 性能: 慢 (比 AES 约慢 100-1000×)
    │   └─ RSA-OAEP: 填充方案，抗选择密文攻击
    ├─ 离散对数类
    │   ├─ Diffie-Hellman (DH): 密钥交换
    │   ├─ DSA: 数字签名
    │   └─ ECDH/ECDSA: 椭圆曲线版本，更短密钥等效安全
    ├─ 椭圆曲线类
    │   ├─ 优势: 256-bit EC ≈ 3072-bit RSA 安全性
    │   ├─ 曲线标准
    │   │   ├─ P-256 (NIST, secp256r1): 广泛支持
    │   │   ├─ P-384: 高安全场景
    │   │   ├─ Curve25519 (X25519): 现代设计，抗侧信道
    │   │   └─ Curve448: 更高安全余量
    │   └─ EdDSA (Ed25519): 确定性签名，无随机数偏差风险
    └─ 数字签名
        ├─ RSA-PSS: 概率签名方案，安全性证明完备
        ├─ ECDSA: 随机数 k 必须唯一，重用 k 导致私钥泄露
        └─ Ed25519: 推荐方案，快速、安全、确定性
```

## 二、核心概念的形式化定义

### 2.1 密码学哈希

```text
定义 (密码学哈希函数):
  H: {0,1}* → {0,1}^n  (任意长度输入 → 固定长度输出)

  安全属性:
    1. 抗原像 (One-Wayness):
       ∀ h ∈ {0,1}^n, Pr[ A(h) → m : H(m)=h ] ≤ negl(n)

    2. 抗弱碰撞 (Second Preimage Resistance):
       ∀ m₁, Pr[ A(m₁) → m₂ : H(m₁)=H(m₂) ∧ m₁≠m₂ ] ≤ negl(n)

    3. 抗强碰撞 (Collision Resistance):
       Pr[ A() → (m₁,m₂) : H(m₁)=H(m₂) ∧ m₁≠m₂ ] ≤ negl(n)

  密码哈希 (Password Hashing) 额外要求:
    4. 慢计算: 单次哈希时间 >> 普通哈希 (bcrypt cost=10: ~100ms)
    5. 加盐: 随机 salt 使相同密码产生不同哈希
    6. 内存-hard: 高内存需求抗 GPU/ASIC 并行

  Argon2 参数:
    memory: 内存使用量 (KB)
    iterations: 时间迭代次数
    parallelism: 并行线程数
    salt: 随机 16+ bytes
```

### 2.2 认证加密 (AEAD)

```text
定义 (Authenticated Encryption with Associated Data):
  算法: (C, T) = AEAD_Encrypt(K, N, P, A)

  输入:
    K: 密钥
    N: Nonce (每个消息唯一，不保密)
    P: 明文 (需加密)
    A: 关联数据 (不加密，但参与认证)

  输出:
    C: 密文
    T: 认证标签

  解密验证:
    P = AEAD_Decrypt(K, N, C, A, T)
    如果验证失败 → ⊥ (拒绝解密，不返回任何明文)

  AES-GCM 约束:
    Nonce 绝不能重用同一密钥!
    Nonce 重用后果: XOR 密文可消去密钥流，恢复明文
    对策: 计数器 Nonce 或 96-bit 随机 Nonce (2^32 消息后风险上升)

  ChaCha20-Poly1305:
    256-bit 密钥, 96-bit Nonce
    软件实现速度优于 AES-GCM (无 AES-NI 时)
    TLS 1.3 强制支持
```

### 2.3 ECDH 密钥交换

```text
定义 (Elliptic Curve Diffie-Hellman):
  公共参数: 椭圆曲线 E, 基点 G, 阶 n

  Alice:
    私钥 a ← random([1, n-1])
    公钥 A = a·G

  Bob:
    私钥 b ← random([1, n-1])
    公钥 B = b·G

  共享密钥:
    Alice 计算: S = a·B = a·(b·G) = ab·G
    Bob 计算:   S = b·A = b·(a·G) = ab·G

  安全性 (CDH 假设):
    给定 (G, A=aG, B=bG), 计算 ab·G 是困难的

  X25519 (Curve25519):
    曲线: y² = x³ + 486662x² + x over GF(2^255 - 19)
    坐标: 仅使用 x 坐标 (Montgomery  ladder)
    特性: 恒定时间实现，抗侧信道攻击

  与 RSA 对比:
    X25519 公钥: 32 bytes
    RSA-2048 公钥: 256 bytes
    同等安全级别下，EC 密钥短 8-12 倍
```

## 五、工程实践与代码示例

### 5.1 Argon2 密码哈希 (Python)

```python
from argon2 import PasswordHasher

# Argon2id 推荐参数 (OWASP 2023)
ph = PasswordHasher(
    time_cost=3,        # 迭代次数
    memory_cost=65536,  # 64 MB
    parallelism=4,      # 4 线程
    hash_len=32,
    salt_len=16
)

# 注册时哈希密码
hash = ph.hash("user_password")
# $argon2id$v=19$m=65536,t=3,p=4$...$

# 登录时验证
ph.verify(hash, "user_password")

# 检查参数是否需要升级
ph.check_needs_rehash(hash)  # 若返回 True，用新参数重新哈希
```

### 5.2 AES-256-GCM 加密 (Node.js)

```javascript
const crypto = require('crypto');

function encrypt(plaintext, key) {
  const iv = crypto.randomBytes(12);  // 96-bit nonce for GCM
  const cipher = crypto.createCipheriv('aes-256-gcm', key, iv);

  const encrypted = Buffer.concat([
    cipher.update(plaintext, 'utf8'),
    cipher.final()
  ]);
  const authTag = cipher.getAuthTag();

  // 返回: iv + authTag + ciphertext
  return Buffer.concat([iv, authTag, encrypted]).toString('base64');
}

function decrypt(ciphertextBase64, key) {
  const data = Buffer.from(ciphertextBase64, 'base64');
  const iv = data.subarray(0, 12);
  const authTag = data.subarray(12, 28);
  const encrypted = data.subarray(28);

  const decipher = crypto.createDecipheriv('aes-256-gcm', key, iv);
  decipher.setAuthTag(authTag);

  return Buffer.concat([
    decipher.update(encrypted),
    decipher.final()
  ]).toString('utf8');
}
```

### 5.3 Ed25519 签名与验证

```python
from cryptography.hazmat.primitives.asymmetric.ed25519 import Ed25519PrivateKey
from cryptography.exceptions import InvalidSignature

# 生成密钥对
private_key = Ed25519PrivateKey.generate()
public_key = private_key.public_key()

# 签名
message = b"important transaction data"
signature = private_key.sign(message)

# 验证
try:
    public_key.verify(signature, message)
    print("Signature valid")
except InvalidSignature:
    print("Signature INVALID")
```

## 六、批判性总结

密码学是计算机科学中最接近**数学真理**的领域之一：其安全性建立在经过严格分析的数学困难问题之上（整数分解、离散对数、椭圆曲线离散对数）。但密码学系统的实际安全性却常常**崩溃于实现层面**：Heartbleed（OpenSSL 缓冲区越界读取）、ROC（RSA 密钥生成时质数共享）、Nonce 重用导致的 GCM 认证失效——这些漏洞与算法本身的数学强度无关，而是**工程实现的人性弱点**。

AES-GCM 的 Nonce 重用灾难是密码学工程教育的经典案例：GCM 的安全性证明**严格依赖每个消息使用唯一 Nonce**。一旦重用，攻击者通过 XOR 两个密文即可消去密钥流，恢复明文并伪造认证标签。2026 年的生产环境中，仍频繁发现随机数生成器种子重复（虚拟机快照克隆、嵌入式设备弱熵源）导致的 GCM Nonce 碰撞。这印证了密码学的一条铁律：**安全证明的条件必须在运行时严格满足，任何假设的违背都可能导致从"可证明安全"到"完全崩溃"的断崖式跌落**。

量子计算对现有非对称密码体系的威胁（Shor 算法可在多项式时间内分解整数和求解离散对数）推动了后量子密码学 (PQC) 的标准化进程。NIST 已于 2024 年发布首批后量子算法标准（CRYSTALS-Kyber 用于密钥封装，CRYSTALS-Dilithium 用于签名），但它们的密钥和签名尺寸（Kyber-768 公钥 1,184 bytes，Dilithium-3 签名 3,293 bytes）远大于传统算法。这意味着 TLS 握手包膨胀、证书链增大、嵌入式设备存储压力——**密码学安全的进化从来不是免费的午餐**。工程团队应在 2026-2030 窗口期内制定"密码敏捷 (Cryptographic Agility)"策略，确保系统能在不重构架构的前提下迁移到新算法。
