## 一、知识体系思维导图

```text
发布策略谱系
│
├─► 蓝绿部署 (Blue-Green Deployment)
│   ├─► 架构
│   │   ├─ 两套完全相同的环境并行运行
│   │   ├─ Blue: 当前生产环境 (活跃)
│   │   ├─ Green: 新版本环境 (待命)
│   │   └─ 负载均衡器: 控制流量切换
│   │
│   ├─► 流程
│   │   1. Green 部署新版本，运行冒烟测试
│   │   2. 负载均衡器将流量从 Blue → Green
│   │   3. 监控关键指标 (错误率、延迟)
│   │   4. 若异常: 秒级回滚 (Green → Blue)
│   │   5. 若正常: Blue 保留为备份，或更新为下一版本
│   │
│   ├─► 优点
│   │   ├─ 回滚即时 (< 1分钟)
│   │   ├─ 零停机发布
│   │   └─ 便于 A/B 测试 (部分流量)
│   │
│   └─► 缺点
│       ├─ 资源成本 2x
│       ├─ 数据库 Schema 变更复杂 (两版本需兼容)
│       └─ 会话状态需外部化
│
├─► 金丝雀发布 (Canary Release)
│   ├─► 架构
│   │   ├─ 单套环境，逐步替换实例
│   │   ├─ 初始: 5% 流量到新版本
│   │   ├─ 渐进: 10% → 25% → 50% → 100%
│   │   └─ 自动回滚: 指标异常时自动切回旧版本
│   │
│   ├─► 流量控制
│   │   ├─ 基于权重: 负载均衡器百分比
│   │   ├─ 基于用户: 按用户ID哈希
│   │   ├─ 基于地域: 特定区域先升级
│   │   └─ 基于特征: 内部员工/付费用户优先
│   │
│   ├─► 监控指标
│   │   ├─ 错误率: 新版本 vs 旧版本对比
│   │   ├─ 延迟: P50/P95/P99 分位数
│   │   ├─ 业务指标: 转化率、点击率
│   │   └─ 资源使用: CPU/内存/网络
│   │
│   └─► 优点
│       ├─ 风险暴露最小化
│       ├─ 真实用户流量验证
│       └─ 资源成本增量小 (1.05x-1.5x)
│
├─► 滚动更新 (Rolling Update)
│   ├─ 逐个替换实例，保持服务可用
│   ├─ Kubernetes 默认策略
│   ├─ 优点: 简单、资源成本低
│   └─ 缺点: 回滚慢、同时存在两个版本
│
└─► 功能开关 (Feature Toggle)
    ├─ 代码部署 ≠ 功能启用
    ├─ 按条件灰度: 用户/地域/百分比
    ├─ 紧急关闭: 无需回滚代码
    └─ 代表: LaunchDarkly, Unleash, 自研开关
```

## 二、核心概念的形式化定义

### 2.1 发布策略的形式化

```text
定义 (发布策略):
  设: 旧版本 V_old, 新版本 V_new, 实例集合 I = {i₁, i₂, ..., iₙ}

  蓝绿部署:
    状态: I_blue ∪ I_green, 其中 |I_blue| = |I_green| = n
    流量函数: traffic(i) = 1 if i ∈ active_set else 0
    切换: active_set = I_blue → I_green (原子操作)
    回滚: active_set = I_green → I_blue (原子操作)

  金丝雀发布:
    状态: I_old ∪ I_new, 其中 |I_old| + |I_new| = n
    流量权重: w_new ∈ [0, 1], w_old = 1 - w_new
    渐进函数: w_new(t) = f(t), 单调递增
    回滚: w_new → 0

  滚动更新:
    状态序列: S₀ → S₁ → ... → Sₙ
    Sₖ: k 个实例为 V_new, n-k 个实例为 V_old
    每次替换: 1 个实例 (maxUnavailable=1)
    回滚: Sₙ → Sₙ₋₁ → ... → S₀ (逐个回退)
```

### 2.2 风险暴露量化

```text
定义 (风险暴露):
  设: 缺陷影响用户数为 U, 缺陷持续时间为 T

  风险暴露 = U × T

  各策略风险暴露对比 (假设 n=100 实例, 缺陷率=1%):

    全量发布: U = 100%, T = 检测时间 (可能 5-30分钟)
              风险 = 1.0 × 30min = 30 用户·分钟

    蓝绿部署: U = 100% (切换后), T ≈ 回滚时间 (< 1分钟)
              风险 = 1.0 × 1min = 1 用户·分钟

    金丝雀(5%): U = 5%, T = 检测时间 (5分钟)
              风险 = 0.05 × 5min = 0.25 用户·分钟

    金丝雀渐进: 若 5% 阶段发现问题，95% 用户不受影响
              总体风险 ≈ 0.25 用户·分钟

  结论: 金丝雀将风险暴露降低约 **100倍**
```

## 五、工程实践

### 5.1 Kubernetes 金丝雀发布 (Argo Rollouts)

```yaml
apiVersion: argoproj.io/v1alpha1
kind: Rollout
spec:
  strategy:
    canary:
      steps:
      - setWeight: 5          # 5% 流量到新版本
      - pause: {duration: 10m} # 观察 10 分钟
      - setWeight: 20
      - pause: {duration: 10m}
      - setWeight: 50
      - pause: {duration: 10m}
      - setWeight: 100

      analysis:               # 自动分析
        templates:
        - templateName: success-rate
        args:
        - name: service-name
          value: my-service

      autoPromotionEnabled: false  # 需手动确认进入下一阶段
```

### 5.2 功能开关实现

```python
# 基于 LaunchDarkly 的功能开关
from ldclient import LDClient

ld_client = LDClient("sdk-key")

def new_feature(user_id):
    # 按用户ID百分比灰度
    if ld_client.variation("new-checkout-flow", {"key": user_id}, False):
        return render_new_checkout()
    else:
        return render_old_checkout()

# 紧急关闭 (无需回滚代码)
def emergency_disable():
    ld_client.variation("new-checkout-flow", {}, False)
```

## 六、批判性总结

金丝雀发布是**风险控制**的数学最优解：它将"全有或全无"的二元风险转化为**可量化、可渐进、可回滚**的连续风险函数。但金丝雀的成功依赖于**精确的监控和自动化的决策**：如果无法准确检测新版本的健康状况（错误率、延迟、业务指标），金丝雀就变成了"缓慢的全量发布"。

蓝绿部署的**2x资源成本**在 Serverless 和容器化时代已大幅降低——Kubernetes 的 ReplicaSet 可以在几分钟内创建完整副本环境。但在数据库 Schema 变更场景中，蓝绿部署面临**不可解的兼容性挑战**：两个版本同时运行时，Schema 必须同时兼容新旧代码，这往往限制了重构的自由度。

功能开关（Feature Toggle）是**发布与部署解耦**的终极形态：代码可以频繁部署到生产环境，但功能通过开关控制逐步启用。但功能开关的诅咒是**技术债务累积**——长期存在的开关增加了代码路径的复杂度，测试矩阵呈指数增长。2026 年的最佳实践是：**每个开关必须有消亡日期**，功能全量上线后 2 周内必须清理开关代码。
