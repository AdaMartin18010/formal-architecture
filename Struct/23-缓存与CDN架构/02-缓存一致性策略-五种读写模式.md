## 一、知识体系思维导图

```text
缓存一致性策略
│
├─► Cache-Aside (旁路缓存 / Lazy Loading)
│   ├─ 读: 先查缓存→miss则查DB→写入缓存
│   ├─ 写: 先写DB→删除缓存
│   ├─ 优点: 简单、缓存仅含热数据
│   ├─ 缺点: 首次访问慢、写后读可能不一致
│   └─ 适用: 读多写少、容忍最终一致
│
├─► Read-Through
│   ├─ 读: 缓存自动从DB加载
│   ├─ 写: 应用写DB→删除缓存
│   ├─ 优点: 应用逻辑简化
│   ├─ 缺点: 缓存 miss 时延迟高、首次访问惩罚
│   └─ 适用: 通用场景
│
├─► Write-Through
│   ├─ 读: 先查缓存
│   ├─ 写: 同时写DB+缓存 (同步)
│   ├─ 优点: 强一致性、无脏读
│   ├─ 缺点: 写延迟增加 (DB+缓存 RTT)、缓存写放大
│   └─ 适用: 读写均衡、强一致要求
│
├─► Write-Behind (Write-Back / 异步写)
│   ├─ 读: 先查缓存
│   ├─ 写: 只写缓存→异步批量刷DB
│   ├─ 优点: 写延迟极低、可批量合并写
│   ├─ 缺点: 数据丢失窗口、缓存与DB不一致
│   └─ 适用: 写密集型、可容忍短暂不一致
│
└─► Write-Around
    ├─ 读: 先查缓存
    ├─ 写: 直接写DB，绕过缓存
    ├─ 优点: 避免缓存污染 (写后立即不读的数据)
    ├─ 缺点: 写后读必 miss
    └─ 适用: 写多极少读 (如日志写入)
```

## 二、核心概念的形式化定义

### 2.1 五种策略的形式化

```text
定义 (缓存策略状态机):
  设: Cache C, Database D, 应用 A

  Cache-Aside:
    read(k):
      v = C.get(k)
      if v is null:
        v = D.get(k)
        C.set(k, v, TTL)
      return v

    write(k, v):
      D.set(k, v)
      C.delete(k)  // 非 C.set，避免并发写覆盖

  Read-Through:
    read(k):
      return C.get(k, loader=λk: D.get(k))  // 缓存自动加载

    write(k, v):
      D.set(k, v)
      C.delete(k)

  Write-Through:
    read(k):
      return C.get(k)  // 假设缓存已有

    write(k, v):
      C.set(k, v)
      D.set(k, v)
      // 原子性挑战: C 成功但 D 失败 → 不一致

  Write-Behind:
    read(k):
      return C.get(k)

    write(k, v):
      C.set(k, v)
      async_flush_to_D(k, v)  // 异步刷盘

  Write-Around:
    read(k):
      return C.get(k, loader=λk: D.get(k))

    write(k, v):
      D.set(k, v)
      // 不操作缓存
```

### 2.2 一致性分析

```text
定义 (一致性强度):
  强一致性: read(k) 始终返回最近一次 write(k) 的结果
  最终一致性: 若不再写入，最终 read 返回最新结果

  策略一致性排序:
    Write-Through > Read-Through = Cache-Aside > Write-Behind > Write-Around

  Cache-Aside 的写后读窗口:
    A: write(k, v₁) → D.set(k, v₁) → C.delete(k)
    B: read(k) → C.miss → D.get(k) → C.set(k, v₁)
    A: write(k, v₂) → D.set(k, v₂) → C.delete(k)
    B: read(k) → C.hit → return v₁ (脏读！)

    条件: A 的 delete 与 B 的 set 发生竞态
    概率: 低但非零，在写密集型场景显著
```

| 维度 | Cache-Aside | Read-Through | Write-Through | Write-Behind | Write-Around |
|------|------------|--------------|---------------|-------------|-------------|
| **读延迟** | 低 (缓存命中) | 低 | 低 | 低 | 低 |

| **一致性** | 最终一致 | 最终一致 | **强一致** | 弱一致 | 最终一致 |

| **缓存污染** | 中 | 中 | 高 | 高 | **最低** |
| **复杂度** | **低** | 中 | 中 | **高** | 低 |
| **适用场景** | **通用** | 通用 | 读写均衡 | 写密集 | 写多极少读 |

## 五、工程实践

### 5.1 Cache-Aside + 延迟双删

```python
# 解决写后读竞态: 延迟双删策略
class CacheAside:
    def read(self, key):
        value = cache.get(key)
        if value is None:
            value = db.get(key)
            cache.set(key, value, ttl=3600)
        return value

    def write(self, key, value):
        db.set(key, value)
        cache.delete(key)
        # 延迟双删: 在旧数据可能写入缓存的时间窗口后再次删除
        async_delayed_delete(key, delay_ms=500)
```

### 5.2 Write-Through 的事务保障

```python
# 使用分布式事务或 Saga 模式保障一致性
class WriteThrough:
    def write(self, key, value):
        try:
            cache.set(key, value)  # 先写缓存
            db.set(key, value)     # 再写DB
        except DBError:
            cache.delete(key)      # DB失败则回删缓存
            raise
        except CacheError:
            db.rollback()          # 缓存失败则回滚DB
            raise
```

## 六、批判性总结

Cache-Aside 是 2026 年最广泛使用的缓存策略，但**"先写DB后删缓存"存在固有竞态窗口**：当写操作删除缓存后，一个并发的读操作可能从DB读取旧值并回填缓存，导致脏数据持续存在。延迟双删（在数百毫秒后再次删除缓存）可以缩小这个窗口，但**无法完全消除**——这是分布式系统中"不存在完美一致性"的又一个例证。

Write-Through 提供强一致性，但引入了**写放大**：每次写操作需要同步更新两个存储系统，延迟增加且故障面扩大。在微服务架构中，缓存和数据库往往由不同团队运维，Write-Through 的跨系统事务协调复杂度呈超线性增长。

工程界的务实选择是：**读用 Cache-Aside + 写用延迟双删 + 短 TTL 兜底**。短 TTL（如 5-10 分钟）作为"最终一致性加速器"，确保即使发生竞态脏读，数据也会在 TTL 后自动失效。这是**概率一致性**的工程智慧——不追求数学完美，追求故障影响可控。

## 八、深度增强：形式化推理链

### 8.1 公理

**公理 A1（并发读写公理）**
在并发环境下，若没有分布式锁，读写操作的重叠必然导致一致窗口：
exists t: write(t) 与 read(t) 重叠 => consistent(t) = false

**公理 A2（网络分区公理）** [Brewer, 2000]
分布式缓存无法同时满足强一致性、可用性和分区容错性。

**公理 A3（延迟-一致性单调性）**
写延迟与一致性强度正相关：
latency(write) ~ C(strategy)

### 8.2 引理

**引理 L1（Cache-Aside 脏读条件）**
设 W: write(k,v2); D: delete(k); R: read(k)
若执行序为 W -> D -> R_miss -> DB_read(v2) -> R_set(v2) -> W'(v1) -> D'
则后续读可能返回 v2 而非 v1（脏读）。

**引理 L2（Write-Through 原子性缺口）**
即使先写缓存后写 DB，若 DB 写入失败，缓存回删操作存在窗口：
window = time(cache_set) -> time(db_fail_detect) -> time(cache_delete)

### 8.3 定理

**定理 T1（延迟双删收敛）**
设第一次删除时刻为 t1，第二次延迟删除时刻为 t2 = t1 + Delta
则竞态脏读概率从 P_dirty 降至：
P'_dirty = P_dirty * P(T_concurrent_read > Delta)
当 Delta > P99(并发读回填时间) 时，P'_dirty -> 0。

**定理 T2（Write-Behind 丢失上界）**
设崩溃间隔服从指数分布 Exp(lambda)，异步刷盘间隔为 T_flush
则单次崩溃的数据丢失期望：
E[loss] = lambda *T_flush* write_rate / 2

### 8.4 推论

**推论 C1**：在读写比 rho >> 1 时，Cache-Aside 的期望不一致时间：
E[T_inconsistent] = (write_rate^{-1} + read_latency) / rho

**推论 C2**：Write-Through 的写放大系数：
amplification = (T_cache + T_db + T_network) / T_db >= 2

## 十一、批判性总结（深度增强版）

缓存一致性策略的选择本质上是**在延迟、一致性和可用性之间进行帕累托权衡**。Write-Through 的强一致性承诺在理论上成立，但在工程实践中往往因跨系统事务的补偿复杂性而名存实亡——当缓存节点在 DB 写入前崩溃时，回删操作本身又引入了新的竞态条件。这揭示了一个更深层的原理：**分布式系统中的强一致性是一种渐进逼近，而非绝对状态**。

Cache-Aside 之所以成为业界默认选择，并非因为它在理论上最优，而是因为它在故障模式下表现最为**可预测**。当缓存层完全失效时，Cache-Aside 退化为直连数据库（性能降级但功能可用），而 Write-Through 在缓存不可达时可能因事务悬挂导致级联故障。这种降级路径的可预测性，是工程韧性（resilience）的核心。

延迟双删策略常被误认为是银弹，但其数学本质只是一个**概率衰减器**——它降低脏读概率，却无法归零。在高并发写入场景（如秒杀库存扣减），延迟双删的失效窗口与写入频率成正比，此时必须退回到分布式锁或数据库乐观锁。这提示我们：**缓存一致性策略必须分层使用——Cache-Aside 作为默认路径，分布式锁作为关键路径的兜底**。

最终一致性在理论上常被贬低为弱一致性，但从用户感知角度审视，5 秒的最终一致与 5 毫秒的强一致在多数业务场景中并无体验差异。缓存策略的选择应当回归业务本质：**不一致的代价是什么？** 对于社交媒体时间线，最终一致完全可接受；对于银行转账，任何可见的不一致都是致命缺陷。
