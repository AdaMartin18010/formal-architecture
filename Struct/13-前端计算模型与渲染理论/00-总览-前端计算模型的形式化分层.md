## 一、知识体系思维导图

```text
前端计算模型 𝓒 = ⟨Obj(𝓒), Hom(𝓒), ∘, id⟩
│
├─► 运行时模型层 (Runtime Computation Models)
│   ├─► Virtual DOM + Diffing
│   │   ├─ 形式化: UI = f(state) + Reconciliation
│   │   ├─ 复杂度: O(n) 启发式 diff, 内存 O(树规模)
│   │   ├─ 公理 A3 满足度: ⚠️ 启发式近似
│   │   └─ 代表: React (≤18), Vue 2
│   │
│   ├─► Fine-grained Signals
│   │   ├─ 形式化: UI = graph(state) —— 依赖图的增量传播
│   │   ├─ 复杂度: O(1) 靶向更新
│   │   ├─ 公理 A3 满足度: ✅ 严格满足
│   │   └─ 代表: SolidJS, Angular 20 Signals, Vue 3, Preact Signals
│   │
│   └─► 直接 DOM (命令式)
│       ├─ 形式化: UI = imperative(DOM API)
│       ├─ 复杂度: O(1) 直接操作
│       ├─ 公理 A1 满足度: ❌ 状态与 UI 同步无保证
│       └─ 代表: jQuery, 原生 WebGL
│
├─► 编译时模型层 (Compile-time Computation Models)
│   ├─► 编译为命令式 DOM 操作
│   │   ├─ 形式化: compile: AST → IR → DOMOps
│   │   ├─ 代表: Svelte 5 (Runes), Vue Vapor
│   │   └─ 本质: 静态依赖分析的提前求值 (AOT Evaluation)
│   │
│   └─► 优化编译器 (Optimizing Compiler)
│       ├─ 形式化: 自动 memoization 插入
│       ├─ 代表: React Compiler (Forget)
│       └─ 本质: 运行时缺陷的编译期弥补
│
├─► 网络计算模型层 (Network-as-Computation)
│   ├─► SSR + Hydration
│   │   ├─ 形式化: 服务端渲染 → 客户端注水
│   │   └─ 代表: Next.js Pages, Nuxt 2
│   │
│   ├─► Islands Architecture
│   │   ├─ 形式化: 静态 HTML ∈ DOM, JS ⊆ ∅ (默认)
│   │   └─ 代表: Astro, Fresh, 11ty
│   │
│   └─► RSC (React Server Components)
│       ├─ 形式化: 跨网络边界的计算单元
│       ├─ 争议: 破坏 A7 组合封闭性公理 (定理 T2)
│       └─ 代表: Next.js App Router, React 19
│
└─► 并发模型层 (Frontend Concurrency)
    ├─► CSP 映射: Web Workers + postMessage
    ├─► Actor 映射: Service Worker + event.waitUntil
    ├─► STM 映射: useTransition / startTransition
    └─► 数据并行: WebGPU Compute Shader
```

## 二、核心公理体系（Λ_frontend）

```text
┌─────────────────────────────────────────────────────────────────────┐
│            Λ_frontend = ⟨Σ_frontend, Γ_frontend, ⊢⟩                 │
│                                                                     │
│  Σ: {Component, State, DOM, Effect, Render, Network, Signal}      │
│  Γ: 以下 7 条不可约命题                                              │
└─────────────────────────────────────────────────────────────────────┘
```

| 公理 | 名称 | 形式化表述 | 自然语言解释 |
|------|------|-----------|------------|
| **A1** | 状态闭包公理 | `∀s ∈ State, ∃!ui ∈ DOM : ui = f(s)` | 任何状态唯一确定一个 UI 表示 |
| **A2** | 渲染幂等公理 | `∀c ∈ Component, f(f(s)) = f(s)` | 组件渲染函数在相同状态下必须幂等 |
| **A3** | 变更最小化公理 | `Δs → min(|ΔDOM|)` | 状态变更应触发最小 DOM 变更集 |
| **A4** | 边界隔离公理 | `Server ∩ Client = ∅` (计算资源层面) | 服务端与客户端计算资源天然隔离 |
| **A5** | 类型结构公理 | `T₁ ≡ T₂ ⇔ shape(T₁) = shape(T₂)` | TypeScript 结构等价 |
| **A6** | 依赖传递公理 | `s₁ → s₂ ∧ s₂ → s₃ ⇒ s₁ →* s₃` | 状态依赖具有传递闭包性 |
| **A7** | 组合封闭公理 | `∀c₁,c₂ ∈ Component, c₁ ∘ c₂ ∈ Component` | 组件组合必须封闭于组件集合 |

> **关键定理 T1**: Virtual DOM **不是**满足 A3 的必要条件。
> **证明**: Signals 直接满足 A3 且无需 VDOM 中间层。∎

> **关键定理 T2**: RSC 引入的网络边界破坏了 A7 的封闭性。
> **证明**: `c_SC ∘ c_CC` 的语义取决于执行环境，同一组合在不同环境产生不同计算结果。∎

## 三、三种计算范式的范畴论本质

| 范式 | 范畴构造 | 态射定义 | 核心函子 | 自然变换 |
|------|---------|---------|---------|---------|
| **Virtual DOM** | 自由范畴 Free Category | `render: State → VDOM`, `diff: VDOM × VDOM → Patch`, `patch: Patch → DOM` | `F: Set → Tree` | Reconciliation 作为 `F ⇒ G` 的自然变换 |
| **Signals** | 图范畴 Graph Category | `signal: A → B` (依赖边), `effect: ΔA → ΔDOM` | `Γ: State → ReactiveGraph` | 传播算法作为 `Γ` 的保持结构映射 |
| **编译时模型** | 语法范畴 Syntax Category | `compile: AST → IR`, `codegen: IR → DOMOps` | `C: Lang → DOM` | 优化 passes 作为 `C ⇒ C'` 的等价变换 |

> **范畴论洞察**: React 的 `UI = f(state)` 是**态射** `f: State → UI` 的声明；Signals 的依赖图是**图表** `D: J → 𝓒` 的极限（Limit）构造；Svelte 编译器是**语法范畴到语义范畴的函子**。

> **定理**: React 与 Solid 的范畴表示**不同构**。
> **证明**: Tree 是有向无环图且每个节点有唯一父节点；Graph 允许环和多父节点。Tree 是 Graph 的真子范畴，非同构。∎

## 八、形式化推理链

```text
公理 A1 (状态闭包): ∀s ∈ State, ∃!ui ∈ DOM : ui = f(s)
        ↓
引理 L1 (渲染确定性): 相同状态输入下，render 函数输出唯一确定
        ↓
引理 L2 (VDOM 充分性): VDOM 作为中间表示满足 A1，但非最小表示
        ↓
定理 T1 (VDOM 非必要性): VDOM 不是满足 A3(最小变更公理)的必要条件
        ↓
推论 C1 (Signals 严格占优): Signals 直接满足 A3 且无需 VDOM 中间层，
                              在状态变更频率 λ → ∞ 时，复杂度优势趋近 ∞
```

```text
公理 A4 (边界隔离): Server ∩ Client = ∅ (计算资源层面)
        ↓
引理 L3 (网络传输不可逆): 跨网络边界的计算结果必须可序列化
        ↓
定理 T2 (RSC 破坏封闭性): c_SC ∘ c_CC 的语义取决于执行环境，
                           同一组合在不同环境产生不同计算结果
        ↓
推论 C2 (RSC 需粘合层): RSC 不是组件代数 ⟨C, ∘⟩ 上的同态映射，
                         而是两个异构代数系统的 Glue Layer
```

## 十一、深度批判性形式化总结（增强版）

从形式化视角审视前端计算模型的四重分层，本质上是在回答「状态到像素的映射应当发生在何时何地」这一核心问题。Dan Abramov (2019) 将 React 定义为「UI Runtime」而非模板系统，这一洞见揭示了 VDOM 的本质：它不是渲染的优化手段，而是将「计算」从浏览器 DOM API 迁移到 JavaScript 运行时的**语义中间层**。然而，这一迁移的代价在数学上是显见的——`render: State → VDOM` 与 `diff: VDOM × VDOM → Patch` 的组合构成了一个**近似最小化函子** `F_approx: State → ΔDOM`，其启发式假设 H1/H2 导致 `F_approx` 并非严格保距（isometry）。Ryan Carniato (2023) 的批判因此具有形式化分量：VDOM 的 O(n) 复杂度在渐进意义上虽优于朴素 O(n³) 树编辑距离，但在常数因子与内存占用上仍是对 A3（最小变更公理）的**工程妥协**而非**理论满足**。

Signals 与编译时模型的崛起并非简单的技术迭代，而是对「计算发生的时空坐标」的根本重定位。Signals 将计算从「整树重执行」转移到「依赖图上的增量传播」，其形式化优势在于将状态变更的语义从**时间维度** `UI(t) = f(state(t))` 转化为**结构维度** `UI = graph(state)`，从而消除了时间作为隐式参数的复杂性。编译时模型则更进一步，将范畴论中的函子计算 `compile: 𝓢 → 𝓓` 前置到构建阶段，使运行时仅保留纯粹的命令式调度。这两种路径的共同点是**将复杂性从运行时迁移到设计时或编译时**，这与 Aho, Sethi & Ullman (1986) 在 Dragon Book 中提出的「优化是保持语义的程序变换」原则高度一致。

然而，2026 年的技术现实拒绝任何单一范式的霸权。RSC 的引入甚至打破了 A7 的组合封闭性，证明当计算跨越网络边界时，经典代数结构的假设不再成立。前端并发模型的映射实验——Hoare (1978) 的 CSP、Hewitt (1973) 的 Actor、Shavit & Touitou (1995) 的 STM——无一能在浏览器环境中获得形式化 purity，它们都是**受约束的隐喻**而非**严格的实现**。这揭示了一个深层定理：**前端计算模型不是对经典理论的直接应用，而是在浏览器安全沙箱、单线程事件循环与网络延迟三重约束下的工程近似集合**。真正的理论进步不在于证明某一种模型最优，而在于建立「模型选择」本身的形式化判定框架——这正是本模块推理判定树试图达成的初步目标。
