## 一、知识体系思维导图

```text
容器化与编排
│
├─► 容器运行时 (Container Runtime)
│   ├─► OCI 标准
│   │   ├─ runtime-spec: runc, crun, kata-runtime
│   │   └─ image-spec: 分层镜像、rootfs、manifest
│   │
│   ├─► 容器隔离机制
│   │   ├─ Linux Namespace: PID/Network/UTS/IPC/Mount/User/Cgroup
│   │   ├─ Cgroups v1/v2: CPU/内存/IO/网络限制
│   │   └─ Capabilities: 细粒度权限控制
│   │
│   └─► 安全容器
│       ├─ Kata Containers: 轻量VM (VM-level隔离)
│       ├─ gVisor: 用户态内核 (Sentry)
│       └─ Firecracker: AWS microVM
│
├─► 容器镜像
│   ├─► 分层存储: UnionFS / OverlayFS
│   ├─► 镜像构建: Dockerfile → 构建缓存 → 多阶段构建
│   └─► 镜像分发: Registry、内容寻址、签名验证
│
├─► Kubernetes 编排
│   ├─► 核心抽象
│   │   ├─ Pod: 最小调度单元、共享网络/存储
│   │   ├─ Deployment: 无状态应用、滚动更新
│   │   ├─ StatefulSet: 有状态应用、稳定网络标识
│   │   ├─ DaemonSet: 每节点一个副本
│   │   ├─ Job/CronJob: 批处理/定时任务
│   │   └─ Service: 服务发现、负载均衡、ClusterIP
│   │
│   ├─► 调度器
│   │   ├─ 调度流程: Predicates → Priorities → Bind
│   │   ├─ 资源模型: Requests/Limits、QoS (Guaranteed/Burstable/BestEffort)
│   │   └─ 亲和性: NodeAffinity / PodAffinity / AntiAffinity
│   │
│   ├─► 网络模型
│   │   ├─ CNI (Container Network Interface): Calico, Cilium, Flannel
│   │   ├─ Service 实现: kube-proxy (iptables/IPVS/eBPF)
│   │   └─ Ingress / Gateway API: L7 路由
│   │
│   └─► 存储模型
│       ├─ CSI (Container Storage Interface)
│       ├─ PV / PVC: 持久卷声明
│       └─ 存储类: 动态供应、快照、扩容
│
└─► 声明式 vs 命令式
    ├─ 命令式: "如何做" (Imperative)
    └─ 声明式: "做什么" (Declarative) —— Kubernetes 核心哲学
```

## 二、容器隔离的形式化

```text
定义 (Linux 容器):
  Container = ⟨Namespace, Cgroup, RootFS, Capabilities⟩

  Namespace (隔离):
    PID: 进程ID空间隔离
    NET: 网络设备/端口隔离
    MNT: 挂载点隔离
    UTS: 主机名隔离
    IPC: 进程间通信隔离
    USER: UID/GID映射隔离
    CGROUP: cgroup根目录隔离

  Cgroup (限制):
    CPU: shares / quota / period / cpuset
    Memory: limit / swap / reservation
    IO: weight / throttle
    Pids: max进程数
```

## 三、Kubernetes 调度形式化

```text
调度问题:
  输入: Pod P, 节点集合 N = {n₁, n₂, ..., nₖ}
  输出: 最优节点 n* ∈ N

  Predicates (过滤):
    ∀predicate p: p(P, n) → {true, false}
    可行集: F = {n ∈ N | ∀p, p(P, n) = true}

  Priorities (评分):
    ∀priority f: f(P, n) → score ∈ [0, 100]
    最优节点: n* = argmaxₙ∈F Σ wᵢ·fᵢ(P, n)

  常见 Predicates:
    - PodFitsResources: 节点资源足够
    - PodFitsHost: 指定节点名匹配
    - PodFitsHostPorts: 端口不冲突
    - NoDiskConflict: 存储不冲突

  常见 Priorities:
    - LeastRequested: 选择资源使用率低的节点
    - BalancedResourceAllocation: CPU/内存均衡
    - NodeAffinity: 节点亲和性匹配度
```

## 六、批判性总结

Kubernetes 的核心设计哲学是**声明式配置 + 控制循环 (Control Loop)**：用户声明期望状态，系统持续调和实际状态。这与 Imperative 编程形成鲜明对比——前者描述"做什么"，后者描述"怎么做"。

但 Kubernetes 的复杂性已被广泛批评：**YAML 工程**、**操作复杂度**、**学习曲线陡峭**。2026年的趋势是：**平台工程 (Platform Engineering)** 团队构建内部开发者平台 (IDP)，将 Kubernetes 的复杂性抽象为自助服务 API。这不是 Kubernetes 的失败，而是其作为"平台构建平台"定位的成功。
