## 一、知识体系思维导图

```text
消费者组与重平衡协议
│
├─► 消费者组 (Consumer Group)
│   ├─ 语义: 一组消费者协同消费一个 Topic 的所有分区
│   ├─ 约束: 每个分区恰好被组内一个消费者持有
│   └─ 动态性: 成员加入/离开/崩溃触发重平衡
│
├─► 分区分配策略 (Partition Assignment Strategy)
│   ├─ Range Assignor (默认)
│   │   └─ 按 Topic 范围分配，可能导致分配不均
│   ├─ RoundRobin Assignor
│   │   └─ 全局轮询，分配更均匀
│   ├─ Sticky Assignor (KIP-54)
│   │   ├─ 目标1: 尽可能均匀
│   │   ├─ 目标2: 尽可能保留现有分配
│   │   └─ 最小化重平衡后的分区迁移
│   └─ Cooperative Sticky Assignor (KIP-429, Kafka 2.4+)
│       ├─ 增量式重平衡 (Incremental Rebalancing)
│       └─ 避免全局停止消费 (Stop-The-World)
│
└─► 重平衡协议 (Rebalance Protocol)
    ├─ Eager Rebalance Protocol (传统)
    │   ├─ 所有消费者先撤销所有分区
    │   ├─ 重新计算分配方案
    │   └─ 所有消费者重新认领分区
    └─ Cooperative Rebalance Protocol (改进)
        ├─ 仅撤销需要迁移的分区
        ├─ 保留大部分现有分配
        └─ 多轮协商，逐步收敛
```

## 二、核心概念的形式化定义

### 2.1 消费者组约束的形式化

```text
定义 (消费者组 G):
  G = {c₁, c₂, ..., cₘ} 为消费者集合
  Topic T 的分区集合 P = {p₁, p₂, ..., pₙ}

  分配函数:
    assign: G × P → 2^P
    满足:
      (1) 完备性: ⋃ assign(cᵢ) = P   (所有分区被分配)
      (2) 互斥性: assign(cᵢ) ∩ assign(cⱼ) = ∅  (i ≠ j)
      (3) 单消费者约束: |assign(cᵢ)| 任意 (可为 0)
```

### 2.2 重平衡协议的形式化

```text
定义 (重平衡 Rebalance):
  状态转换: S_old = ⟨G_old, assignment_old⟩ → S_new = ⟨G_new, assignment_new⟩

  Eager Rebalance:
    Step 1: ∀c ∈ G_old: revoke(assign(c))     (全部撤销)
    Step 2: assignment_new = compute(G_new, P) (重新计算)
    Step 3: ∀c ∈ G_new: assign(c) = assignment_new[c]  (重新分配)
    性质: 在 Step 1→3 期间，全局消费暂停

  Cooperative Rebalance:
    Step 1: 识别变更集 ΔG = G_new △ G_old
    Step 2: 仅撤销受影响的分区: revoke(affected_partitions)
    Step 3: 重新分配受影响分区，保留未受影响分配
    性质: 非受影响分区持续消费，无全局停顿
```

### 2.3 分配策略对比

```text
Range Assignor:
  对每个 Topic: partitions_per_consumer = n_partitions / n_consumers
  分配不均当分区数不能被消费者数整除

RoundRobin Assignor:
  将所有 Topic 的分区全局排序后轮询分配
  分配均匀，但跨 Topic 分配可能导致订阅不匹配

Sticky Assignor:
  优化目标: maximize(保留现有分配) subject to(均匀约束)
  形式化: min Σ |assign_new(c) △ assign_old(c)|
```

## 三、多维矩阵对比

| 维度 | Range | RoundRobin | Sticky | Cooperative Sticky |
|------|-------|-----------|--------|-------------------|
| **分配均匀性** | 差 (Topic 级别) | 优 (全局) | 优 | 优 |
| **重平衡稳定性** | 差 | 差 | **优** | **最优** |
| **重平衡停顿** | 全局停顿 | 全局停顿 | 全局停顿 | **增量式，无全局停顿** |
| **多 Topic 支持** | 每 Topic 独立 | 全局混合 | 支持 | 支持 |
| **实现复杂度** | 低 | 低 | 中 | 高 |
| **Kafka 版本** | 全部 | 全部 | ≥0.11 | ≥2.4 |
| **适用场景** | 单 Topic | 均匀优先 | 稳定优先 | 高可用优先 |

## 四、权威引用

> **KIP-54: Sticky Partition Assignment Strategy** (Apache Kafka, 2017):
> "The sticky assignor serves two purposes: (1) to spread the partitions as evenly as possible among the consumers, and (2) to preserve as many existing assignments as possible during a rebalance."

> **KIP-429: Kafka Consumer Incremental Rebalance Protocol** (Apache Kafka, 2019):
> "The cooperative rebalancing protocol avoids the 'stop-the-world' problem of eager rebalancing by allowing consumers to keep their current partitions during the rebalance process."

> **Neha Narkhede** (Kafka 联合创始人, Confluent):
> "Consumer rebalances are one of the most complex and error-prone parts of the Kafka client. The cooperative protocol is a significant step towards making rebalances less disruptive."

## 五、工程实践与代码示例

### 配置 Cooperative Sticky Assignor

```java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("group.id", "my-consumer-group");
props.put("partition.assignment.strategy",
          "org.apache.kafka.clients.consumer.CooperativeStickyAssignor");
props.put("session.timeout.ms", 10000);
props.put("heartbeat.interval.ms", 3000);

KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(Arrays.asList("topic1", "topic2"));

// 在 poll 循环中优雅处理再均衡
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        process(record);
    }
    consumer.commitSync(); // 同步提交偏移量
}
```

## 六、批判性总结

消费者组重平衡是 Kafka 客户端中最复杂且最容易出问题的机制之一。Eager Rebalance 的"全局停止消费"特性在消费者数量多、分区数量大的场景下会造成严重的吞吐毛刺（throughput hiccup），甚至触发级联故障——重平衡期间消息堆积 → 消费者恢复后大量消费 → 处理延迟增加 → 更多消费者被判定为超时 → 再次触发重平衡。Cooperative Sticky Assignor 通过增量式重平衡显著缓解了这一问题，但它并非银弹：**多轮协商增加了协议复杂度和完成时间**，在某些极端场景下（如网络分区频繁发生），增量重平衡的收敛速度反而比全局重平衡更慢。此外，消费者组的"单分区单消费者"约束本质上是顺序性与并行性的根本冲突——增加消费者不能超过分区数，这迫使架构师必须在 Topic 设计阶段就预估并发需求，过早的分区数量决策会在后期成为扩展瓶颈。

## 八、形式化推理链

### 8.1 公理体系

**公理 1 (分区单持有公理)** — Kafka Consumer Protocol
> 同一消费者组内，任意分区在任一时刻仅能被一个消费者持有。
> ∀p ∈ P, ∀t: |{c ∈ G | owns(c, p, t)}| ≤ 1

**公理 2 (完备性公理)**
> 消费者组分配必须覆盖其订阅的所有分区。
> ∀p ∈ subscribed_partitions(G): ∃c ∈ G: owns(c, p)

**公理 3 (互斥性公理)**
> 不同消费者持有的分区集合互不相交。
> ∀cᵢ, cⱼ ∈ G, i ≠ j: assigned_partitions(cᵢ) ∩ assigned_partitions(cⱼ) = ∅

### 8.2 引理

**引理 1 (消费者数上界)**
> 消费者组的有效消费者数受限于其订阅Topic的分区总数。
> effective_consumers ≤ Σ |partitions(topic)|
> Proof: 由公理1，每分区单消费者；由公理2，所有分区被分配。故消费者数 ≤ 分区数。

**引理 2 (重平衡期间的消费暂停)**
> Eager Rebalance期间，全局消费暂停。
> Proof: Eager协议要求所有消费者在重新分配前撤销全部分区，故在revoke与re-assign之间无分区可消费。

### 8.3 定理

**定理 1 (Cooperative Rebalance的最优性)** — KIP-429, 2019
> Cooperative Sticky Assignor 在增量重平衡场景下，其稳定性优于Eager Rebalance。
>
> 形式化: 设变更集为Δ，Cooperative的迁移量 = |affected(Δ)|，
> Eager的迁移量 = |all_partitions|。
> 因 |affected(Δ)| ≤ |all_partitions|，故 Cooperative 稳定性 ≥ Eager 稳定性。
>
> 等号成立当且仅当 Δ = G（全部消费者变更）。

**定理 2 (重平衡风暴的收敛条件)**
> 若消费者session超时时间 < 重平衡完成时间，则可能触发级联重平衡。
>
> 条件: session_timeout < T_rebalance ⟹ cascading_rebalance_possible
>
> Proof: 重平衡期间消费者若未及时心跳，被判定为dead，触发新一轮重平衡。
> 若该条件持续满足，则重平衡永不收敛。

### 8.4 推论

**推论 1 (分区预分配刚性)**
> Topic创建时的分区数决策是长期的扩展约束。
> 分区数不足 ⟹ 消费者扩展受限 ⟹ 并行度天花板

**推论 2 (粘性分配的局部最优)**
> Sticky Assignor的稳定性优化可能导致全局分配次优。
> 即: 最小化迁移量 ≠ 最大化均匀性

### 9.1 消费者组分配策略选择决策树

```text
分配策略选择
│
├─► 是否订阅多个Topic？
│   ├─ 是 ──► 是否需要跨Topic均匀分布？
│   │           ├─ 是 ──► RoundRobin / Cooperative Sticky
│   │           │           └─ 全局轮询，跨Topic均匀
│   │           └─ 否（按Topic独立分配）──► Range Assignor

│   │
│   └─ 否（单Topic）──► 分区数是否能被消费者数整除？
│               ├─ 是 ──► Range（简单）或 Sticky（稳定）
│               └─ 否 ──► Sticky / Cooperative Sticky
│                           └─ 避免Range分配不均问题
```

### 9.2 重平衡问题诊断决策树

```text
重平衡问题诊断
│
├─► 重平衡频率是否异常高？
│   ├─ 是 ──► 是否有消费者频繁加入/离开？
│   │           ├─ 是 ──► 检查消费者处理逻辑是否耗时过长

│   └─ 否（重平衡慢）──► 消费者数量是否很大？
│               ├─ 是（>100消费者）──► 使用Cooperative Sticky
│               │                       └─ 避免Eager的全局协调开销
│               └─ 否 ──► 检查Broker端Group Coordinator负载
│                           └─ Coordinator热点 ⟹ 分区迁移到其他Broker
```

## 十一、批判性总结

消费者组重平衡是Kafka客户端协议中最复杂且最容易引发生产故障的机制，其复杂性根源在于它试图在动态环境中维护一个分布式不变式——每个分区恰好被一个消费者持有。Eager Rebalance的"全局停止消费"设计是分布式系统中乐观协议的典型代表：假设重平衡是低频事件，因此全局停顿的代价可接受。然而，这一假设在云原生环境中频繁被违反——容器编排平台（如Kubernetes）的自动扩缩容导致消费者组成员的动态变化成为常态，而非异常。Cooperative Sticky Assignor通过增量式重平衡显著缓解了这一问题，但其多轮协商机制引入了新的收敛性挑战：在网络分区频繁发生的环境中，增量重平衡可能永远无法收敛，因为每轮协商都可能因新成员变化而失效。这种"重平衡风暴"（rebalance storm）的级联效应是分布式系统中正反馈循环的典型案例——重平衡期间消息堆积导致处理延迟增加，处理延迟增加导致消费者心跳超时，心跳超时触发新一轮重平衡，形成恶性循环。从理论角度看，消费者组重平衡本质上是分布式系统中的动态资源分配问题，其最优解是NP-Hard的图划分问题（Graph Partitioning）的在线版本。Sticky Assignor采用的贪心启发式策略（最大化保留现有分配）虽然在实践中表现良好，但缺乏全局最优性保证。更深层的矛盾在于"单分区单消费者"约束与弹性扩展需求之间的结构性冲突：分区数在Topic创建时即固定，而消费负载随时间动态变化，这种静态-动态的不匹配迫使架构师在系统启动时就对未来负载做出预测，而这种预测的失误成本极高。最终，消费者组设计的真正教训是：分布式系统的协调协议必须对成员动态性做出显式假设，并将这些假设编码到协议的收敛性边界中，而不是试图隐藏动态性的存在。
