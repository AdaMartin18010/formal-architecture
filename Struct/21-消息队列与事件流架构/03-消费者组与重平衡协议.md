## 一、知识体系思维导图

```text
消费者组与重平衡协议
│
├─► 消费者组 (Consumer Group)
│   ├─ 语义: 一组消费者协同消费一个 Topic 的所有分区
│   ├─ 约束: 每个分区恰好被组内一个消费者持有
│   └─ 动态性: 成员加入/离开/崩溃触发重平衡
│
├─► 分区分配策略 (Partition Assignment Strategy)
│   ├─ Range Assignor (默认)
│   │   └─ 按 Topic 范围分配，可能导致分配不均
│   ├─ RoundRobin Assignor
│   │   └─ 全局轮询，分配更均匀
│   ├─ Sticky Assignor (KIP-54)
│   │   ├─ 目标1: 尽可能均匀
│   │   ├─ 目标2: 尽可能保留现有分配
│   │   └─ 最小化重平衡后的分区迁移
│   └─ Cooperative Sticky Assignor (KIP-429, Kafka 2.4+)
│       ├─ 增量式重平衡 (Incremental Rebalancing)
│       └─ 避免全局停止消费 (Stop-The-World)
│
└─► 重平衡协议 (Rebalance Protocol)
    ├─ Eager Rebalance Protocol (传统)
    │   ├─ 所有消费者先撤销所有分区
    │   ├─ 重新计算分配方案
    │   └─ 所有消费者重新认领分区
    └─ Cooperative Rebalance Protocol (改进)
        ├─ 仅撤销需要迁移的分区
        ├─ 保留大部分现有分配
        └─ 多轮协商，逐步收敛
```

## 二、核心概念的形式化定义

### 2.1 消费者组约束的形式化

```text
定义 (消费者组 G):
  G = {c₁, c₂, ..., cₘ} 为消费者集合
  Topic T 的分区集合 P = {p₁, p₂, ..., pₙ}

  分配函数:
    assign: G × P → 2^P
    满足:
      (1) 完备性: ⋃ assign(cᵢ) = P   (所有分区被分配)
      (2) 互斥性: assign(cᵢ) ∩ assign(cⱼ) = ∅  (i ≠ j)
      (3) 单消费者约束: |assign(cᵢ)| 任意 (可为 0)
```

### 2.2 重平衡协议的形式化

```text
定义 (重平衡 Rebalance):
  状态转换: S_old = ⟨G_old, assignment_old⟩ → S_new = ⟨G_new, assignment_new⟩

  Eager Rebalance:
    Step 1: ∀c ∈ G_old: revoke(assign(c))     (全部撤销)
    Step 2: assignment_new = compute(G_new, P) (重新计算)
    Step 3: ∀c ∈ G_new: assign(c) = assignment_new[c]  (重新分配)
    性质: 在 Step 1→3 期间，全局消费暂停

  Cooperative Rebalance:
    Step 1: 识别变更集 ΔG = G_new △ G_old
    Step 2: 仅撤销受影响的分区: revoke(affected_partitions)
    Step 3: 重新分配受影响分区，保留未受影响分配
    性质: 非受影响分区持续消费，无全局停顿
```

### 2.3 分配策略对比

```text
Range Assignor:
  对每个 Topic: partitions_per_consumer = n_partitions / n_consumers
  分配不均当分区数不能被消费者数整除

RoundRobin Assignor:
  将所有 Topic 的分区全局排序后轮询分配
  分配均匀，但跨 Topic 分配可能导致订阅不匹配

Sticky Assignor:
  优化目标: maximize(保留现有分配) subject to(均匀约束)
  形式化: min Σ |assign_new(c) △ assign_old(c)|
```

## 三、多维矩阵对比

| 维度 | Range | RoundRobin | Sticky | Cooperative Sticky |
|------|-------|-----------|--------|-------------------|
| **分配均匀性** | 差 (Topic 级别) | 优 (全局) | 优 | 优 |
| **重平衡稳定性** | 差 | 差 | **优** | **最优** |
| **重平衡停顿** | 全局停顿 | 全局停顿 | 全局停顿 | **增量式，无全局停顿** |
| **多 Topic 支持** | 每 Topic 独立 | 全局混合 | 支持 | 支持 |
| **实现复杂度** | 低 | 低 | 中 | 高 |
| **Kafka 版本** | 全部 | 全部 | ≥0.11 | ≥2.4 |
| **适用场景** | 单 Topic | 均匀优先 | 稳定优先 | 高可用优先 |

## 四、权威引用

> **KIP-54: Sticky Partition Assignment Strategy** (Apache Kafka, 2017):
> "The sticky assignor serves two purposes: (1) to spread the partitions as evenly as possible among the consumers, and (2) to preserve as many existing assignments as possible during a rebalance."

> **KIP-429: Kafka Consumer Incremental Rebalance Protocol** (Apache Kafka, 2019):
> "The cooperative rebalancing protocol avoids the 'stop-the-world' problem of eager rebalancing by allowing consumers to keep their current partitions during the rebalance process."

> **Neha Narkhede** (Kafka 联合创始人, Confluent):
> "Consumer rebalances are one of the most complex and error-prone parts of the Kafka client. The cooperative protocol is a significant step towards making rebalances less disruptive."

## 五、工程实践与代码示例

### 配置 Cooperative Sticky Assignor

```java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("group.id", "my-consumer-group");
props.put("partition.assignment.strategy",
          "org.apache.kafka.clients.consumer.CooperativeStickyAssignor");
props.put("session.timeout.ms", 10000);
props.put("heartbeat.interval.ms", 3000);

KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(Arrays.asList("topic1", "topic2"));

// 在 poll 循环中优雅处理再均衡
while (true) {
    ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(100));
    for (ConsumerRecord<String, String> record : records) {
        process(record);
    }
    consumer.commitSync(); // 同步提交偏移量
}
```

## 六、批判性总结

消费者组重平衡是 Kafka 客户端中最复杂且最容易出问题的机制之一。Eager Rebalance 的"全局停止消费"特性在消费者数量多、分区数量大的场景下会造成严重的吞吐毛刺（throughput hiccup），甚至触发级联故障——重平衡期间消息堆积 → 消费者恢复后大量消费 → 处理延迟增加 → 更多消费者被判定为超时 → 再次触发重平衡。Cooperative Sticky Assignor 通过增量式重平衡显著缓解了这一问题，但它并非银弹：**多轮协商增加了协议复杂度和完成时间**，在某些极端场景下（如网络分区频繁发生），增量重平衡的收敛速度反而比全局重平衡更慢。此外，消费者组的"单分区单消费者"约束本质上是顺序性与并行性的根本冲突——增加消费者不能超过分区数，这迫使架构师必须在 Topic 设计阶段就预估并发需求，过早的分区数量决策会在后期成为扩展瓶颈。
