# Bullshark：DAG共识与异步优势

> **定位**：Bullshark代表了共识算法的最新前沿——从链式结构到DAG（有向无环图），从同步假设到部分异步，它在不牺牲安全性的前提下实现了更高的吞吐量和更低的延迟。
>
> **核心命题**：链式共识（HotStuff）序列化处理区块，DAG共识并行化处理——这是从"单线程"到"多线程"的范式转变。

---

## 一、思维导图：DAG共识家族

```text
DAG共识家族
│
├─【核心思想】
│   └─ 区块以DAG结构连接，而非单链
│       → 并行提议、并行确认
│
├─【Narwhal & Tusk】
│   ├─ Narwhal：高吞吐量数据可用性层（DAG传播）
│   └─ Tusk：异步共识层（随机Leader选举）
│
├─【Bullshark】
│   ├─ 结合Narwhal的数据层
│   ├─ 确定性共识（非随机）
│   └─ 部分同步优化
│
├─【DAG结构】
│   ├─ 每个区块引用多个父区块
│   ├─ 因果历史形成DAG
│   └─ 共识决定DAG的"锚点"顺序
│
└─【优势】
    ├─ 吞吐量：100K+ TPS（实验室）
    ├─ 延迟：亚秒级最终性
    └─ 抗网络抖动（异步安全）
```

## 二、DAG vs 链式结构

```
链式结构（HotStuff）：
  B1 → B2 → B3 → B4 → ...

  - 每个区块只有一个父区块
  - Leader串行提议
  - 带宽利用率：Leader瓶颈

DAG结构（Bullshark）：
  B1 ──► B3 ◄── B2
  │       │
  ▼       ▼
  B4 ──► B5 ◄── B6

  - 每个区块可引用多个父区块
  - 多节点并行提议
  - 带宽利用率：所有节点同时利用

关键区别：
  链式：共识和排序耦合（先共识再出块）
  DAG：共识和排序解耦（先出块再共识排序）
```

## 三、Narwhal & Tusk架构

```
两层架构：

  Narwhal（数据可用性层）：
    - 构建交易数据的DAG
    - 保证数据可用性（不保证顺序）
    - 高吞吐量：利用所有节点的带宽

    过程：
      1. 节点收集交易 → 形成工作区块
      2. 区块引用多个父区块（DAG边）
      3. 广播区块（gossip）
      4. DAG自然传播

  Tusk（共识层）：
    - 在DAG上建立共识顺序
    - 随机Leader选举（异步安全）
    - 确定哪些区块被提交及顺序

    过程：
      1. 每轮随机选择Leader
      2. Leader的因果历史形成候选提交集
      3. 投票确认（异步二元共识）
      4. 提交的区块按因果顺序排列
```

## 四、Bullshark优化

> **权威来源**：Spiegelman et al., "Bullshark: DAG BFT Protocols Made Practical", 2022

```
Bullshark = Narwhal数据层 + 确定性共识

核心创新：
  1. 确定性共识（非随机）
     - Tusk使用随机Leader（需要公共随机源）
     - Bullshark使用轮询Leader（确定性）
     - 在同步期：低延迟（类似HotStuff）
     - 在异步期：仍安全（回退到异步模式）

  2. 部分同步优化
     - 检测网络同步状态
     - 同步期：快速路径（2-3轮提交）
     - 异步期：慢速路径（保证安全）

  3. 隐式View-Change
     - 无需显式Leader更换协议
     - 超时后自动切换到下一轮
     - 减少消息复杂度

性能：
  - 实验室：160K TPS（20节点，局域网）
  - 广域网：10-50K TPS
  - 延迟：1-3秒最终性
```

## 五、DAG共识对比矩阵

| 维度 | **链式（HotStuff）** | **DAG（Bullshark）** |
|------|---------------------|---------------------|
| **结构** | 单链 | DAG |
| **提议** | Leader串行 | 多节点并行 |
| **吞吐量** | 中（5-20K TPS） | 高（50-160K TPS） |
| **延迟** | 低（1-2秒） | 中（1-3秒） |
| **异步安全** | 需部分同步 | 是（Narwhal层） |
| **实现复杂度** | 中 | 高（DAG管理） |
| **代表系统** | Diem, Celo | Aptos, Sui（变体） |

## 六、概念定义与属性关系

| 概念 | 定义 | 属性 | 示例 | 反例 |
|------|------|------|------|------|
| **DAG共识** | 基于有向无环图的共识算法 | 并行、高吞吐、解耦 | Bullshark, Tusk | 链式共识（HotStuff） |
| **Narwhal** | 高吞吐量数据可用性层 | 无共识、仅传播、DAG结构 | Aptos使用 | 传统 mempool |
| **数据可用性** | 保证数据可被所有节点获取 | 共识的前提、独立优化 | 纠删码、广播 | 假设可用（不保证） |
| **因果历史** | 区块通过引用边形成的依赖图 | 偏序、可确定并发 | DAG中的祖先节点 | 链式结构的全序 |
| **部分同步** | 网络大部分时间同步但偶尔异步 | 实际假设、工程友好 | PBFT, HotStuff | 完全异步（FLP限制） |
