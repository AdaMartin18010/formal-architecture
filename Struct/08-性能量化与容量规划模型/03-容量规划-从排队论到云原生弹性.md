# 容量规划：从排队论到云原生弹性

> **来源映射**: [08-总览] → 容量规划方法论 → 排队论到云原生弹性

> **定位**：容量规划是从"猜需要多少机器"到"数学建模+自动化弹性"的转变。云原生时代，容量规划不再是静态的"买多少台服务器"，而是动态的"何时扩缩、扩缩多少"。
>
> **核心命题**：自动扩缩容（HPA/VPA）是排队论 Little定律 的实时实现——监控 L（当前负载），调整 c（服务台数量），维持 W（延迟）在目标范围内。

## 一、思维导图：容量规划方法论

```text
容量规划
│
├─【经典方法】
│   ├─ 基于峰值流量 × 冗余系数
│   ├─ 基于历史趋势线性外推
│   └─ 基于业务增长预测
│
├─【排队论方法】
│   ├─ M/M/c模型 → 确定最优服务台数c
│   ├─ Little定律 → L = λW 验证系统健康
│   └─ 目标利用率 ρ_target = λ/(cμ) < 0.7
│
├─【云原生弹性】
│   ├─ HPA（Horizontal Pod Autoscaler）
│   │   └─ 基于CPU/内存/自定义指标扩缩Pod数
│   ├─ VPA（Vertical Pod Autoscaler）
│   │   └─ 调整单个Pod的资源请求/限制
│   ├─ Cluster Autoscaler
│   │   └─ 调整节点数
│   └─ KEDA（基于事件驱动扩缩）
│
└─【关键指标】
    ├─ 饱和度（Saturation）
    ├─ 利用率（Utilization）
    ├─ 错误率（Errors）
    └─ 延迟（Latency）—— USE方法
```

## 二、排队论驱动的容量公式

```
目标：确定服务实例数 c，使得延迟满足SLA

已知：
  λ = 请求到达率（req/s）
  μ = 单个实例服务率（req/s）
  W_SLA = 目标平均延迟（秒）

M/M/c模型：
  利用率 ρ = λ / (cμ)

  平均等待时间（Erlang C公式）：
    Wq = P(wait > 0) × [1 / (cμ - λ)]

    其中 P(wait > 0) = ErlangC(c, λ/μ)

近似设计规则（ρ < 0.7时）：
  c ≈ ⌈λ/μ × safety_factor⌉

  safety_factor = 1 / ρ_target

  若 ρ_target = 0.7 → safety_factor ≈ 1.43

示例：
  λ = 1000 req/s, μ = 100 req/s per instance

  最小实例数 = 1000/100 = 10
  目标实例数（ρ=0.7）= 10 / 0.7 ≈ 15 instances
```

## 三、K8s HPA的数学原理

```
HPA目标：维持度量值 = target值

核心公式：
  desiredReplicas = ceil[currentReplicas × (currentMetricValue / targetMetricValue)]

示例（CPU-based）：
  currentReplicas = 10
  currentCPUUtilization = 80%
  targetCPUUtilization = 50%

  desiredReplicas = ceil[10 × (80/50)] = ceil[16] = 16

稳定性机制：
  - scaleUp stabilizationWindow: 防止抖动（默认0s）
  - scaleDown stabilizationWindow: 防止过早缩容（默认300s）
  - 最大扩缩比例限制

自定义指标（KEDA）：
  基于队列深度扩缩：
    desiredReplicas = ceil[queueLength / targetMessagesPerPod]

  示例：
    queueLength = 1000 messages
    targetMessagesPerPod = 100
    desiredReplicas = ceil[1000/100] = 10
```

| 阶段 | **关键问题** | **工具/方法** |
|------|-----------|-------------|
| **需求分析** | 峰值QPS是多少？延迟SLA？ | 业务预估、历史数据分析 |

## 五、概念定义与属性关系

| 概念 | 定义 | 属性 | 示例 | 反例 |
|------|------|------|------|------|
| **HPA** | K8s水平Pod自动扩缩 | 基于指标、反应式、秒级响应 | CPU>70%→扩容 | 手动扩缩（慢、易出错） |
| **VPA** | K8s垂直Pod自动扩缩 | 调整资源请求、需重启Pod | 内存不足→增加请求 | 固定资源分配 |
| **KEDA** | 基于事件驱动的K8s扩缩 | 支持多种事件源、云无关 | Kafka队列深度驱动扩缩 | 仅CPU/内存驱动 |
| **饱和点** | 系统性能开始非线性恶化的负载点 | 需实验确定、随代码变化 | 单实例>500rps时延迟飙升 | 线性扩展区域 |
| **USE方法** | 利用率、饱和度、错误率三指标 | 简洁、全面、可监控 | 资源监控Dashboard | 仅监控CPU |

## 九、批判性总结

云原生弹性（HPA/VPA/KEDA）将容量规划从"静态预算"转变为"动态反馈控制"，但这一转变引入了新的控制理论问题。HPA的核心公式desiredReplicas = ceil[current × (current/target)]本质上是一个比例控制器（P-controller），它缺乏积分项（消除稳态误差）和微分项（抑制超调），因此在面对阶梯式流量增长时必然产生延迟和振荡。排队论模型告诉我们最优利用率应低于100%，但云原生实践常常为了成本优化将目标利用率设得过高（如70-80%），这在流量突发时几乎没有缓冲余量。与传统容量规划（基于峰值预测+冗余）相比，自动扩缩容降低了过度配置成本但增加了响应延迟风险；与无服务器（完全自动）相比，Kubernetes弹性仍需要人工调参和预热。未来，预测性扩缩容（Predictive Autoscaling）结合时间序列预测和强化学习，有望将反应时间从分钟级缩短到秒级，但其稳定性取决于预测模型的准确性——错误的预测可能比无预测造成更严重的资源振荡。

## 十、形式化定义

```text
排队论容量规划形式化：
  设系统为 M/M/c 队列模型：
    λ = 请求到达率（req/s），服从泊松过程
    μ = 单实例服务率（req/s）
    c = 服务实例数量
    ρ = λ/(cμ) 为系统利用率，稳态要求 ρ < 1

  Erlang C 公式（等待概率）：
    P(W > 0) = [ (cρ)^c / c! ] / [ (cρ)^c / c! + (1-ρ) Σ_{k=0}^{c-1} (cρ)^k / k! ]

  平均等待时间：
    W_q = P(W > 0) / (cμ - λ)

  HPA 扩缩容形式化：
    desiredReplicas = ⌈currentReplicas × (currentMetric / targetMetric)⌉
    约束：minReplicas ≤ desiredReplicas ≤ maxReplicas

  目标利用率设计规则：
    c_target = ⌈λ / (μ × ρ_target)⌉
    典型工程取值：ρ_target ∈ [0.6, 0.8]
```
