## 一、知识体系思维导图

```text
云原生与多云架构
│
├─► 云原生原则 (CNCF Definition)
│   ├─ 容器化打包 (Containerized)
│   ├─ 动态管理 (Dynamically managed)
│   ├─ 面向微服务 (Microservices-oriented)
│   └─ 弹性与可观测性 (Resilient & Observable)
│
├─► 云服务模型
│   ├─► IaaS: EC2, GCE, Azure VM (基础设施)
│   ├─► PaaS: Heroku, Cloud Foundry, App Engine (平台)
│   ├─► SaaS: Gmail, Slack, Salesforce (软件)
│   ├─► FaaS: Lambda, Cloud Functions, Azure Functions (函数)
│   └─► CaaS: ECS, GKE, AKS (容器)
│
├─► 多云架构 (Multi-Cloud)
│   ├─► 策略动机
│   │   ├─ 避免厂商锁定 (Vendor Lock-in)
│   │   ├─ 灾难恢复: 跨云冗余
│   │   ├─ 最优服务选择: 各取所长
│   │   └─ 合规要求: 数据主权 (GDPR/中国法)
│   │
│   ├─► 架构模式
│   │   ├─ 多云冗余: 主动-主动 / 主动-被动
│   │   ├─ 云爆发 (Cloud Bursting): 私有云+公有云弹性
│   │   ├─ 最佳服务: 不同服务用不同云
│   │   └─ 数据分层: 热数据AWS/冷数据Azure
│   │
│   └─► 挑战
│       ├─ 技能碎片: 每个云有不同的API/工具链
│       ├─ 网络成本: 跨云数据传输费用
│       ├─ 安全合规: 统一的IAM/审计
│       └─ 可观测性: 跨云监控统一
│
├─► 混合云 (Hybrid Cloud)
│   ├─ 私有云 + 公有云互联
│   ├─ 代表: AWS Outposts, Azure Stack, Google Anthos
│   └─ 适用: 强合规、低延迟、已有数据中心
│
├─► FinOps (云成本管理)
│   ├─ 云成本可视化: 标签化、分摊、归因
│   ├─ 优化策略: 预留实例、Spot实例、自动伸缩
│   ├─ 单位经济: 每请求成本、每用户成本
│   └─ 文化: 工程团队对成本负责
│
└─► 边缘云 (Edge Computing)
    ├─ 将计算推近数据源/用户
    ├─ 代表: Cloudflare Workers, AWS Lambda@Edge, Vercel Edge
    └─ 适用: 低延迟、带宽节省、数据主权
```

## 二、云服务模型对比

| 模型 | 管理责任 | 用户控制 | 灵活性 | 运维负担 | 代表 |
|------|---------|---------|--------|---------|------|
| **On-Premise** | 用户全管 | 最高 | 最高 | 最重 | 自建机房 |
| **IaaS** | 云管硬件/网络 | 高 | 高 | 重 | EC2, GCE |
| **PaaS** | 云管运行时 | 中 | 中 | 轻 | Heroku, GAE |
| **CaaS** | 云管容器编排 | 中高 | 高 | 中 | GKE, EKS |
| **FaaS** | 云管全部运行时 | 低 | 低 | 最轻 | Lambda |
| **SaaS** | 云管全部 | 最低 | 最低 | 无 | Salesforce |

## 六、批判性总结

多云架构的流行常被包装为"避免厂商锁定"，但真实驱动力往往是**组织政治**（不同团队偏好不同云）和**并购历史**（被收购公司使用不同云）。纯粹的技术理性很少驱动多云决策——因为多云的**网络成本、技能碎片、操作复杂度**往往超过厂商锁定的风险。

FinOps 的兴起标志着云计算进入**成本优化时代**：从"云迁移"到"云成本治理"。这类似于 DevOps 的演进——从"快速发布"到"可靠发布"，再到"成本可控的发布"。
