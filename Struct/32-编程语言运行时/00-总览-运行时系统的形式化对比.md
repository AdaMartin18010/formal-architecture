## 一、知识体系思维导图

```text
编程语言运行时
│
├─► JVM (Java Virtual Machine)
│   ├─► 内存模型
│   │   ├─ 堆: 对象分配、分代GC (G1/ZGC/Shenandoah)
│   │   ├─ 栈: 方法调用、本地变量
│   │   ├─ 元空间: 类元数据 (替代永久代)
│   │   └─ 直接内存: NIO 堆外分配
│   │
│   ├─► 执行引擎
│   │   ├─ 解释器: 逐条执行字节码
│   │   ├─ JIT编译器: C1 (客户端) / C2 (服务端) / Graal
│   │   └─ AOT编译: GraalVM Native Image
│   │
│   ├─► 垃圾回收
│   │   ├─ 标记-清除 (Mark-Sweep)
│   │   ├─ 标记-整理 (Mark-Compact)
│   │   ├─ 复制 (Copying): Eden/Survivor
│   │   └─ 并发GC: CMS → G1 → ZGC (亚毫秒暂停)
│   │
│   └─► 并发模型
│       ├─ 线程: OS 线程 1:1 映射
│       ├─ 锁: synchronized / ReentrantLock / StampedLock
│       └─ 虚拟线程 (Project Loom): 轻量级线程、M:N 映射
│
├─► Go Runtime
│   ├─► Goroutine: 用户态线程、M:N 调度
│   ├─► GMP 模型: G(Goroutine) / M(OS线程) / P(逻辑处理器)
│   ├─► Channel: CSP 通信原语
│   ├─► GC: 三色标记+混合写屏障、并发GC
│   └─► 内存分配: TCMalloc 风格、span 管理
│
├─► V8 (JavaScript Engine)
│   ├─► 编译流水线
│   │   ├─ 解析器: 源码 → AST
│   │   ├─ Ignition: AST → 字节码 (解释执行)
│   │   └─ TurboFan: 热点代码 → 优化机器码 (JIT)
│   │
│   ├─► 内存管理
│   │   ├─ 新生代: Scavenge (复制算法)
│   │   └─ 老生代: Mark-Sweep-Compact
│   │
│   ├─► 隐藏类 (Hidden Class): 对象形状优化
│   └─► 内联缓存 (Inline Caching): 属性访问优化
│
├─► .NET Runtime
│   ├─ CLR: 通用语言运行时
│   ├─ 分层编译: Tiered Compilation
│   ├─ GC: 工作站/服务器模式、后台GC
│   └─ async/await: 状态机编译、Task调度
│
└─► WebAssembly (WASM)
    ├─ 栈式虚拟机、线性内存、沙箱执行
    ├─ 无 GC: 手动内存管理 (Rust/C++)
    ├─ WASI: 系统接口标准 (文件/网络/时钟)
    ├─ WASM Component Model: 跨语言组合
    └─ 2026年: 服务端 WASM (Wasmtime/Wasmer)、边缘计算
```

## 二、运行时对比矩阵

| 维度 | JVM | Go | V8 | .NET | WASM |
|------|-----|----|----|------|------|
| **执行模型** | 字节码+JIT | 机器码+AOT | 字节码+JIT | IL+JIT | 字节码 (AOT/JIT) |
| **内存管理** | 分代GC (ZGC) | 并发GC | 分代GC | 分代GC | 手动管理 |
| **并发模型** | 线程 (虚拟线程) | Goroutine+CSP | 事件循环 | async/await | 无内置 |
| **启动时间** | 慢 (JIT预热) | 快 | 中等 | 中等 | 极快 |
| **峰值性能** | 极高 (C2优化) | 高 | 高 | 高 | 接近原生 |
| **内存占用** | 大 | 小 | 中 | 中 | 极小 |
| **跨平台** | 强 (Write Once) | 强 (静态链接) | 强 (浏览器) | 强 | **最强** (沙箱) |

## 三、垃圾回收算法对比

| 算法 | 暂停时间 | 吞吐量 | 内存碎片 | 适用场景 |
|------|---------|--------|---------|---------|
| **Serial GC** | 长 | 高 | 低 | 单线程客户端 |
| **Parallel GC** | 较长 | 最高 | 低 | 批处理、高吞吐 |
| **CMS** | 短 | 中 | 高 | 低延迟应用 (已废弃) |
| **G1** | 可配置 (<200ms) | 高 | 低 | 通用服务端 |
| **ZGC** | 亚毫秒 (<1ms) | 中 | 低 | **超低延迟** |
| **Shenandoah** | 亚毫秒 | 中 | 低 | 超低延迟 |
| **Go GC** | 微秒-毫秒 | 高 | 低 | 通用服务端 |
| **V8 GC** | 毫秒 | 高 | 低 | 浏览器/Node.js |

## 六、批判性总结

运行时系统的演进遵循**抽象层级不断提升**的规律：从手动内存管理 (C) 到自动 GC (Java/Go/JS) 再到**所有权系统** (Rust 编译时内存安全)。每种模型都有 trade-off：GC 简化开发但引入暂停；手动管理精确控制但容易出错；Rust 的借用检查器在编译期消除两类错误，但学习曲线陡峭。

WebAssembly 是 2026 年最具范式意义的技术：它将**沙箱执行**从浏览器带到服务端，实现了"一次编译，到处运行"的真正跨平台。WASM 的组件模型 (Component Model) 正在解决跨语言组合的问题——这是自 POSIX 以来最重要的可移植性抽象。
