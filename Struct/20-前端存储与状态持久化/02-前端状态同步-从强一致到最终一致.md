## 一、知识体系思维导图

```text
前端状态同步
│
├─► 一致性谱系
│   ├─ 强一致性 (Strong Consistency)
│   │   └─ 所有用户同时看到相同状态
│   ├─ 顺序一致性 (Sequential Consistency)
│   │   └─ 操作按某种全局顺序执行
│   ├─ 因果一致性 (Causal Consistency)
│   │   └─ 因果相关的操作对所有用户可见顺序相同
│   ├─ 会话一致性 (Session Consistency)
│   │   └─ 同一用户的操作在自己的会话中一致
│   └─ 最终一致性 (Eventual Consistency)
│       └─ 若无新更新，最终所有副本一致
│
├─► 同步策略
│   ├─► 乐观更新 (Optimistic Update)
│   │   ├─ 本地立即更新 UI
│   │   ├─ 异步发送请求到服务器
│   │   ├─ 成功: 确认状态
│   │   └─ 失败: 回滚并通知用户
│   │
│   ├─► 悲观更新 (Pessimistic Update)
│   │   ├─ 等待服务器确认后才更新 UI
│   │   └─ 体验差但无回滚风险
│   │
│   └─► 离线优先 (Offline-First)
│       ├─ 本地为唯一可信源
│       ├─ 后台同步到服务器
│       └─ 冲突解决: LWW / 手动合并 / CRDT
│
├─► 状态管理架构
│   ├─ 集中式: Redux / Zustand (单一状态树)
│   ├─ 原子化: Jotai / Recoil (细粒度依赖)
│   ├─ 信号式: Solid / Vue Signals (响应式图)
│   └─ 服务端状态: React Query / SWR / Apollo
│
└─► 冲突解决
    ├─ LWW (Last Write Wins): 时间戳决定
    ├─ 版本向量 (Version Vectors): 检测并发冲突
    ├─ 手动合并: 用户选择保留哪个版本
    └─ CRDT: 自动合并，无冲突
```

## 二、核心概念的形式化定义

### 2.1 一致性模型

```text
定义 (一致性模型):
  设系统有 N 个副本，操作集合 Op = {read, write}

  强一致性:
    ∀readᵢ, writeⱼ: 若 writeⱼ 在 readᵢ 之前完成
    则 readᵢ 必须看到 writeⱼ 的结果

  因果一致性:
    定义 Happens-Before (→):
      - 同一进程: op₁ → op₂ 若 op₁ 在 op₂ 之前执行
      - 跨进程: op₁ → op₂ 若 op₂ 读取了 op₁ 的写入

    要求:
      若 op₁ → op₂，则所有进程看到的 op₁ 都在 op₂ 之前

  最终一致性:
    若系统停止写入:
      lim_{t→∞} stateᵢ(t) = stateⱼ(t)  ∀i,j

  前端场景:
    强一致: 银行转账、库存扣减
    因果一致: 聊天应用、评论回复
    最终一致: 社交点赞、浏览计数
```

### 2.3 向量时钟与 Happens-Before 的形式化

```text
定义 (向量时钟):
  设系统有 N 个副本，每个副本 i 维护向量 VCᵢ[1..N]

  初始: VCᵢ[j] = 0, ∀i,j

  本地事件: VCᵢ[i] += 1

  发送消息: 附带当前 VCᵢ

  接收消息 (来自 j):
    VCᵢ[k] = max(VCᵢ[k], VCⱼ[k]), ∀k
    VCᵢ[i] += 1

  Happens-Before 判定:
    a → b ⟺ VCₐ < VC_b (分量-wise 严格小于)
    a ∥ b (并发) ⟺ ¬(a → b) ∧ ¬(b → a)

  前端应用:
    - 检测乐观更新冲突
    - 实现因果一致性广播 (Causal Broadcast)
    - 版本向量比 Lamport 时间戳更能表达并发关系
```

### 2.4 前端 MVCC 快照隔离

```text
定义 (多版本并发控制):
  设状态 S 有版本链: S₁ → S₂ → ... → Sₙ

  读操作:
    事务 T 在开始时获取 read_timestamp
    读取满足: version_timestamp ≤ read_timestamp 的最新版本

  写操作:
    创建新版本 Sₙ₊₁，附带 write_timestamp

  冲突检测 (Write-Write Conflict):
    若事务 T₁, T₂ 都写入同一键:
      - 先提交者获胜 (First-Committer-Wins)
      - 或后提交者重试

  前端映射:
    React Query / SWR 的缓存即多版本快照
    乐观更新 = 在本地创建未提交的版本
    回滚 = 丢弃本地版本，回退到已提交版本
```

### 2.2 乐观更新与回滚

```text
定义 (乐观更新):
  状态转换:
    S₀ --local_update(o)--> S₁ (乐观状态)
    S₁ --server_confirm(o)--> S₂ (确认状态)
    S₁ --server_reject(o)--> S₀ (回滚)

  条件:
    乐观更新适用 ⟺ 操作可逆 ∧ 冲突概率低

  回滚策略:
    1. 撤销 (Undo): 应用逆操作
    2. 重载 (Reload): 从服务器重新获取状态
    3. 合并 (Merge): 本地修改 + 服务器状态 = 合并结果

  用户体验:
    乐观更新 + 回滚 比 悲观等待 响应更快
    但回滚造成的心理冲击 > 等待的耐心消耗
```

## 四、工程实践

### 5.1 React Query 乐观更新

```javascript
import { useMutation, useQueryClient } from '@tanstack/react-query';

function useUpdateTodo() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: updateTodoAPI,

    // 乐观更新
    onMutate: async (newTodo) => {
      await queryClient.cancelQueries(['todos']);
      const previousTodos = queryClient.getQueryData(['todos']);

      queryClient.setQueryData(['todos'], (old) =>
        old.map((todo) =>
          todo.id === newTodo.id ? { ...todo, ...newTodo } : todo
        )
      );

      return { previousTodos }; // 上下文用于回滚
    },

    // 错误回滚
    onError: (err, newTodo, context) => {
      queryClient.setQueryData(['todos'], context.previousTodos);
    },

    // 完成后重新验证
    onSettled: () => {
      queryClient.invalidateQueries(['todos']);
    },
  });
}
```

### 5.2 版本向量冲突检测

```javascript
// 简化的版本向量
class VersionVector {
  constructor() {
    this.versions = {}; // { nodeId: counter }
  }

  increment(nodeId) {
    this.versions[nodeId] = (this.versions[nodeId] || 0) + 1;
  }

  // 比较两个版本向量
  static compare(vv1, vv2) {
    let greater = false, less = false;
    const allNodes = new Set([...Object.keys(vv1.versions), ...Object.keys(vv2.versions)]);

    for (const node of allNodes) {
      const c1 = vv1.versions[node] || 0;
      const c2 = vv2.versions[node] || 0;
      if (c1 > c2) greater = true;
      if (c1 < c2) less = true;
    }

    if (greater && !less) return 'after';      // vv1 在 vv2 之后
    if (!greater && less) return 'before';     // vv1 在 vv2 之前
    if (!greater && !less) return 'equal';     // 相同
    return 'concurrent';                        // 并发冲突！
  }
}
```

### 5.3 Zustand 持久化与跨标签页同步

```javascript
import { create } from 'zustand';
import { persist, createJSONStorage } from 'zustand/middleware';
import { BroadcastChannel } from 'broadcast-channel';

const useStore = create(
  persist(
    (set) => ({
      count: 0,
      increment: () => set((s) => ({ count: s.count + 1 })),
    }),
    {
      name: 'app-storage',
      storage: createJSONStorage(() => localStorage),
      // 持久化前转换 (如日期序列化)
      serialize: (state) => JSON.stringify(state),
      deserialize: (str) => JSON.parse(str),
    }
  )
);

// 跨标签页同步 (最终一致)
const channel = new BroadcastChannel('app-state-sync');
channel.onmessage = (msg) => {
  useStore.setState(msg.data, false); // false = 不触发持久化循环
};

useStore.subscribe((state) => {
  channel.postMessage({ data: state });
});
```

## 六、批判性总结

前端状态管理的演进从**单一可信源**（Redux）到**服务端状态分离**（React Query）再到**细粒度响应式**（Signals），本质是对**状态所有权**的重新界定。Redux 假设所有状态都归前端管理；React Query 认识到大部分状态属于服务端，前端只是缓存；Signals 发现组件级别的局部状态不需要全局管理。

乐观更新是**用户体验与数据一致性**的经典权衡：它让 UI 在 16ms 内响应用户操作（人类感知的"即时"阈值），但承担了回滚的风险。心理学研究表明，**用户更容忍等待而非背叛**——如果系统承诺了更新又撤销，用户的信任损失远大于等待确认。因此，2026 年的最佳实践是**谨慎乐观**：仅在低冲突、可逆的场景使用乐观更新，高价值操作（支付、删除）必须使用悲观确认。

最终一致性在前端场景中的接受度远超后端——用户可以接受点赞数延迟同步，但无法容忍余额延迟更新。这印证了**一致性需求是业务驱动的，而非技术驱动的**：同样的技术方案，在不同业务场景下可能是"完全可接受"或"完全不可接受"。

向量时钟在前端工程中的落地虽然罕见，但其思想已渗透进主流库的设计：React Query 的 `queryClient.setQueryData` 本质上是**无向量时钟的因果广播**——通过 `invalidateQueries` 传播失效事件，而非完整状态。这在大多数场景足够，但在离线优先应用中，缺乏显式的 Happens-Before 关系会导致**静默数据丢失**。Yjs 等 CRDT 库通过状态向量（State Vector）实现了轻量级的向量时钟变体，在保持因果一致的同时避免了向量时钟的存储开销（O(N²) → O(N)）。这揭示了前端分布式系统的一个核心设计原则：**在正确性与工程可行性之间，前端几乎总是选择后者**。
