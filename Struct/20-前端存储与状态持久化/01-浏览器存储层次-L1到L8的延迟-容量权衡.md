## 一、知识体系思维导图

```text
浏览器存储层次 (L1-L8)
│
├─► L1: CPU Register / L1 Cache
│   ├─ 容量: ~64B - 64KB
│   ├─ 延迟: ~1ns
│   └─ 控制: 硬件管理，程序无直接控制
│
├─► L2: Memory (RAM / JS Heap)
│   ├─ 容量: ~MB - GB
│   ├─ 延迟: ~100ns
│   └─ 控制: 垃圾回收管理
│
├─► L3: JS 运行时缓存 (Memoization)
│   ├─ 容量: ~MB
│   ├─ 延迟: ~μs
│   └─ 控制: 应用代码 (useMemo, LRU Cache)
│
├─► L4: Session Storage
│   ├─ 容量: ~5MB
│   ├─ 延迟: ~ms
│   └─ 生命周期: 标签页关闭即清除
│
├─► L5: Local Storage
│   ├─ 容量: ~5-10MB
│   ├─ 延迟: ~ms
│   └─ 生命周期: 永久，同源共享
│
├─► L6: IndexedDB
│   ├─ 容量: ~50MB+ (可请求更多)
│   ├─ 延迟: ~ms - 10ms
│   └─ 特性: 结构化、事务、索引
│
├─► L7: Cache API / Service Worker Cache
│   ├─ 容量: 取决于磁盘空间
│   ├─ 延迟: ~ms
│   └─ 特性: Request/Response 对、离线支持
│
└─► L8: 网络 / 服务器端存储
    ├─ 容量: 无上限
    ├─ 延迟: ~50-500ms
    └─ 特性: 持久化、共享、权威数据源
```

## 二、核心概念的形式化定义

### 2.1 存储层次的延迟-容量权衡

```text
定义 (存储层次):
  设存储层次 L = {L₁, L₂, ..., L₈}

  每个层级 Lᵢ = ⟨capacityᵢ, latencyᵢ, lifetimeᵢ, controlᵢ⟩

  基本性质:
    1. 容量递增: capacity₁ < capacity₂ < ... < capacity₈
    2. 延迟递增: latency₁ < latency₂ < ... < latency₈
    3. 控制递减: control₁(无) < control₈(完全)

  局部性原理:
    时间局部性: 若数据最近被访问，很可能再次被访问
    空间局部性: 若数据被访问，邻近数据很可能被访问

  缓存命中率:
    hit_rate = hits / (hits + misses)
    有效延迟 = hit_rate × latency_cache + (1 - hit_rate) × latency_source
```

### 2.3 存储访问的形式化成本模型

```text
定义 (存储访问成本):
  设访问数据 d 的请求序列 req(d) = (r₁, r₂, ..., rₙ)

  层次选择策略:
    选择层级 Lᵢ 使得总成本最小:
      Cost(Lᵢ) = latencyᵢ + miss_penaltyᵢ × (1 - hit_rateᵢ)

  典型数值 (Chrome @ M1 Mac):
    L1 (Register):    ~0.5 ns, 容量 64B
    L2 (RAM):         ~100 ns, 容量 16 GB
    L3 (Memoization): ~1 μs,   容量 10 MB
    L4 (Session):     ~10 μs,  容量 5 MB
    L5 (Local):       ~100 μs, 容量 5 MB
    L6 (IndexedDB):   ~1 ms,   容量 2 GB
    L7 (Cache API):   ~1 ms,   容量 ∞ (磁盘)
    L8 (Network):     ~50 ms,  容量 ∞

  每层级延迟比 ≈ 10×，形成数量级阶梯

  存储配额请求:
    navigator.storage.estimate() → { usage, quota }
    用户提示时 quota 可达磁盘空间的 60%
```

### 2.4 存储分区与隐私边界

```text
定义 (存储分区):
  现代浏览器 (Safari 2024+, Firefox 2025+) 实施存储分区:
    Partition Key = (top-level site, frame origin)

  影响:
    - 第三方 iframe 的 LocalStorage/IndexedDB 与顶层站点隔离
    - 跨站嵌入的地图/聊天 widget 无法共享存储
    - Service Worker 缓存按分区隔离

  形式化:
    存储键从 origin 扩展为 (top-level-site, origin)
    即: Store(origin) → Store(site, origin)
    导致: 同一 origin 在不同 top-level site 下存储不共享

  工程应对:
    - Storage Access API (requestStorageAccess)
    - Related Website Sets (Chrome)
    - First-Party Sets → 更名为 Related Website Sets
```

### 2.2 前端 CAP 约束

```text
定义 (前端 CAP):
  浏览器存储受以下约束:

  Consistency (一致性):
    - 同一标签页: LocalStorage 事件传播 (不可靠)
    - 跨标签页: BroadcastChannel / SharedWorker
    - 离线/在线: Service Worker 同步策略

  Availability (可用性):
    - LocalStorage: 始终可用 (除非用户禁用)
    - IndexedDB: 可能因隐私模式/存储配额不可用
    - Cache API: 仅 HTTPS + Service Worker 上下文

  Partitioning (分区):
    - 第三方 Cookie 限制
    - Storage Partitioning (Safari/Firefox)
    - 同源策略严格限制跨域存储访问

  权衡:
    强一致 + 高可用 → 需要在线同步 (网络延迟)
    离线可用 + 强一致 → IndexedDB + 同步协议
    离线可用 + 高可用 → 牺牲一致性 (最终一致)
```

## 四、工程实践

### 4.1 分层存储策略

```javascript
// 分层缓存管理器
class TieredStorage {
  async get(key) {
    // L3: 内存缓存
    if (this.memoryCache.has(key)) {
      return this.memoryCache.get(key);
    }

    // L6: IndexedDB
    const dbValue = await this.idb.get(key);
    if (dbValue) {
      this.memoryCache.set(key, dbValue); // 回填内存
      return dbValue;
    }

    // L8: 网络请求
    const networkValue = await fetch(`/api/data/${key}`).then(r => r.json());
    await this.idb.set(key, networkValue); // 回填 IndexedDB
    this.memoryCache.set(key, networkValue);
    return networkValue;
  }
}
```

### 4.2 StorageManager 配额查询

```javascript
// 查询存储配额
if (navigator.storage && navigator.storage.estimate) {
  const { usage, quota } = await navigator.storage.estimate();
  console.log(`Used: ${(usage / 1e6).toFixed(2)} MB`);
  console.log(`Quota: ${(quota / 1e6).toFixed(2)} MB`);
  console.log(`Usage: ${(usage / quota * 100).toFixed(2)}%`);
}

// 请求持久化存储
if (navigator.storage && navigator.storage.persist) {
  const isPersistent = await navigator.storage.persist();
  console.log(`Persistent storage: ${isPersistent}`);
}
```

## 六、批判性总结

浏览器存储层次的 L1-L8 模型是**内存层次结构**在 Web 环境中的映射：从 CPU 寄存器到网络存储，每一层都是**延迟与容量**的权衡。但前端存储有一个独特挑战——**生命周期控制**：与服务器端存储不同，浏览器存储可能随时被用户清除（隐私模式、存储清理、iOS 7天规则），这意味着前端应用必须始终假设本地缓存可能失效。

IndexedDB 的设计是**浏览器存储的巅峰之作**：事务、索引、游标、版本升级——这些数据库级别的特性在浏览器中可用，使离线优先应用成为可能。但 IndexedDB 的 API 是**基于事件的回调地狱**（尽管现代 wrapper 如 idb 提供了 Promise 封装），这反映了 Web 标准的历史包袱：IndexedDB 设计于 2010 年，当时 Promise 尚未成为标准。

Storage Partitioning 是 2024-2026 年浏览器隐私政策的**重大变化**：Safari 和 Firefox 开始将第三方 iframe 的存储与顶层站点隔离，这意味着跨域共享 LocalStorage/IndexedDB 的能力被大幅限制。这是**隐私保护**与**功能需求**之间的新一轮博弈——广告追踪的滥用导致了存储能力的收缩，所有 Web 应用都为此付出代价。

OPFS（Origin Private File System）代表了浏览器存储的**下一代方向**：它提供了一种接近原生文件系统的同步 API（通过 Web Worker），允许对大文件进行随机读写和内存映射。对于视频编辑、游戏资源包、科学计算数据集等大文件场景，OPFS 的吞吐量比 IndexedDB 高一个数量级。但 OPFS 的局限在于**仅在 Worker 中支持同步 API**，主线程仍需使用异步接口，这反映了浏览器架构中"避免阻塞主线程"的根本约束。Storage Buckets API 则是**配额管理的形式化升级**——允许应用为不同功能模块创建独立的存储桶，各自设定配额、持久性级别和过期时间，这比全局的 `navigator.storage` 提供了更细粒度的资源控制。

| 概念节点 | 核心属性 | 依赖节点 | 关联强度 | 形式化映射 |

## 八、形式化推理链

**推理链 R1：从局部性原理到缓存层次最优配置**

```text
前提:
  P1: 局部性原理:
        时间局部性: P(访问x | x 最近被访问) ≫ P(访问随机y)
        空间局部性: 访问x ⟹ 邻近地址 x±Δ 很可能被访问
  P2: 存储层次 L₁..L₈ 满足: latencyᵢ₊₁ ≈ 10 × latencyᵢ, capacityᵢ₊₁ > capacityᵢ
  P3: 缓存命中率 hit_rate = hits / (hits + misses)

推导:
  Step 1: 有效访问延迟公式 (平均内存访问时间 AMAT)
    AMAT = latency_Lᵢ + (1 - hit_rate_Lᵢ) × latency_Lᵢ₊₁

  Step 2: 层次选择优化目标
    最小化总成本: min_{Lᵢ} Σ_{req∈R} Cost(Lᵢ, req)
    其中 Cost(Lᵢ, req) = latencyᵢ 若命中
                        latencyᵢ + penalty_miss 若未命中

  Step 3: 最优放置定理
    数据项 d 应放置于层级 Lᵢ 使得:
      hit_rate(d, Lᵢ) × latencyᵢ + (1-hit_rate(d,Lᵢ)) × latencyᵢ₊₁
      在 i ∈ {1..8} 中最小

  Step 4: 工程推论
    若 hit_rate(L3) > 0.9 (内存缓存): 有效延迟 ≈ 1μs
    若 hit_rate(L5) = 0.5 (IndexedDB): 有效延迟 ≈ 0.5ms + 0.5×50ms ≈ 25ms
    差距: 25,000× —— 证明缓存策略决定性能数量级

结论 (定理 T-MH1):
  存储层次的最优配置由访问模式决定:
    高频小数据 → 尽可能靠近 L1 (Register/Cache)
    中频结构化数据 → L5-L6 (IndexedDB/OPFS)
    低频大文件 → L7-L8 (Cache API/Network)
  违反此原则的"缓存倒置"将导致有效延迟恶化 3-4 个数量级。
```

**推理链 R2：从前端 CAP 到具体存储方案的形式化映射**

```text
前提:
  P1: CAP: 分布式系统同时满足 C, A, P 中至多两项
  P2: 浏览器是多标签页分布式系统，标签页间存在网络式分区
  P3: 各存储 API 的可用性/一致性特征已知

推导:
  Step 1: 各方案的 CAP 分类
    ┌─────────────────┬──────┬──────┬──────┐
    │    方案          │  C   │  A   │  P   │
    ├─────────────────┼──────┼──────┼──────┤
    │ localStorage    │ 强   │ 高   │ 无   │  → CA
    │ IndexedDB       │ 事务 │ 高   │ 单   │  → CA
    │ BroadcastChannel│ 因果 │ 高   │ 同源 │ → AP-ish
    │ Service Worker  │ 最终 │ 高   │ 是   │ → AP
    │ Web Locks       │ 强   │ 中   │ 无   │ → CP
    │ CRDT            │ 最终 │ 高   │ 是   │ → AP
    └─────────────────┴──────┴──────┴──────┘

  Step 2: 场景约束映射
    金融级表单 (强一致+可用) → CP: Web Locks + IndexedDB 事务
    离线优先应用 (可用+分区容错) → AP: Service Worker + BackgroundSync
    协作编辑 (可用+分区容错) → AP: CRDT + BroadcastChannel
    用户偏好 (强一致+可用，无分区) → CA: localStorage

  Step 3: 形式化选择函数
    select(C_need, A_need, P_need) =
      argmax_{scheme} [1(C_scheme ≥ C_need) × 1(A_scheme ≥ A_need) × 1(P_scheme ≥ P_need)]
    若为空集，则必须 relax 某个需求 (通常降级一致性)。

结论 (定理 T-MH2):
  前端存储方案选择的完备算法:
    1. 判定是否存在网络分区风险 (多标签/离线)
    2. 若存在分区风险 → 必须选择 AP 或 CP，放弃 CA
    3. 在 AP/CP 中，根据业务对一致性的容忍度选择:
         容忍最终一致 → AP (SW/CRDT)
         不能容忍不一致 → CP (Web Locks)
  绝大多数前端离线场景选择 AP，因为用户可用性优先于数据瞬时一致性。
```

## 九、推理判定树/决策树

```text
浏览器存储层级选择决策树
│
├─► 数据访问频率判定
│   ├─ [极高频 (>1K ops/s)] ──► JS 变量 / V8 Heap (L1-L2, ~ns)
│   ├─ [高频 (100-1K ops/s)] ──► Memoization / LRU Cache (L3, ~μs)
│   ├─ [中频 (1-100 ops/s)] ──► localStorage / SessionStorage (L4, ~μs-ms)
│   ├─ [低频 (<1 ops/s)] ──► IndexedDB / OPFS (L5-L6, ~ms)
│   └─ [按需加载] ──► Network / Cache API (L7-L8, ~10-100ms)
│
├─► 数据结构化需求判定
│   ├─ [需要索引查询] ──► IndexedDB (B-Tree 索引、范围查询)
│   ├─ [键值对即可] ──► localStorage / SessionStorage
│   ├─ [大文件/二进制] ──► OPFS (字节级随机读写)
│   └─ [网络响应缓存] ──► Cache API (Request/Response 对)
│
├─► 持久化需求判定
│   ├─ [页面刷新后保留] ──► localStorage / IndexedDB / OPFS
│   ├─ [标签页关闭后清除] ──► SessionStorage / JS Heap
│   └─ [请求系统级持久化] ──► navigator.storage.persist()
│       └─  granted=true → 浏览器不会自动清理
│
└─► 多标签页同步需求判定
    ├─ [强同步] ──► BroadcastChannel + 事件监听
    ├─ [事务隔离] ──► Web Locks API (navigator.locks)
    └─ [无需同步] ──► 各标签页独立存储实例
```
