## 1. 修正机制概述

### 1.1 修正触发条件

理论修正由以下情况触发：

1. **可证伪性测试**：假设被[可证伪性测试](./01-可证伪性测试.md)证伪
2. **预测准确性不足**：[预测准确性评估](./02-预测准确性评估.md)显示准确性<80%
3. **置信区间排除**：95%置信区间不包含理论预测值
4. **新证据发现**：实证研究或案例发现与理论矛盾

### 1.2 修正原则

- **最小修正原则**：优先采用影响范围最小的修正
- **可追溯原则**：所有修正需记录原因、内容和影响
- **一致性原则**：修正后需更新所有相关文档

## 2. 反馈循环设计

### 2.1 反馈循环步骤

```
1. 理论预测 → 2. 实验验证 → 3. 结果分析 → 4. 理论修正 → 1. 理论预测
```

### 2.2 循环详解

| 步骤 | 活动 | 产出 |
|------|------|------|
| **理论预测** | 从理论导出可测试假设 | 细化假设、证伪条件 |
| **实验验证** | 执行实验或案例研究 | 原始数据、观测结果 |
| **结果分析** | 统计分析、对比预测 | 准确性评估、证伪判断 |
| **理论修正** | 根据结果修正理论 | 更新文档、版本记录 |

### 2.3 决策节点

**是否需要修正？**

- **是** → 进入修正流程，确定修正类型
- **否** → 继续下一轮验证或扩大验证范围

## 3. 修正类型与流程

### 3.1 修正类型

| 类型 | 范围 | 影响 | 版本变化 | 示例 |
|------|------|------|----------|------|
| **小修正** | 调整参数、补充边界条件 | 不影响核心理论 | v1.0 → v1.0.1 | 将"50%"调整为"40-60%" |
| **中修正** | 修改假设、扩展适用范围 | 影响部分理论 | v1.0 → v1.1 | 将"所有业务系统"改为"90%业务系统" |
| **大修正** | 重构理论、建立新框架 | 影响核心理论 | v1.0 → v2.0 | MSMFIT从四要素扩展为五要素 |

### 3.2 修正示例

**小修正示例**：

- **修正内容**：将开发时间减少预测从"50%"调整为"40-60%"
- **原因**：实际数据表明存在较大方差，95%置信区间为[40%, 50%]
- **影响**：更新预测文档、可证伪性测试的证伪条件

**中修正示例**：

- **修正内容**：将MSMFIT适用范围从"所有业务系统"修改为"90%的业务系统"
- **原因**：发现实时控制系统、纯计算型系统不适用
- **影响**：更新MSMFIT普适性论证、理论边界文档

**大修正示例**：

- **修正内容**：将MSMFIT从{E,R,V,C}扩展为{E,R,V,C,X}（新增要素X）
- **原因**：发现某类系统无法用四要素完整描述
- **影响**：全理论体系更新、形式化证明重写

### 3.3 修正流程步骤

1. **问题识别**
   - 发现预测不准确或假设被证伪
   - 识别理论缺陷
   - 分析根本原因

2. **修正方案设计**
   - 设计修正方案（小/中/大）
   - 评估修正影响范围
   - 制定实施计划

3. **修正实施**
   - 修改理论文档
   - 更新相关文档（交叉引用）
   - 通知相关人员

4. **验证修正效果**
   - 重新运行可证伪性测试
   - 重新进行预测准确性评估
   - 记录修正历史

## 4. 版本管理策略

### 4.1 版本号规则

采用语义化版本（Semantic Versioning）思想：

- **主版本号**：大修正（v1.0 → v2.0）
- **次版本号**：中修正（v1.0 → v1.1）
- **修订号**：小修正（v1.0 → v1.0.1）

### 4.2 版本记录格式

```markdown
## 版本历史

### v1.1.0 (2025-02-03)
- **中修正**：将MSMFIT适用范围从"所有业务系统"修改为"90%的业务系统"
- **原因**：发现实时控制系统不适用
- **影响**：更新相关文档和示例

### v1.0.1 (2025-01-15)
- **小修正**：将开发时间减少预测从"50%"调整为"40-60%"
- **原因**：实际数据表明存在较大方差
- **影响**：更新预测文档
```

## 批判性总结

本文档设计了理论修正的反馈循环、修正类型（小/中/大）和版本管理策略，体现了对知识渐进性和可错论的深刻理解。修正机制的设计在工程层面是实用的，但在科学哲学层面可以更精确。首先，**修正触发条件的量化标准**虽然明确（预测准确性<80%或95%置信区间不包含预测值），但这些标准本身也需要被质疑——为什么选择80%而非70%或90%？这一阈值若未经校准，可能过于敏感或过于迟钝。在拉卡托斯的研究纲领方法论中，理论的"硬核"（hard core）受到"保护带"（protective belt）的防卫，不应因单次异常而被轻易修正。文档未明确区分MSMFIT的哪些部分属于"硬核"（不可修正）、哪些属于"保护带"（可修正），这可能导致核心概念在修正过程中被不必要地弱化。其次，版本管理策略借用了软件工程的语义化版本控制（SemVer），但理论版本与软件版本存在本质差异：软件版本的后向兼容性可以通过测试验证，而理论版本的"兼容性"涉及概念变迁（conceptual change），新旧版本之间的术语可能具有不可通约性（库恩）。例如，若MSMFIT从四要素扩展为五要素，旧文献中的"MSMFIT"与新文献中的"MSMFIT"不再是同一概念，简单的版本号递增可能掩盖了这一深层断裂。第三，反馈循环设计为"理论预测→实验验证→结果分析→理论修正"，这是一个线性模型，但实际科学实践中的反馈往往是非线性的、多路径的——实验设计本身受理论指导，而理论修正又受可获得的技术手段限制。文档未讨论这些非线性反馈如何影响修正的质量和速度。

## 形式化定义

**定义 V.5** (理论修正类型)
设 $T$ 为理论，$\Delta$ 为观测与预测的偏差。修正类型 $\tau(\Delta)$ 定义为：
- **小修正**（$\tau = 1$）：$\Delta$ 仅影响参数估计，$T$ 的核心公理 $A_T$ 不变；
- **中修正**（$\tau = 2$）：$\Delta$ 要求修改辅助假设 $H_{aux} \subset T$，但 $A_T$ 不变；
- **大修正**（$\tau = 3$）：$\Delta$ 与 $A_T$ 矛盾，需重构 $A_T$ 或替换为 $A_{T'}$。

**定义 V.6** (理论硬核与保护带)
设 $T = \langle C, P, H \rangle$，其中：
- $C$ 为**硬核**（hard core）：不可放弃的基本假设集合，修正 $\tau = 3$ 时 $T$ 不再为原理论；
- $P$ 为**保护带**（protective belt）：可调整的辅助假设、初始条件、边界条件集合；
- $H$ 为**启发式**（heuristics）：指导保护带调整的元规则。
修正的优先序为：$P$ 内调整 $\succ$ $H$ 修正 $\succ$ $C$ 替换。MSMFIT的 $C$ 至少包含：四要素最小性、结构同构存在性、语义保真原则。
