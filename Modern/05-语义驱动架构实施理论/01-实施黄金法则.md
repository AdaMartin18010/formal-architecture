## 1. 法则概述

### 1.1 法则定位

三大黄金法则是语义驱动架构实施的**核心原则**，确保：

- **语义优先**：业务语义始终优先于技术实现
- **可逆性保证**：DSL与代码双向同步，语义模型始终是单一可信源
- **上下文显性化**：所有分支逻辑对应有名字的上下文参数

### 1.2 法则关系

三大法则相互支撑：

```text
语义优先（法则1）
    ↓
可逆性保证（法则2）
    ↓
上下文显性化（法则3）
```

## 2. 法则1：语义优先，技术归零

### 2.1 原则阐述

**原则**：所有设计评审**先讨论DSL语义，再讨论技术实现**

**核心思想**：

- 业务语义是**本质复杂性**，技术实现是**偶发复杂性**
- 技术实现应**服务于**业务语义，而非相反
- 技术栈的选择不应影响业务语义的表达

### 2.2 实施要求

**红线**：未经DSL定义的需求，**不允许写第一行代码**

**评审流程**：

1. **需求评审**：业务分析师 + 架构师 → DSL设计评审
2. **语义验证**：验证DSL是否符合业务语义
3. **技术实现**：基于DSL生成技术实现代码

### 2.3 工具支持

**语义契约门禁**：Git Hook阻止无DSL的PR提交

**实现方式**：

```bash
# .git/hooks/pre-commit
if ! grep -q "semantic:" "$(git diff --cached --name-only)"; then
  echo "错误：提交必须包含DSL语义定义"
  exit 1
fi
```

## 3. 法则2：可逆性即生命线

### 3.1 原则阐述

**原则**：任何DSL变更必须**可反向提取**，代码与模型**双向同步**

**核心思想**：

- DSL是**单一可信源（SSOT）**
- 代码与DSL必须**严格同步**
- 任何一端的变更必须能**反向映射**到另一端

### 3.2 实施要求

**实践**：CI流水线每日凌晨执行 **"反向工程校验"**，若提取的DSL与源码DSL不一致，**自动阻塞发布**

**校验流程**：

1. **正向生成**：DSL → 代码生成
2. **反向提取**：代码 → DSL提取
3. **差异对比**：对比原始DSL与提取的DSL
4. **阻塞发布**：如有差异，自动阻塞发布

### 3.3 度量方法

**可逆性得分**：

$$R_{reversible} = \frac{\text{成功提取次数}}{\text{总构建次数}} \times 100\%$$

**目标**：$R_{reversible} = 100\%$

## 4. 法则3：上下文显性化

### 4.1 原则阐述

**原则**：所有`if-else`分支必须对应**一个有名字的上下文参数**

**核心思想**：

- 上下文是[MSMFIT](../01-IT语义世界基础理论/02-最小语义模型MSMFIT.md)的第四要素（C）
- 所有业务场景差异应通过**上下文参数**表达
- 禁止硬编码分支逻辑

### 4.2 实施要求

**错误示范**：

```java
// 错误：硬编码分支
if (user.getVipLevel() > 3) { ... } // 魔法数3
```

**正确示范**：

```java
// 正确：上下文参数
if (context.userSegment() == UserSegment.HIGH_VALUE) { ... }
```

### 4.3 工具支持

**SonarQube自定义规则**：**禁止硬编码分支**

**规则定义**：

```xml
<rule>
  <key>NoHardcodedBranch</key>
  <name>禁止硬编码分支</name>
  <description>所有if-else分支必须对应上下文参数</description>
  <severity>BLOCKER</severity>
</rule>
```

## 5. 三大法则的综合应用

### 5.1 应用场景

**场景**：实现"VIP用户折扣"功能

#### 5.1.1 步骤1：语义优先（法则1）

```dsl
// 先定义DSL语义
rule VipDiscount {
  condition: context.userSegment == VIP
  action: applyDiscount(rate: 0.1)
}
```

#### 5.1.2 步骤2：可逆性保证（法则2）

```bash
# 生成代码
./gradlew generate

# 反向提取验证
./gradlew extractSemanticModel

# 验证一致性
git diff src/semantic/ || exit 1
```

#### 5.1.3 步骤3：上下文显性化（法则3）

```java
// 生成的代码（自动包含上下文）
if (context.userSegment() == UserSegment.VIP) {
  applyDiscount(0.1);
}
```

### 5.2 综合效果

- **语义保真**：业务规则直接映射到代码
- **可逆同步**：代码与DSL严格同步
- **上下文显性**：所有分支逻辑都有明确的上下文参数

## 批判性总结

三大黄金法则（语义优先、可逆性即生命线、上下文显性化）构成了语义驱动架构实施的道德律令，其设计意图在于通过强制约束防止业务语义在技术实现过程中被稀释。语义优先法则有效应对了"需求漂移"问题，将 DSL 作为需求评审的交付物而非事后文档；可逆性法则通过 CI 校验确保单一可信源，避免了模型与代码的长期不一致；上下文显性化法则直接针对"魔法数"与硬编码分支这一软件维护的沉疴。然而，严格的"无 DSL 不编码"红线在探索性开发、技术 Spike 与紧急补丁场景下可能降低团队响应速度。此外，可逆性校验的 100% 目标在引入第三方库或框架升级时可能产生大量伪差异（false positives）。因此，建议将三大法则视为"默认严格、例外可审"的治理原则，而非不可违背的物理定律。

## 形式化定义

**定义** (黄金法则约束系统). 设代码提交为 $c$，DSL 定义库为 $\mathcal{D}$，语义注解集合为 $\mathcal{A}$。定义提交合规谓词：

$$Compliant(c) \iff (\exists d \in \mathcal{D}, d \leadsto c) \lor (\exists a \in \mathcal{A}, a \in c)$$

其中 $d \leadsto c$ 表示提交 $c$ 可由 DSL 定义 $d$ 生成或解释。定义可逆性得分：

$$R_{reversible} = \frac{|\{c \in Build \mid ext(c) \sim_S \mathcal{D}\}|}{|Build|} \times 100\%$$

目标 $R_{reversible} = 100\%$ 为理想约束，实际运行中允许容差 $\epsilon = 5\%$ 以应对框架生成代码等例外。

### 2. 形式化推理链

**公理 A1** (Parnas模块化律, 1972): 系统分解应隐藏设计决策。

Module_i \perp Module_j \iff DesignDecision_i \cap DesignDecision_j = \emptyset


**公理 A2** (Evans领域律, 2003): 领域模型是软件的核心资产。

Value(Software) \propto Fidelity(DomainModel, BusinessDomain)


**引理 L1** (SSOT一致性引理): 唯一可信源保证所有派生制品的一致性：

SSOT \models \phi \implies \forall Artifact_{derived}: Artifact_{derived} \models \phi


**引理 L2** (防腐层隔离引理): 反闪光层将遗留系统语义与目标系统语义解耦：

Legacy_{sem} \xrightarrow{Translate} AntiFlash \xrightarrow{DSL_{sem}} Target_{sem}


**定理 T1** (黄金法则完备性定理): 五条黄金法则构成语义驱动实施的充分约束集：

\mathcal{G} \models Correctness(SDA_{implementation})


**推论 C1** (可逆性下界推论): 在容差ε内，系统的可逆成本有界：

Cost_{revert} \leq \epsilon \cdot |Code_{total}| \cdot UnitCost_{manual}


**推论 C2** (上下文完备性推论): 由MSMFIT模型，运行时上下文C必须覆盖所有业务场景差异：

\forall Scenario: \exists c \in C: Scenario \models c

2. **Eric Evans** (2003). *Domain-Driven Design: Tackling Complexity in the Heart of Software*. Addison-Wesley. DDD创始人，提出防腐层(Anti-Corruption Layer)与限界上下文(Bounded Context)模式，为反闪光层设计提供直接理论来源。

3. **Martin Fowler** (2010). *Domain-Specific Languages*. Addison-Wesley. 系统阐述DSL的设计与实现，为"DSL即代码"法则提供技术实现路径。

4. **Lynn Beighley** (2009). "Database Abstraction Layers". *O'Reilly Media*. 数据库抽象层设计模式，为SSOT在数据持久化层的实现提供工程参考。

### 批判性总结

实施黄金法则以五条约束为语义驱动转型提供了简洁的行为规范，但其背后隐藏着深刻的实践悖论。SSOT原则在实践中面临"谁拥有真相"的组织政治学难题：当业务规则分散在多个部门的Excel表格、邮件约定和遗留代码中时，强制建立单一DSL作为SSOT往往遭遇权力阻力——某些部门可能将数据控制权视为组织资本，拒绝将其纳入统一的DSL治理体系。反闪光层的设计虽然借鉴了Evans(2003)的防腐层模式，但忽视了模式适用的语境差异：DDD中的防腐层用于不同限界上下文之间的翻译，而语义驱动中的反闪光层需要处理的是"声明式语义"与"命令式实现"之间的根本性范式冲突，这种冲突的复杂度远超接口适配。更为关键的是，可逆性约束的95%阈值缺乏统计学依据——在复杂系统中，5%的手工代码可能集中在关键路径（如性能优化、安全加密）上，其业务重要性可能远超95%的生成代码；将可逆性定义为代码比例而非业务价值比例，是一种形而上学的度量方式。从形式化视角，这五条法则虽然看似正交，实则存在隐性依赖：没有强大的治理机制（法则5），SSOT（法则1）和DSL即代码（法则3）将在组织政治中迅速腐化。因此，法则之间的权重并非均等，当前框架对此未加区分，可能误导实施者优先级排序。
