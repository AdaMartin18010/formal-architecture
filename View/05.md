# 一、范畴论视角的计算模型

## 1.1 前端框架的范畴构造

将前端计算模型严格表述为**范畴**（Category）：

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    𝓒 = ⟨Obj(𝓒), Hom(𝓒), ∘, id⟩  // 前端渲染范畴              │
│                                                                             │
│  Obj(𝓒): {State, VDOM, DOM, Effect, Component, Event}                       │
│  Hom(𝓒): 态射集合                                                            │
│  ∘: 态射复合                                                                 │
│  id: 恒等态射                                                                │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 三种响应式范式的范畴论本质

| 范式 | 范畴构造 | 态射定义 | 核心函子 | 自然变换 |
|-----|---------|---------|---------|---------|
| **Virtual DOM** | **自由范畴** Free Category | `render: State → VDOM`, `diff: VDOM × VDOM → Patch`, `patch: Patch → DOM` | `F: Set → Tree` (状态集到树范畴) | Reconciliation 作为 `F ⇒ G` 的自然变换 |
| **Signals** | **图范畴** Graph Category | `signal: A → B` (依赖边), `effect: ΔA → ΔDOM` | `Γ: State → ReactiveGraph` (状态到依赖图) | 传播算法作为 `Γ` 的保持结构映射 |
| **编译时模型** | **语法范畴** Syntax Category | `compile: AST → IR`, `codegen: IR → DOMOps` | `C: Lang → DOM` (语言到 DOM 操作) | 优化 passes 作为 `C ⇒ C'` 的等价变换 |

> **范畴论洞察**：React 的 `UI = f(state)` 是**态射** `f: State → UI` 的声明；Signals 的依赖图是**图表**（Diagram）`D: J → 𝓒` 的极限（Limit）构造；Svelte 编译器是**语法范畴到语义范畴的函子**。

## 1.2 函子与自然变换的形式化

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    框架演进的函子序列                                          │
└─────────────────────────────────────────────────────────────────────────────┘

                    [范畴 𝓢: 状态范畴]
                         │
                         ▼
              ┌─────────────────────┐
              │ 函子 F_react: 𝓢 → 𝓥  │
              │  (状态 → VDOM 树)    │
              │  对象映射: s ↦ ReactElement │
              │  态射映射: Δs ↦ Reconciliation│
              └──────────┬──────────┘
                         │
                         ▼
              ┌─────────────────────┐
              │ 函子 F_solid: 𝓢 → 𝓖  │
              │  (状态 → 依赖图)      │
              │  对象映射: s ↦ SignalNode │
              │  态射映射: Δs ↦ GraphPropagation│
              └──────────┬──────────┘
                         │
                         ▼
              ┌─────────────────────┐
              │ 函子 F_svelte: 𝓢 → 𝓓  │
              │  (状态 → DOM 操作)    │
              │  对象映射: s ↦ DOMInstruction │
              │  态射映射: Δs ↦ CompiledUpdate │
              └─────────────────────┘

自然变换 η: F_react ⇒ F_solid
  定义: η_s: F_react(s) → F_solid(s)
  即: 对于每个状态 s, 存在一个从 VDOM 树到依赖图的映射
  但此自然变换不存在!
  ∵ VDOM 是树结构, Signals 是图结构
  ∴ 二者不是自然等价的 (Not Naturally Isomorphic)

定理: React 与 Solid 的范畴表示不同构
证明:
  假设存在同构函子 I: 𝓥 → 𝓖
  则 I 必须保持结构:
    I(Tree) = Graph
    I(Node) = SignalNode
    I(ChildRelation) = DependencyEdge

  但 Tree 是有向无环图且每个节点有唯一父节点
  Graph 允许环 (circular dependencies) 和多父节点
  ∴ Tree 是 Graph 的真子范畴, 非同构
  ∴ I 不是同构函子    ∎
```

## 3.2 React Concurrent 的半可判定性分析

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    React 18+ Concurrent 特性的形式化分析                     │
└─────────────────────────────────────────────────────────────────────────────┘

定理 T5: React Concurrent 的优先级调度是优先级翻转的启发式解

证明:
  设任务集合 T = {t₁, t₂, ..., tₙ}, 优先级 p: T → ℕ
  React 使用 Lane 模型: Lane = 2^k, k ∈ [0, 31]

  优先级翻转场景:
    t_high: 用户输入 (TransitionPriority = 1)
    t_low: 数据获取 (DefaultPriority = 16)
    t_lock: 渲染锁 (由 t_low 持有, 因为 Suspense 边界)

  经典优先级翻转:
    t_high 等待 t_lock, t_lock 被 t_low 持有
    ∴ t_high 间接等待 t_low

  React 的解决: Time Slicing + Interruptible Reconciliation
    1. t_low 在执行过程中可被 t_high 中断
    2. t_low 的渲染状态保存为 "WorkInProgress" 树
    3. t_high 完成后, t_low 从断点恢复

  形式化:
    设渲染函数 R: State → Tree 是可中断的
    R(s) = R₁(s) ⊕ R₂(s) ⊕ ... ⊕ Rₙ(s)
    其中 ⊕ 是状态拼接操作, 每个 Rᵢ 是一个时间片

    对于高优先级任务 t_h:
      ∃k: R(s) = R₁(s) ⊕ ... ⊕ Rₖ(s) ⊕ t_h ⊕ Rₖ₊₁(s) ⊕ ...

    这等价于协程 (Coroutine) 的 yield 语义

  但此方案非形式化正确:
    1. 状态一致性: 中断时 Rₖ(s) 可能处于不一致状态
    2. 副作用原子性: useEffect 可能在中断边界处部分执行
    3. 内存开销: 需维护多棵 WIP 树

  ∴ React Concurrent 是优先级翻转的实用解, 非形式化解    ∎
```

# 六、设计系统的代数结构

## 6.1 设计系统的群论模型

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    设计令牌 (Design Tokens) 的代数结构                       │
│                                                                             │
│  定义: 设计系统 𝓓 = ⟨T, ⊕, ⊗, 𝟎, 𝟏⟩                                         │
│    T: 设计令牌集合 (colors, spacing, typography)                             │
│    ⊕: 组合操作 (例如: color + opacity)                                        │
│    ⊗: 应用操作 (例如: token → component)                                     │
│    𝟎: 中性元素 (透明, 零间距)                                                 │
│    𝟏: 单位元素 (默认主题)                                                     │
└─────────────────────────────────────────────────────────────────────────────┘

定理 T7: 设计令牌在 ⊕ 操作下构成阿贝尔群 (Abelian Group)

证明:
  封闭性: ∀a,b ∈ T, a ⊕ b ∈ T
    例: color(#000) ⊕ opacity(0.5) = color(rgba(0,0,0,0.5)) ∈ T

  结合律: (a ⊕ b) ⊕ c = a ⊕ (b ⊕ c)
    例: (spacing(4) ⊕ spacing(8)) ⊕ spacing(16) = spacing(28)
       spacing(4) ⊕ (spacing(8) ⊕ spacing(16)) = spacing(28)

  交换律: a ⊕ b = b ⊕ a
    例: color ⊕ opacity = opacity ⊕ color (结果相同)

  单位元: ∃𝟎 ∈ T: a ⊕ 𝟎 = a
    𝟎 = transparent, spacing(0), font-size(inherit)

  逆元: ∀a ∈ T, ∃a⁻¹ ∈ T: a ⊕ a⁻¹ = 𝟎
    例: spacing(8) 的逆元是 spacing(-8) (在相对定位中)
       color 的逆元是 mix-blend-mode: difference 中的补色

∴ ⟨T, ⊕⟩ 是阿贝尔群    ∎

应用: CSS 变量 (Custom Properties) 是群的线性表示
  --color-primary: #000;
  --spacing-unit: 4px;
  这些变量是群 T 的生成元 (Generators)
```

## 6.2 组件的组合代数

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    组件组合的幺半群 (Monoid) 结构                              │
└─────────────────────────────────────────────────────────────────────────────┘

定义: 组件组合 ⟨C, ∘, ε⟩
  C: 组件集合
  ∘: 组合操作 (嵌套/包裹)
  ε: 空组件/Fragment

公理验证:
  结合律: (A ∘ B) ∘ C = A ∘ (B ∘ C)
    <A><B><C /></B></A> = <A><B><C /></B></A>

  单位元: A ∘ ε = ε ∘ A = A
    <A><></></A> = <A /> = <></><A /></>

但交换律不成立:
  A ∘ B ≠ B ∘ A (组件嵌套顺序影响渲染树结构)

∴ ⟨C, ∘, ε⟩ 是幺半群, 非阿贝尔群

高阶组件 (HOC) 是群上的自同态:
  HOC: C → C
  withTheme(A) = ThemeProvider ∘ A ∘ ThemeConsumer
  即: HOC 是组件群上的群同态 (保持组合结构)
```

## 7.2 视觉回归的度量空间

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    像素空间的度量与等价                                        │
└─────────────────────────────────────────────────────────────────────────────┘

定义: 像素空间 P = [0,255]^(W×H×3) (RGB 图像)
  度量 d: P × P → ℝ⁺

常用度量:
  1. L0 (汉明距离): d₀(p₁, p₂) = |{i | p₁[i] ≠ p₂[i]}|
     用于: 像素级精确比较 (过于严格)

  2. L1 (曼哈顿): d₁(p₁, p₂) = Σ|p₁[i] - p₂[i]|
     用于: 颜色差异累积

  3. L2 (欧几里得): d₂(p₁, p₂) = √(Σ(p₁[i] - p₂[i])²)
     用于: 感知颜色差异

  4. SSIM (结构相似): d_SSIM = f(亮度, 对比度, 结构)
     用于: 人眼感知相似度 (更符合视觉系统)

工程决策:
  精确像素比较 (L0) 是反人类的
  ∵ 子像素渲染、字体抗锯齿、GPU 差异导致 L0 差异 ≠ 视觉差异

  ∴ 应使用 SSIM + 阈值 ε:
    若 d_SSIM(p₁, p₂) < ε, 则判定为 "视觉等价"

  这与前端工程中的 "视觉快照测试" 实践一致:
    Chromatic 使用 1px 容差 + 忽略区域
    Percy 使用稳定化 (Stabilization) 预处理
```

## 8.2 Tree Shaking 的死亡代码消除

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                    Tree Shaking 的形式化分析                                   │
└─────────────────────────────────────────────────────────────────────────────┘

问题: 给定模块图 M, 标记所有从入口可达的节点 (Live Code)
      删除不可达节点 (Dead Code)

形式化:
  设入口集合 E ⊆ V
  可达集 R = {v ∈ V | ∃e ∈ E, e →* v} (从入口可达)
  死亡代码 D = V \ R

算法 (标记-清除):
  1. 标记: 从 E 开始 BFS/DFS, 标记所有可达节点
  2. 清除: 删除未标记节点及其导入边

副作用分析 (Side Effect):
  问题: 某些模块有全局副作用 (polyfill, CSS inject)
        即使无显式导出, 也不能删除

  形式化: 设副作用函数 side: V → {0, 1}
          若 side(v) = 1, 则 v ∈ R (强制保留)

  ES Module 的 pure 标记:
    /*#__PURE__*/ 提示编译器: 此调用无副作用
    形式化: side(v) = 0 (可被 Tree Shaking)

ESM vs CSM 的 Tree Shaking 差异:
  ESM: 静态导入, 编译期可确定依赖图
  CSM: 动态 require, 运行时确定, 不可静态分析

  ∴ ESM 的 Tree Shaking 是编译期优化 (AOT)
     CSM 的 Tree Shaking 是运行期近似 (近似死代码消除)
```

```
  操作系统定义 (Silberschatz):
    OS = 资源管理器 + 程序执行环境 + 用户接口

  浏览器满足:
    1. 资源管理: 内存 (V8 heap), CPU (Scheduler), 网络 (HTTP stack)
    2. 程序执行: JS 引擎 (V8), WASM 运行时
    3. 用户接口: DOM 渲染, 事件系统

  但浏览器是 "受限" 的:

    - 无原始网络套接字 (CORS, CSP)
    - 无共享内存并发 (Spectre/Meltdown 后禁用 SharedArrayBuffer 默认)

    每个 "应用" (Origin) 只有被授予的能力 (Capabilities)
```

## 11.2 终极定理：前端技术的不可能三角

```
┌─────────────────────────────────────────────────────────────────────────────────────┐
│                    定理 T10: 前端技术的不可能三角 (Frontend Impossibility Trinity)     │
└─────────────────────────────────────────────────────────────────────────────────────┘

命题: 对于任何前端技术栈 T, 以下三项最多同时满足两项:

  A: 最大生态兼容性 (Ecosystem) —— 可复用现有库, 人才供给充足
  B: 理论性能最优 (Performance) —— 运行时开销最小, A3 严格满足
  C: 心智模型极简 (Simplicity) —— 学习曲线平缓, 概念数量少

证明:
  假设存在 T 满足 A ∧ B ∧ C

  由 A (生态兼容):
    T 必须兼容 React 生态 (最大库集合)
    ∴ T 必须支持 VDOM 或 React API 兼容层

  由 B (性能最优):
    T 必须使用 Signals 或编译时优化 (无 VDOM 开销)

  但 VDOM 与 Signals 范畴不同构 (已证定理)
  React API (Hooks) 假设组件可重执行
  Signals 假设组件单次执行

  兼容层 (如 Preact Signals in React) 引入概念复杂度:
    - 两套状态系统并存
    - 边界转换规则
    - 调试工具需理解两种模型

  ∴ 兼容层破坏了 C (心智模型极简)

  反证: 不存在 T 满足 A ∧ B ∧ C    ∎

实例验证:
┌─────────────────┬─────┬─────┬─────┬─────────────────────────────────────────┐
│    技术栈        │  A   │  B   │  C   │      说明                               │
├─────────────────┼─────┼─────┼─────┼─────────────────────────────────────────┤
│ React 19        │  ✅  │  ❌  │  ❌  │  A最大, B/C因VDOM+Hooks复杂度牺牲        │
├─────────────────┼─────┼─────┼─────┼─────────────────────────────────────────┤

├─────────────────┼─────┼─────┼─────┼─────────────────────────────────────────┤
│ Angular 20      │  ✅  │  ✅  │  ❌  │  A/B强, C因概念数量多(模块/依赖注入/Signals│
│                 │     │     │     │  /RxJS/Zone)牺牲                         │
├─────────────────┼─────┼─────┼─────┼─────────────────────────────────────────┤
│ Svelte 5        │  ❌  │  ✅  │  ✅  │  B/C最优, A因生态 niche 牺牲             │
├─────────────────┼─────┼─────┼─────┼─────────────────────────────────────────┤

工程推论:
  选择技术栈 = 选择牺牲哪一项
  企业级项目: 牺牲 C (接受复杂度) → Angular / React
```

## 11.4 最终批判：技术选择的非计算性

```
┌─────────────────────────────────────────────────────────────────────────────────────┐
│                    最终批判: 技术选择是非计算性决策 (Non-computable Decision)         │
└─────────────────────────────────────────────────────────────────────────────────────┘

形式化批判:
  设技术选择问题 P = ⟨Constraints, Objectives, Alternatives⟩

  若 P 是计算性问题:
    ∃算法 A: P → OptimalSolution
    即: 给定约束和目标, 算法可输出最优技术栈

  但 P 包含非形式化要素:
    1. 团队认知负荷 (Team Cognitive Load) —— 不可度量
    2. 组织政治 (Org Politics) —— 不可形式化
    3. 技术债务历史 (Tech Debt) —— 不可完全枚举
    4. 未来需求不确定性 (Future Uncertainty) —— 不可预测

  ∴ P ∉ 可计算问题集合 (Not in Class P, NP, or any computable class)

  更严格地, P 是 " wicked problem " (Rittel & Webber, 1973):
    - 无明确问题定义
    - 无终止规则
    - 解决方案即测试
    - 无唯一正确解

  ∴ 所有 "技术选型矩阵" 都是启发式 (Heuristic), 非算法 (Algorithm)

  实践智慧 (Phronesis):
    优秀架构师的价值不在于选择 "正确" 技术
    而在于理解:
      1. 每种选择的权衡空间 (Trade-off Space)
      2. 组织的承受边界 (Absorptive Capacity)
      3. 迁移的过渡路径 (Transition Path)
      4. 回滚的逃生舱口 (Egress Hatch)

    这是亚里士多德的 "实践智慧", 非笛卡尔的 "计算理性"
```

## 十三、形式化推理链

### 13.1 公理系统与定理演绎

**公理系统 Π = {Π₁, Π₂, Π₃, Π₄}**

- **Π₁（范畴公理）**: 前端渲染构成范畴𝓒，对象=计算实体，态射=转换函数
- **Π₂（Curry-Howard公理）**: 逻辑命题、类型、范畴对象三者同构
- **Π₃（线性性公理）**: 资源必须恰好使用一次（线性类型）
- **Π₄（同构公理）**: 浏览器是Capability-based OS的受限实现

**推理链 I：React与Solid的范畴不同构性**

```
Π₁ ⊢ 定理T1（React与Solid范畴不同构）:
     证明:
       假设存在同构函子 I: 𝓥 → 𝓖
       则 I 必须保持结构:
         I(Tree) = Graph
         I(Node) = SignalNode
         I(ChildRelation) = DependencyEdge
       
       但 Tree是有向无环图且每个节点有唯一父节点
       Graph允许环（circular dependencies）和多父节点
       ∴ Tree是Graph的真子范畴，非同构
       ∴ I不是同构函子。    ∎

     自然变换 η: F_react ⇒ F_solid 不存在!
     ∵ ∀s∈State, η_s: F_react(s) → F_solid(s)
       要求从VDOM树到依赖图的映射保持结构
       但树结构 ≠ 图结构，η_s无法定义为自然变换。
```

**推理链 II：Curry-Howard-Lambek到Rust所有权的演绎**

```
Π₂ ⊢ 引理L1（Curry-Howard-Lambek三维扩展）:
       逻辑命题 ──► TypeScript类型 ──► 范畴对象 ──► 前端概念
       A∧B      ↔    [A,B]      ↔    A×B      ↔   Props组合
       A∨B      ↔    A|B        ↔    A+B      ↔   条件渲染
       A→B      ↔    (a:A)=>B   ↔    B^A      ↔   回调函数
       ¬A       ↔    never      ↔    0        ↔   不可达代码
       ∀x.P(x)  ↔    <T>(x:T)=>P ↔   Π       ↔   泛型组件
       ∃x.P(x)  ↔    {x:T&P}    ↔    Σ       ↔   Branded Types
       ◇A       ↔    Promise<A>  ↔    T(A)    ↔   异步Effect
       □A       ↔    ()=>A      ↔    W(A)    ↔   纯函数

Π₃ ⊢ 定理T2（React Compiler ≈ 线性类型推断器）:
       证明:
         useEffect的依赖数组 ≈ 线性类型的"使用上下文"
         
         违反线性性:
           const [count, setCount] = useState(0)
           useEffect(() => { console.log(count) }, [])
           // count未在依赖数组声明 → 线性错误
         
         React Compiler的自动memo:
           自动推断"资源使用图"，插入正确依赖声明
           等价于线性类型系统的类型推断算法。
           
         Rust借用检查器:
           编译期编码Separation Logic子集
           区域(Region) + 所有权(Ownership)
           零运行时开销消除Use-after-free与Out-of-bounds。∎

Π₂ + Π₃ ⊢ 推论C1（前端类型的资源敏感性）:
           现代前端正从传统幂等类型（A→A→A）
           向线性资源类型（A⊸A⊸A）演进。
           useEffect cleanup、AbortController、
           Suspense边界均体现"资源必须恰好处理一次"的线性思维。
```

**推理链 III：设计令牌的阿贝尔群结构**

```
Π₁ ⊢ 定理T3（设计令牌构成阿贝尔群）:
       定义: 设计系统 𝓓 = ⟨T, ⊕, ⊗, 𝟎, 𝟏⟩
       
       证明⟨T, ⊕⟩是阿贝尔群:
         封闭性: ∀a,b∈T, a⊕b∈T
           color(#000)⊕opacity(0.5)=rgba(0,0,0,0.5)∈T
         
         结合律: (a⊕b)⊕c = a⊕(b⊕c)
           (spacing(4)⊕spacing(8))⊕spacing(16)=spacing(28)
           spacing(4)⊕(spacing(8)⊕spacing(16))=spacing(28)
         
         交换律: a⊕b = b⊕a
           color⊕opacity = opacity⊕color
         
         单位元: ∃𝟎∈T: a⊕𝟎=a
           𝟎 = transparent, spacing(0), font-size(inherit)
         
         逆元: ∀a∈T, ∃a⁻¹∈T: a⊕a⁻¹=𝟎
           spacing(8)的逆元是spacing(-8)
           color的逆元是mix-blend-mode:difference中的补色
       
       ∴ ⟨T, ⊕⟩是阿贝尔群。    ∎

定理T3 ⊢ 推论C2（CSS变量是群的线性表示）:
         --color-primary: #000;
         --spacing-unit: 4px;
         这些变量是群T的生成元（Generators）。
         设计系统的所有令牌可由生成元通过⊕运算组合得到。
```

### 13.2 批判性总结

上述推理链将前端工程从实用技术提升到了**形式科学**的审视高度，但其真正的学术价值不在于证明了React与Solid"不同构"这一显而易见的工程事实，而在于揭示了**范畴论作为元语言**的诊断能力：当两个框架的态射结构无法保持时，任何试图在它们之间建立"兼容层"的努力都必然引入概念泄漏。Curry-Howard-Lambek三维对应则提供了更深层的教育学洞察——前端工程师每天都在无意识地进行逻辑推理：条件渲染（A∨B）、Props组合（A∧B）、回调传递（A→B）、异步操作（◇A）。类型系统不是编译器的专属领域，而是**逻辑命题的工程化身**；当TypeScript报告类型错误时，它实际上是在指出工程师的逻辑推理中存在矛盾。线性逻辑的引入尤为关键：传统类型系统假设值可以无限复制（A→A→A），但前端中的副作用（Effect）、订阅（Subscription）、网络请求（Request）都是**不可复制的资源**——React的useEffect依赖数组本质上是在模拟线性类型的使用上下文，而React Compiler正是在做线性类型推断。设计系统的群论模型则表明，CSS变量、主题切换、暗色模式等工程实践背后隐藏着严格的代数结构：设计令牌在组合操作下构成阿贝尔群，这意味着主题系统具有可逆性（逆元）、中性（单位元）和交换性——这些性质保证了设计系统的一致性和可预测性。最终，前端技术的不可能三角（T10）证明了一个令人不安的结论：**技术选型不是计算问题**，而是包含不可量化要素（团队认知、组织政治、历史债务）的Wicked Problem。所有"技术选型矩阵"都是启发式而非算法，优秀架构师的价值不在于选择"正确"技术，而在于理解每种选择的权衡空间、组织的承受边界、迁移的过渡路径和回滚的逃生舱口——这正是亚里士多德的"实践智慧"（Phronesis），而非笛卡尔的"计算理性"。

### 决策树A：并发模型与并行策略选择

```text

│   └─ YES → React Concurrent (STM模型)

│       └─ 警告: 中断时状态可能不一致

│
└─ 任务类型 = 矩阵运算/神经网络?
    └─ YES → WebGPU Compute Shader (SIMD/SPMD)
        ├─ Data-Parallel: map(f, [x...])

        └─ 适用: 浏览器内AI推理、数据可视化
```

### 决策树B：存储层级与一致性策略选择

```text
[根] 状态需要持久化或跨会话共享?
│
├─ 生命周期 = 单页面会话?
│   └─ YES → 内存存储
│       ├─ 简单状态 → JS变量/闭包 (~1ns)
│       ├─ 复杂对象 → V8 Heap (~100ns)
│       └─ 跨标签共享? → 不可行（同源隔离）
│
├─ 生命周期 = 浏览器会话?
│   └─ YES → sessionStorage (~1μs, ~5MB)
│       └─ 强一致（同源同标签），页面关闭清除
│
├─ 生命周期 = 永久本地存储?
│   └─ YES → 容量与延迟权衡
│       ├─ 小数据(<10MB) + 键值访问? → localStorage (~10μs)
│       ├─ 大数据(<1GB) + 索引查询? → IndexedDB (~1ms)
│       ├─ 文件操作 + 大容量? → OPFS (~5ms, ~磁盘容量)
│       └─ SQL关系查询? → SQLite-WASM (~10ms)
│           └─ sql.js / absurd-sql
│
└─ 生命周期 = 跨设备/跨用户?
    └─ YES → 网络存储 + 一致性模型选择
        ├─ 强一致必要? → 服务端数据库 + 乐观锁
        ├─ 因果一致足够? → COPS / ChainReaction
        └─ 最终一致可接受? → CRDT (Yjs)
            ├─ 协作编辑 → YATA / RGA
            ├─ 购物车同步 → OR-Set
            └─ 计数器 → G-Counter / PN-Counter
```
